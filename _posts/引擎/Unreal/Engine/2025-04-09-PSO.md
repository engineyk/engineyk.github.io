# Unreal Engine PSO 预热改造

## 一、PSO 基础概念

### 1.1 什么是 PSO？

**PSO**（Pipeline State Object，管线状态对象）是现代图形 API（DX12 / Vulkan / Metal）中将渲染管线所有状态一次性打包的对象。

**D3D11 vs D3D12 的变化：**
```
D3D11: 拥有很多小的状态对象 → GPU 硬件不匹配开销大
D3D12: 将管线状态分组到单个 PSO → 直接拷贝到硬件寄存器
```

**Graphics PSO 包含的状态：**

| Pipeline State  | Description                                                                |
| --------------- | -------------------------------------------------------------------------- |
| DepthStencil    | Depth/Stencil comparison functions and write masks                         |
| Sampler         | Filter states, addressing modes, LOD state                                 |
| Render Pipeline | VS/PS functions, Vertex layout, Multisample, Blend state, Color write mask |

**Compute PSO 包含的状态：**

| Pipeline State | Description                                |
| -------------- | ------------------------------------------ |
| Compute State  | Compute functions, workgroup configuration |
| Sampler        | Filter states, addressing modes, LOD state |

### 1.2 PSO 的设计宗旨

- 创建 PSO 时就已生成大部分硬件指令（编译进硬件寄存器）
- PSO 的 Shader 输入是二进制的，对 Shader Cache 非常友好
- 切换 PSO 时，硬件只需直接拷贝最少的预计算状态到寄存器，而非实时计算
- Draw Call 开销显著减少，每帧可以有更多 Draw Call

### 1.3 PSO 的核心问题

```
首次使用某个 PSO 时：
  驱动需要 编译 Shader + 组装管线状态 → 耗时几百毫秒 → 卡顿（Hitching）

表现：
  - 进入新场景时明显掉帧
  - 首次看到某种材质时卡顿
  - 首次使用某个渲染路径时掉帧
```

1. **PSO预热**
   1. 初次预编译：初次启动游戏时，把PSO list中的PSO编译一遍，会产生渲染API的本地Cache。通常十分缓慢
   2. PSO的预热（Warm-up）是识别需求
   3. 源码与编译：你（或材质编辑器）首先编写HLSL等高级语言写成的Shader源码。在项目打包时，UE4的着色器编译器会将源码编译成特定图形API（如DX12的DXBC或Vulkan的SPIR-V）所需的Shader二进制码。
2. **PSO预编译**
   1. 后续预编译（预载）：后续启动游戏时，仍然会触发预编译流程，但此时因为已有缓存，速度大大加快
   2. PSO预编译是载体：预编译（Precompile）则是实际执行创建PSO对象的步骤。
   3. 而这个创建过程，必然触发GPU驱动将中间字节码进行最终的JIT编译。没有这个步骤，PSO对象本身是无法创建的。
   4. 状态组合与PSO创建：当一个模型需要渲染时，引擎会根据其材质、顶点格式、混合模式等，确定一整套渲染状态组合，并查找对应的Shader二进制码。在现代图形API（如DX12、Vulkan）下，引擎会使用这些信息创建一个PSO对象。这个创建PSO的过程非常耗时，因为它需要驱动层进行验证和优化。PSO对象中包含众多数据，其中就包含了Shader，但是是引用的关系，Shader二进制码存放在另外的地方（硬盘、内存中的 ShaderCodeLibrary 中）。
3. **SetPSO**
   1. 运行时加载：提交渲染时，从内存或磁盘中获取缓存的PSO
   2. Set PSO命令的作用，是将一个已经创建（即已经完成最终编译）的、立即可用的PSO对象绑定到当前的渲染状态。
   3. 正因为最耗时的编译工作已经在预编译阶段完成，Set PSO才能成为一个高效、瞬时完成的操作，从而避免在提交绘制指令时产生卡顿。
   4. 绘制提交：在实际的绘制调用（DrawCall）前，渲染线程将所需的PSO设置到GPU。由于PSO已经包含了所有状态，GPU驱动无需在运行时进行繁琐的状态检查和设置，大大提升了效率。

## 二、Unreal Engine PSO 缓存架构

### 2.1 Shader 缓存体系

UE 的 Shader 缓存分两种：
1. **离线缓存（DDC）**：存于 Derived Data Cache，加速编辑器/开发阶段，参考 `FGlobalShaderMap`
2. **运行时缓存**：早期由 `FShaderCache` 承担，UE4.26 后由 `FShaderPipelineCache` 取代

### 2.2 核心类：FShaderPipelineCache

`FShaderPipelineCache` 提供 PSO 日志记录、序列化和预编译机制。缓存 PSO 并将初始化器序列化到磁盘，下次运行时预编译，减少卡顿。

**依赖关系：**
```
FShaderPipelineCache
  ├── FShaderCodeLibrary         (Shader 代码库)
  ├── Share Material Shader Code (共享材质 Shader 代码)
  └── FPipelineFileCache (RHI)   (RHI 侧管线文件缓存)
```

**源码结构：**
```cpp
// Engine\Source\Runtime\RenderCore\Public\ShaderPipelineCache.h

class FShaderPipelineCache : public FTickableObjectRenderThread
{
    // Compile job structure
    struct CompileJob
    {
        FPipelineCacheFileFormatPSO PSO;

public:

    static void PauseBatching();
    static void ResumeBatching();
    
    // Batch modes with different throughput
    enum class BatchMode
    {
        Background, // r.ShaderPipelineCache.BackgroundBatchSize
        Fast,       // r.ShaderPipelineCache.BatchSize
        Precompile  // r.ShaderPipelineCache.PrecompileBatchSize
    };
    
    static void SetBatchMode(BatchMode Mode);
    static uint32 NumPrecompilesRemaining();
    static uint32 NumPrecompilesActive();
    
    // File cache operations
    static bool OpenPipelineFileCache(EShaderPlatform Platform);
    static bool SavePipelineFileCache(FPipelineFileCache::SaveMode Mode);
    static void ClosePipelineFileCache();
    
    // Tick on render thread - drives batch precompilation
    void Tick(float DeltaTime);
    
    // Precompile lifecycle delegates
    static FShaderPrecompilationBeginDelegate& GetPrecompilationBeginDelegate();
    static FShaderPrecompilationCompleteDelegate& GetPrecompilationCompleteDelegate();

private:
    TArray<CompileJob> ReadTasks;
    TArray<CompileJob> CompileTasks;
    TArray<FPipelineCachePSOHeader> OrderedCompileTasks;
    TDoubleLinkedList<FPipelineCacheFileFormatPSORead*> FetchTasks;
    TSet<uint32> CompiledHashes;
    
    // RHI state caches
    TMap<FBlendStateInitializerRHI, FRHIBlendState*> BlendStateCache;
    TMap<FRasterizerStateInitializerRHI, FRHIRasterizerState*> RasterizerStateCache;
    TMap<FDepthStencilStateInitializerRHI, FRHIDepthStencilState*> DepthStencilStateCache;
};
```

### 2.3 引擎初始化流程

PSO 缓存在引擎启动时分两阶段初始化：

```cpp
// Phase 1: PreInitPreStartupScreen - Initialize
int32 FEngineLoop::PreInitPreStartupScreen(const TCHAR* CmdLine)
{
    bool bUseCodeLibrary = FPlatformProperties::RequiresCookedData() 
                           || GAllowCookedDataInEditorBuilds;
    if (bUseCodeLibrary)
    {
        // Initialize shader code library first
        FShaderCodeLibrary::InitForRuntime(GMaxRHIShaderPlatform);

    #if !UE_EDITOR
        if (FPlatformProperties::RequiresCookedData())
        {
            // Initialize PSO cache system
            FShaderPipelineCache::Initialize(GMaxRHIShaderPlatform);
        }
    #endif
    }
}

// Phase 2: PreInitPostStartupScreen - Open cache file
int32 FEngineLoop::PreInitPostStartupScreen(const TCHAR* CmdLine)
{
    // Open shader code library
    FShaderCodeLibrary::OpenLibrary(FApp::GetProjectName(), 
                                    FPaths::ProjectContentDir());
    
    // Open PSO pipeline cache - triggers precompilation
    FShaderPipelineCache::OpenPipelineFileCache(GMaxRHIShaderPlatform);
}
```

## 三、PSO 预热策略

### 3.1 BatchMode 三档预热

UE 提供三种预热模式，适用于不同场景：

```
┌─────────────────────────────────────────────────────────────────┐
│  BatchMode::Precompile                                          │
│  - 最大吞吐量，用于 Loading 界面                                 │
│  - r.ShaderPipelineCache.PrecompileBatchSize 控制               │
│  - 尽快编译完所有 PSO                                            │
│                                                                 │
│  BatchMode::Fast                                                │
│  - 中等吞吐量，用于游戏启动后快速预热                              │
│  - r.ShaderPipelineCache.BatchSize 控制                         │
│  - 平衡预热速度和游戏流畅度                                       │
│                                                                 │
│  BatchMode::Background                                          │
│  - 最低吞吐量，用于游戏运行中后台持续预热                          │
│  - r.ShaderPipelineCache.BackgroundBatchSize 控制               │
│  - 不影响游戏帧率                                                │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 按设备档位分级预热

```
低档设备策略：
  ┌─ Loading 界面 → BatchMode::Precompile（全量预热）
  ├─ 预热完成后才进入游戏
  └─ 避免游戏中出现任何 PSO 编译卡顿

高档设备策略：
  ┌─ Loading 界面 → 预热少量核心 PSO
  ├─ 进入游戏后 → BatchMode::Background（后台持续预热）
  └─ 高档设备编译速度快，后台预热不影响体验
```

### 3.3 按 Usage 分组预热

```
将 PSO 按使用场景分组：
  ├── Core PSO（核心渲染）：BasePass、Shadow、PostProcess
  │   └── 优先级最高，Loading 时必须预热完成
  ├── Gameplay PSO（玩法相关）：特效、UI、技能
  │   └── 进入对应玩法前预热
  ├── Rare PSO（低频使用）：特殊天气、过场动画
  │   └── 后台预热或按需预热
  └── Debug PSO（调试用）：线框、碰撞可视化
      └── 仅开发版本预热
```

## 四、PSO 预热改造实践

### 4.1 PSO 收集流程

```
开发阶段收集：
  1. 启用 PSO 日志记录
     r.ShaderPipelineCache.Enabled=1
     r.ShaderPipelineCache.LogPSO=1
     r.ShaderPipelineCache.SaveAfterPSOsLogged=1
  
  2. 遍历所有游戏场景/玩法，触发所有渲染路径
     - 自动化测试脚本遍历所有关卡
     - 切换所有画质档位
     - 触发所有材质/特效
  
  3. 导出 PSO 缓存文件
     - rec.upipelinecache（原始录制）
     - stablepc.csv（展开的稳定列表）
  
  4. 打包到游戏资源中随包分发
```

### 4.2 OpenGL ES PSO 预热特殊处理

```
OpenGL ES 的 PSO 预热核心：
  将 Shader 二进制文件（Program Binary）转换并缓存到本地

流程：
  1. 首次运行：编译 GLSL → 获取 Program Binary → 缓存到本地文件
  2. 后续运行：直接加载 Program Binary → 跳过编译
  3. glProgramBinary() 比 glCompileShader() + glLinkProgram() 快 10-100x

注意：
  - Program Binary 与 GPU 驱动版本绑定
  - 驱动更新后缓存失效，需要重新编译
  - 不同设备的 Binary 不通用
```

### 4.3 PSO PreCache 辅助 RHI 线程方案

```
问题：
  默认 PSO 编译在渲染线程执行 → 阻塞渲染 → 卡顿

改造方案：RHI 编译和链接放到独立 RHI 线程

┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  Game Thread │    │ Render Thread│    │  RHI Thread  │
│              │    │              │    │              │
│  提交渲染命令 │───→│  处理渲染命令 │───→│  PSO 编译     │
│              │    │  不阻塞等待   │    │  Shader 链接 │
│              │    │              │    │  异步完成通知 │
└──────────────┘    └──────────────┘    └──────────────┘

注意事项：
  - 编译期间该 PSO 对应的物体不渲染 → 可能出现短暂黑屏/物体消失
  - 需要 Fallback 机制：编译中使用简化 Shader 临时替代
  - 编译完成后自动切换回正确的 PSO
```

### 4.4 Shader 变体精简

```
变体爆炸是 PSO 数量膨胀的根源：

项目实例：总编译量 450 万变体！

优化策略：
  1. 全局变体单独管理
     - 全局 Keyword 独立开关
     - 避免全局 × 局部的组合爆炸
  
  2. 尽早剔除未使用的排列
     - 分析实际使用的变体组合
     - 移除从未触发的变体
  
  3. Uber Shader 策略
     - 将多个小 Shader 合并为一个 Uber Shader
     - 用动态分支替代静态变体（适用于分支代价低的情况）
  
  4. Specialization Constants（Vulkan）
     - 使用 Vulkan Specialization Constants
     - 运行时特化，减少编译时变体数
  
  5. D3D12 Root Constants
     - 将常量放到 Root Signature
     - 减少因常量不同导致的 PSO 差异
```

### 4.5 PSO 排序优化

```
减少 PSO 切换开销：

1. 按 PSO 相似性排序 Draw Call
   - UE 默认按 PS、VS 等键值对绘制指令排序
   - 相似 PSO 的 Draw Call 连续执行
   - 减少硬件状态切换

2. 相同 Shader 不同状态的 PSO 复用
   - 不同 BlendState 但 VS/PS 相同 → 复用 Shader 编译结果
   - 同一线程编译类似 PSO → 自动复用

3. 无关状态使用默认值
   - 深度测试关闭时，DepthStencil 参数无关紧要
   - 统一使用相同默认值 → 减少 PSO 种类

4. 状态哈希优化
   - 为每个状态创建唯一哈希
   - 将状态块放入 ID 池
   - 用块 ID 构造状态哈希
   - UE 采用 16 个固定采样器状态
```

## 七、源码级三阶段详解：触发预热 → 执行预编译 → 渲染时设置

### 7.1 阶段一：触发预热（Trigger Warm-up）

预热的触发入口是 `OpenPipelineFileCache`，在引擎启动的 `PreInitPostStartupScreen` 阶段调用：

```cpp
// Engine\Source\Runtime\Launch\Private\LaunchEngineLoop.cpp

int32 FEngineLoop::PreInitPostStartupScreen(const TCHAR* CmdLine)
{
    // 1. 先打开 Shader 代码库
    FShaderCodeLibrary::OpenLibrary(FApp::GetProjectName(), FPaths::ProjectContentDir());
    
    // 2. 打开 PSO 缓存文件 → 触发预编译流程
    FShaderPipelineCache::OpenPipelineFileCache(GMaxRHIShaderPlatform);
}
```

**`OpenPipelineFileCache` 内部流程：**

```
OpenPipelineFileCache(Platform)
  │
  ├─ 1. 读取 .upipelinecache 文件（反序列化 PSO 描述列表）
  │     FPipelineFileCache::OpenPipelineFileCache()
  │     → 解析文件头、版本校验、UsageMask 过滤
  │
  ├─ 2. 构建 OrderedCompileTasks（按优先级排序的待编译 PSO 列表）
  │     → 根据 Usage 分组排序（Core > Gameplay > Rare）
  │     → 过滤已编译的 PSO（CompiledHashes 去重）
  │
  ├─ 3. 广播 PrecompilationBeginDelegate
  │     → Vulkan: OnShaderPipelineCacheOpened() 注册回调
  │     → 通知 RHI 层准备接收 PSO 编译请求
  │
  └─ 4. 启动 Batch 预编译（由 Tick 驱动）
        → 设置初始 BatchMode（通常为 Precompile）
        → FShaderPipelineCache 作为 FTickableObjectRenderThread 开始 Tick
```

**关键数据结构：**

```cpp
// PSO 缓存文件格式中的单个 PSO 条目
struct FPipelineCacheFileFormatPSO
{
    FPipelineCacheFileFormatPSODesc Desc;  // PSO 完整描述
    // 包含：
    //   - VertexShader / PixelShader / GeometryShader 的 SHA Hash
    //   - BlendState / RasterizerState / DepthStencilState 初始化器
    //   - VertexDeclaration / PrimitiveType
    //   - RenderTarget 格式信息
    //   - UsageMask（标记该 PSO 属于哪个使用场景）
};

// 编译作业
struct CompileJob
{
    FPipelineCacheFileFormatPSO PSO;           // 要编译的 PSO 描述
    FShaderPipelineCacheArchive* ReadRequests;  // 关联的 Shader 代码读取请求
};
```

### 7.2 阶段二：执行预编译（Execute Precompilation）

预编译由 `FShaderPipelineCache::Tick()` 在**渲染线程**上每帧驱动：

```cpp
// Engine\Source\Runtime\RenderCore\Private\ShaderPipelineCache.cpp

void FShaderPipelineCache::Tick(float DeltaTime)
{
    if (IsBatchingPaused()) return;
    
    // 1. 确定本帧编译数量（根据 BatchMode）
    int32 LocalBatchSize = 0;
    switch (CurrentBatchMode)
    {
        case BatchMode::Background:  LocalBatchSize = BackgroundBatchSize;  break;  // 默认 5
        case BatchMode::Fast:        LocalBatchSize = BatchSize;            break;  // 默认 50
        case BatchMode::Precompile:  LocalBatchSize = PrecompileBatchSize;  break;  // 默认 200
    }
    
    // 2. Fetch 阶段：从磁盘异步读取 Shader 二进制代码
    //    OrderedCompileTasks → FetchTasks（异步 IO）→ ReadTasks
    ProcessFetchTasks();
    
    // 3. Read 阶段：将读取完成的数据转为 CompileJob
    //    ReadTasks → CompileTasks
    ProcessReadTasks();
    
    // 4. Compile 阶段：实际创建 PSO（核心！）
    int32 NumCompiled = 0;
    while (NumCompiled < LocalBatchSize && CompileTasks.Num() > 0)
    {
        CompileJob& Job = CompileTasks[0];
        
        // 4a. 从 PSO 描述中查找/创建 RHI 状态对象
        FRHIBlendState* BlendState = GetOrCreateBlendState(Job.PSO.Desc.BlendState);
        FRHIRasterizerState* RasterizerState = GetOrCreateRasterizerState(Job.PSO.Desc.RasterizerState);
        FRHIDepthStencilState* DepthStencilState = GetOrCreateDepthStencilState(Job.PSO.Desc.DepthStencilState);
        
        // 4b. 从 ShaderCodeLibrary 查找 Shader 二进制
        FRHIVertexShader* VS = FShaderCodeLibrary::CreateVertexShader(Job.PSO.Desc.VertexShaderHash);
        FRHIPixelShader* PS = FShaderCodeLibrary::CreatePixelShader(Job.PSO.Desc.PixelShaderHash);
        
        // 4c. 组装 FGraphicsPipelineStateInitializer
        FGraphicsPipelineStateInitializer Initializer;
        Initializer.BlendState = BlendState;
        Initializer.RasterizerState = RasterizerState;
        Initializer.DepthStencilState = DepthStencilState;
        Initializer.BoundShaderState.VertexShaderRHI = VS;
        Initializer.BoundShaderState.PixelShaderRHI = PS;
        Initializer.PrimitiveType = Job.PSO.Desc.PrimitiveType;
        // ... 其他状态 ...
        
        // 4d. 调用 RHI 创建 PSO（关键调用！）
        //     → DX12: ID3D12Device::CreateGraphicsPipelineState()
        //     → Vulkan: vkCreateGraphicsPipelines()
        //     → Metal: [MTLDevice newRenderPipelineStateWithDescriptor:]
        SetGraphicsPipelineState(RHICmdList, Initializer);
        // 或直接：RHICreateGraphicsPipelineState(Initializer);
        
        // 4e. 记录已编译的 Hash，避免重复编译
        CompiledHashes.Add(GetTypeHash(Job.PSO));
        
        CompileTasks.RemoveAt(0);
        NumCompiled++;
    }
    
    // 5. 检查是否全部完成
    if (OrderedCompileTasks.Num() == 0 && CompileTasks.Num() == 0 && FetchTasks.Num() == 0)
    {
        // 广播预编译完成委托
        GetPrecompilationCompleteDelegate().Broadcast(TotalCompiled, ElapsedTime, ShaderCachePrecompileContext);
    }
}
```

### 7.3 阶段三：渲染时设置（Runtime PSO Binding）

渲染时 PSO 的设置发生在 `FMeshDrawCommand::SubmitDraw` 中，这是 UE 网格绘制管线的核心提交路径：

```cpp
// Engine\Source\Runtime\Renderer\Private\MeshDrawCommands.cpp

void FMeshDrawCommand::SubmitDraw(
    const FMeshDrawCommand& RESTRICT MeshDrawCommand, 
    const FGraphicsMinimalPipelineStateSet& GraphicsMinimalPipelineStateSet,
    FRHIVertexBuffer* ScenePrimitiveIdsBuffer,
    int32 PrimitiveIdOffset,
    uint32 InstanceFactor,
    FRHICommandList& RHICmdList,
    FMeshDrawCommandStateCache& RESTRICT StateCache)
{
    // 1. 从 MeshDrawCommand 获取缓存的 PSO 状态
    const FGraphicsMinimalPipelineStateInitializer& MeshPipelineState = 
        MeshDrawCommand.CachedPipelineId.GetPipelineState(GraphicsMinimalPipelineStateSet);

    // 2. PSO 状态变更检测（避免冗余设置）
    if (MeshDrawCommand.CachedPipelineId.GetId() != StateCache.PipelineId)
    {
        // 2a. 将 Minimal 状态扩展为完整的 PSO 初始化器
        FGraphicsPipelineStateInitializer GraphicsPSOInit = 
            MeshPipelineState.AsGraphicsPipelineStateInitializer();
        
        // 2b. 应用当前 RenderTarget 信息到 PSO
        RHICmdList.ApplyCachedRenderTargets(GraphicsPSOInit);
        
        // 2c. 设置 PSO（查找缓存或创建新的）
        SetGraphicsPipelineState(RHICmdList, GraphicsPSOInit);
        
        // 2d. 更新状态缓存（下次相同 PSO 跳过设置）
        StateCache.SetPipelineState(MeshDrawCommand.CachedPipelineId.GetId());
    }

    // 3. 设置其他渲染状态
    if (MeshDrawCommand.StencilRef != StateCache.StencilRef)
    {
        RHICmdList.SetStencilRef(MeshDrawCommand.StencilRef);
        StateCache.StencilRef = MeshDrawCommand.StencilRef;
    }

    // 4. 设置顶点流
    for (int32 i = 0; i < MeshDrawCommand.VertexStreams.Num(); i++)
    {
        const FVertexInputStream& Stream = MeshDrawCommand.VertexStreams[i];
        if (StateCache.VertexStreams[Stream.StreamIndex] != Stream)
        {
            RHICmdList.SetStreamSource(Stream.StreamIndex, Stream.VertexBuffer, Stream.Offset);
            StateCache.VertexStreams[Stream.StreamIndex] = Stream;
        }
    }

    // 5. 设置 Shader 绑定资源（Uniform Buffer 等）
    MeshDrawCommand.ShaderBindings.SetOnCommandList(RHICmdList, ...);

    // 6. 执行绘制调用
    RHICmdList.DrawIndexedPrimitive(...);
}
```

```
│                     PSO 两级缓存体系                                 │
```

## 八、面试回答要点

### Q: Unreal PSO 预热改造做了哪些工作？（含三阶段源码级理解）

```
1. PSO 收集与分发
   - 开发阶段通过自动化测试遍历所有场景/玩法，收集 PSO
   - 导出 rec.upipelinecache 和 stablepc.csv
   - 随游戏包分发，玩家首次运行即可预热

2. 分级预热策略
   - 低档设备：Loading 界面全量预热（Precompile 模式）
   - 高档设备：Loading 预热核心 PSO，游戏中后台持续预热（Background 模式）

3. 按 Usage 分组
   - 核心渲染 PSO 优先预热
   - 玩法相关 PSO 按需预热
   - 低频 PSO 后台预热

4. 辅助 RHI 线程方案
   - PSO 编译从渲染线程移到独立 RHI 线程
   - 避免编译阻塞渲染
   - 配合 Fallback Shader 避免黑屏

5. 变体精简
   - 总变体量从 450 万优化
   - 全局变体独立管理
   - 剔除未使用排列
   - Uber Shader 合并

6. OpenGL ES 特殊处理
   - Program Binary 本地缓存
   - 跳过重复编译，加速 10-100x

效果：
  - Loading 卡顿消除
  - 游戏中 PSO 编译卡顿率降低 95%+
  - 首次进入场景无明显掉帧
```
