## 1. 概述

### 1.1 Why Multi-Threading

```
单线程瓶颈：
┌──────────────────────────────────────────────────────┐
│  Input → Logic → Physics → Animation → Render → GPU  │
│  ←──────────────── 33ms (30fps) ──────────────────→  │
└──────────────────────────────────────────────────────┘

多线程并行：
┌─ Game Thread ──── Logic ──── Animation ─────────────┐
├─ Render Thread ──────── Command Build ──────────────┤  16ms (60fps)
├─ RHI Thread ────────────── GPU Submit ──────────────┤
├─ Worker 0~N ──── Physics ── AI ── Particles ────────┤
└─────────────────────────────────────────────────────┘
```

### 1.2 Core Challenges

| Challenge      | Description                                            |
| -------------- | ------------------------------------------------------ |
| Data Race      | Multiple threads read/write shared data simultaneously |
| Deadlock       | Circular wait on locks                                 |
| False Sharing  | Cache line contention between cores                    |
| Load Imbalance | Some threads idle while others overloaded              |
| Determinism    | Floating-point order-dependent results                 |
| Debugging      | Non-reproducible bugs, Heisenbug                       |

## 2. 线程模型演进

### 2.1 Evolution Timeline

```
Gen 1: Single Thread (2000s)
  └─ Everything on main thread

Gen 2: Dedicated Threads (2005~2010)
  └─ Main + Render + Audio + Loading
  └─ Fixed thread assignment, manual sync

Gen 3: Thread Pool + Task Queue (2010~2015)
  └─ Worker threads pull tasks from queue
  └─ Better utilization, still coarse-grained

Gen 4: Job System + Dependency Graph (2015~now)
  └─ Fine-grained jobs with DAG dependencies
  └─ Work stealing, fiber-based (Naughty Dog)
  └─ Near-optimal core utilization

Gen 5: Fiber / Coroutine Based (Emerging)
  └─ Stackful fibers or C++20 coroutines
  └─ Suspend/resume without blocking thread
  └─ Naughty Dog, Our Machinery
```

### 2.2 Dedicated Threads vs Job System

| Aspect         | Dedicated Threads            | Job System                   |
| -------------- | ---------------------------- | ---------------------------- |
| Thread Count   | Fixed (Main/Render/Audio...) | N workers (= CPU cores)      |
| Granularity    | Coarse (whole subsystem)     | Fine (small task)            |
| Load Balance   | Poor (some threads idle)     | Good (work stealing)         |
| Scalability    | Bad (hardcoded thread count) | Excellent (auto-scale)       |
| Complexity     | Low (simple sync)            | High (dependency management) |
| Cache Locality | Good (data stays on thread)  | Needs careful design         |

### 3.2 Frame Pipeline (3-Thread Overlap)

```
Frame N:     Frame N+1:    Frame N+2:

Game ──────►Game ─────────►Game ──────────►
       \         \              \
        Render ──►Render ──────►Render ────►
              \         \              \
               RHI ─────►RHI ─────────►RHI►
                    \         \
                     GPU ─────►GPU ────────►

│◄── 1 frame latency ──►│
│◄──── 2 frame latency ────────►│ (Game to Screen)
```

**Key Insight**: Game Thread is always 1~2 frames ahead of what's displayed on screen. This is the **input latency** cost of pipelining.

## 4. Task / Job System 原理

### 4.1 Core Concepts

```cpp
// A Job/Task is a small unit of work
struct Job {
    void (*function)(void* data);   // Work function
    void* data;                      // Payload
    std::atomic<int> unfinished;     // Dependency counter
    Job* parent;                     // Parent job (for completion)
    Job* dependency;                 // Must complete before this runs
};
```

### 4.2 DAG (Directed Acyclic Graph) Dependencies

```
                ┌───────────┐
                │ FrameStart│
                └─────┬─────┘
          ┌───────────┼───────────┐
          ▼           ▼           ▼
    ┌───────────┐ ┌──────────┐ ┌───────────┐
    │ Physics   │ │Animation │ │   AI      │
    │ (200 jobs)│ │(100 jobs)│ │(50 jobs)  │
    └─────┬─────┘ └────┬─────┘ └───┬───────┘
          │            │           │
          ▼            ▼           ▼
    ┌──────────────────────────────────┐
    │         Sync Barrier             │
    └──────────────┬───────────────────┘
                   ▼
    ┌──────────────────────────────────┐
    │    Render Command Generation     │
    └──────────────┬───────────────────┘
                   ▼
    ┌──────────────────────────────────┐
    │          Frame End               │
    └──────────────────────────────────┘
```

### 4.3 Job Lifecycle

```
1. Create Job
   └─ Allocate from pool, set function + data

2. Set Dependencies
   └─ job_B.depends_on(job_A)
   └─ Increment job_B.unfinished counter

3. Schedule / Kick
   └─ Push to thread-local queue

4. Execute
   └─ Worker pops job, runs function
   └─ On complete: decrement dependents' counters
   └─ If counter reaches 0 → dependent becomes ready

5. Wait / Sync
   └─ Caller can wait on a job
   └─ While waiting: steal and execute other jobs (don't block!)
```

### 4.4 Parallel-For Pattern

```cpp
// Split N items across M jobs
void ParallelFor(int count, int batchSize, std::function<void(int start, int end)> body) {
    int jobCount = (count + batchSize - 1) / batchSize;
    Job* root = CreateJob(EmptyJob);
    
    for (int i = 0; i < jobCount; i++) {
        int start = i * batchSize;
        int end = std::min(start + batchSize, count);
        Job* child = CreateChildJob(root, [=](){ body(start, end); });
        Schedule(child);
    }
    
    Schedule(root);
    WaitForJob(root);  // Steal work while waiting
}

// Usage: Update 10000 particles across all cores
ParallelFor(10000, 256, [&](int start, int end) {
    for (int i = start; i < end; i++) {
        UpdateParticle(particles[i], deltaTime);
    }
});
```

## 5. Work Stealing 调度算法

### 5.1 Why Work Stealing

```
Problem with single shared queue:
┌──────────────────────────────────────┐
│  Shared Queue (contention!)          │
│  ┌─┬─┬─┬─┬─┬─┬─┬─┬─┬─┐               │
│  │J│J│J│J│J│J│J│J│J│J│               │
│  └─┴─┴─┴─┴─┴─┴─┴─┴─┴─┘               │
│    ▲   ▲   ▲   ▲   ▲                 │
│    W0  W1  W2  W3  W4  (all compete) │
└──────────────────────────────────────┘

Work Stealing solution:
┌──────────────────────────────────────┐
│  Each worker has LOCAL deque         │
│                                      │
│  W0: [J J J J]  ← push/pop (LIFO)    │
│  W1: [J J]                           │
│  W2: [J J J J J J]                   │
│  W3: []  ──steal──► W2: [J J J J J]  │
│  W4: [J]           (steal from FIFO) │
└──────────────────────────────────────┘
```

### 5.2 Deque Operations

```
Local Thread (Owner):
  push() → bottom end (LIFO)
  pop()  → bottom end (LIFO)  ← cache-friendly!

Stealing Thread:
  steal() → top end (FIFO)    ← oldest/largest jobs

┌─────────────────────────┐
│  TOP (steal end)        │
│  ┌───┐                  │
│  │ J │ ← steal here     │
│  ├───┤                  │
│  │ J │                  │
│  ├───┤                  │
│  │ J │                  │
│  ├───┤                  │
│  │ J │ ← push/pop here  │
│  └───┘                  │
│  BOTTOM (local end)     │
└─────────────────────────┘
```

### 5.3 Why LIFO Local + FIFO Steal

| Strategy       | Reason                                                                    |
| -------------- | ------------------------------------------------------------------------- |
| **Local LIFO** | Most recently pushed job is likely still in L1/L2 cache → better locality |
| **Steal FIFO** | Oldest jobs tend to be larger (parent jobs) → amortizes steal overhead    |
| **Steal FIFO** | Reduces contention: local and stealer access different ends of deque      |

### 5.4 Lock-Free Work Stealing Deque (Chase-Lev)

```cpp
class WorkStealingDeque {
    std::atomic<int64_t> top;     // Steal end
    std::atomic<int64_t> bottom;  // Local end
    std::atomic<Job**> buffer;    // Circular buffer
    
    // Owner thread only
    void push(Job* job) {
        int64_t b = bottom.load(relaxed);
        int64_t t = top.load(acquire);
        // ... resize if full ...
        buffer.load(relaxed)[b & mask] = job;
        std::atomic_thread_fence(release);
        bottom.store(b + 1, relaxed);
    }
    
    // Owner thread only
    Job* pop() {
        int64_t b = bottom.load(relaxed) - 1;
        bottom.store(b, relaxed);
        std::atomic_thread_fence(seq_cst);
        int64_t t = top.load(relaxed);
        if (t <= b) {
            Job* job = buffer.load(relaxed)[b & mask];
            if (t == b) {
                // Last element: race with steal
                if (!top.compare_exchange_strong(t, t + 1, seq_cst, relaxed))
                    job = nullptr;  // Lost race
                bottom.store(b + 1, relaxed);
            }
            return job;
        }
        bottom.store(b + 1, relaxed);
        return nullptr;  // Empty
    }
    
    // Any thread
    Job* steal() {
        int64_t t = top.load(acquire);
        std::atomic_thread_fence(seq_cst);
        int64_t b = bottom.load(acquire);
        if (t < b) {
            Job* job = buffer.load(relaxed)[t & mask];
            if (top.compare_exchange_strong(t, t + 1, seq_cst, relaxed))
                return job;
        }
        return nullptr;  // Empty or lost race
    }
};
```

## 6. 同步原语与无锁编程

### 6.1 Synchronization Primitives Comparison

| Primitive              | Cost                | Use Case                             | Danger                       |
| ---------------------- | ------------------- | ------------------------------------ | ---------------------------- |
| **Mutex**              | ~25ns (uncontended) | Protecting shared data               | Deadlock, priority inversion |
| **SpinLock**           | ~5ns (uncontended)  | Very short critical sections         | CPU waste if contended       |
| **RWLock**             | ~30ns               | Read-heavy, rare writes              | Writer starvation            |
| **Semaphore**          | ~25ns               | Producer-consumer, resource counting | Complex reasoning            |
| **Atomic**             | ~5ns                | Counters, flags, CAS loops           | ABA problem, memory ordering |
| **Condition Variable** | ~50ns               | Wait for condition                   | Spurious wakeup              |
| **Fence/Barrier**      | ~1ns                | Memory ordering guarantee            | Subtle bugs                  |
| **Event**              | ~50ns               | One-shot signal                      | Miss signal if not waiting   |

### 6.2 Memory Ordering (C++11)

```cpp
// From weakest to strongest:

memory_order_relaxed    // No ordering guarantee, only atomicity
                        // Use: counters, statistics

memory_order_acquire    // Reads after this see writes before matching release
                        // Use: lock acquisition, reading shared data

memory_order_release    // Writes before this are visible after matching acquire
                        // Use: lock release, publishing shared data

memory_order_acq_rel    // Both acquire and release
                        // Use: read-modify-write operations

memory_order_seq_cst    // Total ordering across all threads (default)
                        // Use: when in doubt (but slower)
```

### 6.3 Common Lock-Free Patterns

```cpp
// 1. Lock-Free Queue (MPSC - Multiple Producer Single Consumer)
template<typename T>
class MPSCQueue {
    struct Node {
        T data;
        std::atomic<Node*> next{nullptr};
    };
    std::atomic<Node*> head;  // Producers push here
    Node* tail;               // Consumer pops here (single thread)
    
    void push(T value) {
        Node* node = new Node{std::move(value)};
        Node* prev = head.exchange(node, std::memory_order_acq_rel);
        prev->next.store(node, std::memory_order_release);
    }
};

// 2. Double-Buffered Data (common in game engines)
class DoubleBuffer {
    FrameData buffers[2];
    std::atomic<int> readIndex{0};
    
    // Writer (Game Thread)
    void write(const FrameData& data) {
        int writeIdx = 1 - readIndex.load(relaxed);
        buffers[writeIdx] = data;
        readIndex.store(writeIdx, release);  // Publish
    }
    
    // Reader (Render Thread)
    const FrameData& read() {
        int readIdx = readIndex.load(acquire);  // Consume
        return buffers[readIdx];
    }
};

// 3. Atomic Flag for One-Shot Event
std::atomic<bool> dataReady{false};
SharedData sharedData;

// Producer
sharedData = ComputeData();                    // Write data
dataReady.store(true, std::memory_order_release); // Publish

// Consumer
while (!dataReady.load(std::memory_order_acquire)) { /* spin or yield */ }
UseData(sharedData);  // Safe: acquire sees release's writes
```

### 6.4 False Sharing

```cpp
// BAD: Two atomics on same cache line (64 bytes)
struct BadLayout {
    std::atomic<int> counterA;  // Thread 0 writes
    std::atomic<int> counterB;  // Thread 1 writes
    // Both on same 64-byte cache line → constant invalidation!
};

// GOOD: Pad to separate cache lines
struct GoodLayout {
    alignas(64) std::atomic<int> counterA;  // Own cache line
    alignas(64) std::atomic<int> counterB;  // Own cache line
};

// UE4 uses:
struct alignas(PLATFORM_CACHE_LINE_SIZE) FCacheLineAligned {
    std::atomic<int> counter;
};
```

## 7. 帧同步与流水线并行

### 7.1 Frame Overlap Model

```
Time ──────────────────────────────────────────────────►

         Frame N              Frame N+1            Frame N+2
        ┌─────────┐          ┌─────────┐          ┌─────────┐

        └─────────┘          └─────────┘          └─────────┘
```

### 7.2 Sync Points & Fences

```cpp
// UE4 Frame Sync
void FEngineLoop::Tick() {
    // 1. Game Thread: Start frame
    GEngine->Tick(DeltaTime);
    
    // 2. Wait for Render Thread to finish PREVIOUS frame
    //    (so we can safely update scene proxies)
    FRenderCommandFence fence;
    fence.BeginFence();
    fence.Wait();  // Block until render thread catches up
    
    // 3. Enqueue new render commands for THIS frame
    ENQUEUE_RENDER_COMMAND(UpdateScene)([](FRHICommandListImmediate& RHICmdList) {
        // Runs on Render Thread
        UpdateSceneProxies();
        RenderFrame();
    });
}
```

### 7.3 Triple Buffering (CPU-GPU)

```
Buffer 0: GPU reading (Frame N-2)
Buffer 1: Render Thread writing (Frame N-1)
Buffer 2: Game Thread preparing (Frame N)

Rotate each frame:
  Frame N:   Game→Buf2, Render→Buf1, GPU→Buf0
  Frame N+1: Game→Buf0, Render→Buf2, GPU→Buf1
  Frame N+2: Game→Buf1, Render→Buf0, GPU→Buf2

Benefit: No stall as long as GPU finishes within 2 frames
Cost: 2 frames of input latency, 3x buffer memory
```

### 8.3 Dependency Chain

```csharp
void Update() {
    // Job A: Update positions
    var jobA = new MoveJob { dt = Time.deltaTime, positions = posArray }
        .Schedule(posArray.Length, 64);  // batchSize = 64
    
    // Job B: Depends on A, update bounds
    var jobB = new BoundsJob { positions = posArray, bounds = boundsArray }
        .Schedule(posArray.Length, 64, jobA);  // dependency!
    
    // Job C: Depends on B, collision detection
    var jobC = new CollisionJob { bounds = boundsArray }
        .Schedule(jobB);
    
    // Complete at end of frame (or next frame)
    jobC.Complete();  // Blocks main thread until done
}
```

### 8.4 NativeContainer Safety

```csharp
// Safety system prevents data races at schedule time (Editor only)
var array = new NativeArray<int>(100, Allocator.TempJob);

var jobA = new WriteJob { data = array }.Schedule();
var jobB = new WriteJob { data = array }.Schedule(); // ERROR! Two writers!

// Correct: chain dependency
var jobB = new WriteJob { data = array }.Schedule(jobA); // OK

// Or mark as ReadOnly
struct ReadJob : IJobParallelFor {
    [ReadOnly] public NativeArray<int> data;  // Multiple readers OK
    public void Execute(int i) { var x = data[i]; }
}
```

### 10.2 Why Separate RHI Thread?

```
Without RHI Thread:
  Render Thread: [Build Cmds][Translate to API][Submit]  ← bottleneck
                                                          API calls are slow!

With RHI Thread:
  Render Thread: [Build Cmds Frame N+1]                  ← free to work
  RHI Thread:    [Translate Frame N][Submit Frame N]     ← parallel!
```

| Benefit          | Description                                               |
| ---------------- | --------------------------------------------------------- |
| Parallelism      | Render thread builds next frame while RHI submits current |
| API Abstraction  | RHI thread handles DX12/Vulkan/Metal differences          |
| Command Batching | RHI thread can optimize/reorder API calls                 |
| Reduced Stalls   | Render thread never waits for GPU driver                  |

## 13. 常见 Bug 与调试

### 13.1 Common Multi-Threading Bugs

```
1. DATA RACE
   ┌─────────────────────────────────────────┐
   │ Thread A: x = 1;     Thread B: x = 2;   │
   │ Result: x = 1 or 2 (non-deterministic)  │
   │ Fix: Atomic or mutex                    │
   └─────────────────────────────────────────┘

2. DEADLOCK
   ┌─────────────────────────────────────────┐
   │ Thread A: Lock(M1) → Lock(M2)          │
   │ Thread B: Lock(M2) → Lock(M1)          │
   │ Fix: Consistent lock ordering           │
   └─────────────────────────────────────────┘

3. USE-AFTER-FREE (Async)
   ┌─────────────────────────────────────────┐
   │ Game Thread: Destroy Actor              │
   │ Worker: Still processing Actor's data   │
   │ Fix: Reference counting, completion     │
   │      fence before destruction           │
   └─────────────────────────────────────────┘

4. TORN READ
   ┌─────────────────────────────────────────┐
   │ Thread A writes 128-bit struct:         │
   │   Write high 64 bits → (preempted)      │
   │ Thread B reads: sees half-old half-new  │
   │ Fix: Atomic or mutex for >64bit data    │
   └─────────────────────────────────────────┘

5. ABA PROBLEM
   ┌─────────────────────────────────────────┐
   │ Thread A: Read ptr = A                  │
   │ Thread B: Change A→B→A (same address)   │
   │ Thread A: CAS succeeds (thinks unchanged)│
   │ Fix: Tagged pointer (counter + pointer)  │
   └─────────────────────────────────────────┘
```

## 14. 面试高频问题

### Q1: Game Thread 和 Render Thread 如何通信？

**Answer:**

```
通信机制：Command Queue (生产者-消费者模式)

Game Thread (Producer):
  ENQUEUE_RENDER_COMMAND(Name)([CapturedData](FRHICommandListImmediate& RHICmdList) {
      // Lambda captured data is the "message"
      // Runs on Render Thread
  });

Render Thread (Consumer):
  - Dequeues and executes commands in order
  - Commands are type-erased lambdas stored in ring buffer

Sync mechanisms:
  1. FRenderCommandFence: Game waits for Render to catch up
  2. FlushRenderingCommands(): Hard sync (stalls Game Thread)
  3. Scene Proxy pattern: Game Thread owns UPrimitiveComponent,
     Render Thread owns FPrimitiveSceneProxy (mirror object)

Data ownership rule:
  - Game Thread: UObject, UActorComponent, AActor
  - Render Thread: FSceneProxy, FMaterialRenderProxy
  - Never cross-access! Copy data through render commands.
```

### Q2: Work Stealing 为什么本地用 LIFO，偷取用 FIFO？

**Answer:**

```
Local LIFO:
  - Most recently pushed job is still in CPU cache (temporal locality)
  - Child jobs pushed after parent → execute children first (DFS)
  - DFS reduces peak memory usage of pending jobs

Steal FIFO:
  - Oldest jobs tend to be parent/root jobs → larger work units
  - Stealing a large job amortizes the overhead of cross-thread sync
  - Stealer accesses TOP of deque, owner accesses BOTTOM
    → minimal contention (different ends of data structure)
  - BFS from stealer's perspective → better load distribution

Combined effect:
  - Local execution: DFS (cache-friendly, low memory)
  - Global distribution: BFS (large chunks, low contention)
  - This is provably optimal: O(T₁/P + T∞) expected time
    where T₁ = total work, P = processors, T∞ = critical path
```

### Q3: 如何避免多线程中的 False Sharing？

**Answer:**

```
False Sharing: Two threads modify different variables that share
the same cache line (64 bytes), causing constant cache invalidation.

Detection:
  - perf c2c (Linux): identifies cache line contention
  - VTune: shows cache line sharing between threads
  - Symptom: parallel code slower than expected, high L1 miss rate

Solutions:
  1. Padding: alignas(64) to separate cache lines
  2. Thread-local accumulation: each thread writes to own buffer,
     merge at end (reduce pattern)
  3. Data layout: SoA instead of AoS for parallel iteration
  4. Avoid shared counters: use per-thread counters, sum at sync point

Example:
  // BAD: 4 threads updating adjacent array elements
  int results[4];  // All on same cache line!
  ParallelFor(4, [&](int i) { results[i] = Compute(i); });

  // GOOD: Padded
  struct alignas(64) PaddedInt { int value; };
  PaddedInt results[4];  // Each on own cache line
  ParallelFor(4, [&](int i) { results[i].value = Compute(i); });
```

### Q4: Fiber-Based Job System 的优势是什么？

**Answer:**

```
Traditional Thread-Based:
  Worker thread executes Job A → Job A needs to wait for Job B
  → Thread BLOCKS (wastes a core!) or must return and re-schedule

Fiber-Based (Naughty Dog approach):
  Fiber executes Job A → Job A needs to wait for Job B
  → Fiber SUSPENDS (saves stack to memory)
  → Worker thread picks up another Fiber immediately
  → When Job B completes, Job A's Fiber is RESUMED on any worker

Advantages:
  1. No thread blocking: Workers never idle
  2. Natural wait semantics: Code reads like synchronous
  3. No callback hell: WaitForCounter() suspends, resumes later
  4. Better utilization: N fibers on M threads (N >> M)

Disadvantages:
  1. Stack memory: Each fiber needs its own stack (~64KB)
  2. Complexity: Fiber migration between threads
  3. TLS issues: Thread-local storage doesn't work with fibers
  4. Debugging: Call stacks span multiple fibers
  5. Platform support: Not all platforms have native fiber API

Naughty Dog's implementation:
  - 160 fibers, 6 worker threads
  - 64KB stack per fiber
  - WaitForCounter() = suspend current fiber
  - ~1μs context switch (vs ~10μs thread switch)
```

### Q5: Unity Job System 的 Safety System 如何工作？

**Answer:**

```
Unity's Safety System (Editor/Development builds only):

1. Each NativeContainer has an AtomicSafetyHandle
2. When scheduling a job, the system records:
   - Which containers the job reads ([ReadOnly])
   - Which containers the job writes (default)
3. At Schedule() time, checks:
   - No two jobs write to same container without dependency
   - No job reads while another writes without dependency
   - Container not disposed while job is running
4. At runtime, checks:
   - Container access from correct thread
   - Container not accessed after Dispose()
   - Index bounds checking

This catches 90%+ of threading bugs at development time.
Zero overhead in Release builds (checks compiled out).

Limitation:
  - Only works with NativeContainers (not raw pointers)
  - Cannot detect races in unmanaged memory
  - Burst-compiled code bypasses some checks
```

### Q6: 渲染线程和 RHI 线程为什么要分开？

**Answer:**

```
Render Thread responsibilities:
  - Scene traversal, visibility culling
  - Material sorting, draw call batching
  - Generate platform-agnostic FRHICommands
  - CPU-intensive, benefits from running ahead

RHI Thread responsibilities:
  - Translate FRHICommands to DX12/Vulkan/Metal API calls
  - Manage GPU resources (buffers, textures, PSOs)
  - Submit command buffers to GPU
  - Often blocked by GPU driver

Why separate:
  1. Render Thread can build Frame N+1 commands while
     RHI Thread submits Frame N to GPU
  2. GPU driver calls can stall (shader compilation, resource creation)
     → only RHI Thread stalls, Render Thread continues
  3. DX12/Vulkan allow parallel command recording:
     Multiple workers record, RHI Thread submits
  4. Platform abstraction: RHI Thread handles API differences

Without RHI Thread (DX11 model):
  Render Thread: [Build][Translate][Submit][Wait]  = 12ms

With RHI Thread:
  Render Thread: [Build N+1]                       = 6ms
  RHI Thread:    [Translate N][Submit N]            = 6ms (parallel!)
  Total frame time: 6ms (not 12ms)
```

### Q7: 如何处理多线程中的确定性问题？

**Answer:**

```
Problem: Floating-point addition is not associative
  (a + b) + c ≠ a + (b + c) in floating point
  → Different thread scheduling = different accumulation order
  → Non-deterministic results

Solutions:

1. Fixed accumulation order:
   - ParallelFor produces per-thread partial results
   - Single thread reduces in fixed order
   - Deterministic but slightly slower

2. Kahan summation:
   - Compensated summation reduces order-dependency
   - Not fully deterministic but much more stable

3. Integer/fixed-point math:
   - Used in lockstep networking (RTS games)
   - Fully deterministic across platforms
   - Less precision, more complex code

4. Deterministic scheduling:
   - Assign work by index, not by thread
   - Same input → same thread assignment → same result
   - Requires careful job system design

Game engine approach:
  - Physics: Deterministic mode (fixed iteration order)
  - Networking: Server-authoritative (only server result matters)
  - Replay: Record inputs, not state (requires determinism)
```

### Q8: ParallelFor 的 BatchSize 如何选择？

**Answer:**

```
BatchSize = number of items processed per job

Too small (e.g., 1):
  - Overhead: Job creation + scheduling > actual work
  - Cache thrashing: each job touches different memory
  - Typical job overhead: 1-5μs

Too large (e.g., N/2):
  - Only 2 jobs → poor load balance
  - Some cores idle while others still working

Optimal formula:
  BatchSize ≈ TotalItems / (NumCores × 4)
  
  Why ×4? Oversubscription factor for load balancing.
  If one batch finishes early, thread steals another.

Practical guidelines:
  ┌──────────────────────────────────────────────┐
  │ Work per item  │ Recommended BatchSize        │
  ├────────────────┼──────────────────────────────┤
  │ Trivial (<1μs) │ 256-1024 (amortize overhead) │
  │ Light (1-10μs) │ 64-256                       │
  │ Medium (10-100μs)│ 16-64                      │
  │ Heavy (>100μs) │ 1-16                         │
  └──────────────────────────────────────────────┘

Profile and adjust! Use Tracy/Superluminal to see
if workers are idle (batch too large) or thrashing
(batch too small).
```

## 15. 实践检查清单

### Architecture
- [ ] Identified dedicated threads (Game, Render, RHI, Audio)
- [ ] Worker thread count = max(1, NumCores - 3) on mobile
- [ ] Frame pipeline overlap configured (Game ahead of Render)
- [ ] Triple buffering for CPU-GPU data exchange

### Job System
- [ ] Fine-grained jobs with DAG dependencies
- [ ] ParallelFor for data-parallel workloads
- [ ] Appropriate batch sizes (profiled, not guessed)
- [ ] Work stealing enabled for load balancing
- [ ] No blocking waits inside jobs (use fiber/suspend or split)

### Synchronization
- [ ] Minimal lock scope (FScopeLock, not manual lock/unlock)
- [ ] Consistent lock ordering to prevent deadlock
- [ ] Prefer atomics over mutexes for simple counters/flags
- [ ] Memory ordering correct (acquire/release, not all seq_cst)
- [ ] No false sharing (alignas(64) for per-thread data)

### Data Safety
- [ ] Clear ownership: Game Thread data vs Render Thread data
- [ ] Scene Proxy pattern for cross-thread object access
- [ ] Render command fence before destroying shared resources
- [ ] NativeContainer safety (Unity) or thread assertions (UE)

### Mobile Specific
- [ ] Thread affinity for critical threads (Game → big core)
- [ ] Background work on efficiency cores
- [ ] Thermal budget considered (don't max all cores)
- [ ] Reduced worker count on low-end devices
- [ ] Larger batch sizes (higher context switch cost)

### Performance
- [ ] Thread timeline profiled (no idle gaps)
- [ ] Lock contention measured and minimized
- [ ] Context switch count acceptable (<1000/frame)
- [ ] Cache miss rate acceptable per thread
- [ ] No FlushRenderingCommands() in hot path

### Debugging
- [ ] ThreadSanitizer enabled in CI builds
- [ ] Thread assertions (IsInGameThread()) at API boundaries
- [ ] Deterministic mode available for replay/debugging
- [ ] Tracy/Insights markers on all significant tasks

---

> **Key Takeaway**: Modern game engine multi-threading is not about "more threads = faster". It's about **minimizing synchronization**, **maximizing data locality**, and **balancing load** across heterogeneous cores. The job system with work stealing is the industry standard, but the real challenge is designing data flow to minimize cross-thread dependencies.
