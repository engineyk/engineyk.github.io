# Unreal 网络同步

---

## 1. 网络架构总览

### 1.1 Client-Server 权威模型

Unreal 采用 **Server 权威（Server Authoritative）** 架构，Server 是唯一的真相来源（Source of Truth）。

```
┌───────────────────────────────────────────────────┐
│              Network Topology                     │
│                                                   │
│              ┌──────────────┐                     │
│              │   Server     │                     │
│              │  (Authority) │                     │
│              │              │                     │
│              │ • 游戏逻辑仲裁│                     │
│              │ • 状态同步源  │                     │
│              │ • 反作弊验证  │                     │
│              └──────┬───────┘                     │
│                     │                             │
│          ┌──────────┼──────────┐                  │
│          │          │          │                  │
│    ┌─────▼────┐ ┌───▼──────┐ ┌▼─────────┐         │
│    │ Client 0 │ │ Client 1 │ │ Client 2 │         │
│    │(Listen   │ │(Remote)  │ │(Remote)  │         │
│    │ Server)  │ │          │ │          │         │
│    └──────────┘ └──────────┘ └──────────┘         │
└───────────────────────────────────────────────────┘
```

### 1.2 网络角色（Network Role）

每个 Actor 有两个角色属性，决定了它在网络中的身份：

```cpp
// Actor.h
ENetRole GetLocalRole() const;   // 本地角色
ENetRole GetRemoteRole() const;  // 远端角色
```

| ENetRole               | 值  | 含义                                              |
| ---------------------- | --- | ------------------------------------------------- |
| `ROLE_None`            | 0   | 无网络角色，不参与同步                            |
| `ROLE_SimulatedProxy`  | 1   | 模拟代理 - 客户端上非本地控制的 Actor 副本        |
| `ROLE_AutonomousProxy` | 2   | 自治代理 - 客户端上本地控制的 Actor（如玩家角色） |
| `ROLE_Authority`       | 3   | 权威 - Server 上的 Actor，拥有最终决定权          |

**各端角色分布：**

```
Server 端:
  所有 Actor → LocalRole = ROLE_Authority

Owning Client 端（拥有者客户端）:
  自己的 Pawn → LocalRole = ROLE_AutonomousProxy
  其他 Actor  → LocalRole = ROLE_SimulatedProxy

Other Client 端（其他客户端）:
  所有 Actor  → LocalRole = ROLE_SimulatedProxy
```

### 1.3 网络模式（Network Mode）

```cpp
ENetMode GetNetMode() const;
```

| ENetMode             | 说明                            | HasAuthority() | IsNetMode(NM_Client) |
| -------------------- | ------------------------------- | -------------- | -------------------- |
| `NM_Standalone`      | 单机模式                        | true           | false                |
| `NM_DedicatedServer` | 专用服务器（无渲染）            | true           | false                |
| `NM_ListenServer`    | 监听服务器（Server + Client 0） | true           | false                |
| `NM_Client`          | 纯客户端                        | false          | true                 |

### 1.4 连接与通道架构

```
┌─────────────────────────────────────────────────┐
│              UNetDriver                         │
│  ┌───────────────────────────────────────────┐  │
│  │         UNetConnection (per client)       │  │
│  │  ┌─────────────────────────────────────┐  │  │
│  │  │  UChannel (多种类型)                 │  │  │
│  │  │                                     │  │  │
│  │  │  ControlChannel (Ch 0) - 连接控制    │  │  │
│  │  │  VoiceChannel   (Ch 1) - 语音数据    │  │  │
│  │  │  ActorChannel   (Ch 2+) - Actor 同步 │  │  │
│  │  │    每个被同步的 Actor 占一个 Channel  │  │  │
│  │  └─────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────┘  │
│                                                 │
│  底层传输: UDP (可靠/不可靠)                      │
│  序列化:   FBitWriter / FBitReader               │
│  可靠性:   自定义 ACK 机制 (非 TCP)               │
└─────────────────────────────────────────────────┘
```

**关键数据流：**

```
Server Tick:
  1. UWorld::Tick()
  2. UNetDriver::TickFlush()
  3. UNetDriver::ServerReplicateActors()     ← 核心入口
     3.1 BuildConsiderList()                 ← 收集需要同步的 Actor
     3.2 ServerReplicateActors_PrioritizeActors() ← 优先级排序
     3.3 ServerReplicateActors_ProcessPrioritizedActors() ← 执行同步
         → UActorChannel::ReplicateActor()
             → FObjectReplicator::ReplicateProperties() ← 属性同步
             → Actor::ReplicateSubobjects()             ← 子对象同步
```

## 2. Actor 同步（Replication）基础

### 2.1 开启同步

```cpp
// 构造函数中
AMyActor::AMyActor()
{
    // 必须：开启 Actor 同步
    bReplicates = true;
    
    // 可选：开启移动同步
    bReplicateMovement = true;
    
    // 可选：同步频率（每秒最大同步次数）
    NetUpdateFrequency = 100.0f;    // 默认 100Hz
    MinNetUpdateFrequency = 2.0f;   // 最低 2Hz（自适应）
    
    // 可选：网络优先级（越高越优先同步）
    NetPriority = 1.0f;             // 默认 1.0
    // PlayerController = 3.0, Pawn = 3.0, PlayerState = 1.0
    
    // 可选：网络剔除距离
    NetCullDistanceSquared = 225000000.0f; // 15000 units
    
    // 可选：是否只同步给 Owner
    bOnlyRelevantToOwner = false;
    
    // 可选：是否在初始同步时使用 NetLoadOnClient
    bNetLoadOnClient = true;
}
```

### 2.2 Actor 生命周期与网络

```
┌─────────────────────────────────────────────────────────┐
│              Actor 网络生命周期                          │
│                                                         │
│  Server 端:                                             │
│    SpawnActor() → BeginPlay() → Tick() → Destroy()      │
│         │                                    │          │
│         ▼ (同步 Spawn 到客户端)               ▼          │
│    Open ActorChannel              Close ActorChannel    │
│         │                                    │          │
│  Client 端:                                             │
│    PostNetInit() → BeginPlay() → Tick() → EndPlay()     │
│         │                                               │
│         ▼                                               │
│    OnRep_xxx() 回调触发                                  │
│    (初始属性同步完成后)                                   │
└─────────────────────────────────────────────────────────┘

关键时序:
  1. Server: SpawnActor()
  2. Server: 属性初始化
  3. Server: BeginPlay()
  4. --- 网络传输 ---
  5. Client: Actor 创建（PostNetInit）
  6. Client: 初始属性同步 + OnRep 回调
  7. Client: BeginPlay()
```

### 2.3 Relevancy（相关性）

Server 不会把所有 Actor 同步给所有 Client，而是通过 **Relevancy** 判断：

```cpp
// 判断 Actor 是否与某个 Connection 相关
virtual bool IsNetRelevantFor(
    const AActor* RealViewer,      // PlayerController
    const AActor* ViewTarget,      // 视角目标（通常是 Pawn）
    const FVector& SrcLocation     // 观察位置
) const;
```

**Relevancy 判断规则（按优先级）：**

```
1. bAlwaysRelevant = true                       → 始终相关（如 GameState）
2. bNetUseOwnerRelevancy = true                 → 使用 Owner 的相关性
3. bOnlyRelevantToOwner = true                  → 仅对 Owner 相关（如 PlayerController）
4. Actor 在 Viewer 的 HiddenActors 中            → 不相关
5. Actor 有 APlayerState Owner                   → 相关
6. 距离检查: Dist² < NetCullDistanceSquared      → 相关
7. 以上都不满足                                  → 不相关
```

**Dormancy（休眠）优化：**

```cpp
// 休眠状态 - 减少不必要的同步检查
ENetDormancy NetDormancy;

// DORM_Never           - 永不休眠（始终检查同步）
// DORM_Awake           - 当前唤醒状态
// DORM_DormantAll      - 对所有连接休眠
// DORM_DormantPartial  - 对最近无变化的连接休眠
// DORM_Initial         - 初始休眠，首次变为相关时唤醒

// 手动唤醒
FlushNetDormancy();  // 唤醒并强制同步

// 手动设置休眠
SetNetDormancy(DORM_DormantAll);
```

## 3. 属性同步（Property Replication）

### 3.1 基本用法

```cpp
// .h 文件
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()
    
    // 方式1: 仅同步，无回调
    UPROPERTY(Replicated)
    float Health;
    
    // 方式2: 同步 + 值变化时触发回调
    UPROPERTY(ReplicatedUsing = OnRep_Armor)
    float Armor;
    
    UFUNCTION()
    void OnRep_Armor();  // 仅在客户端调用
    
    // 方式3: 同步 + 回调带旧值参数（UE5.1+）
    UPROPERTY(ReplicatedUsing = OnRep_Score)
    int32 Score;
    
    UFUNCTION()
    void OnRep_Score(int32 OldScore);  // 可获取变化前的值
    
    // 必须重写
    virtual void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps
    ) const override;
};

// .cpp 文件
void AMyCharacter::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);
    
    // 基本同步
    DOREPLIFETIME(AMyCharacter, Health);
    
    // 带条件的同步
    DOREPLIFETIME_CONDITION(AMyCharacter, Armor, COND_OwnerOnly);
    
    // 自定义条件
    DOREPLIFETIME_CONDITION(AMyCharacter, Score, COND_None);
}
```

### 3.2 同步条件（Replication Condition）

```cpp
// ELifetimeCondition 枚举
COND_None                // 无条件，始终同步
COND_InitialOnly         // 仅初始同步一次（如角色名字）
COND_OwnerOnly           // 仅同步给 Owner（如弹药数）
COND_SkipOwner           // 同步给除 Owner 外的所有人
COND_SimulatedOnly       // 仅同步给 SimulatedProxy
COND_AutonomousOnly      // 仅同步给 AutonomousProxy
COND_SimulatedOrPhysics  // SimulatedProxy 或物理模拟的 Actor
COND_InitialOrOwner      // 初始同步 或 Owner
COND_Custom              // 自定义条件（需配合 DOREPLIFETIME_ACTIVE_OVERRIDE）
COND_ReplayOrOwner       // Replay 或 Owner
COND_ReplayOnly          // 仅 Replay
COND_SkipReplay          // 跳过 Replay
COND_Dynamic             // 动态条件（运行时可变）UE5
COND_Never               // 永不同步（运行时可改为其他条件）
```

**动态条件控制（UE5 Push Model）：**

```cpp
// UE5 Push Model - 手动标记属性脏
// 比传统的每帧比较更高效

// .h
UPROPERTY(ReplicatedUsing = OnRep_Health)
float Health;

// .cpp - 修改属性时手动标脏
void AMyCharacter::SetHealth(float NewHealth)
{
    if (Health != NewHealth)
    {
        Health = NewHealth;
        
        // Push Model: 手动标记脏
        MARK_PROPERTY_DIRTY_FROM_NAME(AMyCharacter, Health, this);
        
        // 如果 Actor 处于休眠状态，需要唤醒
        ForceNetUpdate();
    }
}

// GetLifetimeReplicatedProps 中
FDoRepLifetimeParams Params;
Params.bIsPushBased = true;  // 启用 Push Model
DOREPLIFETIME_WITH_PARAMS_FAST(AMyCharacter, Health, Params);
```

### 3.3 属性同步内部原理

```
┌─────────────────────────────────────────────────────────┐
│              属性同步流程                                │
│                                                         │
│  Server 端:                                             │
│  ┌─────────────────────────────────────────────┐        │
│  │ FObjectReplicator::ReplicateProperties()    │        │
│  │                                             │        │
│  │  1. 遍历 FRepLayout 中的属性列表             │        │
│  │  2. 对比当前值 vs Shadow State（上次同步值）  │        │
│  │     ┌──────────────┐  ┌──────────────┐     │         │
│  │     │ Current Data │  │ Shadow State │     │         │
│  │     │ Health = 80  │  │ Health = 100 │      │        │
│  │     └──────┬───────┘  └──────┬───────┘      │        │
│  │            │    比较 ≠        │              │       │
│  │            └────────┬────────┘              │        │
│  │                     ▼                       │        │
│  │  3. 值不同 → 序列化差异数据到 FBitWriter     │         │
│  │  4. 更新 Shadow State = Current Data        │        │
│  │  5. 通过 UActorChannel 发送                  │       │
│  └─────────────────────────────────────────────┘        │
│                                                         │
│  Client 端:                                             │
│  ┌─────────────────────────────────────────────┐        │
│  │ FObjectReplicator::ReceivedBunch()          │       │
│  │                                             │       │
│  │  1. 反序列化属性数据                          │       │
│  │  2. 写入 Actor 属性                          │       │
│  │  3. 调用 OnRep_xxx() 回调                    │       │
│  │     (仅当值确实发生变化时)                     │       │
│  └─────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────┘

Push Model vs 传统模型:

传统模型 (UE4 默认):
  每次 ReplicateProperties() 都遍历所有属性做 memcmp
  O(N) per actor per frame，N = 属性数量
  
Push Model (UE5):
  仅检查 DirtyBit 标记
  O(1) 检查，仅序列化标脏的属性
  大量 Actor 时性能提升 30-50%
```

### 3.4 支持同步的类型

```cpp
// 基本类型 - 直接支持
UPROPERTY(Replicated) bool bFlag;
UPROPERTY(Replicated) int32 Count;
UPROPERTY(Replicated) float Value;

UPROPERTY(Replicated) FMyData Data;

### 3.5 FFastArraySerializer（高效数组同步）

普通 TArray 同步时，任何元素变化都会重新发送整个数组。FFastArraySerializer 支持 **增量同步**：

```cpp
// 定义数组元素
USTRUCT()
struct FInventoryItem : public FFastArraySerializerItem
{
    GENERATED_BODY()
    
    UPROPERTY()
    int32 ItemID;
    
    UPROPERTY()
    int32 Count;
    
    UPROPERTY()
    float Durability;
    
    // 元素变化回调
    void PreReplicatedRemove(const FInventoryArray& InArraySerializer);
    void PostReplicatedAdd(const FInventoryArray& InArraySerializer);
    void PostReplicatedChange(const FInventoryArray& InArraySerializer);
};

// 定义数组容器
USTRUCT()
struct FInventoryArray : public FFastArraySerializer
{
    GENERATED_BODY()
    
    UPROPERTY()
    TArray<FInventoryItem> Items;
    
    // 必须实现
    bool NetDeltaSerialize(
        FNetDeltaSerializeInfo& DeltaParms)
    {
        return FFastArraySerializer::FastArrayDeltaSerialize<
            FInventoryItem, FInventoryArray>(Items, DeltaParms, *this);
    }
};

// 在 Actor 中使用
UPROPERTY(Replicated)
FInventoryArray Inventory;

// 修改时必须调用 MarkItemDirty
void AddItem(int32 ID, int32 Num)
{
    FInventoryItem& NewItem = Inventory.Items.AddDefaulted_GetRef();
    NewItem.ItemID = ID;
    NewItem.Count = Num;
    Inventory.MarkItemDirty(NewItem);  // 关键！标记脏
}

void RemoveItem(int32 Index)
{
    Inventory.Items.RemoveAt(Index);
    Inventory.MarkArrayDirty();  // 删除时标记整个数组
}
```

**FFastArraySerializer vs TArray 同步对比：**

| 特性     | TArray                     | FFastArraySerializer              |
| -------- | -------------------------- | --------------------------------- |
| 增量同步 | ❌ 全量发送                 | ✅ 仅发送变化的元素                |
| 元素回调 | ❌ 仅 OnRep 整个数组        | ✅ Add/Remove/Change 独立回调      |
| 带宽消耗 | 高（100个元素改1个也全发） | 低（仅发送1个元素的变化）         |
| 适用场景 | 小数组（<10个元素）        | 大数组（背包、技能列表等）        |
| 复杂度   | 简单                       | 需要继承 FFastArraySerializerItem |

## 4. RPC（Remote Procedure Call）

### 4.1 三种 RPC 类型

```cpp
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()
    
    // ═══════════════════════════════════════════
    // Server RPC: Client → Server
    // 在客户端调用，在服务器执行
    // ═══════════════════════════════════════════
    UFUNCTION(Server, Reliable, WithValidation)
    void ServerFireWeapon(FVector_NetQuantize AimLocation);
    
    // 验证函数 - 防作弊
    bool ServerFireWeapon_Validate(FVector_NetQuantize AimLocation);
    
    // 实现函数
    void ServerFireWeapon_Implementation(FVector_NetQuantize AimLocation);
    
    // ═══════════════════════════════════════════
    // Client RPC: Server → Owning Client
    // 在服务器调用，在拥有者客户端执行
    // ═══════════════════════════════════════════
    UFUNCTION(Client, Reliable)
    void ClientShowDamageNumber(float Damage, FVector WorldLocation);
    
    void ClientShowDamageNumber_Implementation(
        float Damage, FVector WorldLocation);
    
    // ═══════════════════════════════════════════
    // Multicast RPC: Server → All Clients + Server
    // 在服务器调用，在所有客户端和服务器执行
    // ═══════════════════════════════════════════
    UFUNCTION(NetMulticast, Unreliable)
    void MulticastPlayExplosionFX(FVector Location, float Radius);
    
    void MulticastPlayExplosionFX_Implementation(
        FVector Location, float Radius);
};
```

### 4.2 RPC 执行矩阵

```
┌──────────────────────────────────────────────────────────────┐
│                    RPC 调用 → 执行 矩阵                       │
│                                                              │
│  调用端 \ RPC类型  │  Server RPC  │  Client RPC  │ Multicast │
│  ─────────────────┼──────────────┼──────────────┼───────────│
│  Server           │  Server执行   │  Client执行   │ 全部执行   │
│  Owning Client    │  Server执行   │  Client执行   │ Client执行 │
│  Other Client     │  丢弃❌       │  丢弃❌       │ 丢弃❌     │
│  Unowned Actor    │  丢弃❌       │  丢弃❌       │ 丢弃❌     │
└──────────────────────────────────────────────────────────────┘

关键规则:
  • Server RPC 只能由 Owning Client 调用
  • Client RPC 只能由 Server 调用
  • Multicast RPC 只能由 Server 调用
  • 非 Owner 调用 Server RPC → 静默丢弃
```

### 4.3 Reliable vs Unreliable

```cpp
// Reliable - 保证送达，有序
// 适用: 关键游戏逻辑（开火、购买、技能释放）
UFUNCTION(Server, Reliable)
void ServerUseAbility(int32 AbilityID);

// Unreliable - 不保证送达，可能丢失
// 适用: 高频非关键数据（特效、音效、UI提示）
UFUNCTION(NetMulticast, Unreliable)
void MulticastPlayHitEffect(FVector Location);
```

**⚠️ Reliable RPC 的陷阱：**

```
问题: Reliable RPC 队列溢出

  Reliable RPC 使用有限大小的可靠缓冲区
  如果短时间内发送大量 Reliable RPC:
    → 缓冲区溢出
    → 连接断开！

典型错误:
  // ❌ 错误：每帧发送 Reliable RPC
  void Tick(float DeltaTime)
  {
      ServerUpdatePosition(GetActorLocation()); // Reliable
  }
  
  // ✅ 正确：高频数据用属性同步或 Unreliable RPC
  UPROPERTY(Replicated)
  FVector ReplicatedPosition;

经验法则:
  • Reliable RPC: < 10次/秒/Actor
  • 高频数据: 用属性同步
  • 视觉效果: 用 Unreliable Multicast
```

### 4.4 WithValidation（反作弊验证）

```cpp
// Server RPC 可以添加 WithValidation
// 验证失败 → 踢出玩家
UFUNCTION(Server, Reliable, WithValidation)
void ServerPurchaseItem(int32 ItemID, int32 Quantity);

bool ServerPurchaseItem_Validate(int32 ItemID, int32 Quantity)
{
    // 验证参数合法性
    if (ItemID < 0 || ItemID > MAX_ITEM_ID)
        return false;  // → 踢出玩家
    
    if (Quantity <= 0 || Quantity > 99)
        return false;  // → 踢出玩家
    
    return true;  // 验证通过，执行 Implementation
}

void ServerPurchaseItem_Implementation(int32 ItemID, int32 Quantity)
{
    // Server 端执行购买逻辑
    // 这里还需要做服务器端验证（金币是否足够等）

## 6. 角色移动同步（Character Movement Replication）

### 6.1 UCharacterMovementComponent 同步架构

这是 Unreal 网络同步中最复杂也最精妙的部分：

```
┌─────────────────────────────────────────────────────────────┐
│           Character Movement 网络同步流程                     │
│                                                             │
│  Owning Client (AutonomousProxy):                          │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 1. 收集输入 (WASD, Jump, etc.)                        │  │
│  │ 2. 本地预测执行移动 (PerformMovement)                  │  │
│  │ 3. 保存移动到 SavedMoves 列表                         │  │
│  │ 4. 发送 ServerMove RPC 到 Server                      │  │
│  │    (包含: 时间戳, 加速度, 位置, 旋转)                   │  │
│  └───────────────────────────────────────────────────────┘  │
│                          │                                  │
│                          ▼                                  │
│  Server (Authority):                                        │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 1. 收到 ServerMove                                    │  │
│  │ 2. 用客户端的输入重新执行移动                           │  │
│  │ 3. 比较 Server 结果 vs Client 报告的位置               │  │
│  │ 4. if (误差 > 容忍阈值):                               │  │
│  │       发送 ClientAdjustPosition (纠正)                 │  │
│  │    else:                                              │  │
│  │       发送 ClientAckGoodMove (确认)                    │  │
│  │ 5. 属性同步位置给其他客户端                             │  │
│  └───────────────────────────────────────────────────────┘  │
│                          │                                  │
│                          ▼                                  │
│  Owning Client (收到纠正时):                                │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 1. 收到 ClientAdjustPosition                          │  │
│  │ 2. 将位置设为 Server 的权威位置                         │  │
│  │ 3. 从 SavedMoves 中找到对应时间戳                      │  │
│  │ 4. 重放 (Replay) 该时间戳之后的所有输入                 │  │
│  │    → 这就是 "回滚重放" (Rollback & Replay)             │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  Other Clients (SimulatedProxy):                            │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 1. 收到属性同步的位置/速度                              │  │
│  │ 2. 插值平滑到目标位置                                  │  │
│  │    (SimulatedProxy 不做预测，只做插值)                   │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

│              客户端预测时间线                                  │
│                                                             │
│  Client Timeline:                                           │
│  ──┬────┬────┬────┬────┬────┬────┬────┬──→ Time            │
│    │ M1 │ M2 │ M3 │ M4 │ M5 │ M6 │ M7 │                   │
│    │    │    │    │    │    │    │    │                      │
│    └─┬──┘    │    │    │    │    │    │                      │
│      │       │    │    │    │    │    │                      │
│      ▼       │    │    │    │    │    │                      │
│  Send M1     │    │    │    │    │    │                      │

## 8. Replication Graph

### 8.1 概述

默认的 Actor 同步系统对每个连接遍历所有 Actor，复杂度为 **O(Actors × Connections)**。Replication Graph 通过空间划分和策略分组大幅优化：

```
┌─────────────────────────────────────────────────────────────┐
│              默认 vs Replication Graph                       │
│                                                             │
│  默认系统:                                                   │
│    每帧: 遍历所有 Actor × 所有 Connection                    │
│    1000 Actors × 100 Players = 100,000 次检查               │
│    大部分检查结果是 "不相关"，浪费 CPU                        │
│                                                             │
│  Replication Graph:                                         │
│    Actor 按策略分组到不同 Node                               │
│    每个 Node 有自己的高效查询逻辑                             │
│    空间网格: O(1) 查询附近 Actor                             │
│    AlwaysRelevant: 直接返回，无需距离检查                     │
│                                                             │
│  性能提升: Fortnite 实测 Server CPU 降低 30-50%              │
└─────────────────────────────────────────────────────────────┘
```

### 8.2 内置 Node 类型

```cpp

// MyReplicationGraph.h
UCLASS()
class UMyReplicationGraph : public UReplicationGraph
{
    GENERATED_BODY()
    
public:
    virtual void InitGlobalActorClassSettings() override;
    virtual void InitGlobalGraphNodes() override;
    virtual void InitConnectionGraphNodes(
        UNetReplicationGraphConnection* Conn) override;
    virtual void RouteAddNetworkActorToNodes(
        const FNewReplicatedActorInfo& ActorInfo,
        FGlobalActorReplicationInfo& GlobalInfo) override;
    virtual void RouteRemoveNetworkActorToNodes(
        const FNewReplicatedActorInfo& ActorInfo) override;
    
private:
    // 全局节点
    UReplicationGraphNode_GridSpatialization2D* GridNode;
    UReplicationGraphNode_ActorList* AlwaysRelevantNode;
    
    // 类别映射
    TMap<UClass*, EClassRepPolicy> ClassRepPolicies;
};

// MyReplicationGraph.cpp
enum class EClassRepPolicy : uint8
{
    NotRouted,           // 不通过 RepGraph 路由
    RelevantAllConnections, // 对所有连接相关
    Spatialize_Static,   // 空间化 - 静态（不移动）
    Spatialize_Dynamic,  // 空间化 - 动态（会移动）
    Spatialize_Dormancy, // 空间化 + 休眠优化
};

void UMyReplicationGraph::InitGlobalActorClassSettings()
{
    Super::InitGlobalActorClassSettings();
    
    // 为每个类设置同步策略
    ClassRepPolicies.Add(APlayerState::StaticClass(), 
        EClassRepPolicy::RelevantAllConnections);
    ClassRepPolicies.Add(AGameStateBase::StaticClass(), 
        EClassRepPolicy::RelevantAllConnections);
    
    // 静态物体（宝箱、门等）
    ClassRepPolicies.Add(AInteractableActor::StaticClass(), 
        EClassRepPolicy::Spatialize_Static);
    
    // 动态物体（NPC、投射物等）
    ClassRepPolicies.Add(AEnemyCharacter::StaticClass(), 
        EClassRepPolicy::Spatialize_Dynamic);
    
    // 玩家角色 - 动态 + 休眠
    ClassRepPolicies.Add(AMyCharacter::StaticClass(), 
        EClassRepPolicy::Spatialize_Dormancy);
    
    // 设置全局 Actor 信息
    for (auto& Pair : ClassRepPolicies)
    {
        FClassReplicationInfo& Info = 
            GlobalActorReplicationInfoMap.FindOrAdd(Pair.Key);
        
        // 根据策略设置不同的剔除距离
        switch (Pair.Value)
        {
        case EClassRepPolicy::Spatialize_Static:
            Info.SetCullDistanceSquared(150000.f * 150000.f);
            break;
        case EClassRepPolicy::Spatialize_Dynamic:
            Info.SetCullDistanceSquared(100000.f * 100000.f);
            break;
        default:
            break;
        }
    }
}

void UMyReplicationGraph::InitGlobalGraphNodes()
{
    Super::InitGlobalGraphNodes();
    
    // 创建空间网格节点
    GridNode = CreateNewNode<
        UReplicationGraphNode_GridSpatialization2D>();
    GridNode->CellSize = 10000.f;  // 网格单元大小
    GridNode->SpatialBias = FVector2D(-200000.f, -200000.f);
    
    // 创建始终相关节点
    AlwaysRelevantNode = CreateNewNode<
        UReplicationGraphNode_ActorList>();
    
    AddGlobalGraphNode(GridNode);
    AddGlobalGraphNode(AlwaysRelevantNode);
}

void UMyReplicationGraph::RouteAddNetworkActorToNodes(
    const FNewReplicatedActorInfo& ActorInfo,
    FGlobalActorReplicationInfo& GlobalInfo)
{
    EClassRepPolicy Policy = GetClassPolicy(ActorInfo.Class);
    
    switch (Policy)
    {
    case EClassRepPolicy::RelevantAllConnections:
        AlwaysRelevantNode->NotifyAddNetworkActor(ActorInfo);
        break;
        
    case EClassRepPolicy::Spatialize_Static:
    case EClassRepPolicy::Spatialize_Dynamic:
    case EClassRepPolicy::Spatialize_Dormancy:
        GridNode->AddActor_Dormancy(ActorInfo, GlobalInfo);
        break;
        
    default:
        break;
    }
}

USTRUCT()
struct FMyCompressedVector
{
    GENERATED_BODY()
    
    float X, Y, Z;
    
    bool NetSerialize(FArchive& Ar, UPackageMap* Map, bool& bOutSuccess)
    {
        // 量化为 16 位整数，节省带宽
        // 范围 [-32768, 32767] 映射到 [-327.68, 327.67]
        if (Ar.IsSaving())
        {
            int16 QX = FMath::Clamp(
                FMath::RoundToInt(X * 100.f), -32768, 32767);
            int16 QY = FMath::Clamp(
                FMath::RoundToInt(Y * 100.f), -32768, 32767);
            int16 QZ = FMath::Clamp(
                FMath::RoundToInt(Z * 100.f), -32768, 32767);
            Ar << QX << QY << QZ;
        }
        else
        {
            int16 QX, QY, QZ;
            Ar << QX << QY << QZ;
            X = QX / 100.f;
            Y = QY / 100.f;
            Z = QZ / 100.f;
        }
        
        bOutSuccess = true;
        return true;
    }
};

FVector_NetQuantizeNormal Normal;  // 2 bytes (16-bit)

// FRotator 网络压缩
// CompressedRotation: 每轴 1 byte (256级 = 1.4° 精度)
// ShortCompressedRotation: 每轴 2 bytes (65536级 = 0.005° 精度)

void SerializeRotation(FArchive& Ar)
{
    // 手动压缩旋转
    FRotator Rot = GetActorRotation();
    
    // Byte 压缩: 3 bytes total
    uint8 ByteYaw = FRotator::CompressAxisToByte(Rot.Yaw);
    uint8 BytePitch = FRotator::CompressAxisToByte(Rot.Pitch);
    uint8 ByteRoll = FRotator::CompressAxisToByte(Rot.Roll);
    
    Ar << ByteYaw << BytePitch << ByteRoll;
}

### 12.2 Lag Compensation（延迟补偿）

```cpp
// 射击游戏中的延迟补偿
// Server 需要 "回到过去" 验证客户端的命中

class ULagCompensationComponent : public UActorComponent
{
    // 保存历史位置快照
    struct FPositionSnapshot
    {
        float Timestamp;
        FVector Location;
        FRotator Rotation;
        // 碰撞体信息...
    };
    
    TArray<FPositionSnapshot> PositionHistory;
    static constexpr int32 MAX_HISTORY = 64;  // 约1秒 @60fps
    
    void SaveSnapshot()
    {
        FPositionSnapshot Snap;
        Snap.Timestamp = GetWorld()->GetTimeSeconds();
        Snap.Location = GetOwner()->GetActorLocation();
        Snap.Rotation = GetOwner()->GetActorRotation();
        
        PositionHistory.Add(Snap);
        if (PositionHistory.Num() > MAX_HISTORY)
            PositionHistory.RemoveAt(0);
    }
    
    // Server 端：回滚到客户端开火时的状态
    FPositionSnapshot GetSnapshotAtTime(float Time) const
    {
        // 二分查找最近的快照
        // 在两个快照之间插值
        for (int32 i = PositionHistory.Num() - 1; i > 0; --i)
        {
            if (PositionHistory[i].Timestamp <= Time)
            {
                float Alpha = (Time - PositionHistory[i].Timestamp) /
                    (PositionHistory[i+1].Timestamp - 
                     PositionHistory[i].Timestamp);
                
                FPositionSnapshot Result;
                Result.Location = FMath::Lerp(
                    PositionHistory[i].Location,
                    PositionHistory[i+1].Location, Alpha);
                return Result;
            }
        }
        return PositionHistory.Last();
    }
};

// Server 验证命中
void AMyCharacter::ServerVerifyHit_Implementation(
    float ClientTimestamp, FVector TraceStart, FVector TraceEnd)
{
    // 计算客户端开火时的 Server 时间
    float Ping = GetNetConnection()->AvgLag;
    float ServerTimeAtShot = ClientTimestamp; // 已经是 Server 时间
    
    // 回滚所有目标到那个时间点
    for (AMyCharacter* Target : AllTargets)
    {
        FPositionSnapshot Snap = 
            Target->LagComp->GetSnapshotAtTime(ServerTimeAtShot);
        Target->SetActorLocation(Snap.Location);  // 临时回滚
    }
    
    // 在回滚状态下做射线检测
    FHitResult Hit;
    GetWorld()->LineTraceSingleByChannel(
        Hit, TraceStart, TraceEnd, ECC_Pawn);
    
    // 恢复所有目标到当前位置
    for (AMyCharacter* Target : AllTargets)
    {
        Target->SetActorLocation(Target->GetReplicatedLocation());
    }
    
    // 处理命中结果
    if (Hit.bBlockingHit)
    {
        ApplyDamage(Hit.GetActor(), Damage);
    }
}

## 13. 面试高频问题

### 13.1 概念题

**Q1: Unreal 的网络模型是什么？为什么选择 Server 权威？**

> Server Authoritative（服务器权威）模型。所有游戏逻辑的最终决定权在 Server。
> 原因：
> 1. 防作弊 - Client 不能自行决定游戏状态
> 2. 一致性 - 所有 Client 看到的是 Server 的同一份真相
> 3. 简化冲突解决 - Server 说了算，无需复杂的冲突合并

**Q2: ROLE_SimulatedProxy 和 ROLE_AutonomousProxy 的区别？**

> - AutonomousProxy：本地控制的角色（自己的 Pawn），有客户端预测和输入
> - SimulatedProxy：远程角色的本地副本，只做插值平滑，不做预测
> - 关键区别：AutonomousProxy 可以发送 Server RPC，SimulatedProxy 不能

**Q3: 属性同步和 RPC 的区别？什么时候用哪个？**

> - 属性同步：持续状态（血量、位置、状态），引擎自动检测变化并同步
> - RPC：瞬时事件（开火、技能释放、聊天消息），手动调用
> - 原则：状态用属性同步，事件用 RPC
> - 新加入的玩家能收到属性同步的当前值，但收不到之前的 RPC

**Q4: 为什么不能高频发送 Reliable RPC？**

> Reliable RPC 使用有限大小的可靠传输缓冲区。高频发送会导致：
> 1. 缓冲区溢出 → 连接断开
> 2. 每个 Reliable 包需要 ACK 确认，增加延迟
> 3. 丢包时需要重传，阻塞后续数据
> 解决：高频数据用属性同步或 Unreliable RPC

**Q5: 解释 Unreal 的角色移动预测和纠正机制。**

> 1. Client 本地预测执行移动（即时响应）
> 2. 同时发送输入到 Server
> 3. Server 用相同输入重新执行移动
> 4. 比较 Server 结果和 Client 报告的位置
> 5. 误差超过阈值 → 发送纠正
> 6. Client 收到纠正 → 回滚到 Server 位置 → 重放之后的所有输入
> 7. 重放过程对玩家几乎不可见（如果预测准确）

### 13.2 实践题

**Q6: 如何优化 100 人大逃杀的网络性能？**

> 1. **Replication Graph** - 空间网格剔除远处玩家
> 2. **动态同步频率** - 远处玩家降低到 2-5Hz，近处 30-60Hz
> 3. **Dormancy** - 不活跃的 Actor 休眠
> 4. **属性条件** - COND_OwnerOnly 减少不必要同步
> 5. **FFastArraySerializer** - 背包等大数组增量同步
> 6. **网络量化** - 压缩位置/旋转数据
> 7. **Interest Management** - 只同步玩家视野内的信息
> 8. **带宽预算** - 每连接限制 30-50 KB/s

**Q7: 如何实现一个网络同步的技能系统？**

> ```
> 1. Client 按下技能键
> 2. Client 本地预测播放动画和特效（即时反馈）
> 3. Client 发送 Server RPC: ServerActivateAbility(AbilityID)
> 4. Server 验证:
>    - CD 是否就绪？
>    - 资源（蓝量）是否足够？
>    - 状态是否允许？
> 5a. 验证通过:
>    - Server 执行技能逻辑
>    - 属性同步状态变化（CD、蓝量）
>    - Multicast RPC 播放特效（给其他玩家）
> 5b. 验证失败:
>    - Client RPC 通知取消
>    - Client 回滚预测效果
> ```

**Q8: Push Model 相比传统属性同步有什么优势？**

> 传统模型：每帧对每个 Actor 的每个同步属性做 memcmp
> - 1000 Actor × 20 属性 = 20000 次比较/帧
> 
> Push Model：只在属性修改时标记 dirty bit
> - 仅序列化标脏的属性
> - 未修改的属性零开销
> - 实测：大量 Actor 场景 Server CPU 降低 30-50%
> - 代价：需要手动调用 MARK_PROPERTY_DIRTY
