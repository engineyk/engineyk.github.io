## 一、什么是循环引用

循环引用（Circular Dependency）是指两个或多个资源包（Package）之间形成了相互依赖的闭环：

```
简单循环:
PackageA → PackageB → PackageA

复杂循环:
PackageA → PackageB → PackageC → PackageA

跨Chunk循环:
Chunk1/MapA ←→ Chunk2/MapB
```

### 危害

| 危害类型      | 说明                             |
| ------------- | -------------------------------- |
| **内存暴涨**  | 加载一个包时被迫加载整个依赖链   |
| **分包失败**  | Chunking 失败或产生重复资源      |
| **热更膨胀**  | 热更新时 Patch 包体积膨胀        |
| **Cook 缓慢** | Cook 时间显著增加                |
| **崩溃风险**  | 极端情况下导致死锁或无限递归崩溃 |

## 二、诊断工具

### 1. Reference Viewer（引用查看器）

```
操作路径：
Content Browser → 右键资源 → Reference Viewer

关键设置：
├── Search Depth Limit: 设为 3-5 层即可发现大部分循环
├── Show Referencers: ✓ (谁引用了我)
├── Show Dependencies: ✓ (我引用了谁)
├── Show Soft References: ✓ (软引用也要检查)
└── Compact Mode: 便于查看大量节点

循环标识：
- 图中出现双向箭头或闭环路径即为循环引用
- 红色/橙色连线通常表示硬引用
- 蓝色连线表示软引用（不会造成加载依赖循环）
```

### 2. Size Map（大小地图）

```
操作路径：
Content Browser → 右键资源 → Size Map

用途：
- 可视化一个资源"拖带"了多少依赖
- 如果一个小蓝图的 Size Map 异常庞大，很可能存在循环引用
```

### 3. Asset Audit（资源审计）

```
操作路径：
Window → Developer Tools → Asset Audit

关键列：
├── Disk Size vs Memory Size
├── Total Size with Dependencies
├── Chunk ID assignment
└── Primary Asset info

异常信号：
- Total Size with Dependencies 远大于 Disk Size
- 同一资源出现在多个 Chunk 中
```

### 5. C++ 检测脚本

```cpp
// Editor Utility Widget 或 Commandlet 中使用
#include "AssetRegistry/AssetRegistryModule.h"

void DetectCircularReferences()
{
    FAssetRegistryModule& AssetRegistryModule = 
        FModuleManager::LoadModuleChecked<FAssetRegistryModule>("AssetRegistry");
    IAssetRegistry& AssetRegistry = AssetRegistryModule.Get();

    TArray<FAssetIdentifier> Dependencies;
    FAssetIdentifier AssetId(FName("/Game/Blueprints/BP_Player"));
    
    // Get all dependencies recursively
    AssetRegistry.GetDependencies(
        AssetId, 
        Dependencies, 
        UE::AssetRegistry::EDependencyCategory::Package
    );

    // Check if any dependency references back to the original
    for (const FAssetIdentifier& Dep : Dependencies)
    {
        TArray<FAssetIdentifier> SubDeps;
        AssetRegistry.GetDependencies(Dep, SubDeps, 
            UE::AssetRegistry::EDependencyCategory::Package);
        
        for (const FAssetIdentifier& SubDep : SubDeps)
        {
            if (SubDep.PackageName == AssetId.PackageName)
            {
                UE_LOG(LogTemp, Warning, 
                    TEXT("CIRCULAR: %s <-> %s"), 
                    *AssetId.PackageName.ToString(),
                    *Dep.PackageName.ToString());
            }
        }
    }
}
```

## 三、循环引用的常见成因与解决方案

### 场景 1：蓝图之间互相引用

```
问题：
BP_Player 引用 BP_Enemy（需要获取敌人信息）
BP_Enemy 引用 BP_Player（需要获取玩家位置）

BP_Player ←→ BP_Enemy  ✗ 循环引用
```

#### 解决方案 A：引入公共接口（Interface）

```
重构后：
BPI_Damageable (Blueprint Interface)
├── GetHealth()
├── GetLocation()
└── TakeDamage()

BP_Player implements BPI_Damageable
BP_Enemy implements BPI_Damageable

BP_Player → BPI_Damageable (不直接引用 BP_Enemy)
BP_Enemy → BPI_Damageable (不直接引用 BP_Player)

依赖方向：
BP_Player → BPI_Damageable ← BP_Enemy  ✓ 无循环
```

#### 解决方案 B：使用软引用（Soft Reference）

```
硬引用 vs 软引用：

硬引用 (Hard Reference):
  UPROPERTY()
  TSubclassOf<ABP_Enemy> EnemyClass;
  → 加载 BP_Player 时立即加载 BP_Enemy
  → 形成加载依赖，导致循环

软引用 (Soft Reference):
  UPROPERTY()
  TSoftClassPtr<AActor> EnemyClass;
  → 加载 BP_Player 时不加载 BP_Enemy
  → 需要时异步加载，打破循环
```

```cpp
// C++ 软引用使用
TSoftClassPtr<AActor> EnemyClassSoft;

void SpawnEnemy()
{
    // Async load when needed
    UClass* EnemyClass = EnemyClassSoft.LoadSynchronous();
    if (EnemyClass)
    {
        GetWorld()->SpawnActor<AActor>(EnemyClass, SpawnTransform);
    }
}

// 蓝图中：
// 将变量类型从 "BP_Enemy (Class Reference)" 
// 改为 "BP_Enemy (Soft Class Reference)"
// 使用 "Async Load Class Asset" 节点加载
```

#### 解决方案 C：事件驱动解耦（Event Dispatcher / Delegate）

```c++
重构前（直接调用）：
BP_Player::OnHit() { BP_Enemy->TakeDamage(); }  // Player → Enemy
BP_Enemy::OnAttack() { BP_Player->ReceiveHit(); } // Enemy → Player

重构后（事件驱动）：
// GameMode 或 EventManager 作为中介
BP_GameMode:
  Event Dispatcher: OnPlayerHitEnemy(AActor* Target, float Damage)
  Event Dispatcher: OnEnemyAttackPlayer(AActor* Source, float Damage)

BP_Player:
  OnHit() → GameMode->OnPlayerHitEnemy.Broadcast(Enemy, Damage)
  Bind to: GameMode->OnEnemyAttackPlayer → ReceiveHit()

BP_Enemy:
  OnAttack() → GameMode->OnEnemyAttackPlayer.Broadcast(Self, Damage)
  Bind to: GameMode->OnPlayerHitEnemy → TakeDamage()

依赖方向：
BP_Player → BP_GameMode ← BP_Enemy  ✓ 无循环
```

### 场景 2：Widget 与 Actor 互相引用

```
问题：
WBP_HUD 引用 BP_Character（显示血量）
BP_Character 引用 WBP_HUD（创建/更新 HUD）

WBP_HUD ←→ BP_Character  ✗ 循环引用
```

#### 解决方案：MVC / 中介者模式

```
重构后架构：

BP_PlayerController (中介者/Controller)
├── 持有 BP_Character 引用 (硬引用，同 Chunk)
├── 持有 WBP_HUD 引用 (硬引用，同 Chunk)
├── 监听 Character 事件 → 更新 HUD
└── 监听 HUD 输入事件 → 控制 Character

BP_Character:
├── Event Dispatcher: OnHealthChanged(float NewHealth, float MaxHealth)
├── Event Dispatcher: OnAmmoChanged(int32 Current, int32 Max)
└── 不引用任何 UI 类  ✓

WBP_HUD:
├── Function: UpdateHealth(float Percent)
├── Function: UpdateAmmo(int32 Current, int32 Max)
└── 不引用任何 Gameplay 类  ✓

BP_PlayerController:
├── BeginPlay():
│   Character->OnHealthChanged.AddDynamic(this, &UpdateHUD_Health)
│   HUD->OnButtonClicked.AddDynamic(this, &HandleInput)
└── 单向引用 Character 和 HUD

依赖方向：
WBP_HUD ← BP_PlayerController → BP_Character  ✓ 无循环
```

### 场景 3：跨关卡/跨 Chunk 循环

```
问题：
Chunk1 (Level_Forest):
  BP_ForestPortal → references Level_Castle (in Chunk2)

Chunk2 (Level_Castle):
  BP_CastlePortal → references Level_Forest (in Chunk1)

Chunk1 ←→ Chunk2  ✗ 跨 Chunk 循环
```

#### 解决方案：软引用 + DataTable/DataAsset

```cpp
// 创建一个轻量级的关卡数据资产（放在共享 Chunk0 中）
USTRUCT(BlueprintType)
struct FLevelPortalData : public FTableRowBase
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere)
    TSoftObjectPtr<UWorld> TargetLevel;  // Soft reference!

    UPROPERTY(EditAnywhere)
    FName SpawnPointTag;

    UPROPERTY(EditAnywhere)
    FText DisplayName;
};

// DataTable: DT_LevelPortals (in Chunk0 - shared)
// Row: Forest_To_Castle | /Game/Maps/Level_Castle | SpawnPoint_FromForest
// Row: Castle_To_Forest | /Game/Maps/Level_Forest | SpawnPoint_FromCastle

// BP_GenericPortal (in Chunk0 - shared)
// - References DT_LevelPortals (same chunk, no cross-chunk dependency)
// - Uses row name to look up target level
// - Async loads target level via soft reference
```

```
依赖方向：
Chunk1 → Chunk0 (DT_LevelPortals) ← Chunk2  ✓ 无循环
```

### 场景 4：材质与蓝图循环

```
问题：
M_CharacterSkin 引用 BP_Character（通过 Material Parameter Collection 或直接引用）
BP_Character 引用 M_CharacterSkin（设置材质）

常见于动态材质切换场景
```

#### 解决方案：Material Parameter Collection + 间接引用

```
重构后：

MPC_CharacterParams (Material Parameter Collection)
├── Scalar: Health_Percent
├── Scalar: Shield_Active
└── Vector: Team_Color

M_CharacterSkin:
├── 引用 MPC_CharacterParams (轻量级，无蓝图依赖)
└── 使用 MPC 参数驱动视觉效果

BP_Character:
├── 引用 MPC_CharacterParams (设置参数值)
├── 材质通过 Soft Reference 或 DataAsset 指定
└── 不直接引用具体材质资产
```

```cpp
// C++ 设置 MPC 参数
UMaterialParameterCollectionInstance* MPCInstance = 
    GetWorld()->GetParameterCollectionInstance(MPC_CharacterParams);
MPCInstance->SetScalarParameterValue(FName("Health_Percent"), 0.75f);
```

```
依赖方向：
M_CharacterSkin → MPC ← BP_Character  ✓ 无循环
```

## 四、架构层面的预防策略

### 1. 分层依赖原则

```
Layer Architecture (依赖只能向下，不能向上):

┌─────────────────────────────────────────┐
│  Layer 4: Game-Specific Content         │  Maps, Levels, Game Modes
│  (Can reference Layer 0-3)              │
├─────────────────────────────────────────┤
│  Layer 3: Feature Modules               │  Combat, Inventory, Dialog
│  (Can reference Layer 0-2)              │
├─────────────────────────────────────────┤
│  Layer 2: Gameplay Framework            │  Characters, Weapons, Items
│  (Can reference Layer 0-1)              │
├─────────────────────────────────────────┤
│  Layer 1: Core Systems                  │  Interfaces, DataAssets,
│  (Can reference Layer 0 only)           │  Enums, Structs, MPC
├─────────────────────────────────────────┤
│  Layer 0: Engine & Third-Party          │  UE Modules, Plugins
│  (No project references)               │
└─────────────────────────────────────────┘
```

### 推荐目录结构

```
Content/
├── _Core/              ← Layer 1: Interfaces, DataAssets, Enums
│   ├── Interfaces/
│   ├── DataAssets/
│   ├── DataTables/
│   └── Enums/
├── Gameplay/           ← Layer 2: Base gameplay classes
│   ├── Characters/
│   ├── Weapons/
│   └── Items/
├── Features/           ← Layer 3: Feature modules
│   ├── Combat/
│   ├── Inventory/
│   └── Dialog/
└── Maps/               ← Layer 4: Game content
    ├── Forest/
    └── Castle/

Rule: Arrows only point DOWN. Never UP or SIDEWAYS between siblings.
```

### 2. 模块化插件架构

```
Plugin-based Modular Architecture:

MyGame.uproject
├── Plugins/
│   ├── CoreTypes/           ← Interfaces, Structs, Enums (no dependencies)
│   │   └── CoreTypes.uplugin
│   │       Dependencies: []
│   │
│   ├── CombatSystem/        ← Combat logic
│   │   └── CombatSystem.uplugin
│   │       Dependencies: [CoreTypes]
│   │
│   ├── InventorySystem/     ← Inventory logic
│   │   └── InventorySystem.uplugin
│   │       Dependencies: [CoreTypes]
│   │
│   └── UIFramework/         ← UI widgets
│       └── UIFramework.uplugin
│           Dependencies: [CoreTypes]
│
└── Source/MyGame/           ← Game module (glues everything together)
    Dependencies: [CoreTypes, CombatSystem, InventorySystem, UIFramework]

Key benefit: .uplugin enforces dependency direction at compile time.
CombatSystem CANNOT #include InventorySystem headers (build error).
```

### 3. 资源命名与目录规范

```
Naming Convention for Dependency Tracking:

Prefix system:
├── BP_     Blueprint
├── WBP_    Widget Blueprint
├── BPI_    Blueprint Interface        ← Shared, no dependencies
├── DA_     Data Asset                 ← Shared, lightweight
├── DT_     Data Table                 ← Shared, lightweight
├── E_      Enum                       ← Shared, no dependencies
├── S_      Struct                     ← Shared, no dependencies
├── MPC_    Material Parameter Collection ← Shared bridge
├── M_      Material
├── MI_     Material Instance
├── T_      Texture
└── SM_     Static Mesh

Rule: BPI_, DA_, DT_, E_, S_, MPC_ prefixed assets go in _Core/ folder
      and must NEVER reference BP_, WBP_, or map-specific assets.
```

## 五、自动化检测流水线

### CI/CD 集成检测脚本（Python）

```python
# check_circular_deps.py
# Run as part of CI/CD pipeline after cook

import json
import sys
from collections import defaultdict

def load_dependency_graph(dep_file):
    """Load dependency graph from UE asset registry dump"""
    graph = defaultdict(set)
    with open(dep_file, 'r') as f:
        for line in f:
            data = json.loads(line)
            pkg = data['package']
            for dep in data['dependencies']:
                graph[pkg].add(dep)
    return graph

def find_cycles(graph):
    """DFS-based cycle detection"""
    WHITE, GRAY, BLACK = 0, 1, 2
    color = defaultdict(int)
    cycles = []
    
    def dfs(node, path):
        color[node] = GRAY
        path.append(node)
        for neighbor in graph.get(node, []):
            if color[neighbor] == GRAY:
                # Found cycle
                cycle_start = path.index(neighbor)
                cycle = path[cycle_start:] + [neighbor]
                cycles.append(cycle)
            elif color[neighbor] == WHITE:
                dfs(neighbor, path)
        path.pop()
        color[node] = BLACK
    
    for node in graph:
        if color[node] == WHITE:
            dfs(node, [])
    
    return cycles

if __name__ == '__main__':
    graph = load_dependency_graph(sys.argv[1])
    cycles = find_cycles(graph)
    
    if cycles:
        print(f"ERROR: Found {len(cycles)} circular dependencies!")
        for i, cycle in enumerate(cycles):
            print(f"\nCycle {i+1}:")
            print(" → ".join(cycle))
        sys.exit(1)
    else:
        print("OK: No circular dependencies found.")
        sys.exit(0)
```

### UE Editor Commandlet（C++）

```cpp
// CycleDetectionCommandlet.h
UCLASS()
class UCycleDetectionCommandlet : public UCommandlet
{
    GENERATED_BODY()
public:
    virtual int32 Main(const FString& Params) override;
    
private:
    enum class EVisitState : uint8 { White, Gray, Black };
    
    bool DetectCycles(
        const TMap<FName, TArray<FName>>& Graph,
        TArray<TArray<FName>>& OutCycles);
    
    void DFS(
        FName Node,
        const TMap<FName, TArray<FName>>& Graph,
        TMap<FName, EVisitState>& State,
        TArray<FName>& Path,
        TArray<TArray<FName>>& OutCycles);
};
```

```cpp
// CycleDetectionCommandlet.cpp
int32 UCycleDetectionCommandlet::Main(const FString& Params)
{
    FAssetRegistryModule& ARM = 
        FModuleManager::LoadModuleChecked<FAssetRegistryModule>("AssetRegistry");
    IAssetRegistry& AR = ARM.Get();
    AR.SearchAllAssets(true);

    // Build dependency graph
    TMap<FName, TArray<FName>> Graph;
    TArray<FAssetData> AllAssets;
    AR.GetAllAssets(AllAssets, true);

    for (const FAssetData& Asset : AllAssets)
    {
        TArray<FAssetIdentifier> Deps;
        AR.GetDependencies(
            FAssetIdentifier(Asset.PackageName), Deps,
            UE::AssetRegistry::EDependencyCategory::Package,
            UE::AssetRegistry::EDependencyQuery::Hard); // Hard refs only
        
        TArray<FName>& DepNames = Graph.FindOrAdd(Asset.PackageName);
        for (const FAssetIdentifier& Dep : Deps)
        {
            DepNames.Add(Dep.PackageName);
        }
    }

    // Detect cycles
    TArray<TArray<FName>> Cycles;
    DetectCycles(Graph, Cycles);

    if (Cycles.Num() > 0)
    {
        UE_LOG(LogTemp, Error, TEXT("Found %d circular dependencies!"), Cycles.Num());
        for (int32 i = 0; i < Cycles.Num(); ++i)
        {
            FString CycleStr;
            for (const FName& Node : Cycles[i])
            {
                CycleStr += Node.ToString() + TEXT(" → ");
            }
            UE_LOG(LogTemp, Error, TEXT("Cycle %d: %s"), i + 1, *CycleStr);
        }
        return 1;
    }

    UE_LOG(LogTemp, Display, TEXT("No circular dependencies found."));
    return 0;
}
```

## 六、解决循环引用的决策流程图

```
发现循环引用后的决策流程：

                    ┌─────────────────────┐
                    │ 检测到循环引用       │
                    │ A ←→ B              │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │ A 和 B 是否属于      │
                    │ 同一功能模块？        │
                    └──┬──────────────┬───┘
                       │ Yes          │ No
                       ▼              ▼
              ┌────────────┐  ┌──────────────────┐
              │ 合并到同一  │  │ 是否可以用接口    │
              │ Package    │  │ 替代直接引用？    │
              └────────────┘  └──┬───────────┬───┘
                                 │ Yes       │ No
                                 ▼           ▼
                        ┌────────────┐ ┌──────────────────┐
                        │ 提取公共    │ │ 是否只在运行时    │
                        │ Interface  │ │ 需要引用？        │
                        │ 到 _Core/  │ └──┬───────────┬───┘
                        └────────────┘    │ Yes       │ No
                                          ▼           ▼
                                 ┌────────────┐ ┌──────────────┐
                                 │ 改为软引用  │ │ 引入中介者    │
                                 │ TSoftPtr   │ │ (GameMode/   │
                                 │ + 异步加载  │ │  Subsystem/  │
                                 └────────────┘ │  EventBus)   │
                                                └──────────────┘
```

## 七、核心原则总结

| 原则             | 说明                                                        |
| ---------------- | ----------------------------------------------------------- |
| **依赖单向流动** | 上层可以引用下层，下层绝不引用上层                          |
| **接口隔离**     | 用 Blueprint Interface / C++ Interface 替代具体类引用       |
| **软引用优先**   | 跨模块引用一律使用 `TSoftObjectPtr` / `TSoftClassPtr`       |
| **中介者解耦**   | 需要双向通信时，引入第三方（GameMode、Subsystem、EventBus） |
| **共享资源下沉** | 被多方引用的资源放入 `_Core/` 或 `Chunk0`                   |
| **定期审计**     | CI/CD 中集成循环依赖检测，防止回归                          |
| **编译期约束**   | 使用 Plugin 模块化，让编译器强制执行依赖方向                |

- [ ] 插件模块的 `.uplugin` 依赖声明是否正确且单向？
