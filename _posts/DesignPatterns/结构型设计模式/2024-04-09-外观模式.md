# 外观模式（Facade Pattern）—— C# 实现

外观模式是一种结构型设计模式，提供一个统一的接口，用来访问子系统中的一群接口。外观模式定义了一个高层接口，让子系统更易使用。

## 结构

- **Facade（外观类）**：为子系统中的一组接口提供统一的高层接口。
- **Subsystem Classes（子系统类）**：实现子系统的功能，外观对象可调用它们的方法。

## UML 类图

```
[Client] ——> [Facade] ——> [SubsystemA]
                         └──> [SubsystemB]
                         └──> [SubsystemC]
```

## C# 示例代码

```csharp
using System;

// 子系统A
class SubsystemA
{
    public void OperationA() => Console.WriteLine("SubsystemA: OperationA");
}

// 子系统B
class SubsystemB
{
    public void OperationB() => Console.WriteLine("SubsystemB: OperationB");
}

// 子系统C
class SubsystemC
{
    public void OperationC() => Console.WriteLine("SubsystemC: OperationC");
}

// 外观类
class Facade
{
    private SubsystemA a = new SubsystemA();
    private SubsystemB b = new SubsystemB();
    private SubsystemC c = new SubsystemC();

    public void DoWork()
    {
        Console.WriteLine("Facade: DoWork start");
        a.OperationA();
        b.OperationB();
        c.OperationC();
        Console.WriteLine("Facade: DoWork end");
    }
}

// 使用示例
class Program
{
    static void Main()
    {
        Facade facade = new Facade();
        facade.DoWork();
    }
}
```

## 输出结果

```
Facade: DoWork start
SubsystemA: OperationA
SubsystemB: OperationB
SubsystemC: OperationC
Facade: DoWork end
```

## 适用场景

- 为复杂子系统提供简单接口。
- 客户端与子系统解耦。
- 需要统一和简化对一组类或系统的调用。

## 优缺点

**优点：**
- 简化复杂系统的使用。
- 降低系统间的耦合度。
- 更好的划分访问层次。

**缺点：**
- 不符合开闭原则，新增子系统时需要修改外观类。
- 过多地使用外观类可能导致系统结构不清晰。

## 总结

外观模式为子系统提供统一入口，简化了客户端与子系统的交互，使系统更易于使用和维护。
