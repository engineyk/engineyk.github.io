# C++ 编译原理

## 一、编译的四个阶段

```
源代码 (.cpp/.h)
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 1. 预处理 (Preprocessing)                       │
│    - 展开 #include（递归插入头文件内容）          │
│    - 展开宏 #define（文本替换）                  │
│    - 处理条件编译 #if / #ifdef / #ifndef         │
│    - 删除注释                                    │
│    - 生成 .i 文件（翻译单元 Translation Unit）    │
│    - 命令: g++ -E main.cpp -o main.i            │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 2. 编译 (Compilation)                            │
│    - 词法分析 → Token 流                          │
│    - 语法分析 → AST（抽象语法树）                  │
│    - 语义分析 → 类型检查、重载决议、模板实例化      │
│    - 中间代码生成 → IR（如 LLVM IR）              │
│    - 优化 → 常量折叠、死代码消除、内联展开等        │
│    - 目标代码生成 → 汇编代码                      │
│    - 生成 .s 文件                                │
│    - 命令: g++ -S main.i -o main.s              │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 3. 汇编 (Assembly)                              │
│    - 将汇编代码翻译为机器码                       │
│    - 生成目标文件 .o / .obj                       │
│    - 包含：机器码 + 符号表 + 重定位信息            │
│    - 命令: g++ -c main.s -o main.o               │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 4. 链接 (Linking)                                │
│    - 符号解析：将未定义符号与其他 .o 中的定义匹配   │
│    - 重定位：修正代码中的地址引用                   │
│    - 合并段：.text(代码) .data(数据) .bss(未初始化) │
│    - 生成可执行文件 .exe / .out                    │
│    - 命令: g++ main.o utils.o -o main.exe         │
└─────────────────────────────────────────────────┘
```

## 二、编译阶段详解

## 2.1 词法分析 (Lexical Analysis)

```
源代码: int x = 42 + y;

Token 流:
  [keyword: int] [identifier: x] [operator: =] 
  [literal: 42] [operator: +] [identifier: y] [punctuation: ;]

工具: Lexer / Scanner（如 flex）
作用: 将字符流分割为有意义的 Token
```

## 2.2 语法分析 (Syntax Analysis / Parsing)

```
Token 流 → AST（抽象语法树）

        Declaration
        /    |    \
     Type  Name  InitExpr
      |     |       |
     int    x    BinaryOp(+)
                  /       \
              Literal(42)  Identifier(y)

工具: Parser（如 bison / ANTLR）
作用: 检查语法正确性，构建树形结构
```

## 2.3 语义分析 (Semantic Analysis)

```
检查内容：
  - 类型检查：int + float → 隐式转换
  - 作用域解析：变量 y 是否已声明？
  - 重载决议：foo(int) vs foo(float) 选哪个？
  - 模板实例化：vector<int> → 生成具体类型代码
  - 访问控制：private 成员是否可访问？
  - const 正确性检查
```

## 2.4 中间代码与优化

```
LLVM IR 示例：
  define i32 @add(i32 %a, i32 %b) {
    %result = add i32 %a, %b
    ret i32 %result
  }

常见优化（按级别）：
  -O0: 无优化（调试用）
  -O1: 基本优化（常量折叠、死代码消除）
  -O2: 标准优化（循环优化、内联展开、指令调度）
  -O3: 激进优化（自动向量化、循环展开）
  -Os: 优化代码体积

关键优化技术：
  - 常量折叠: 3 + 5 → 8（编译期计算）
  - 死代码消除: 删除永远不会执行的代码
  - 内联展开: 将小函数体直接插入调用处
  - 循环展开: 减少循环控制开销
  - 寄存器分配: 将变量尽量放在寄存器中
  - 尾调用优化: 递归 → 循环
  - RVO/NRVO: 返回值优化，避免拷贝
```

## 三、链接详解

## 3.1 符号与符号表

```
每个 .o 文件包含：
  - 已定义符号（Defined）：本文件实现的函数/全局变量
  - 未定义符号（Undefined）：本文件引用但未实现的
  - 局部符号（Local）：static 函数/变量，仅本文件可见

示例：
  // a.cpp
  int globalVar = 10;        // 已定义符号
  extern int otherVar;       // 未定义符号
  static int localVar = 5;   // 局部符号
  void foo() { ... }         // 已定义符号
```

## 3.2 静态链接 vs 动态链接

```
静态链接 (.lib / .a)：
  - 编译时将库代码复制到可执行文件中
  - 优点：独立运行，无依赖
  - 缺点：体积大，更新需重新编译

动态链接 (.dll / .so)：
  - 运行时加载共享库
  - 优点：体积小，多进程共享，热更新
  - 缺点：运行时依赖，加载开销

引擎中的应用：
  - UE 模块系统：每个模块编译为 .dll（Editor）或静态链接（Shipping）
  - 插件热重载：基于动态链接实现
```

## 3.3 常见链接错误

```
LNK2001 / LNK2019: 未解析的外部符号
  原因：声明了但没实现，或 .lib 未链接

LNK2005: 符号已定义
  原因：多个 .cpp 中定义了同名全局符号
  解决：使用 inline / static / 匿名命名空间

LNK1169: 多个符号定义
  原因：头文件中定义了非 inline 函数且被多个 .cpp 包含
  解决：声明放 .h，定义放 .cpp，或使用 inline
```

## 四、翻译单元 (Translation Unit)

```
一个 .cpp 文件经过预处理后 = 一个翻译单元

关键概念：
  - 每个翻译单元独立编译
  - #include 只是文本插入，不是模块导入
  - 头文件被多个 .cpp 包含 → 每个翻译单元各有一份副本
  - ODR（One Definition Rule）：同一实体在所有翻译单元中定义必须一致

头文件保护：
  #pragma once          // 编译器扩展，推荐
  #ifndef MY_HEADER_H   // 传统方式
  #define MY_HEADER_H
  ...
  #endif
```

## 五、C++ 特有的编译复杂性

## 5.1 模板编译模型

```
模板的特殊性：
  - 模板不是代码，是代码的"蓝图"
  - 实例化时才生成真正的代码
  - 模板定义必须对使用它的翻译单元可见 → 通常放在头文件中

编译影响：
  - 头文件膨胀：模板代码全在头文件 → 预处理后翻译单元巨大
  - 重复实例化：vector<int> 在 100 个 .cpp 中使用 → 编译 100 次
  - 链接器去重：最终链接时去除重复的模板实例（COMDAT folding）

优化手段：
  - extern template：显式声明不在本翻译单元实例化
    extern template class vector<int>;  // 声明
    template class vector<int>;         // 在某个 .cpp 中显式实例化
  - PCH（预编译头）：将常用头文件预编译为二进制
  - C++20 Modules：从根本上解决头文件模型的问题
```

## 5.2 Name Mangling（名称修饰）

```
C++ 支持重载 → 编译器需要将函数签名编码到符号名中

示例：
  void foo(int)    → _Z3fooi        (GCC/Clang)
  void foo(float)  → _Z3foof
  void foo(int, int) → _Z3fooii

  MSVC 使用不同的修饰方案：
  void foo(int)    → ?foo@@YAXH@Z

extern "C" 的作用：
  - 禁用 Name Mangling
  - 使 C++ 函数可被 C 代码调用
  - DLL 导出接口常用 extern "C"
```

## 5.3 虚函数表 (vtable) 的编译期生成

```
编译器为每个含虚函数的类生成 vtable：

class Base {
    virtual void foo();  // vtable[0] = &Base::foo
    virtual void bar();  // vtable[1] = &Base::bar
};

class Derived : public Base {
    void foo() override; // vtable[0] = &Derived::foo（覆盖）
                         // vtable[1] = &Base::bar（继承）
};

编译期：
  - 编译器为每个类生成 vtable（函数指针数组）
  - 每个对象头部插入 vptr（指向 vtable 的指针）
  - vtable 存放在 .rodata 段（只读数据）

链接期：
  - vtable 是全局符号，需要链接器解析
  - 如果虚函数只声明未定义 → 链接错误
```

## 六、引擎中的编译优化实践

## 6.1 UE 的编译体系

```
UBT (Unreal Build Tool)：
  - 读取 .uproject / .uplugin / Target.cs / Build.cs
  - 自动调用编译器（MSVC / Clang / GCC）
  - 管理模块依赖、增量编译、并行编译

UHT (Unreal Header Tool)：
  - 编译前扫描 UCLASS / UPROPERTY / UFUNCTION 等反射宏
  - 生成 .generated.h 和 .gen.cpp
  - 本质是一个 C++ 预处理器 + 代码生成器

编译流程：
  UBT 分析依赖 → UHT 生成反射代码 → 编译器编译 → 链接器链接
```

## 6.2 编译加速手段

```
1. PCH（预编译头）
   - 将 Engine.h 等大头文件预编译为二进制
   - 每个模块的 PCH 只编译一次
   - UE 中：SharedPCH / PrivatePCH

2. Unity Build（合并编译）
   - 将多个 .cpp 合并为一个翻译单元编译
   - 减少头文件重复解析和模板重复实例化
   - UE 中：bUseUnityBuild = true（默认开启）
   - 注意：static 变量/函数可能冲突

3. 增量编译
   - 只重新编译修改过的文件
   - 依赖分析：头文件修改 → 所有包含它的 .cpp 重编

4. 分布式编译
   - IncrediBuild / FASTBuild / Distcc
   - 将编译任务分发到多台机器

5. C++20 Modules（未来方向）
   - 替代 #include 的头文件模型
   - 模块只编译一次，导入时使用二进制接口
   - 从根本上解决编译速度问题
```

## 七、面试高频问题

```
Q: C++ 从源代码到可执行文件经历了哪些步骤？
A: 预处理 → 编译（词法/语法/语义分析 + 优化 + 代码生成）→ 汇编 → 链接

Q: 头文件为什么需要 #pragma once？
A: 防止同一头文件在一个翻译单元中被多次包含，导致重复定义错误。

Q: 为什么模板定义要放在头文件中？
A: 模板在使用时才实例化，编译器需要看到完整定义才能生成代码。
   每个翻译单元独立编译，如果定义在 .cpp 中，其他翻译单元看不到。

Q: extern "C" 的作用是什么？
A: 禁用 C++ 的 Name Mangling，使函数符号与 C 兼容，
   常用于 DLL 导出接口和 C/C++ 混合编程。

Q: 静态链接和动态链接的区别？
A: 静态链接在编译时将库代码复制到可执行文件，独立但体积大；
   动态链接在运行时加载共享库，体积小可共享但有运行时依赖。

Q: 什么是 ODR（One Definition Rule）？
A: 每个实体（函数/变量/类）在整个程序中只能有一个定义。
   违反 ODR 是未定义行为，可能导致链接错误或运行时异常。

Q: Unity Build 是什么？有什么优缺点？
A: 将多个 .cpp 合并为一个翻译单元编译。
   优点：减少头文件重复解析，加速编译。
   缺点：static 变量/函数可能命名冲突，增量编译粒度变粗。

Q: 编译器做了哪些优化？
A: 常量折叠、死代码消除、内联展开、循环展开/向量化、
   寄存器分配、尾调用优化、RVO/NRVO 等。
```
