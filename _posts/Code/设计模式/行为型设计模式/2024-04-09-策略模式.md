# 策略模式（Strategy Pattern）—— C# 实现

策略模式是一种行为型设计模式，定义一系列算法，把它们一个个封装起来，并且使它们可以互相替换。此模式使得算法可以独立于使用它的客户端变化。

## 结构

- **Strategy（策略接口）**：定义所有支持的算法的公共接口。
- **ConcreteStrategy（具体策略）**：实现策略接口的具体算法。
- **Context（上下文）**：持有一个策略对象，并可以切换策略。

## UML 类图

```
[Context] ——> [Strategy]
                  ↑
        [ConcreteStrategyA]
        [ConcreteStrategyB]
```

## C# 示例代码

```csharp
using System;

// 策略接口
public interface IStrategy
{
    void DoAlgorithm();
}

// 具体策略A
public class ConcreteStrategyA : IStrategy
{
    public void DoAlgorithm()
    {
        Console.WriteLine("Strategy A is used.");
    }
}

// 具体策略B
public class ConcreteStrategyB : IStrategy
{
    public void DoAlgorithm()
    {
        Console.WriteLine("Strategy B is used.");
    }
}

// 上下文
public class Context
{
    private IStrategy _strategy;

    public Context(IStrategy strategy)
    {
        _strategy = strategy;
    }

    public void SetStrategy(IStrategy strategy)
    {
        _strategy = strategy;
    }

    public void ExecuteAlgorithm()
    {
        _strategy.DoAlgorithm();
    }
}

// 使用示例
class Program
{
    static void Main()
    {
        Context context = new Context(new ConcreteStrategyA());
        context.ExecuteAlgorithm(); // 输出：Strategy A is used.

        context.SetStrategy(new ConcreteStrategyB());
        context.ExecuteAlgorithm(); // 输出：Strategy B is used.
    }
}
```

## 输出结果

```
Strategy A is used.
Strategy B is used.
```

## 适用场景

- 多个类只在算法或行为上略有不同。
- 需要动态地在几种算法中切换。
- 希望把算法的实现和使用算法的客户端分开。

## 优缺点

**优点：**
- 策略可自由切换，扩展性强。
- 避免多重条件判断（如if-else或switch）。
- 代码复用性和可维护性好。

**缺点：**
- 客户端必须了解所有策略，自己决定使用哪一个。
- 会增加系统中类的数量。

## 总结

策略模式使算法独立于使用它的客户端，易于扩展和维护，是行为变化频繁场景下的优雅选择。
