## 0. 渲染管线原理

### 0.1 GPU 渲染管线阶段

GPU 渲染管线是将 3D 场景数据转换为 2D 屏幕像素的完整处理流水线。理解每个阶段的职责和数据流向，是掌握 URP 的基础。

```mermaid
flowchart LR
    subgraph Application Stage["应用阶段 (CPU)"]
        A1["场景管理\n可见性判断"] --> A2["裁剪 Culling\n视锥体/遮挡剔除"]
        A2 --> A3["排序 Sorting\n按材质/深度排序"]
        A3 --> A4["提交 DrawCall\nCommandBuffer → GPU"]
    end

    subgraph Geometry Stage["几何阶段 (GPU)"]
        G1["Input Assembler\n顶点数据装配"] --> G2["Vertex Shader ★\n顶点变换 MVP"]
        G2 --> G3["Hull Shader\n曲面细分控制"]
        G3 --> G4["Tessellator\n曲面细分"]
        G4 --> G5["Domain Shader\n细分后顶点处理"]
        G5 --> G6["Geometry Shader\n图元生成/销毁"]
        G6 --> G7["Stream Output\n几何数据回写"]
        G7 --> G8["Clipping\n裁剪超出视锥体的图元"]
        G8 --> G9["Screen Mapping\nNDC → 屏幕坐标"]
    end 
    ApplicationStage["ApplicationStage"] --> GeometryStage["GeometryStage"] --> RasterizationStage["RasterizationStage"]
```


> **★ 标记**的阶段为可编程阶段，开发者通过 Shader 控制其行为。

#### 0.1.1 各阶段详解

| 阶段                   | 类型     | 职责                                                     | 性能关注点                     |
| ---------------------- | -------- | -------------------------------------------------------- | ------------------------------ |
| **Input Assembler**    | 固定功能 | 从顶点/索引缓冲区读取数据，组装图元（三角形/线/点）      | 顶点数据格式、索引缓冲区命中率 |
| **Vertex Shader**      | 可编程 ★ | 逐顶点变换（Object→World→View→Clip），计算法线、UV、切线 | 顶点数量、指令复杂度           |
| **Hull/Domain Shader** | 可编程 ★ | 曲面细分（Tessellation），动态增加几何细节               | 细分因子、移动端慎用           |
| **Geometry Shader**    | 可编程 ★ | 生成/销毁图元（草地、粒子），输入一个图元输出多个        | 性能差，移动端避免使用         |
| **Clipping**           | 固定功能 | 裁剪超出 CVV（Canonical View Volume）的图元              | 自动执行，无需干预             |
| **Screen Mapping**     | 固定功能 | NDC 坐标 → 屏幕像素坐标                                  | 视口设置                       |
| **Triangle Setup**     | 固定功能 | 计算三角形边方程、插值梯度                               | 自动执行                       |
| **Triangle Traversal** | 固定功能 | 遍历三角形覆盖的像素，生成 Fragment                      | 三角形大小影响效率             |
| **Early-Z**            | 固定功能 | 在 Fragment Shader 前剔除被遮挡的像素                    | Alpha Test/Clip 会破坏 Early-Z |
| **Fragment Shader**    | 可编程 ★ | 逐像素计算颜色（光照、纹理采样、PBR）                    | 指令数、纹理采样次数、Overdraw |
| **Depth/Stencil Test** | 固定功能 | 深度比较、模板测试                                       | ZWrite/ZTest 配置              |
| **Color Blending**     | 固定功能 | Alpha 混合、加法混合等                                   | 透明物体的 Overdraw            |


```hlsl
// Vertex Shader: the core transformation
VertexOutput vert(VertexInput v) {
    VertexOutput o;
    // Object → World
    float3 positionWS = TransformObjectToWorld(v.positionOS.xyz);
    // World → Clip (View + Projection combined)
    o.positionCS = TransformWorldToHClip(positionWS);
    // Normal: Object → World (using inverse transpose)
    o.normalWS = TransformObjectToWorldNormal(v.normalOS);
    return o;
}
```

#### 0.1.3 NDC 平台差异

```
NDC (Normalized Device Coordinates) platform differences:

  DirectX:   Z range [0, 1]   (near=0, far=1)    Y-axis: top-down
  OpenGL:    Z range [-1, 1]  (near=-1, far=1)   Y-axis: bottom-up
  Vulkan:    Z range [0, 1]   (near=0, far=1)    Y-axis: top-down
  Metal:     Z range [0, 1]   (near=0, far=1)    Y-axis: top-down

  Unity abstracts these differences:
    - UNITY_REVERSED_Z      : 1 on DX/Vulkan/Metal (reversed-Z for better precision)
    - UNITY_UV_STARTS_AT_TOP: 1 on DX/Vulkan/Metal
    - Use UNITY_NEAR_CLIP_VALUE macro for near plane value
```

### 0.3 SRP 设计原理

SRP（Scriptable Render Pipeline）的核心设计思想是将渲染流程的控制权从引擎内部交给开发者。

#### 0.3.1 SRP 核心抽象


#### 0.3.2 ScriptableRenderContext — The Bridge

`ScriptableRenderContext` is the critical bridge between Unity's C++ engine core and the C# SRP layer. It does NOT execute commands immediately — it records them.

```csharp
// ScriptableRenderContext key APIs
public struct ScriptableRenderContext {
    // Culling: ask engine to perform visibility culling
    public CullingResults Cull(ref ScriptableCullingParameters parameters);

    // Drawing: record draw commands (NOT immediate execution)
    public void DrawRenderers(CullingResults cullingResults,
                              ref DrawingSettings drawingSettings,
                              ref FilteringSettings filteringSettings);
    public void DrawSkybox(Camera camera);

    // Command Buffer: execute recorded GPU commands
    public void ExecuteCommandBuffer(CommandBuffer cmd);

    // Submit: flush all recorded commands to GPU
    // ⚠️ Nothing actually executes until Submit() is called
    public void Submit();

    // Setup: configure camera properties for shader globals
    public void SetupCameraProperties(Camera camera, bool stereoSetup = false);
}
```

```

3. SortCameras
4. Foreach : Camera
    1. BeginCameraRendering
    2. UpdateVolumeFramework
    3. RenderSingleCamera
        1. InitializeCameraData 拿到CameraData 设置摄像机需要的相关数据
        2. RenderSingleCamera

            2. ScriptableRenderer.SetupCullingParameters 拿到ScriptableCullingParameters
            3. 执行Cmd
```

> **CPU-GPU Parallelism**: The CPU and GPU work in parallel. While the CPU prepares Frame N+1, the GPU is still rendering Frame N. This is why `WaitForLastPresentation` appears in profilers — the CPU may need to wait for the GPU to finish before presenting.

#### 0.3.3 CommandBuffer vs ScriptableRenderContext

```
CommandBuffer:
  - Low-level GPU command recording
  - SetRenderTarget, ClearRenderTarget, Blit, DrawMesh, etc.
  - Must be submitted via context.ExecuteCommandBuffer(cmd)
  - Can be reused across frames

ScriptableRenderContext:
  - High-level rendering API
  - DrawRenderers (batch draw with culling results)
  - DrawSkybox, SetupCameraProperties
  - Manages command ordering and submission
  - context.Submit() flushes everything to GPU

Execution order within a frame:
  context.ExecuteCommandBuffer(cmd1)  // recorded at position 0
  context.DrawRenderers(...)          // recorded at position 1
  context.ExecuteCommandBuffer(cmd2)  // recorded at position 2
  context.Submit()                    // execute 0 → 1 → 2 in order
```

Culling is the first and most critical optimization — objects not visible to the camera are never submitted to the GPU.

### 0.4 DrawCall Anatomy

A DrawCall is the fundamental unit of GPU work submission. Understanding its internal structure is essential for optimization.

#### 0.4.1 What Happens in a Single DrawCall

```mermaid
flowchart TD
    DC["DrawCall"] --> S1["1. Set Pipeline State\nShader program\nBlend/Depth/Stencil state\nRasterizer state"]
    S1 --> S2["2. Bind Resources\nCBuffers (material params)\nTextures + Samplers\nStructuredBuffers"]
    S2 --> S3["3. Set Vertex/Index Buffer\nBind VBO + IBO\nSet vertex layout"]
    S3 --> S4["4. Set Render Target\nColor attachment(s)\nDepth attachment"]
    S4 --> S5["5. Draw Command\nDrawIndexed(indexCount,\n  startIndex, baseVertex)"]
    S5 --> S6["6. GPU Executes\nVertex → Rasterize → Fragment"]
```

#### 0.4.2 DrawCall Cost Breakdown

```
CPU cost of a DrawCall (approximate):

  State Validation:     2-5 μs    (driver validates pipeline state)
  Resource Binding:     1-3 μs    (bind CBuffers, textures)
  Command Recording:    0.5-1 μs  (write to command buffer)
  Driver Overhead:      3-10 μs   (driver translates to GPU commands)
  ─────────────────────────────────
  Total per DrawCall:   8-20 μs   (varies by platform/driver)

  At 60 FPS (16.6ms budget):
    Theoretical max DrawCalls: ~800-2000 (CPU-limited)
    Practical budget: 200-500 DrawCalls (leave room for game logic)

  SRP Batcher reduces per-DrawCall cost to ~1-3 μs
    by eliminating redundant state changes
```

### 0.5 Render Target Management

Render Targets (RT) are GPU textures that receive rendering output. URP manages multiple RTs throughout a frame.

#### 0.5.1 RT Lifecycle in a Frame

```mermaid
flowchart TD
    Start["Frame Start"] --> Alloc["Allocate RTs\n_CameraColorAttachment\n_CameraDepthAttachment\n_ShadowmapTexture"]

    Alloc --> Shadow["Shadow Pass\nRT: _ShadowmapTexture\nWrite: depth only"]
    Shadow --> Depth["Depth Prepass\nRT: _CameraDepthAttachment\nWrite: depth only"]
    Depth --> Opaque["Opaque Pass\nColor RT: _CameraColorAttachment\nDepth RT: _CameraDepthAttachment\nWrite: color + depth"]
    Opaque --> CopyDepth["Copy Depth\nSrc: _CameraDepthAttachment\nDst: _CameraDepthTexture\n(for shader sampling)"]
    CopyDepth --> CopyColor["Copy Color\nSrc: _CameraColorAttachment\nDst: _CameraOpaqueTexture\n(for transparent refraction)"]
    CopyColor --> Trans["Transparent Pass\nColor RT: _CameraColorAttachment\nRead: _CameraDepthTexture\nRead: _CameraOpaqueTexture"]
    Trans --> Post["Post Processing\nPing-pong between temp RTs"]
    Post --> Blit["Final Blit\nSrc: _CameraColorAttachment\nDst: Backbuffer (screen)"]
    Blit --> Release["Release temp RTs"]
```

#### 0.5.2 Why Copy Depth / Copy Color?

```
Problem:
  GPU cannot read and write the same RT simultaneously.
  Transparent objects need to sample scene depth (soft particles)
  and scene color (refraction/distortion).

Solution:
  Copy the depth/color attachment to a separate texture
  before the transparent pass.

  _CameraDepthAttachment  →  Copy  →  _CameraDepthTexture (read-only)
  _CameraColorAttachment  →  Copy  →  _CameraOpaqueTexture (read-only)

Optimization:
  ✅ Only enable CopyDepth if shaders use _CameraDepthTexture
  ✅ Only enable CopyColor if shaders use _CameraOpaqueTexture
  ✅ On Vulkan/Metal: use Subpass for zero-cost depth access
```

#### 0.5.3 Load/Store Actions (Mobile TBR Optimization)

On Tile-Based Rendering (TBR) GPUs (Mali, Adreno, Apple), RT load/store actions critically affect bandwidth.

```csharp
// URP configures load/store actions per pass
ConfigureTarget(colorAttachment, depthAttachment);
ConfigureClear(ClearFlag.All, Color.clear);

// Translates to:
//   Color: LoadAction.DontCare (clear anyway) → saves bandwidth
//   Depth: LoadAction.DontCare (clear anyway) → saves bandwidth
//   Color: StoreAction.Store (need result for next pass)
//   Depth: StoreAction.DontCare (if not needed later) → saves bandwidth
```

```
Load/Store Action impact on bandwidth:

  LoadAction.Load      → Read RT from DRAM to Tile Memory (costly)
  LoadAction.Clear     → Initialize Tile Memory to clear value (cheap)
  LoadAction.DontCare  → Skip loading, undefined content (cheapest)

  StoreAction.Store    → Write Tile Memory back to DRAM (costly)
  StoreAction.Resolve  → MSAA resolve + store (costly)
  StoreAction.DontCare → Skip writing, discard content (cheapest)

  Example bandwidth savings for 1080p RGBA8:
    Load + Store:     1920×1080×4 × 2 = ~16 MB per RT per frame
    DontCare + Store: 1920×1080×4 × 1 = ~8 MB (save 50%)
    DontCare + DontCare: 0 MB (transient RT, stays on-chip)
```

### 0.6 Rendering Equation and Shading Models

The rendering pipeline ultimately evaluates the rendering equation at each visible pixel.

#### 0.6.1 The Rendering Equation

```
Lo(p, ωo) = Le(p, ωo) + ∫Ω fr(p, ωi, ωo) · Li(p, ωi) · (n · ωi) dωi

Where:
  Lo = outgoing radiance at point p in direction ωo
  Le = emitted radiance (emissive materials)
  fr = BRDF (Bidirectional Reflectance Distribution Function)
  Li = incoming radiance from direction ωi
  n  = surface normal
  Ω  = hemisphere above the surface

In real-time rendering, we approximate the integral:
  - Direct lighting: sum over discrete light sources
  - Indirect lighting: precomputed (lightmaps, probes, SSAO)
  - Environment: reflection probes, skybox sampling
```

#### 0.6.2 URP PBR Shading (Lit Shader)

```hlsl
// URP Lit shader uses Cook-Torrance BRDF
// Simplified PBR equation:
//   f = kd * diffuse + ks * specular
//   diffuse = baseColor / PI
//   specular = D * G * F / (4 * NdotL * NdotV)

// D = Normal Distribution Function (GGX/Trowbridge-Reitz)
float D_GGX(float NdotH, float roughness) {
    float a2 = roughness * roughness;
    float d = (NdotH * NdotH) * (a2 - 1.0) + 1.0;
    return a2 / (PI * d * d);
}

// G = Geometry Function (Smith-Schlick)
float G_SmithSchlick(float NdotV, float NdotL, float roughness) {
    float k = (roughness + 1.0) * (roughness + 1.0) / 8.0;
    float gv = NdotV / (NdotV * (1.0 - k) + k);
    float gl = NdotL / (NdotL * (1.0 - k) + k);
    return gv * gl;
}

// F = Fresnel (Schlick approximation)
float3 F_Schlick(float VdotH, float3 F0) {
    return F0 + (1.0 - F0) * pow(1.0 - VdotH, 5.0);
}
```

### 0.8 Depth Buffer Precision and Reversed-Z

#### 0.8.1 The Depth Precision Problem

```
Standard Z-buffer (non-reversed):
  Near plane (z=0.1) maps to NDC z ≈ 0.0
  Far plane  (z=1000) maps to NDC z = 1.0

  Problem: depth distribution is highly non-linear
    50% of depth precision is used for the first 1% of the depth range
    Objects far from camera suffer severe Z-fighting

  Depth precision at distance d:
    precision ∝ 1/d²  (gets exponentially worse with distance)

Reversed-Z (used by URP on DX/Vulkan/Metal):
  Near plane (z=0.1) maps to NDC z = 1.0
  Far plane  (z=1000) maps to NDC z = 0.0

  Combined with floating-point precision (more bits near 0):
    Depth precision becomes nearly uniform across the entire range
    Z-fighting virtually eliminated
```

```hlsl
// URP handles reversed-Z automatically
// In shader, check with:
#if UNITY_REVERSED_Z
    // Near = 1.0, Far = 0.0
    // Depth comparison: Greater = closer
#else
    // Near = 0.0, Far = 1.0
    // Depth comparison: Less = closer
#endif

// Linear depth reconstruction
float LinearEyeDepth(float rawDepth, float4 _ZBufferParams) {
    // _ZBufferParams: (1-far/near, far/near, x/far, y/far)
    return 1.0 / (_ZBufferParams.z * rawDepth + _ZBufferParams.w);
}
```

## 5. SRP Batcher 与批次系统

### 5.1 StandardBatch vs SRPBatcher 对比

```mermaid
graph LR
    subgraph StandardBatch["StandardBatch（传统）"]
        S1[创建 built-in CBuffer] --> S2[创建 Material CBuffer]
        S2 --> S3[创建 BindObject CBuffer]
        S3 --> S4[Draw]
    end

    subgraph SRPBatcher["SRPBatcher（优化）"]
        P1[Bind PersistentMaterial CBuffer<br/>持久化，不重复上传] --> P2[Bind offset Object CBuffer<br/>仅更新偏移量]
        P2 --> P3[Draw]
    end
```

| 特性         | StandardBatch        | SRPBatcher                       |
| ------------ | -------------------- | -------------------------------- |
| 材质数据上传 | 每次 Draw 都上传     | 持久化存储在 GPU，仅变化时更新   |
| 每物体数据   | 每次 Draw 都上传     | 专用大型 GPU CBuffer，仅更新偏移 |
| 合批条件     | 相同材质 + 相同 Mesh | 相同 Shader 变体（材质可不同）   |
| 性能         | 材质越多越慢         | 大量不同材质时性能优势明显       |

### 5.2 SRPBatcher 打断合批的原因

```
kSRPBatchBreakDifferentShader          // Shader 不同
kSRPBatchBreakCauseMultiPassShader     // 多 Pass Shader
kSRPBatchkeywordsChange                // Keyword 变化
kSRPBatchEndOfBatchFlush               // 批次结束强制 Flush
kSRPBatchNotCompatibleNode             // 节点不兼容
kSRPBatchMaterialNeedDeviceStateChange // 材质需要改变设备状态
kSRPBatchFirstCall                     // 第一次调用
kSRPBatchMaterialBufferOverride        // 材质 Buffer 被覆盖
```

### 5.3 SRPBatcher 兼容条件

```
✅ 支持 SRPBatcher 的条件：
  - Mesh / SkinnedMesh 渲染器
  - Shader 兼容 SRP Batcher（在 Shader Inspector 中可查看）
  - 对象不使用 MaterialPropertyBlock

❌ 不支持 SRPBatcher 的情况：
  - 使用 MaterialPropertyBlock
  - 多 Pass Shader
  - 不兼容的 Shader 变体
```

### 5.4 RenderLoopNewBatcher.Draw 流程

```
RenderLoopNewBatcher.Draw
    → ObjectData 不一致 → Flush（提交当前批次）
    → 填充 Buffer（写入 CBuffer 数据）
    → SRPBatcher.Flush
        → 填充 PersistentMaterial CBuffer
        → 填充 PerObject CBuffer（偏移量）
        → 提交 DrawCall
```

### 5.5 SRPBatcher 完整提交流程

```mermaid
flowchart TD
    Start["ScriptableRenderer.Execute"] --> Sort["SortStable\n按 RenderPassEvent 排序"]
    Sort --> Loop["遍历所有 RenderPass"]
    Loop --> ExecPass["ExecuteRenderPass"]
    ExecPass --> DrawCall["DrawRenderers / DrawMesh"]

    DrawCall --> Check{"检查是否可合批"}
    Check -->|"相同 Shader 变体"| Batch["加入当前 Batch"]
    Check -->|"Shader/Keyword 不同"| Flush1["Flush 当前 Batch\n提交 DrawCall"]
    Flush1 --> NewBatch["开启新 Batch"]
    NewBatch --> Batch

    Batch --> WriteCB["写入 CBuffer 数据"]
    WriteCB --> MatCB["PersistentMaterial CBuffer\n首次上传后 GPU 持久驻留\n仅数据变化时重新上传"]
    WriteCB --> ObjCB["PerObject CBuffer\n写入 ObjectData\n（变换矩阵等）"]

    MatCB --> BindOffset["Bind CBuffer + 更新 Offset\n不同物体只改偏移量"]
    ObjCB --> BindOffset

    BindOffset --> GPU["GPU 执行 DrawCall"]
    GPU --> NextObj{"还有物体？"}
    NextObj -->|"是"| Check
    NextObj -->|"否"| FinalFlush["Final Flush\n提交剩余 Batch"]

    subgraph 传统 StandardBatch
        S1["每次 Draw"] --> S2["上传 built-in CBuffer"]
        S2 --> S3["上传 Material CBuffer"]
        S3 --> S4["上传 BindObject CBuffer"]
        S4 --> S5["DrawCall"]
    end

    subgraph SRPBatcher优化
        P1["首次"] --> P2["上传 PersistentMaterial CBuffer\n驻留 GPU"]
        P2 --> P3["后续只更新 PerObject Offset"]
        P3 --> P4["DrawCall（CPU 开销大幅降低）"]
    end
```

### 5.6 SRPBatcher 内存布局

```mermaid
graph TD
    subgraph GPU显存
        MCB["PersistentMaterial CBuffer\n材质A参数 | 材质B参数 | 材质C参数\n（连续存储，不重复上传）"]
        OCB["PerObject CBuffer\nObj0数据 | Obj1数据 | Obj2数据\n（大型 Buffer，按 offset 访问）"]
    end

    subgraph CPU每帧
        D1["Draw Obj0"] -->|"Bind offset=0"| OCB
        D2["Draw Obj1"] -->|"Bind offset=256B"| OCB
        D3["Draw Obj2"] -->|"Bind offset=512B"| OCB
        D1 -->|"Bind 材质A"| MCB
        D2 -->|"Bind 材质A（无需重传）"| MCB
        D3 -->|"Bind 材质B（无需重传）"| MCB
    end
```

> **核心优化点**：传统管线每次 DrawCall 都要上传材质数据；SRPBatcher 将材质数据持久化在 GPU，CPU 只需更新 PerObject 的偏移量，大幅减少 CPU→GPU 数据传输带宽。

## 6. CBuffer 与数据传递

### 6.1 URP 内置 CBuffer 分类

| CBuffer 名称       | 更新频率     | 包含数据                       |
| ------------------ | ------------ | ------------------------------ |
| `UnityPerFrame`    | 每帧一次     | 时间、全局光照                 |
| `UnityPerCamera`   | 每摄像机一次 | 视图矩阵、投影矩阵、摄像机位置 |
| `UnityPerDraw`     | 每 DrawCall  | 物体变换矩阵（M矩阵）          |
| `UnityPerMaterial` | 材质变化时   | 材质参数（颜色、贴图等）       |
| `_LightBuffer`     | 每帧/摄像机  | 灯光颜色、方向、衰减           |

### 6.6 CBuffer 与 UniformBuffer 原理详解

#### 6.6.1 概念对应关系

```mermaid
graph TD
    HLSL["HLSL / Unity ShaderLab\nCBUFFER_START / CBUFFER_END"] --> DX["DirectX 11/12\nConstant Buffer\nb0, b1, b2..."]
    HLSL --> GL["OpenGL / OpenGL ES\nUniform Buffer Object\nUBO binding point"]
    HLSL --> VK["Vulkan\nUniform Buffer\nDescriptor Set"]
    HLSL --> MT["Metal\nArgument Buffer\nbuffer(0), buffer(1)..."]
```

> Unity 通过 `CBUFFER_START/CBUFFER_END` 宏统一抽象，编译时自动映射到各平台实现，开发者无需关心底层差异。

#### 6.6.2 CBuffer vs UniformBuffer 对比

| 特性            | CBuffer（DirectX）           | UniformBuffer / UBO（OpenGL/Vulkan）    |
| --------------- | ---------------------------- | --------------------------------------- |
| 命名            | Constant Buffer              | Uniform Buffer Object (UBO)             |
| 绑定方式        | 寄存器槽 `b0~b13`            | Binding Point（0, 1, 2...）             |
| 最大大小        | 64KB（DX11）/ 无限制（DX12） | 16KB（最低保证）/ 通常 64KB+            |
| 数据对齐        | 16字节对齐（float4 边界）    | std140 / std430 布局规则                |
| 更新方式        | `UpdateSubresource` / `Map`  | `glBufferSubData` / `vkCmdUpdateBuffer` |
| GPU 缓存        | 专用常量缓存（L1 Cache）     | 通用显存，部分平台有专用缓存            |
| SRPBatcher 支持 | ✅ 持久化 CBuffer             | ✅ 持久化 UBO                            |

#### 6.6.3 内存对齐规则（std140 / HLSL）

```hlsl
// ⚠️ HLSL CBuffer 对齐规则：每个变量不能跨 float4（16字节）边界

// ❌ 错误示例：float3 + float 会被编译器自动填充，但手动排列可能出错
CBUFFER_START(UnityPerMaterial)
    float3 _Color;   // 占 12 字节
    float  _Alpha;   // 紧接在后，共 16 字节 ✅
    float  _Value1;  // 新的 float4 起始
    float  _Value2;
    float  _Value3;
    // 此处有 4 字节隐式 padding！
    float4 _NextVec; // 下一个 float4
CBUFFER_END

// ✅ 推荐写法：手动补齐，避免隐式 padding
CBUFFER_START(UnityPerMaterial)
    float4 _ColorAlpha;   // xyz=Color, w=Alpha
    float4 _Values;       // xyzw 全部使用
CBUFFER_END
```

```
std140 对齐规则（OpenGL UBO）：
  float        → 4 字节对齐
  vec2         → 8 字节对齐
  vec3 / vec4  → 16 字节对齐
  mat4         → 16 字节对齐（每列视为 vec4）
  数组元素      → 每个元素强制 16 字节对齐（即使是 float）

HLSL CBuffer 规则：
  - 变量不能跨 16 字节（float4）边界
  - 数组元素每个占 16 字节（与 std140 一致）
  - 结构体内部按成员最大对齐
```

#### 6.6.4 CBuffer 更新流程

```mermaid
flowchart TD
    CPU["CPU 侧数据\n（C# / C++）"] --> |"Shader.SetGlobalVector\nMaterial.SetFloat\ncmd.SetGlobalBuffer"| Upload["上传到 GPU 显存"]
    Upload --> CB["CBuffer / UBO\n（GPU 常量缓存）"]
    CB --> VS["Vertex Shader\n读取变换矩阵"]
    CB --> FS["Fragment Shader\n读取材质参数"]

    subgraph SRPBatcher优化
        PersistCB["PersistentMaterial CBuffer\n首次上传后持久驻留 GPU"] --> |"仅数据变化时更新"| CB
        OffsetCB["PerObject CBuffer\n只更新 offset 偏移量"] --> CB
    end
```

#### 6.6.5 各平台 CBuffer 绑定槽位

```hlsl
// DirectX：使用寄存器槽 b0~b13
CBUFFER_START(UnityPerFrame)    // → register(b0)
CBUFFER_START(UnityPerCamera)   // → register(b1)
CBUFFER_START(UnityPerDraw)     // → register(b2)
CBUFFER_START(UnityPerMaterial) // → register(b3)

// OpenGL / Vulkan：使用 Binding Point
layout(std140, binding = 0) uniform UnityPerFrame { ... };
layout(std140, binding = 1) uniform UnityPerCamera { ... };
layout(std140, binding = 2) uniform UnityPerDraw { ... };
layout(std140, binding = 3) uniform UnityPerMaterial { ... };

// Metal：使用 buffer 索引
// vertex shader: buffer(0), buffer(1)...
// Unity 自动处理，开发者无需手动指定
```

#### 6.6.6 CBuffer 大小限制与最佳实践

```
各平台 CBuffer 大小限制：
  DirectX 11：单个 CBuffer 最大 64KB（4096 个 float4）
  DirectX 12：无硬性限制，但建议 < 64KB
  OpenGL ES：最低保证 16KB，通常 64KB
  Vulkan：Push Constants 最大 128~256 字节（极小数据用）
           Uniform Buffer 通常 64KB+
  Metal：Argument Buffer 无严格限制

最佳实践：
  ✅ 将高频更新数据（每帧/每摄像机）与低频数据（材质）分开放不同 CBuffer
  ✅ 使用 float4 对齐，避免隐式 padding 浪费空间
  ✅ 大量只读数据（灯光列表）使用 StructuredBuffer 而非 CBuffer
  ✅ 极少量高频数据（< 128字节）考虑 Push Constants（Vulkan）
  ❌ 避免在单个 CBuffer 中混合不同更新频率的数据
  ❌ 避免 CBuffer 超过 64KB
```

#### 6.6.7 Push Constants（Vulkan 特有）

```hlsl
// Vulkan Push Constants：极低延迟的小数据传递（< 128 字节）
// 适合每 DrawCall 变化的少量数据（如物体 ID、材质索引）

// Vulkan GLSL 写法
layout(push_constant) uniform PushConstants {
    mat4 modelMatrix;   // 64 字节
    int  materialIndex; // 4 字节
} pc;

// Unity 中通过 SRPBatcher 的 PerObject CBuffer 实现类似效果
// 不需要手动使用 Push Constants
```

## 8. 注意事项

### 8.1 SRPBatcher 兼容性

> ⚠️ **MaterialPropertyBlock 会破坏 SRPBatcher**
> - 使用 `MaterialPropertyBlock` 设置属性时，SRPBatcher 自动失效
> - 替代方案：将属性写入 `CBUFFER_START(UnityPerMaterial)` 块

```csharp
// ❌ 破坏 SRPBatcher
var mpb = new MaterialPropertyBlock();
mpb.SetColor("_Color", Color.red);
renderer.SetPropertyBlock(mpb);

// ✅ 兼容 SRPBatcher（通过材质实例）
renderer.material.SetColor("_Color", Color.red);
```

> ⚠️ **Texture2DArray 会打断 SRP Batcher**
> - 不同材质绑定不同的 `Texture2DArray` 资源时，每次切换都会触发 Batch Flush
> - 原因：`Texture2DArray` 的 GPU Descriptor 切换属于设备状态变更，无法通过 CBuffer offset 优化
> - FrameDebugger 中显示中断原因：`SRP: Material needs device state change`

```hlsl
// ❌ 打断 SRP Batcher：不同材质绑定不同的 Texture2DArray
TEXTURE2D_ARRAY(_DiffuseArray);  // Material A binds ArrayTex_1
SAMPLER(sampler_DiffuseArray);   // Material B binds ArrayTex_2
                                  // → Batch break on every switch!

CBUFFER_START(UnityPerMaterial)
    float _SliceIndex;
    float4 _BaseColor;
CBUFFER_END

half4 frag(Varyings input) : SV_Target {
    // Sample from array using per-material slice index
    return SAMPLE_TEXTURE2D_ARRAY(_DiffuseArray, sampler_DiffuseArray,
                                  input.uv, _SliceIndex);
}
```

```csharp
// ✅ 正确做法：所有材质共享同一个 Texture2DArray 实例，仅 slice index 不同
Texture2DArray sharedArray = CreateMergedTextureArray(); // merge all textures

foreach (var mat in allMaterials) {
    mat.SetTexture("_DiffuseArray", sharedArray);  // same GPU resource
    mat.SetFloat("_SliceIndex", sliceIndex);        // different index in CBuffer
    // → SRP Batcher compatible! CBuffer offset only
}
```

### 8.2 颜色空间与 Gamma 校正

> ⚠️ **HDR 开启时颜色格式变化**
> - 开启 HDR：颜色格式为 `B10G11R11_UFloatPack32`
> - 关闭 HDR：颜色格式为 `R8G8B8A8_SRGB`
> - UI 使用线性空间时，Blit 到 FrameBuffer 会自动做 Linear→SRGB 转换
> - 如需关闭 Gamma 校正，需修改源码关闭 RenderTexture GammaCorrection

### 8.3 FrameDebug 调试

> ⚠️ **FrameDebug 不显示 RenderTarget 名称**
> - 原因：变量用函数返回值，FrameDebug 无法识别名称
> - 解决：将 RT 名称改为常量字符串

### 8.4 灯光数量限制

> - 默认每摄像机最多 16 个可见光
> - 修改 `MAX_VISIBLE_LIGHTS` 宏后必须重启 Unity 才能生效

### 8.5 Camera Stack 注意事项

> ⚠️ **堆叠摄像机（Camera Stack）的渲染目标**
> - Base Camera：创建 `CreateCameraRenderTarget`，设置 AA
> - Overlay Camera：复用 Base Camera 的 Color/Depth Attachment

### 8.6 Volume 性能问题

> ⚠️ **多个 Volume 混合开销较大**
> - 每帧 `UpdateVolumeFramework` 会遍历所有 Volume 并混合参数
> - 优化：减少 Volume 数量，合并相同区域的 Volume
> - 对于静态场景，可以缓存 Volume 混合结果

### 8.7 资源冗余问题

> ⚠️ **ScriptableRendererData 更新问题**
> - 在 `ScriptableRendererData` 增加 Override 数组覆盖数据
> - 避免运行时频繁修改 RendererData，会触发管线重建

## 9. 优化方案

### 9.1 SRP Batcher 最大化利用

```
优化策略：
1. 所有材质参数放入 CBUFFER_START(UnityPerMaterial) 块
2. 避免使用 MaterialPropertyBlock（改用材质实例）
3. 减少 Shader 变体数量（Shader Stripping）
4. 同一 Shader 的物体尽量连续渲染（减少 Keyword 切换）
```

### 9.2 Shader Stripping（变体裁剪）

```csharp
// 在 IPreprocessShaders 中裁剪不需要的变体
public class ShaderVariantStripper : IPreprocessShaders {
    public void OnProcessShader(Shader shader, ShaderSnippetData snippet,
                                IList<ShaderCompilerData> data) {
        // 移除不需要的平台变体
        for (int i = data.Count - 1; i >= 0; i--) {
            if (data[i].shaderKeywordSet.IsEnabled(keyword_not_needed))
                data.RemoveAt(i);
        }
    }
}
```

### 9.3 渲染目标优化

```
1. 避免不必要的 CopyDepthPass / CopyColorPass
   - 仅在 Shader 中使用 _CameraDepthTexture 时才开启 CopyDepth
   - 仅在 Shader 中使用 _CameraOpaqueTexture 时才开启 CopyColor

2. 使用 MSAA 替代 TAA（移动端）
   - MSAA 在 TBR 架构上几乎免费
   - TAA 需要额外的 RT 和 Blit，移动端开销大

3. 减少 RT 切换
   - 合并相邻 Pass 的 RT（相同 Color/Depth 目标）
   - 使用 SubPass（Vulkan/Metal）合并 Pass
```

### 9.4 后处理优化

```
1. Bloom 优化：
   - 降低 Bloom 迭代次数（Iterations）
   - 使用半分辨率 RT 进行模糊
   - 使用 Mask 限制 Bloom 影响范围（只影响特定物体）

2. Volume 优化：
   - 减少 Volume 数量，合并相邻 Volume
   - 静态场景缓存 Volume 混合结果
   - 使用 Volume Layer Mask 限制影响范围

3. 后处理 Pass 合并：
   - 将多个后处理效果合并到一个 Pass（UberPost）
   - 避免每个效果单独 Blit
```

### 9.5 灯光优化

```
1. 减少逐像素光源数量（移动端保持 ≤ 4）
2. 使用烘焙光照替代动态光（静态场景）
3. 使用 Light Layer 限制灯光影响范围
4. Forward+ 模式下合理设置 Tile 大小
5. 关闭不需要的阴影（Per-Light Shadow 设置）
```

### 9.6 移动端 TBR 优化

```
TBR（Tile-Based Rendering）架构优化：
1. 避免 Framebuffer Fetch 之外的 RT 读取
2. 减少 Depth/Stencil 的 Load/Store 操作
3. 使用 MSAA（TBR 上几乎免费）
4. 避免 Compute Shader 写入 Framebuffer（破坏 TBR 优化）
5. 使用 SubPass 合并 Pass（Vulkan/Metal）
```

## 10. 平台适配

### 10.1 各平台渲染 API 对应

| 平台        | 渲染 API           | CBuffer 实现    | 特殊注意                |
| ----------- | ------------------ | --------------- | ----------------------- |
| Windows PC  | DirectX 11/12      | Constant Buffer | 支持 StructuredBuffer   |
| macOS / iOS | Metal              | Argument Buffer | 需要 Metal Shader 编译  |
| Android     | Vulkan / OpenGL ES | Uniform Buffer  | 低端设备用 OpenGL ES    |
| WebGL       | WebGL 2.0          | Uniform Buffer  | 不支持 StructuredBuffer |
| PS5 / Xbox  | GNM / DX12         | Constant Buffer | 主机专用优化            |

### 10.2 CBuffer vs UniformBuffer 平台映射

```
Unity ShaderLab CBUFFER_START/CBUFFER_END
    → DirectX：Constant Buffer (b0, b1, ...)
    → OpenGL：Uniform Buffer Object (UBO)
    → Vulkan：Uniform Buffer / Push Constants
    → Metal：Argument Buffer
（Unity 自动处理平台映射，开发者只需写 CBUFFER 语法）
```

### 10.3 StructuredBuffer 平台支持

```csharp
// 检查是否支持 StructuredBuffer（用于 Forward+ 灯光数据）
bool useStructuredBuffer = SystemInfo.supportsComputeShaders
                        && !Application.isMobilePlatform
                        && SystemInfo.graphicsDeviceType != GraphicsDeviceType.OpenGLES2;

if (useStructuredBuffer) {
    // 使用 StructuredBuffer 传递灯光数据（PC/主机）
    _AdditionalLightsBuffer = new ComputeBuffer(maxLights, stride);
} else {
    // 使用数组传递灯光数据（移动端/WebGL）
    Shader.SetGlobalVectorArray("_AdditionalLightsPosition", positions);
}
```

### 10.4 HDR 与颜色格式

```csharp
// 根据平台选择颜色格式
RenderTextureFormat colorFormat;
if (cameraData.isHdrEnabled) {
    // HDR 格式
    colorFormat = SystemInfo.SupportsRenderTextureFormat(RenderTextureFormat.ARGBHalf)
        ? RenderTextureFormat.ARGBHalf      // 移动端
        : RenderTextureFormat.DefaultHDR;   // PC（B10G11R11）
} else {
    colorFormat = RenderTextureFormat.Default; // R8G8B8A8_SRGB
}
```

### 10.5 质量分级配置

```
高配（PC / 主机）:
  - 渲染模式：Forward+
  - 最大附加光源：8+
  - 阴影：级联阴影（4级）+ 软阴影
  - 后处理：全效果
  - MSAA：4x 或 TAA
  - StructuredBuffer 传递灯光数据

中配（移动高端）:
  - 渲染模式：Forward+
  - 最大附加光源：4
  - 阴影：级联阴影（2级）
  - 后处理：精简效果
  - MSAA：2x

低配（移动低端）:
  - 渲染模式：Forward
  - 最大附加光源：1-2（逐顶点）
  - 阴影：简单阴影或无阴影
  - 后处理：禁用或仅 Bloom
  - 无 MSAA
```

### 10.6 Bloom 后处理适配

```
高端设备：
  - 全分辨率 Bloom
  - 迭代次数：6
  - 高斯模糊

低端设备：
  - 半分辨率 Bloom
  - 迭代次数：3
  - 双线性模糊（更快）
  - 降低 Threshold，减少参与 Bloom 的像素
```
