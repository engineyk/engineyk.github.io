# TBR 与 TBDR 架构深度解析

---

## 1. 三种 GPU 渲染架构总览

### 1.1 架构分类

```
┌─────────────────────────────────────────────────────────────────┐
│                    GPU 渲染架构分类                              │
├──────────────────┬──────────────────┬───────────────────────────┤
│   IMR            │   TBR            │   TBDR                    │
│   Immediate      │   Tile-Based     │   Tile-Based              │
│   Mode           │   Rendering      │   Deferred                │
│   Rendering      │                  │   Rendering               │
│                  │                  │                           │
│   立即模式渲染    │   基于瓦片的渲染  │   基于瓦片的延迟渲染        │
├──────────────────┼──────────────────┼───────────────────────────┤
│   NVIDIA         │   Qualcomm       │   Apple (A-series)        │
│   AMD            │   Adreno         │   Imagination             │
│   Intel          │                  │   PowerVR                 │
│   (桌面/主机)     │   ARM Mali       │                           │
│                  │   Samsung Xclipse│                           │
└──────────────────┴──────────────────┴───────────────────────────┘
```

### 1.2 一句话区分

```
IMR  : 每个三角形立即完整处理，直接读写显存中的 Framebuffer
TBR  : 先收集所有三角形，按 Tile 分组，每个 Tile 在片上 SRAM 中完成渲染
TBDR : 在 TBR 基础上，Tile 内先做隐面消除（HSR），只着色可见像素

核心差异：
IMR  → 逐三角形处理，带宽大
TBR  → 逐 Tile 处理，带宽小
TBDR → 逐 Tile + 隐面消除，带宽小 + 零 Overdraw
```

## 2. IMR（Immediate Mode Rendering）立即模式渲染

### 2.1 工作流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    IMR 渲染流程                                  │
│                                                                 │
│   CPU 提交 DrawCall                                             │
│       │                                                         │
│       ▼                                                         │
│   ┌──────────┐                                                  │
│   │ 顶点着色器│ <-- 逐三角形处理                                  │
│   └────┬─────┘                                                  │
│        ▼                                                        │
│   ┌───────────┐                                                 │
│   │ 光栅化     │                                                │
│   └────┬──────┘                                                │
│        ▼                                                       │
│   ┌───────────┐    ┌──────────────────────────┐                │
│   │ 片元着色器 │--> │ Framebuffer（显存 DRAM）  │                │
│   └────┬──────┘ <--│ Color + Depth + Stencil  │                │
│        │    每个像素│ 每次读写都走总线          │                │
│        │    都要读写│                          │                │
│        ▼           └──────────────────────────┘                │
│   处理下一个三角形...                                            │
│                                                                │
│   特点：                                                        │
│   • 三角形提交后立即完整走完管线                                  │
│   • 每个像素的 Depth Test 需要读显存                              │
│   • 每个像素的 Color Write 需要写显存                             │
│   • 被遮挡的像素也会执行片元着色器（Overdraw）                     │
│   • 带宽消耗 = f(像素数 × Overdraw × 每像素字节数)                │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 IMR 的优势与劣势

```
✅ 优势：
  • 延迟低：三角形提交后立即开始处理
  • 不需要存储中间几何数据（无 Tile List）
  • 适合大量 Geometry Shader / Tessellation 输出
  • 桌面 GPU 有大容量 L2 Cache 缓解带宽问题

❌ 劣势：
  • 带宽消耗巨大：每个像素操作都访问显存
  • Overdraw 浪费：被遮挡像素也执行了完整着色
  • 功耗高：DRAM 访问功耗 >> SRAM 访问功耗
  • 不适合移动端：移动端带宽和功耗预算有限
```

## 3. TBR（Tile-Based Rendering）基于瓦片的渲染

### 3.1 核心思想

```
核心思想：将屏幕分成小块（Tile），每个 Tile 在片上 SRAM 中完成渲染

为什么要分 Tile？
─────────────────────────────────────────
  1080p Framebuffer = 1920×1080×4 bytes = 8.3 MB
  GPU 片上 SRAM 通常只有 32-128 KB
  8.3 MB >> 128 KB → 无法将整个 Framebuffer 放入 SRAM

  但如果分成 32×32 的 Tile：
  单个 Tile = 32×32×4 bytes = 4 KB（Color）
  + 32×32×4 bytes = 4 KB（Depth）
  = 8 KB → 轻松放入 SRAM！

  1080p 共需 60×34 = 2040 个 Tile
  逐个 Tile 在 SRAM 中渲染，渲染完写回显存
```

### 3.2 分 Tile 后，多个 Tile 为什么能全放到 SRAM？

```
⚠️ 关键误区澄清：多个 Tile 不是"同时"放到 SRAM 中的！

常见疑问：
  1080p 分成 2040 个 Tile（32×32），每个 Tile 8 KB
  2040 × 8 KB = 15.9 MB
  GPU 片上 SRAM 只有 32-128 KB
  15.9 MB >> 128 KB → 怎么放得下？

答案：不需要同时放下所有 Tile，而是逐个（或少量并行）处理

═══ 核心原理：时间换空间 ═══

  SRAM 是被复用的，流程如下：

  ┌─────────────────────────────────────────────────────────┐
  │  时间线                                                  │
  │                                                         │
  │  T0: ┌──────────┐                                       │
  │      │ Tile(0,0)│ <- 加载到 SRAM，渲染，写回显存          │
  │      └──────────┘                                       │
  │                                                         │
  │  T1: ┌──────────┐                                       │
  │      │ Tile(1,0)│ <- 同一块 SRAM，覆盖上一个 Tile 数据    │
  │      └──────────┘   渲染，写回显存                       │
  │                                                         │
  │  T2: ┌──────────┐                                       │
  │      │ Tile(2,0)│ <- 同一块 SRAM，再次覆盖               │
  │      └──────────┘   渲染，写回显存                       │
  │                                                         │
  │  ...重复 2040 次，所有 Tile 处理完毕                      │
  │                                                         │
  │  SRAM 始终只存放 1 个（或少量）Tile 的数据                │
  │  就像只有 1 个碗，但可以洗完一碗再盛下一碗                 │
  └─────────────────────────────────────────────────────────┘

═══ 实际情况：流水线并行处理多个 Tile ═══

  现代 GPU 不是严格串行处理 Tile，而是有多个 Shader Core
  每个 Shader Core 可以独立处理不同的 Tile

  ┌─────────────────────────────────────────────────────────┐
  │  GPU 内部多核并行                                        │
  │                                                         │
  │  Shader Core 0: ┌──────────┐ → ┌──────────┐ → ...       │
  │                 │ Tile(0,0)│   │ Tile(4,0)│             │
  │                 └──────────┘   └──────────┘             │
  │                                                         │
  │  Shader Core 1: ┌──────────┐ → ┌──────────┐ → ...       │
  │                 │ Tile(1,0)│   │ Tile(5,0)│             │
  │                 └──────────┘   └──────────┘             │
  │                                                         │
  │  Shader Core 2: ┌──────────┐ → ┌──────────┐ → ...       │
  │                 │ Tile(2,0)│   │ Tile(6,0)│             │
  │                 └──────────┘   └──────────┘             │
  │                                                         │
  │  Shader Core 3: ┌──────────┐ → ┌──────────┐ → ...       │
  │                 │ Tile(3,0)│   │ Tile(7,0)│             │
  │                 └──────────┘   └──────────┘             │
  │                                                         │
  │  每个 Core 有自己的 Tile SRAM 区域                        │
  │  同一时刻 SRAM 中只有 N 个 Tile（N = Core 数量）          │
  │  而不是 2040 个                                          │
  └─────────────────────────────────────────────────────────┘

═══ SRAM 容量 vs 同时处理的 Tile 数量 ═══

  以 Apple A16 为例：
  • Tile SRAM 总容量：约 128 KB（推测值，Apple 未公开）
  • 单个 Tile（32×32，RGBA8 + D32F）：
    Color: 32×32×4 = 4 KB
    Depth: 32×32×4 = 4 KB
    合计: 8 KB
  • 可同时容纳: 128 / 8 = 16 个 Tile
  • 实际并行处理: 取决于 Shader Core 数量和调度

  以 Qualcomm Adreno 740 为例：
  • GMEM（Graphics Memory / Tile SRAM）：3 MB
  • 使用大 Tile（如 512×256）：
    Color: 512×256×4 = 512 KB
    Depth: 512×256×4 = 512 KB
    合计: 1 MB
  • 可同时容纳: 3 个大 Tile
  • Adreno 策略：用更大的 Tile 减少 Tile 总数

═══ 类比理解 ═══

  把整个屏幕想象成一幅巨大的拼图（2040 块）
  SRAM 是你的工作台（只能放几块拼图）
  显存是你的储物柜（可以放所有拼图）

  工作流程：
  1. 从储物柜取出 Tile(0,0) 需要的材料 → 放到工作台
  2. 在工作台上完成 Tile(0,0) 的绘制
  3. 把完成的 Tile(0,0) 放回储物柜
  4. 清空工作台
  5. 取出 Tile(1,0) 的材料 → 放到工作台
  6. 重复...

  工作台（SRAM）永远只需要放得下 1 块拼图
  而不是 2040 块

═══ 这也解释了 Tile Split 的原因 ═══

  如果单个 Tile 的数据太大（多 MRT + MSAA + 高精度）
  超出了 SRAM 中分配给单个 Tile 的容量
  GPU 就必须把一个 Tile 分成多次渲染（Tile Split）
  每次只处理部分 RT → 性能严重下降

  所以优化的关键不是 Tile 总数，而是单个 Tile 的 SRAM 占用
```

### 3.3 两阶段工作流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    TBR 两阶段流程                                │
│                                                                 │
│  ═══════════════ 阶段 1：Binning / Tiling ═══════════════        │
│                                                                 │
│   CPU 提交所有 DrawCall                                          │
│       │                                                         │
│       ▼                                                         │
│   ┌──────────┐                                                  │
│   │ 顶点着色器│ <- 处理所有三角形的顶点                            │
│   └────┬─────┘                                                  │
│        ▼                                                        │
│   ┌──────────────────────────────────────────┐                  │
│   │           Tiling / Binning               │                  │
│   │                                          │                  │
│   │   确定每个三角形覆盖了哪些 Tile            │                  │
│   │   生成 Tile List（每个 Tile 的三角形列表） │                  │
│   │                                          │                  │
│   │   ┌─────┬──────┬─────┬─────┐             │                  │
│   │   │Tile │Tile  │Tile │Tile │             │                  │
│   │   │0,0  │1,0   │2,0  │3,0  │  ...        │                  │
│   │   │△1,3 │△2,5 │△1  │△4,7 │             │                  │
│   │   │△5   │△8   │△6  │△9   │             │                  │
│   │   └──────┴─────┴─────┴─────┘             │                  │
│   │                                          │                  │
│   └──────────────┬───────────────────────────┘                  │
│                  │ Tile List 写入显存                            │
│                  ▼                                              │
│  ═══════════════ 阶段 2：Rendering ═══════════════               │
│                                                                 │
│   对每个 Tile 依次执行：                                         │
│   ┌──────────────────────────────────────────┐                  │
│   │  1. Load Tile（如需要，从显存加载旧数据）  │                  │
│   │  2. 从 Tile List 读取该 Tile 的三角形     │                  │
│   │  3. 光栅化 + 片元着色                     │                  │
│   │     ↕ 所有读写都在 Tile SRAM 中完成        │                  │
│   │     ↕ Depth Test → SRAM                  │                  │
│   │     ↕ Color Write → SRAM                 │                  │
│   │     ↕ Blend → SRAM                       │                  │
│   │  4. Store Tile（将结果写回显存）           │                  │
│   └──────────────────────────────────────────┘                  │
│                                                                 │
│   重复直到所有 Tile 处理完毕                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 Tile List（Polygon List / Visibility Stream）

```
Tile List 是 TBR 的关键中间数据结构：

┌──────────────────────────────────────────────────┐
│  Tile (2,3) 的 Tile List                         │
│  ┌────────────────────────────────────────────┐  │
│  │ DrawCall #0: Triangle 5, 12, 17            │  │
│  │ DrawCall #1: Triangle 3                    │  │
│  │ DrawCall #3: Triangle 8, 9, 22, 45         │  │
│  │ DrawCall #7: Triangle 1                    │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  存储内容：                                       │
│  • 三角形索引 / 顶点数据引用                       │
│  • 对应的 DrawCall / 状态信息                     │
│  • 裁剪后的三角形几何数据                          │
│                                                  │
│  存储位置：系统内存（显存 DRAM）                   │
│  这是 TBR 相比 IMR 额外的开销                      │
│                                                  │
│  大小估算（1080p，100K 三角形）：                  │
│  • 每个三角形平均覆盖 2-4 个 Tile                  │
│  • 每条记录约 32-64 bytes                         │
│  • 总计约 6-25 MB                                 │
│  • 极端情况（大三角形覆盖多 Tile）可能更大          │
└──────────────────────────────────────────────────┘
```

### 3.5 TBR 的带宽节省分析

```
以 1080p RGBA8 + D24S8 为例：

═══ IMR 每帧带宽（假设 Overdraw = 2x）═══
  Color Read:   8.3 MB × 2 (blend/test) = 16.6 MB
  Color Write:  8.3 MB × 2              = 16.6 MB
  Depth Read:   8.3 MB × 2              = 16.6 MB
  Depth Write:  8.3 MB × 2              = 16.6 MB
  总计: ≈ 66.4 MB

═══ TBR 每帧带宽 ═══
  Tile List Write (Binning):  ~10 MB（额外开销）
  Tile List Read  (Render):   ~10 MB（额外开销）
  Color Store:  8.3 MB × 1 = 8.3 MB（每 Tile 只写回一次）
  Depth Store:  0 MB（如果不需要后续读取，DontCare）
  总计: ≈ 28.3 MB

═══ 节省 ═══
  66.4 - 28.3 = 38.1 MB/frame
  节省比例: ~57%
  
  关键：Overdraw 越高，TBR 节省越多
  因为 TBR 的 Overdraw 读写发生在 SRAM 中（免费）
  而 IMR 的 Overdraw 读写发生在 DRAM 中（昂贵）
```

### 3.6 TBR 的 Load/Store Action

```
Load/Store Action 是 TBR 性能优化的核心控制点：

┌──────────────────────────────────────────────────────────────┐
│  Tile 生命周期                                                │
│                                                              │
│  ┌─────────┐    ┌──────────────────┐     ┌──────────┐        │
│  │  Load   │───→│  Tile 内渲染      │───→│  Store   │        │
│  │  Action │    │  (片上 SRAM)      │    │  Action  │        │
│  └─────────┘    └──────────────────┘     └──────────┘        │
│                                                              │
│  Load Action 选项：                                          │
│  ┌────────────────┬──────────────────────────────────────┐   │
│  │ Load           │ 从显存读取旧数据到 SRAM（8.3MB 带宽）  │   │
│  │ Clear          │ 用固定颜色填充 SRAM（0 带宽）          │   │
│  │ DontCare       │ 不初始化 SRAM（0 带宽，内容未定义）     │   │
│  └────────────────┴──────────────────────────────────────┘   │
│                                                              │
│  Store Action 选项：                                         │
│  ┌────────────────┬──────────────────────────────────────┐   │
│  │ Store          │ 将 SRAM 结果写回显存（8.3MB 带宽）     │   │
│  │ Resolve        │ MSAA Resolve 后写回（8.3MB 带宽）     │   │
│  │ StoreAndResolve│ 同时保存 MSAA 和 Resolved             │   │
│  │ DontCare       │ 丢弃 SRAM 数据（0 带宽）               │   │
│  └────────────────┴──────────────────────────────────────┘   │
│                                                              │
│  最优组合示例：                                               │
│  Color:  Clear + Store     = 8.3 MB（只写不读）               │
│  Depth:  Clear + DontCare  = 0 MB（完全免费！）               │
│  MSAA:   DontCare + Resolve = 8.3 MB（Memoryless）           │
│                                                              │
│  最差组合：                                                   │
│  Color:  Load + Store      = 16.6 MB（读+写）                 │
│  Depth:  Load + Store      = 16.6 MB（读+写）                 │
└──────────────────────────────────────────────────────────────┘
```

## 4. TBDR（Tile-Based Deferred Rendering）基于瓦片的延迟渲染

### 4.1 TBDR vs TBR 的关键区别

```
⚠️ 注意：这里的 "Deferred" 不是 Deferred Shading（延迟着色）
   而是 "Deferred" 片元着色到隐面消除之后

TBR  流程：  Binning → Tile 内光栅化 → 片元着色 → Depth Test → 写入
                                       ↑ 被遮挡的像素也执行了着色（Overdraw）

TBDR 流程：  Binning → Tile 内光栅化 → HSR → 片元着色 → 写入
                                       ↑ Hidden Surface Removal
                                       ↑ 先确定哪些像素可见
                                       ↑ 只对可见像素执行着色
                                       ↑ Overdraw = 0（理想情况）
```

### 4.2 HSR（Hidden Surface Removal）隐面消除

```
┌─────────────────────────────────────────────────────────────────┐
│                    TBDR HSR 工作原理                             │
│                                                                 │
│   Tile 内所有三角形已知（来自 Binning 阶段）                       │
│       │                                                         │
│       ▼                                                         │
│   ┌──────────────────────────────────────────┐                  │
│   │  对 Tile 内每个像素：                     │                  │
│   │                                          │                  │
│   │  1. 遍历所有覆盖该像素的三角形             │                  │
│   │  2. 只做光栅化 + Depth Test（不着色）      │                 │
│   │  3. 记录每个像素最终可见的三角形 ID         │                 │
│   │                                           │                 │
│   │  ┌─────────────────────────────────┐      │                 │
│   │  │ Pixel(3,5): Triangle #7 wins    │      │                 │
│   │  │ Pixel(3,6): Triangle #2 wins    │      │                 │
│   │  │ Pixel(3,7): Triangle #7 wins    │      │                 │
│   │  │ ...                             │      │                 │
│   │  └─────────────────────────────────┘      │                 │
│   │                                           │                 │
│   │  4. 只对 "wins" 的像素执行片元着色器        │                 │
│   │     → 每个像素只着色一次                   │                 │
│   │     → Overdraw = 0                       │                  │
│   └──────────────────────────────────────────┘                  │
│                                                                 │
│   HSR 的硬件实现（以 PowerVR/Apple 为例）：                       │
│   • 使用 Tag Buffer 记录每个像素的可见三角形                      │
│   • ISP（Image Synthesis Processor）专用硬件执行                 │
│   • 在 Depth/Stencil Test 阶段完成，不消耗 Shader Core           │
│   • 处理速度：每周期可处理多个像素的深度测试                       │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 TBDR 的 Overdraw 消除效果

```
场景：1080p，平均 3 层 Overdraw（典型游戏场景）

═══ IMR ═══
  片元着色器执行次数: 1920×1080×3 = 6,220,800
  浪费的着色: 6,220,800 - 2,073,600 = 4,147,200（66.7% 浪费）

═══ TBR ═══
  片元着色器执行次数: 6,220,800（与 IMR 相同！）
  TBR 只节省带宽，不减少 Overdraw
  但 Overdraw 的读写在 SRAM 中，功耗低

═══ TBDR ═══
  片元着色器执行次数: 2,073,600（每像素只着色一次）
  节省: 4,147,200 次片元着色器执行
  ALU 节省: 66.7%
  功耗节省: 显著（片元着色器是功耗大户）

量化对比（假设片元着色器 20 cycles/pixel）：
  IMR:  6.2M × 20 = 124M cycles
  TBR:  6.2M × 20 = 124M cycles
  TBDR: 2.1M × 20 = 41.5M cycles  <- 节省 66.7% ALU
```

### 4.4 HSR 失效的情况

```
⚠️ 以下操作会破坏 HSR，导致 TBDR 退化为 TBR：

┌──────────────────────────────────────────────────────────────┐
│  1. Alpha Test / Discard                                     │
│     ─────────────────────────────────────────                │
│     片元着色器中使用 discard/clip                              │
│     HSR 无法预知哪些像素会被 discard                           │
│     必须执行着色器才能确定可见性                                │
│     → 该像素之后的所有三角形都无法做 HSR                        │
│                                                              │
│  2. Alpha Blend（半透明）                                     │
│     ─────────────────────────────────────────                │
│     半透明物体需要按顺序混合，不能跳过                          │
│     HSR 只对不透明物体有效                                     │
│     → 半透明物体仍然有 Overdraw                                │
│                                                              │
│  3. 片元着色器写入 Depth（gl_FragDepth / SV_Depth）            │
│     ─────────────────────────────────────────                │
│     深度值在着色器中被修改                                     │
│     HSR 使用的是光栅化插值深度                                 │
│     着色器修改后深度不一致 → HSR 失效                          │
│                                                              │
│  4. 读取 Framebuffer（Programmable Blending）                 │
│     ─────────────────────────────────────────                │
│     着色器读取当前像素颜色                                     │
│     需要先渲染之前的三角形 → 打断 HSR 流水线                    │
│                                                              │
│  5. SSBO / Image Store 有副作用的着色器                       │
│     ─────────────────────────────────────────                │
│     着色器有写入操作（UAV/SSBO）                               │
│     即使像素被遮挡，副作用也不能跳过                            │
│     → HSR 必须保守处理                                        │
└──────────────────────────────────────────────────────────────┘

最佳实践：
  ✅ 先渲染所有不透明物体（HSR 完全生效）
  ✅ 不透明物体不需要排序（HSR 自动处理遮挡）
  ✅ 尽量用 Alpha to Coverage 替代 Alpha Test
  ✅ 减少使用 discard/clip
  ❌ 不要在不透明 Pass 中使用 discard
  ❌ 不要在片元着色器中写入深度
```

## 5. 三种架构完整对比

### 5.1 特性对比表

| 特性                 | IMR                | TBR                       | TBDR                                |
| -------------------- | ------------------ | ------------------------- | ----------------------------------- |
| **代表厂商**         | NVIDIA, AMD, Intel | Qualcomm Adreno, ARM Mali | Apple A-series, Imagination PowerVR |
| **渲染单位**         | 逐三角形           | 逐 Tile                   | 逐 Tile                             |
| **Framebuffer 位置** | 显存 DRAM          | Tile SRAM                 | Tile SRAM                           |
| **Binning 阶段**     | 无                 | 有                        | 有                                  |
| **HSR（隐面消除）**  | 无（依赖 Early-Z） | 无                        | 有（硬件级）                        |
| **Overdraw 处理**    | 全部着色           | 全部着色（但读写在 SRAM） | 只着色可见像素                      |
| **带宽消耗**         | 高                 | 低                        | 最低                                |
| **ALU 效率**         | 有 Overdraw 浪费   | 有 Overdraw 浪费          | 接近零 Overdraw                     |
| **延迟**             | 低（立即处理）     | 较高（需等 Binning 完成） | 较高（需等 Binning + HSR）          |
| **Tile List 开销**   | 无                 | 有（额外内存+带宽）       | 有（额外内存+带宽）                 |
| **MSAA 成本**        | 带宽 ×N            | 几乎免费                  | 几乎免费                            |
| **适合场景**         | 桌面/主机          | 移动端                    | 移动端（iOS）                       |
| **功耗**             | 高                 | 低                        | 最低                                |

### 5.2 渲染流程对比图

```
═══ IMR ═══
  Triangle → Vertex Shader → Rasterize → Fragment Shader → DRAM
  Triangle → Vertex Shader → Rasterize → Fragment Shader → DRAM
  Triangle → Vertex Shader → Rasterize → Fragment Shader → DRAM
  ...（逐三角形，每次都访问 DRAM）

═══ TBR ═══
  Phase 1 (Binning):
    All Triangles → Vertex Shader → Tile Assignment → Tile List (DRAM)
  
  Phase 2 (Rendering):
    For each Tile:
      Load Tile → Rasterize all triangles → Fragment Shader → Store Tile
                  ↑ 全部在 SRAM 中 ↑

═══ TBDR ═══
  Phase 1 (Binning):
    All Triangles → Vertex Shader → Tile Assignment → Tile List (DRAM)
  
  Phase 2 (Rendering):
    For each Tile:
      Load Tile → Rasterize all → HSR → Fragment Shader(visible only) → Store
                                  ↑ 隐面消除
                                  ↑ 只着色可见像素
```

### 6.2 ARM Mali（TBR）

```
┌──────────────────────────────────────────────────────────────┐
│  ARM Mali 架构特点                                            │
│                                                              │
│  类型：TBR（非 TBDR，但有 FPK 优化）                           │
│  Tile 大小：16×16 像素（固定）                                 │
│  特殊优化：                                                   │
│                                                              │
│  1. FPK（Forward Pixel Kill）                                │
│     ─────────────────────────────────────────                │
│     Mali 的 "准 HSR" 机制                                     │
│     当一个不透明像素通过 Early-Z 后：                          │
│     • 检查管线中是否有更早提交但尚未完成着色的同位置像素          │
│     • 如果有，Kill 掉那个更早的像素（因为它会被当前像素遮挡）     │
│     • 减少正在执行的无用着色                                   │
│                                                              │
│     与 TBDR HSR 的区别：                                      │
│     • HSR：先确定可见性，再着色（零 Overdraw）                  │
│     • FPK：已经开始着色，发现被遮挡后 Kill（减少但非零）         │
│     • FPK 依赖提交顺序，前后排序可提高效率                      │
│                                                              │
│  2. AFBC（ARM Frame Buffer Compression）                     │
│     ─────────────────────────────────────────                │
│     硬件级无损带宽压缩                                         │
│     对 Tile Store/Load 和纹理采样都生效                        │
│     典型压缩率：30-60%                                        │
│                                                              │
│  3. Transaction Elimination                                  │
│     ─────────────────────────────────────────                │
│     比较当前 Tile 渲染结果与上一帧同位置 Tile                   │
│     如果内容完全相同，跳过 Store（节省写带宽）                  │
│     对静态 UI、天空盒等不变区域非常有效                         │
│     节省比例取决于场景动态程度                                 │
│                                                              │
│  Tile 大小 16×16 的影响：                                     │
│     • Tile 数量多：1080p = 120×68 = 8160 个 Tile              │
│     • Binning 开销较大                                        │
│     • 但 SRAM 需求小，适合低功耗设计                           │
│     • 大三角形覆盖多 Tile → Tile List 膨胀                     │
│                                                              │
│  Mali Streamline Profiler 关键指标：                          │
│     • GPU External Read/Write Bytes                          │
│     • Tiles (count)                                          │
│     • Fragment FPK Kill Rate                                 │
│     • Transaction Elimination Rate                           │
└──────────────────────────────────────────────────────────────┘
```

## 7. TBR/TBDR 对渲染管线设计的影响

### 7.1 RenderPass 设计原则

```
┌─────────────────────────────────────────────────────────────┐
│  核心原则：最小化 Tile Load/Store 次数                        │
│                                                             │
│  每次切换 RenderTarget = 一次 Tile Flush                     │
│  Tile Flush = Store 当前 Tile + Load 新 Tile                │
│  = 大量带宽消耗                                              │
│                                                             │
│  ❌ 错误做法（频繁切换 RT）：                                 │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐             │
│  │ RT: A  │→ │ RT: B  │→ │ RT: A  │→ │ RT: C  │             │
│  │ Store A│  │ Store B│  │ Load A │  │ Store A│             │
│  │        │  │ Load A │  │ Store A│  │ Load C │             │
│  └────────┘  └────────┘  └────────┘  └────────┘             │
│  带宽：Store×4 + Load×3 = 7 次 Tile 传输                     │
│                                                             │
│  ✅ 正确做法（合并 Pass，使用 SubPass）：                     │
│  ┌──────────────────────────────────────────┐                │
│  │ RenderPass                               │                │
│  │ ┌──────────┬──────────┬──────────┐       │                │
│  │ │ SubPass0 │ SubPass1 │ SubPass2 │       │                │
│  │ │ G-Buffer │ Lighting │ Tonemap  │       │                │
│  │ │          │ (fetch)  │ (fetch)  │       │                │
│  │ └──────────┴──────────┴──────────┘       │                │
│  │ 所有 SubPass 在同一个 Tile 内完成          │                │
│  │ G-Buffer 数据不离开 SRAM                  │                │
│  └──────────────────────────────────────────┘                │
│  带宽：Store×1（最终结果）+ Load×0（Clear）= 1 次 Tile 传输     │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 SubPass 与 Framebuffer Fetch

```
SubPass 是 Vulkan/Metal 的关键特性，专为 TBR/TBDR 设计：

┌──────────────────────────────────────────────────────────────┐
│  SubPass 工作原理                                             │
│                                                              │
│  同一个 RenderPass 内的多个 SubPass：                         │
│  • 共享同一个 Tile SRAM                                       │
│  • 后续 SubPass 可以直接读取前面 SubPass 的输出                │
│  • 不需要 Store → Load 的往返                                 │
│  • 中间 Attachment 可以声明为 Memoryless                      │
│                                                              │
│  典型应用：单 Pass 延迟渲染                                    │
│                                                              │
│  SubPass 0: 输出 G-Buffer（Albedo, Normal, Depth）            │
│    → G-Buffer 数据留在 Tile SRAM                              │
│                                                              │
│  SubPass 1: 读取 G-Buffer，执行光照计算                        │
│    → 通过 input_attachment / framebuffer_fetch 读取           │
│    → 数据来自 SRAM，零带宽                                     │
│    → G-Buffer 声明为 Memoryless，不分配显存                    │
│                                                               │
│  带宽节省：                                                    │
│  传统延迟渲染：G-Buffer Store + Load = 2 × (8.3MB × 4) = 66MB  │
│  SubPass 延迟渲染：G-Buffer 不离开 SRAM = 0 MB                 │
│  节省：66 MB/frame                                            │
└──────────────────────────────────────────────────────────────┘

API 支持：
┌──────────┬──────────────────────────────────────┐
│ Vulkan   │ VkSubpassDescription + input_attachment│
│ Metal    │ MTLRenderPassDescriptor + [[color(n)]] │
│ OpenGL ES│ GL_EXT_shader_framebuffer_fetch        │
│ Unity URP│ NativeRenderPass API (2021.2+)         │
└──────────┴──────────────────────────────────────┘
```

### 7.3 破坏 TBR/TBDR 优化的常见操作

```
┌──────────────────────────────────────────────────────────────┐
│  ⚠️ 以下操作会导致 Tile Flush，增加带宽消耗                   │
│                                                              │
│  1. 中途读取 RenderTarget 内容                               │
│     ─────────────────────────────────────────                │
│     CopyTexture(currentRT, tempRT)                           │
│     → 强制 Store 当前所有 Tile 到显存                        │
│     → 然后才能执行 Copy                                      │
│     替代方案：使用 SubPass / Framebuffer Fetch               │
│                                                              │
│  2. 频繁切换 RenderTarget                                    │
│     ─────────────────────────────────────────                │
│     SetRenderTarget(A) → Draw → SetRenderTarget(B) → Draw   │
│     → SetRenderTarget(A) → Draw                             │
│     每次切换 = Store + Load                                  │
│     替代方案：合并到同一个 RenderPass                         │
│                                                              │
│  3. 在渲染中途使用 Compute Shader 写入 RT                    │
│     ─────────────────────────────────────────                │
│     Compute Shader 不走 Tile 管线                            │
│     写入 RT 需要先 Flush Tile                                │
│     替代方案：将 Compute 放在 RenderPass 之间                 │
│                                                              │
│  4. 不正确的 Load/Store Action                               │
│     ─────────────────────────────────────────                │
│     忘记设置 LoadAction.Clear → 默认 Load（浪费带宽）        │
│     忘记设置 StoreAction.DontCare → 默认 Store（浪费带宽）   │
│     Depth 不需要后续使用却 Store → 浪费 8.3 MB               │
│                                                              │
│  5. glReadPixels / ReadbackTexture                           │
│     ─────────────────────────────────────────                │
│     强制 GPU 完成所有渲染 + Store                            │
│     然后 CPU 读取 → 完全打断流水线                           │
│     替代方案：异步 Readback（延迟几帧读取）                   │
│                                                              │
│  6. 过大的 RenderTarget                                      │
│     ─────────────────────────────────────────                │
│     RT 越大，每个 Tile 需要的 SRAM 越多                      │
│     超出 SRAM 容量 → 需要分多次渲染（Tile Split）            │
│     RGBA16F + Depth32F + MSAA 4x 可能超出 SRAM              │
│     替代方案：减少 RT 数量或精度                              │
└──────────────────────────────────────────────────────────────┘
```

## 8. MSAA 在 TBR/TBDR 上几乎免费的原理

```
┌──────────────────────────────────────────────────────────────┐
│  为什么 MSAA 在 TBR/TBDR 上几乎免费？                        │
│                                                              │
│  IMR 架构 MSAA 4x：                                         │
│  ┌──────┐    带宽 ×4    ┌──────────────┐                    │
│  │Shader│ ────────────→ │ 4x MSAA RT   │                    │
│  │ Core │ <-──────────── │ (显存 DRAM)  │                    │
│  └──────┘    带宽 ×4    │ 33.2 MB      │                    │
│                         └──────────────┘                    │
│  每个采样点都要读写显存 → 带宽 ×4                             │
│                                                              │
│  TBR/TBDR 架构 MSAA 4x：                                    │
│  ┌──────┐  SRAM 读写  ┌──────────┐  1x Store  ┌─────────┐  │
│  │Shader│ ──────────→ │ Tile     │ ─────────→ │ 1x RT   │  │
│  │ Core │ <-────────── │ SRAM     │  Resolve   │ (DRAM)  │  │
│  └──────┘  零带宽成本  │ 4x 采样  │  自动完成   │ 8.3 MB  │  │
│                       └──────────┘            └─────────┘  │
│                                                              │
```

## 13. 面试高频问题

### Q1: 请解释 IMR、TBR、TBDR 三种架构的区别

```
答：
IMR（立即模式渲染）：逐三角形处理，每个像素操作直接读写显存 DRAM，
带宽消耗大，但延迟低，适合桌面 GPU（NVIDIA/AMD）。

TBR（基于瓦片的渲染）：分两阶段，先 Binning（将三角形分配到 Tile），
再逐 Tile 在片上 SRAM 中完成渲染。带宽大幅降低，因为 Tile 内的
Depth Test、Color Write、Blend 都在 SRAM 中完成。代表：Qualcomm Adreno、ARM Mali。

TBDR（基于瓦片的延迟渲染）：在 TBR 基础上增加 HSR（隐面消除），
Tile 内先确定每个像素的可见三角形，只对可见像素执行片元着色器，
实现零 Overdraw。代表：Apple A-series、Imagination PowerVR。

注意：TBDR 的 "Deferred" 不是 Deferred Shading，而是指延迟着色到
隐面消除之后。
```

### Q2: 为什么 MSAA 在移动端几乎免费？

```
答：
因为 TBR/TBDR 架构中，多采样数据只存在于 Tile 片上 SRAM 中：
1. MSAA 的多个采样点在 SRAM 中读写，不消耗显存带宽
2. Tile 渲染完成后，硬件自动 Resolve（取平均），只将 1x 数据写回显存
3. MSAA RT 可声明为 Memoryless，不分配显存

因此带宽和显存开销与不开 MSAA 几乎相同。
而桌面 IMR 架构的 MSAA 需要在显存中分配 4x 大小的 RT，
每个采样点都要读写显存，带宽 ×4。
```

### Q3: 什么操作会破坏 TBR 的优化？

```
答：
1. 中途读取 RenderTarget（CopyTexture）→ 强制 Store 所有 Tile
2. 频繁切换 RenderTarget → 每次切换 = Store + Load
3. 不正确的 Load/Store Action → 不必要的带宽消耗
4. 在 RenderPass 中间插入 Compute Shader → 强制 Tile Flush
5. glReadPixels / ReadbackTexture → 完全打断流水线
6. 过多 MRT 或过高精度 → 超出 Tile SRAM → Tile Split
```

### Q4: SubPass 是什么？为什么对移动端很重要？

```
答：
SubPass 是 Vulkan/Metal 中 RenderPass 内的子阶段。
同一 RenderPass 内的多个 SubPass 共享 Tile SRAM，
后续 SubPass 可以通过 input_attachment / framebuffer_fetch
直接读取前面 SubPass 的输出，数据不离开 SRAM。

对移动端的重要性：
传统延迟渲染需要将 G-Buffer Store 到显存再 Load 回来，
带宽消耗巨大（1080p 4个 G-Buffer ≈ 66 MB/frame）。
使用 SubPass 后，G-Buffer 留在 Tile SRAM 中，带宽 = 0。
这使得延迟渲染在移动端变得可行。
```

### Q5: Mali 的 FPK 和 Apple 的 HSR 有什么区别？

```
答：
HSR（Apple/PowerVR）：
  在着色之前完成隐面消除
  Tile 内所有三角形已知，全局判断每像素可见性
  只着色可见像素 → 零 Overdraw
  不依赖三角形提交顺序

FPK（ARM Mali）：
  在着色过程中进行
  当新像素通过 Early-Z 后，检查管线中是否有同位置的旧像素正在着色
  如果有，Kill 掉旧像素
  减少但不能完全消除 Overdraw
  从前到后排序可以提高 FPK 效率

本质区别：HSR 是 "先判断再着色"，FPK 是 "边着色边 Kill"。
HSR 更彻底但需要专用硬件（ISP），FPK 实现更简单。
```

### Q6: Adreno 的 FlexRender 和 LRZ 是什么？

```
答：
FlexRender：Adreno 可以在 TBR 和 IMR 模式之间动态切换。
  简单场景用 IMR（避免 Binning 开销），
  复杂场景用 TBR（节省带宽）。
  由驱动自动决策。

LRZ（Low Resolution Z）：在 Binning 阶段生成低分辨率深度图，
  Rendering 阶段用 LRZ 做早期剔除。
  类似 TBDR 的 HSR，但精度是块级别而非像素级别。
  可以减少部分 Overdraw，但不如 HSR 彻底。
```

### Q7: 如何在 Unity URP 中优化 TBR 性能？

```
答：
1. 开启 NativeRenderPass（Use Native RenderPass = true）
   → 利用 SubPass 减少 Tile Load/Store

2. 正确设置 Load/Store Action
   → Color: Clear + Store
   → Depth: Clear + DontCare（如果后续不需要）

3. 使用 Memoryless RT
   → Depth、MSAA RT 声明为 Memoryless

4. 避免中途 Blit / CopyTexture
   → 使用 SetRenderTarget + Draw 替代

5. 减少 RenderTarget 切换
   → 合并 Pass，减少 Tile Flush

6. 开启 MSAA（移动端几乎免费）
   → 4x MSAA 推荐

7. 控制 MRT 数量和精度
   → 避免 Tile Split
```

### Q8: Tile Split 是什么？如何避免？

```
答：
Tile Split 发生在 Tile 内的 RT 数据超出 GPU 片上 SRAM 容量时。
GPU 被迫将一个 Tile 分成多次渲染，每次只处理部分 RT。

触发条件：
  多个 MRT + 高精度格式 + MSAA
  例如：4 个 RGBA16F MRT + D32F + MSAA 4x
  = 4×8×4 + 4×4 + 1×4 = 148 KB per 32×32 Tile
  可能超出某些 GPU 的 SRAM

避免方法：
  1. 减少 MRT 数量（Pack 数据到更少的 RT）
  2. 降低 RT 精度（RGBA16F → RGBA8 / R11G11B10F）
  3. 降低 MSAA 采样数（4x → 2x）
  4. 使用 SubPass 让中间 RT 为 Memoryless
```
