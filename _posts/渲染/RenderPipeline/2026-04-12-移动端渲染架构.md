## 1. 移动端 GPU 架构概述

### 1.1 桌面端 vs 移动端 GPU 根本差异

移动端 GPU 的设计目标与桌面端截然不同：

```
┌─────────────────────────────────────────────────────────────────┐
│                    设计目标对比                                  │
├──────────────────┬──────────────────┬───────────────────────────┤
│     维度         │    桌面端 GPU     │      移动端 GPU           │
├──────────────────┼──────────────────┼───────────────────────────┤
│  首要目标         │  最大吞吐量       │  功耗效率 (perf/watt)      │
│  散热预算         │  200-450W TDP    │  3-8W 整机功耗             │
│  内存带宽         │  256-1024 GB/s   │  25-80 GB/s               │
│  内存类型         │  GDDR6/HBM       │  LPDDR4X/LPDDR5           │
│  缓存策略         │  大 L2 Cache      │  小 L2 + On-Chip Tile Mem │
│  渲染架构         │  Immediate Mode   │  Tile-Based (TBR/TBDR)    │
│  ALU 数量         │  数千-上万        │  数百-数千                 │
│  时钟频率         │  1.5-2.5 GHz     │  0.5-1.0 GHz              │
└──────────────────┴──────────────────┴───────────────────────────┘
```

### 1.2 为什么移动端选择 Tile-Based 架构

**核心原因：带宽是移动端最大的性能瓶颈和功耗杀手。**

```
功耗对比 (相对值):

  操作类型              功耗 (相对 ALU 运算)
  ─────────────────────────────────────────
  ALU 运算 (FP32)       ~1x     (基准)
  L1 Cache 读取         ~2x
  L2 Cache 读取         ~5x
  On-Chip Tile Memory   ~3x
  DRAM 读取 (LPDDR)     ~100-200x  ← 这是关键！
  DRAM 写入 (LPDDR)     ~100-200x
  ─────────────────────────────────────────

  结论：一次 DRAM 访问的功耗 ≈ 100-200 次 ALU 运算
  → 减少 DRAM 访问是移动端优化的第一要务
```

### 1.3 Immediate Mode vs Tile-Based 架构

```
=== Immediate Mode Rendering (IMR) - 桌面端 ===

  Draw Call 1 ──▶ Vertex ──▶ Rasterize ──▶ Fragment ──▶ Framebuffer (DRAM)
  Draw Call 2 ──▶ Vertex ──▶ Rasterize ──▶ Fragment ──▶ Framebuffer (DRAM)
  Draw Call 3 ──▶ Vertex ──▶ Rasterize ──▶ Fragment ──▶ Framebuffer (DRAM)
  ...

  特点：
  - 每个三角形立即处理到最终像素
  - 频繁读写 Framebuffer (DRAM)
  - Depth Buffer 也在 DRAM 中频繁读写
  - 依赖大带宽和大 Cache 来缓解


=== Tile-Based Rendering (TBR) - 移动端 ===

  Phase 1: Geometry Processing (所有 Draw Calls)
  ┌──────────────────────────────────────────────┐
  │ Draw Call 1 ──▶ Vertex Shader ──▶ Tile Bin  │
  │ Draw Call 2 ──▶ Vertex Shader ──▶ Tile Bin  │
  │ Draw Call 3 ──▶ Vertex Shader ──▶ Tile Bin  │
  │ ...                                          │
  │ Output: Per-tile triangle lists (in memory)  │
  └──────────────────────────────────────────────┘
                        │
                        ▼
  Phase 2: Fragment Processing (per tile)
  ┌──────────────────────────────────────────────┐
  │ For each tile (e.g., 16x16 or 32x32):        │
  │   1. Load tile's triangle list               │
  │   2. Rasterize + Fragment shade              │
  │      (all in On-Chip Tile Memory!)           │
  │   3. Resolve: Write final color to DRAM      │
  │                                              │
  │ Tile Memory: Color + Depth + Stencil         │
  │ Size: ~几十 KB (足够一个 Tile)                │
  └──────────────────────────────────────────────┘

  优势：
  - Depth/Stencil 测试在 On-Chip 完成，零 DRAM 带宽
  - Color Blending 在 On-Chip 完成
  - 只有最终结果写回 DRAM
  - MSAA 几乎免费（resolve 在 On-Chip）
```

## 2. TBR 与 TBDR 核心原理

### 2.1 TBR (Tile-Based Rendering)

代表：**ARM Mali**, **Qualcomm Adreno** (部分)

```
TBR 完整流程：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ┌──────────────┐                                           │
│  │ Vertex       │  处理所有 Draw Call 的顶点                 │
│  │ Processing   │  输出裁剪空间坐标                          │
│  └──────┬───────┘                                           │
│         ▼                                                   │
│  ┌──────────────┐                                           │
│  │ Tiling /     │  确定每个三角形覆盖哪些 Tile                │
│  │ Binning      │  生成 Per-Tile Polygon List               │
│  │              │  存储到 System Memory (Tile List Buffer)  │
│  └──────┬───────┘                                           │
│         ▼                                                   │
│  ┌──────────────┐  ← 对每个 Tile 重复                        │
│  │ Tile         │                                           │
│  │ Rendering    │  1. 从内存加载 Tile 的三角形列表            │
│  │              │  2. 光栅化 + Early-Z + Fragment Shader    │
│  │              │  3. 所有操作在 On-Chip Tile Memory 中      │
│  │              │  4. 写回最终颜色到 Framebuffer (DRAM)      │
│  └──────────────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘

TBR 的 Overdraw 问题：
  - 三角形按提交顺序处理
  - 被遮挡的片段仍然执行 Fragment Shader
  - Early-Z 可以剔除部分，但不完美
  - 依赖应用层的 front-to-back 排序
```

### 2.2 TBDR (Tile-Based Deferred Rendering)

代表：**Apple GPU (A-Series, M-Series)**, **Imagination PowerVR**

```
TBDR 完整流程：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ┌──────────────┐                                           │
│  │ Vertex       │  同 TBR                                   │
│  │ Processing   │                                           │
│  └──────┬───────┘                                           │
│         ▼                                                   │
│  ┌──────────────┐                                           │
│  │ Tiling /     │  同 TBR                                   │
│  │ Binning      │                                           │
│  └──────┬───────┘                                           │
│         ▼                                                   │
│  ┌──────────────┐  ← 对每个 Tile                             │
│  │ Hidden       │                                           │
│  │ Surface      │  ★ TBDR 的核心创新 ★                      │
│  │ Removal      │  在执行 Fragment Shader 之前：              │
│  │ (HSR)        │  1. 对 Tile 内所有三角形做光栅化             │
│  │              │  2. 逐像素确定最终可见的三角形               │
│  │              │  3. 只有可见片段才执行 Fragment Shader      │
│  │              │  → 零 Overdraw (理想情况)                  │
│  └──────┬───────┘                                           │
│         ▼                                                   │
│  ┌──────────────┐                                           │
│  │ Fragment     │  只处理可见片段                            │
│  │ Shading      │  On-Chip Tile Memory                      │
│  └──────┬───────┘                                           │
│         ▼                                                   │
│  ┌──────────────┐                                           │
│  │ Tile         │  写回最终颜色到 DRAM                       │
│  │ Writeback    │                                           │
│  └──────────────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘

TBDR 的 HSR 优势：
  - 不依赖提交顺序，硬件自动确定可见性
  - 理论上零 Overdraw
  - Fragment Shader 只执行一次/像素
  - 但 Alpha Test / Alpha Blend 会破坏 HSR
```

### 2.3 TBR vs TBDR 关键差异

| 特性                     | TBR (Mali/Adreno)       | TBDR (Apple/PowerVR) |
| ------------------------ | ----------------------- | -------------------- |
| **Overdraw 处理**        | 依赖 Early-Z + 应用排序 | 硬件 HSR，自动消除   |
| **排序依赖**             | 需要 front-to-back      | 不需要（但仍推荐）   |
| **Alpha Test 影响**      | 破坏 Early-Z            | 破坏 HSR             |
| **Fragment Shader 执行** | 可能多次/像素           | 理想情况一次/像素    |
| **复杂度**               | 较简单                  | 硬件更复杂           |
| **功耗效率**             | 好                      | 更好                 |

### 2.4 Tile Memory 详解

```
On-Chip Tile Memory 布局 (典型):

┌─────────────────────────────────────────┐
│           On-Chip Tile Memory            │
│           (SRAM, ~几十 KB)               │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────────┐  ┌─────────────┐       │
│  │ Color Buffer│  │ Depth Buffer│       │
│  │ (Tile-sized)│  │ (Tile-sized)│       │
│  │ RGBA8/RGB10 │  │ D24S8/D32F  │       │
│  └─────────────┘  └─────────────┘       │
│                                         │
│  ┌─────────────┐  ┌─────────────┐       │
│  │ Stencil Buf │  │ MSAA Samples│       │
│  │ (Tile-sized)│  │ (if enabled)│       │
│  └─────────────┘  └─────────────┘       │
│                                         │
│  Tile Size: 16x16 or 32x32 pixels      │
│  Total: ~4-64 KB per tile               │
│  Access latency: ~1-3 cycles            │
│  vs DRAM: ~100-300 cycles               │
│                                         │
└─────────────────────────────────────────┘

Tile Size 与 MSAA 的关系：
┌──────────┬──────────┬──────────┬──────────┐
│ MSAA     │ Tile Size│ Memory/  │ 说明      │
│ Level    │ (pixels) │ Tile     │           │
├──────────┼──────────┼──────────┼──────────┤
│ No MSAA  │ 32x32    │ ~16 KB   │ 最大 Tile │
│ 2x MSAA  │ 32x16    │ ~16 KB   │ Tile 缩小 │
│ 4x MSAA  │ 16x16    │ ~16 KB   │ 常见配置  │
│ 8x MSAA  │ 16x8     │ ~16 KB   │ 很少使用  │
└──────────┴──────────┴──────────┴──────────┘

注意：MSAA 级别越高，Tile 越小，Tile 数量越多
      但 MSAA resolve 在 On-Chip 完成，几乎免费
```

### 3.1 ARM Mali

```
Mali GPU 架构演进：

  Utgard (Mali-400)     → 固定管线，OpenGL ES 2.0
      ↓
  Midgard (Mali-T600+)  → 统一着色器，OpenGL ES 3.0/3.1
```

#### Mali 架构特点

```
Mali Valhall 架构 (G710/G715/G720):

┌─────────────────────────────────────────────────┐
│                 Mali GPU                        │
├─────────────────────────────────────────────────┤
│                                                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐         │
│  │ Shader   │ │ Shader   │ │ Shader   │ ...     │
│  │ Core 0   │ │ Core 1   │ │ Core 2   │         │
│  │          │ │          │ │          │         │
│  │ ┌──────┐ │ │ ┌──────┐ │ │ ┌──────┐ │         │
│  │ │ FMA  │ │ │ │ FMA  │ │ │ │ FMA  │ │         │
│  │ │ Unit │ │ │ │ Unit │ │ │ │ Unit │ │         │
│  │ ├──────┤ │ │ ├──────┤ │ │ ├──────┤ │         │
│  │ │ CVT  │ │ │ │ CVT  │ │ │ │ CVT  │ │         │
│  │ │ Unit │ │ │ │ Unit │ │ │ │ Unit │ │         │
│  │ ├──────┤ │ │ ├──────┤ │ │ ├──────┤ │         │
│  │ │ SFU  │ │ │ │ SFU  │ │ │ │ SFU  │ │         │
│  │ │      │ │ │ │      │ │ │ │      │ │         │
│  │ ├──────┤ │ │ ├──────┤ │ │ ├──────┤ │         │
│  │ │ Load │ │ │ │ Load │ │ │ │ Load │ │         │
│  │ │ Store│ │ │ │ Store│ │ │ │ Store│ │         │
│  │ └──────┘ │ │ └──────┘ │ │ └──────┘ │         │
│  └──────────┘ └──────────┘ └──────────┘         │
│                                                 │
│  ┌─────────────────────────────────────┐        │
│  │         Tiler / Binning Unit        │        │
│  └─────────────────────────────────────┘        │
│  ┌─────────────────────────────────────┐        │
│  │         L2 Cache (256KB-2MB)        │        │
│  └─────────────────────────────────────┘        │
│                                                 │
└─────────────────────────────────────────────────┘

Mali 关键特性：
- TBR 架构（非 TBDR，无硬件 HSR）
- 依赖 Early-Z 和 FPK (Forward Pixel Kill) 减少 Overdraw
- FPK: 在 Fragment Shader 执行前检查深度，杀死被遮挡的线程
- Warp Size: 16 threads (Valhall)
- Tile Size: 16x16 pixels
- 支持 AFBC (ARM Frame Buffer Compression)
- Transaction Elimination: 跳过未变化的 Tile 写回
```

#### Mali 特有优化技术

```
1. AFBC (ARM Frame Buffer Compression)
   - 无损压缩 Framebuffer 数据
   - 减少 50-60% 带宽
   - 对应用透明，硬件自动处理
   - 支持 Color / Depth / Stencil

2. Transaction Elimination
   - 比较当前 Tile 与上一帧同位置 Tile
   - 如果内容相同，跳过写回
   - 静态场景可节省大量带宽
   - 需要 CRC 校验

3. FPK (Forward Pixel Kill)
   - 在 Fragment Shader 执行期间
   - 如果后续三角形的深度测试确定当前片段被遮挡
   - 终止当前片段的 Shader 执行
   - 部分缓解 Overdraw 问题

4. IDVS (Index-Driven Vertex Shading)
   - 先处理 Position，再处理 Varying
   - 被裁剪的顶点不计算 Varying
   - 节省 Vertex Shader 开销
```

```
Adreno GPU 架构演进：

  Adreno 300 系列  → OpenGL ES 3.0
```

#### Adreno 架构特点

```
Adreno 架构独特之处：

┌────────────────────────────────────────────────────┐
│              Adreno GPU                            │
├────────────────────────────────────────────────────┤
│                                                    │
│  ★ Flex Render (灵活渲染模式) ★                    │
│                                                    │
│  ┌─────────────┐    ┌─────────────┐                │
│  │ Direct Mode │    │ Binning Mode│                │
│  │ (IMR-like)  │    │ (TBR)       │                │
│  └──────┬──────┘    └──────┬──────┘                │
│         │                  │                       │
│         ▼                  ▼                       │
│  ┌─────────────────────────────────┐               │
│  │    Driver 自动选择最优模式       │               │
│  │    基于场景复杂度和带宽压力       │               │
│  └─────────────────────────────────┘               │
│                                                    │
│  Shader Processors (SP):                           │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐                   │
│  │ SP0 │ │ SP1 │ │ SP2 │ │ SP3 │ ...               │
│  │     │ │     │ │     │ │     │                   │
│  │ ALU │ │ ALU │ │ ALU │ │ ALU │                   │
│  │ EFU │ │ EFU │ │ EFU │ │ EFU │ (Elementary Func) │
│  │ TEX │ │ TEX │ │ TEX │ │ TEX │ (Texture Unit)    │
│  └─────┘ └─────┘ └─────┘ └─────┘                   │
│                                                    │
│  ┌─────────────────────────────────┐               │
│  │    UCHE (Unified Cache)         │               │
│  │    L2 Cache (512KB-4MB)         │               │
│  └─────────────────────────────────┘               │
│                                                    │
└────────────────────────────────────────────────────┘

Adreno 关键特性：
- Flex Render: 可在 TBR 和 Direct (IMR) 模式间切换
- 驱动自动判断哪种模式更优
- Warp Size: 64 threads (Fiber)
- 支持 UBWC (Universal Bandwidth Compression)
- Low Resolution Z (LRZ): 硬件级粗粒度深度剔除
- Visibility Stream: 优化 Binning Pass
```

#### Adreno 特有优化技术

```
1. UBWC (Universal Bandwidth Compression)
   - 类似 Mali 的 AFBC
   - 压缩所有 Buffer (Color, Depth, Texture)
   - 减少 40-60% 带宽
   - 需要特定内存布局

2. LRZ (Low Resolution Z)
   - 低分辨率深度缓冲 (1/4 或 1/16 分辨率)
   - 在 Binning Pass 中生成
   - Fragment 阶段用于快速剔除
   - 类似硬件级 Hi-Z
   - ⚠️ Alpha Test / Shader Discard 会禁用 LRZ

3. Flex Render
   - Binning Mode: 标准 TBR，适合复杂场景
   - Direct Mode: 类 IMR，适合简单场景或 Compute 密集
   - Sysmem Mode: 完全绕过 Tiling，用于特殊情况
   - 驱动自动选择，也可通过 API hint 控制

4. Visibility Stream
   - Binning Pass 中生成可见性信息
   - 完全不可见的 Draw Call 在 Rendering Pass 中跳过
   - 减少不必要的 Fragment 处理
```

#### Apple GPU 架构特点

```
Apple GPU (A15+ / M1+):

┌─────────────────────────────────────────────────┐
│              Apple GPU (TBDR)                   │
├─────────────────────────────────────────────────┤
│                                                 │
│  ★ TBDR + HSR (Hidden Surface Removal) ★       │
│                                                 │
│  ┌──────────────────────────────────┐           │
│  │ Vertex Processing                │           │
│  │ → Tiling → Parameter Buffer      │           │
│  └──────────────┬───────────────────┘           │
│                 ▼                               │
│  ┌──────────────────────────────────┐           │
│  │ Per-Tile Processing:             │           │
│  │                                  │           │
│  │  1. Load Tile's triangles        │           │
│  │  2. Rasterize ALL triangles      │           │
│  │  3. HSR: Determine visibility    │ ← 核心    │
│  │     (per-pixel, per-sample)      │           │
│  │  4. Fragment Shade ONLY visible  │           │
│  │  5. Tile Memory → DRAM           │           │
│  └──────────────────────────────────┘           │
│                                                 │
│  GPU Cores:                                     │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐             │
│  │ C0 │ │ C1 │ │ C2 │ │ C3 │ │ C4 │ ...         │
│  └────┘ └────┘ └────┘ └────┘ └────┘             │
│                                                 │
│  ┌─────────────────────────────────┐            │
│  │ Unified Memory Architecture     │            │
│  │ (CPU + GPU 共享内存，零拷贝)     │            │
│  └─────────────────────────────────┘            │
│                                                 │
└─────────────────────────────────────────────────┘

Apple GPU 关键特性：
- TBDR 架构，硬件 HSR
- 理论零 Overdraw（不透明物体）
- Tile Size: 32x32 pixels
- SIMD Width: 32 threads
- Unified Memory: CPU/GPU 共享，零拷贝
- Lossless Compression: 自动压缩
- Tile Shading: 可在 Tile 粒度执行自定义计算
- Imageblock: 直接访问 Tile Memory 的 Metal 特性
```

#### Apple GPU 特有优化技术

```
1. HSR (Hidden Surface Removal)
   - 硬件自动完成，无需应用干预
   - 在 Fragment Shader 执行前确定可见性
   - 不透明物体：完美零 Overdraw
   - ⚠️ 以下情况破坏 HSR:
     - Alpha Test (discard/clip)
     - Alpha Blend (半透明)
     - Shader 中写入深度
     - 使用 programmable blending 读取 framebuffer

2. Imageblock
   - Metal 特有 API
   - 直接在 Shader 中访问 Tile Memory
   - 可实现 On-Chip Deferred Shading
   - 无需将 G-Buffer 写入 DRAM

3. Tile Shading (Metal)
   - 在 Tile 粒度执行 Compute
   - 利用 Tile Memory 做中间计算
   - 适合 Tile-local 后处理

4. Memoryless Render Targets
   - 标记 RT 为 memoryless
   - 只存在于 Tile Memory，不分配 DRAM
   - 适合临时 RT (Depth, MSAA resolve)
   - 节省大量内存和带宽
```

### 4.1 推荐渲染管线架构

```

├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                   Render Pass 0                         │    │
│  │                   Shadow Map Pass                       │    │

│  │  │ Cascade │  低分辨率, 1-2 级 Cascade                   │    │

│  │  │ Output: Color (On-Chip)                     │        │    │

│  │  └─────────────────────────────────────────────┘        │    │
│  │                                                         │    │

关键原则：
✅ 尽量在单个 Render Pass 内完成所有操作
✅ 使用 Subpass 在 Tile Memory 中传递数据
✅ Depth/Stencil 标记为 Don't Care (不写回 DRAM)
✅ 只有最终 Color 写回 DRAM
❌ 避免多个 Render Pass 之间的依赖
❌ 避免中途读取 Framebuffer
```

### 4.2 Render Pass 设计原则

```
=== 移动端 Render Pass 黄金法则 ===

规则 1: 最小化 Render Pass 数量
  ❌ 错误: Shadow Pass → Depth Pass → GBuffer Pass → Light Pass → Post Pass
  ✅ 正确: Shadow Pass → Main Pass (Subpass 0 + 1 + 2)

规则 2: 正确设置 Load/Store Actions
  ┌──────────────┬──────────────┬──────────────────────────┐
  │ Attachment   │ Load Action  │ Store Action             │
  ├──────────────┼──────────────┼──────────────────────────┤
  │ Color        │ Clear        │ Store (最终输出)          │
  │ Depth        │ Clear        │ Don't Care (不需要保留!)  │
  │ Stencil      │ Clear/DC     │ Don't Care               │
  │ MSAA Color   │ Don't Care   │ Don't Care (resolve 后)  │
  │ MSAA Depth   │ Clear        │ Don't Care               │
  └──────────────┴──────────────┴──────────────────────────┘

  ⚠️ Store Action = Store 意味着 Tile Memory → DRAM 写回
  ⚠️ 每个不必要的 Store 都浪费大量带宽和功耗

规则 3: 使用 Subpass 替代多 Render Pass
  Vulkan: VkSubpassDescription
  Metal:  MTLRenderPassDescriptor with tile shading
  OpenGL ES: GL_EXT_shader_framebuffer_fetch

规则 4: 避免 Render Pass 中途 Flush
  以下操作会强制 Tile Flush:
  - 切换 Render Target
  - 读取当前 Render Target 作为纹理
  - 某些 Compute Shader dispatch
  - glReadPixels / glCopyTexImage
```

```cpp
// Subpass 0: G-Buffer
VkSubpassDescription subpass0 = {};
subpass0.pipelineBindPoint = VK_PIPELINE_BIND_POINT_GRAPHICS;
subpass0.colorAttachmentCount = 3; // Albedo, Normal, PBR
subpass0.pColorAttachments = gbufferAttachmentRefs;
subpass0.pDepthStencilAttachment = &depthAttachmentRef;

// Subpass 1: Lighting
VkSubpassDescription subpass1 = {};
subpass1.pipelineBindPoint = VK_PIPELINE_BIND_POINT_GRAPHICS;
subpass1.colorAttachmentCount = 1; // Final color
subpass1.pColorAttachments = &finalColorRef;
subpass1.inputAttachmentCount = 4; // Read G-Buffer + Depth
subpass1.pInputAttachments = inputAttachmentRefs; // ← On-Chip read!

// Subpass dependency: ensure G-Buffer is written before lighting reads
VkSubpassDependency dependency = {};
dependency.srcSubpass = 0;
dependency.dstSubpass = 1;

dependency.dstStageMask = VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT;
```

#### Metal Imageblock 实现

```metal
// Metal: On-Chip Deferred with Imageblock

// Define Imageblock structure (lives in Tile Memory)
struct GBufferData {
    half4 albedo   [[color(0)]];
    half4 normal   [[color(1)]];
    half4 pbr      [[color(2)]];
    float depth    [[color(3)]];
};

// Subpass 0: Write G-Buffer to Imageblock
fragment GBufferData gbufferFragment(VertexOut in [[stage_in]],
                                      texture2d<half> albedoTex [[texture(0)]])
{
    GBufferData out;
    out.albedo = albedoTex.sample(sampler, in.uv);
    out.normal = half4(encodeNormal(in.normalWS), 0, 1);
    out.pbr    = half4(in.metallic, in.roughness, in.ao, 1);
    out.depth  = in.positionCS.z;
    return out;
}

// Subpass 1: Read G-Buffer from Imageblock (On-Chip!)
fragment half4 lightingFragment(VertexOut in [[stage_in]],
                                 GBufferData gbuffer [[imageblock_data]])
{
    // gbuffer read is FREE - directly from Tile Memory
    half3 albedo = gbuffer.albedo.rgb;
    half3 normal = decodeNormal(gbuffer.normal.rg);
    float depth  = gbuffer.depth;

    // Calculate lighting...
    half3 color = calculateLighting(albedo, normal, depth);
    return half4(color, 1.0);
}

// Render Pass Descriptor
let rpd = MTLRenderPassDescriptor()
rpd.colorAttachments[0].storeAction = .store       // Final color → DRAM
rpd.colorAttachments[1].storeAction = .dontCare    // G-Buffer: Tile only
rpd.colorAttachments[2].storeAction = .dontCare    // G-Buffer: Tile only
rpd.depthAttachment.storeAction     = .dontCare    // Depth: Tile only

// Use memoryless textures for G-Buffer
let gbufferDesc = MTLTextureDescriptor()
gbufferDesc.storageMode = .memoryless  // ← No DRAM allocation!
```

## 5. 带宽与内存优化

### 5.1 带宽是移动端的第一瓶颈

```
移动端带宽预算分析 (典型 SoC):

  总系统带宽: ~50 GB/s (LPDDR5, 共享 CPU+GPU+ISP+...)
  GPU 可用带宽: ~25-35 GB/s (约 50-70%)
  目标帧率: 60 FPS
  每帧带宽预算: ~420-580 MB/frame

  1080p 单帧 Framebuffer:
  - Color (RGBA8):  1920 × 1080 × 4 = 8.3 MB
  - Depth (D32F):   1920 × 1080 × 4 = 8.3 MB
  - 读 + 写:        ~33 MB (Color + Depth 各读写一次)

  如果使用传统 Deferred:
  - G-Buffer 写入:  ~50 MB
  - G-Buffer 读取:  ~50 MB
  - Lighting 写入:  ~8 MB
  - 总计:           ~108 MB (仅 G-Buffer 相关)
  - 加上纹理采样、Shadow Map 等: ~200-300 MB
  - 占预算的 50-70%!

  如果使用 TBR + On-Chip:
  - 最终 Color 写入: ~8 MB (压缩后 ~4 MB)
  - 纹理采样:        ~50-100 MB
  - Shadow Map:      ~10-20 MB
  - 总计:            ~70-130 MB
  - 占预算的 15-30%  ← 巨大改善!
```

### 5.3 Load/Store Action 详解

```
Load/Store Action 对带宽的影响：

┌──────────────┬──────────────────────────────────────────────┐
│ Action       │ 说明                                         │
├──────────────┼──────────────────────────────────────────────┤
│ Load: Clear  │ Tile Memory 清零，不从 DRAM 加载              │
│              │ 带宽: 0 (最优)                                │
├──────────────┼──────────────────────────────────────────────┤
│ Load: Load   │ 从 DRAM 加载上一帧/Pass 的内容到 Tile Memory  │
│              │ 带宽: Width × Height × BPP (昂贵!)            │
├──────────────┼──────────────────────────────────────────────┤
│ Load: DC     │ Don't Care, Tile Memory 内容未定义            │
│              │ 带宽: 0 (最优，但内容不可预测)                  │
├──────────────┼──────────────────────────────────────────────┤
│ Store: Store │ 将 Tile Memory 写回 DRAM                     │
│              │ 带宽: Width × Height × BPP (昂贵!)            │
├──────────────┼──────────────────────────────────────────────┤
│ Store: DC    │ Don't Care, 不写回 DRAM                      │
│              │ 带宽: 0 (最优)                                │
├──────────────┼──────────────────────────────────────────────┤
│ Store:       │ MSAA → 单采样 Resolve，写回 DRAM              │
│ Resolve      │ 带宽: 只写 Resolve 后的大小                    │
└──────────────┴──────────────────────────────────────────────┘

最优配置示例：
  Color:   Load=Clear,  Store=Store    (必须保留最终颜色)
  Depth:   Load=Clear,  Store=DontCare (渲染后不需要深度)
  Stencil: Load=Clear,  Store=DontCare
  MSAA:    Load=DC,     Store=Resolve  (Resolve 到单采样)

常见错误：
  ❌ Depth Store=Store (不需要保留深度时浪费带宽)
  ❌ Color Load=Load (新帧应该 Clear，不需要加载上一帧)
  ❌ 忘记设置 Don't Care (默认可能是 Store)
```

### 5.4 MSAA 在移动端的特殊优势

```
移动端 MSAA 几乎免费的原因：

桌面端 (IMR) MSAA:
  - MSAA 样本存储在 DRAM
  - 每个像素 4x 样本 = 4x 带宽
  - Resolve 需要额外 Pass
  - 开销显著

移动端 (TBR) MSAA:
  - MSAA 样本存储在 Tile Memory (On-Chip)
  - 不增加 DRAM 带宽
  - Resolve 在 Tile 写回时自动完成
  - 只写回 Resolve 后的单采样结果
  - 开销: 仅 Tile Memory 使用增加 → Tile 变小

性能影响：
┌──────────┬──────────────┬──────────────┐
│ MSAA     │ 桌面端开销    │ 移动端开销    │
├──────────┼──────────────┼──────────────┤
│ No MSAA  │ 基准          │ 基准          │
│ 2x MSAA  │ +30-50%      │ +5-10%       │
│ 4x MSAA  │ +60-100%     │ +10-20%      │
│ 8x MSAA  │ +100-200%    │ +30-50%      │
└──────────┴──────────────┴──────────────┘

推荐：移动端优先使用 4x MSAA 而非后处理 AA
  - 4x MSAA 质量优于 FXAA
  - 带宽开销远低于 FXAA (FXAA 需要额外 Pass)
  - 不会模糊画面
```

```
│ PBR 参数              │ half     │ 0-1 范围足够          │
```

### 6.2 分支与流控制

```hlsl
// ❌ 避免：动态分支 (divergent branch)
if (materialType == 1)
    color = calculatePBR(...);
else if (materialType == 2)
    color = calculateToon(...);
else
    color = calculateUnlit(...);
// 问题：Warp 内线程走不同分支，所有分支都执行

// ✅ 推荐：使用 lerp/step 替代简单分支
half mask = step(0.5h, materialMask);
color = lerp(colorA, colorB, mask);

// ✅ 推荐：Uniform 分支 (所有线程走同一分支)
if (_EnableFog) // uniform variable
    color = applyFog(color);
// 这是安全的，因为所有线程走同一路径

// ❌ 避免：Shader 中的循环次数不确定
for (int i = 0; i < dynamicCount; i++) // dynamicCount varies per pixel
    color += sampleLight(i);

// ✅ 推荐：固定上限循环
for (int i = 0; i < MAX_LIGHTS; i++)
{
    if (i >= lightCount) break; // uniform break is OK
    color += sampleLight(i);
}
```

### 6.3 纹理采样优化

```hlsl
// 移动端纹理采样最佳实践

// ✅ 使用合适的采样器
// Bilinear: 4 texels, 1 cycle
// Trilinear: 8 texels, 2 cycles (2x cost!)
// Anisotropic 2x: 8 texels, 2 cycles
// Anisotropic 4x: 16 texels, 4 cycles

// ✅ 推荐：大多数情况使用 Bilinear + Mipmap
sampler2D _MainTex; // Bilinear + Mipmap
half4 color = tex2D(_MainTex, uv); // 1 cycle

// ⚠️ 谨慎使用 Trilinear
// 只在 Mipmap 过渡明显时使用
// 地面、远处物体可以用 Trilinear
// 近处物体 Bilinear 足够

// ❌ 避免：dependent texture read
half2 distortedUV = tex2D(_DistortionTex, uv).rg; // 第一次采样
half4 color = tex2D(_MainTex, distortedUV);        // 依赖第一次结果
// 问题：第二次采样必须等第一次完成，无法预取

// ✅ 推荐：尽量使用 non-dependent reads
// UV 只依赖 vertex shader 输出，不依赖其他纹理
half4 color = tex2D(_MainTex, input.uv); // 可以预取

// ✅ 使用纹理 Atlas 减少切换
// 将多张小纹理合并到一张大纹理
// 减少 texture binding 切换
// 提高 cache 命中率
```

### 6.4 ALU 优化

```hlsl
// 移动端 ALU 优化技巧

// ✅ 使用内置函数 (硬件加速)
half result = saturate(x);        // 免费 (clamp unit)
half result = abs(x);             // 免费 (modifier)
half result = -x;                 // 免费 (modifier)
half result = max(a, b);          // 1 cycle
half result = min(a, b);          // 1 cycle
half result = clamp(x, a, b);     // 1 cycle
half3 result = normalize(v);      // 硬件 SFU
half result = rsqrt(x);           // 硬件 SFU
half result = exp2(x);            // 硬件 SFU
half result = log2(x);            // 硬件 SFU

// ❌ 避免：昂贵操作
float result = pow(x, y);         // 展开为 exp2(y * log2(x))
float result = sin(x);            // SFU, 多 cycle
float result = cos(x);            // SFU, 多 cycle
float result = tan(x);            // 非常昂贵
float result = asin(x);           // 非常昂贵
float result = atan2(y, x);       // 非常昂贵

// ✅ 近似替代
// pow(x, 2) → x * x
// pow(x, 4) → x2 = x * x; x2 * x2
// pow(x, 5) → x2 = x * x; x2 * x2 * x
half specular = x * x * x * x; // 比 pow(x, 4) 快

// ✅ MAD (Multiply-Add) 优化
// 大多数 GPU 的 MAD 是单 cycle
half result = a * b + c; // 1 MAD = 1 cycle
// 而不是:
half temp = a * b;       // 1 cycle
half result = temp + c;  // 1 cycle (2 cycles total)
```

### 9.2 Bloom 移动端优化实现

```hlsl
// Mobile-optimized Bloom: Dual Kawase Blur
// 比 Gaussian Blur 更高效，更少采样次数

// Downsample (Kawase)
half4 KawaseDownsample(sampler2D tex, float2 uv, float2 texelSize)
{
    half4 sum = tex2D(tex, uv) * 4.0h;
    sum += tex2D(tex, uv + texelSize * float2(-1, -1));
    sum += tex2D(tex, uv + texelSize * float2( 1, -1));
    sum += tex2D(tex, uv + texelSize * float2(-1,  1));
    sum += tex2D(tex, uv + texelSize * float2( 1,  1));
    return sum * (1.0h / 8.0h);
}

// Upsample (Kawase)
half4 KawaseUpsample(sampler2D tex, float2 uv, float2 texelSize)
{
    half4 sum = 0;
    sum += tex2D(tex, uv + texelSize * float2(-1,  0)) * 2.0h;
    sum += tex2D(tex, uv + texelSize * float2( 1,  0)) * 2.0h;
    sum += tex2D(tex, uv + texelSize * float2( 0, -1)) * 2.0h;
    sum += tex2D(tex, uv + texelSize * float2( 0,  1)) * 2.0h;
    sum += tex2D(tex, uv + texelSize * float2(-1, -1));
    sum += tex2D(tex, uv + texelSize * float2( 1, -1));
    sum += tex2D(tex, uv + texelSize * float2(-1,  1));
    sum += tex2D(tex, uv + texelSize * float2( 1,  1));
    return sum * (1.0h / 12.0h);
}
```

### 11.3 常见性能问题诊断

```
问题诊断流程：

  帧率低？
  ├── GPU Bound?
  │   ├── Fragment Bound?
  │   │   ├── Overdraw 高? → front-to-back 排序, 减少透明
  │   │   ├── Shader 复杂? → 简化 Shader, 使用 half
  │   │   ├── 纹理采样多? → 减少采样, 使用压缩纹理
  │   │   └── 分辨率高?   → 降低渲染分辨率
  │   │
  │   ├── Vertex Bound?
  │   │   ├── 顶点数多?   → LOD, 减面
  │   │   └── VS 复杂?    → 简化顶点着色器
  │   │
  │   ├── Bandwidth Bound?
  │   │   ├── Store Actions 错误? → 检查 Don't Care
  │   │   ├── Render Pass 过多?   → 合并 Pass
  │   │   ├── 纹理未压缩?         → 使用 ASTC
  │   │   └── RT 格式过大?        → 降低精度
  │   │
  │   └── Tiling Bound?
  │       ├── 三角形过多?   → 减少小三角形
  │       └── 顶点数据大?   → 压缩顶点属性
  │
  └── CPU Bound?
      ├── Draw Call 多?    → Batching, Instancing
      ├── 排序开销?        → 减少透明物体
      └── 逻辑计算?        → 优化游戏逻辑
```

## 13. 最佳实践检查清单

### 13.1 渲染架构检查清单

```
□ 架构选择
  □ 使用 Forward 或 Forward+ 渲染路径
  □ 如需 Deferred，使用 On-Chip Deferred (Subpass)
  □ 避免传统 Deferred (G-Buffer 写回 DRAM)

□ Render Pass 优化
  □ 最小化 Render Pass 数量
  □ 使用 Subpass 替代多 Pass
  □ 所有不需要的 Attachment 设置 Store = Don't Care
  □ Depth/Stencil 默认 Store = Don't Care
  □ 使用 Memoryless/Transient 纹理
  □ 避免中途读取 Framebuffer

□ MSAA
  □ 优先使用 4x MSAA (移动端几乎免费)
  □ MSAA Resolve 在 Tile 内完成
  □ 不需要额外的后处理 AA

□ 排序
  □ 不透明物体 front-to-back 排序
  □ 透明物体 back-to-front 排序
  □ 按材质/Shader 分组减少状态切换
```

### 13.2 Shader 检查清单

```
□ 精度
  □ Fragment Shader 默认使用 half (mediump)
  □ 只在必要时使用 float (highp)
  □ 位置/深度/矩阵使用 float
  □ 颜色/法线/光照使用 half

□ 复杂度
  □ Fragment Shader < 100 ALU cycles
  □ 避免动态分支 (divergent branch)
  □ 避免 dependent texture reads
  □ 避免昂贵数学函数 (pow, sin, cos)
  □ 使用 LUT 替代复杂计算

□ Alpha Test
  □ 最小化 Alpha Test / discard 使用
  □ Alpha Test 物体单独排序
  □ 考虑 Alpha to Coverage 替代
  □ 了解 Alpha Test 对 HSR/LRZ/Early-Z 的影响
```

### 13.3 资源检查清单

```
□ 纹理
  □ 使用 ASTC 压缩 (4x4 / 6x6 / 8x8)
  □ 启用 Mipmap (3D 场景纹理)
  □ 使用纹理 Atlas 减少切换
  □ 合理的纹理分辨率 (不要过大)
  □ Streaming Mipmap 按需加载

□ Render Target
  □ 使用最小够用的格式
  □ 避免 RGBA16F (除非必要)
  □ 后处理使用降分辨率 RT
  □ 临时 RT 使用 Memoryless

□ 网格
  □ 合理的 LOD 系统
  □ 避免微小三角形 (< 2 pixels)
  □ 压缩顶点属性 (half, short)
  □ 使用 Index Buffer
```
