## 1. 三条管线总览

```
═══ Unreal 三条主要渲染管线 ═══

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐   │
  │  │  Forward HDR     │  │  Forward LDR     │  │  Deferred        │   │
  │  │  (Mobile HDR)    │  │  (Simple Forward)│  │  (Desktop/PC)    │   │
  │  │                  │  │                  │  │                  │   │
  │  │ • 移动端高配      │  │ • 移动端低配      │  │ • PC/主机        │   │
  │  │ • 支持后处理      │  │ • 无后处理        │  │ • 完整后处理     │   │
  │  │ • HDR 渲染        │  │ • LDR 直出       │  │ • 多光源高效     │   │
  │  │ • MSAA 可选       │  │ • MSAA 可选      │  │ • 不支持 MSAA    │   │
  │  │ • 带宽中等        │  │ • 带宽最低        │  │ • 带宽最高       │   │
  │  └──────────────────┘  └──────────────────┘  └──────────────────┘   │
  │                                                                     │
  │  选择依据：                                                          │
  │  • 平台性能 → 低配移动端用 LDR，高配移动端用 HDR                       │
  │  • 光源数量 → 多光源场景用 Deferred                                   │
  │  • 后处理需求 → 需要 Bloom/DOF 等用 HDR 或 Deferred                   │
  │  • 带宽预算 → TBR 架构优先 Forward                                    │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

═══ 核心入口函数 ═══

  FDeferredShadingSceneRenderer::Render()        // Deferred 入口
  FMobileSceneRenderer::Render()                 // Forward HDR/LDR 入口
    ├── bGammaSpace == true  → LDR 路径
    └── bGammaSpace == false → HDR 路径
```

## 2. Forward HDR 管线

### 2.1 完整流程

```
═══ Forward HDR 完整 Pass 顺序 ═══

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  ① InitView                                                 │
  │  │  • 可见性判定（Frustum Culling / Occlusion Culling）      │
  │  │  • 排序（前后排序 / 材质排序）                             │
  │  │  • 收集动态光源信息                                       │
  │  │  • 构建 GPU Scene（UE5）                                  │
  │  ▼                                                          │
  │  ② ShadowDepthMaps                                          │
  │  │  • 渲染所有光源的 Shadow Map                              │
  │  │  • CSM（Cascaded Shadow Map）for 方向光                   │
  │  │  • Spot/Point Light Shadow Map                           │
  │  │  • Modulated Shadow（移动端优化）                         │
  │  ▼                                                          │
  │  ③ RenderForward                                            │
  │  │  ┌─────────────────────────────────────────────────┐     │
  │  │  │  3.1  PrePass（Depth Only / Early-Z）            │    │
  │  │  │       • 只写深度，不写颜色                        │    │
  │  │  │       • 用于后续深度测试优化                      │    │
  │  │  │       • Masked 物体在此处理 Alpha Test            │    │
  │  │  │                                                  │    │
  │  │  │  3.2  Opaque And Mask                            │    │
  │  │  │       • 不透明物体 + Masked 物体                  │    │
  │  │  │       • Forward Shading: 光照在此计算             │    │
  │  │  │       • 每个物体最多受 N 个光源影响                │    │
  │  │  │       • 输出 HDR 颜色到 SceneColor RT             │    │
  │  │  │                                                  │    │
  │  │  │  3.3  Xray                                       │    │
  │  │  │       • X 光/透视效果渲染                         │    │
  │  │  │       • 自定义 Pass（项目特定）                   │    │
  │  │  │                                                  │    │
  │  │  │  3.4  Translucency PreDepth                      │    │
  │  │  │       • 半透明物体的预深度                        │    │
  │  │  │       • 用于半透明排序和深度测试                   │    │
  │  │  │                                                  │    │
  │  │  │  3.5  Copy Depth / Color（条件触发）             │    │
  │  │  │       • 某些效果需要当前帧的深度/颜色副本          │    │
  │  │  │       • 如: 折射、软粒子、深度淡出                │    │
  │  │  │       • 在 TBR 上尽量避免（触发 Tile Flush）      │    │
  │  │  │                                                  │    │
  │  │  │  3.6  Decal                                      │    │
  │  │  │       • 贴花渲染                                  │    │
  │  │  │       • Forward 中 Decal 需要特殊处理             │    │
  │  │  │       • 通常使用 DBuffer Decal 或 Mesh Decal      │    │
  │  │  │                                                  │    │
  │  │  │  3.7  Translucency                               │    │
  │  │  │       • 半透明物体渲染                            │    │
  │  │  │       • 从后往前排序（Back-to-Front）             │    │
  │  │  │       • 支持 Separate Translucency               │    │
  │  │  │                                                  │    │
  │  │  │  3.8  Highlight                                  │    │
  │  │  │       • 选中高亮效果                              │    │
  │  │  │       • 编辑器/自定义高亮                         │    │
  │  │  │                                                  │    │
  │  │  │  3.9  OutLine                                    │    │
  │  │  │       • 描边效果                                  │   │
  │  │  │       • 自定义 Pass（项目特定）                   │    │
  │  │  └─────────────────────────────────────────────────┘    │
  │  ▼                                                         │
  │  ④ Resolve                                                 │
  │  │  • MSAA Resolve（如果启用 MSAA）                         │
  │  │  • 多采样 → 单采样                                       │
  │  ▼                                                         │
  │  ⑤ PostProcess                                             │
  │  │  ┌─────────────────────────────────────────────────┐    │
  │  │  │  5.1  Distortion                                │    │
  │  │  │       • 屏幕扭曲效果（热浪、水面折射）             │    │
  │  │  │                                                 │    │
  │  │  │  5.2  Bloom                                     │    │
  │  │  │       • 多级降采样 + 高斯模糊 + 升采样            │    │
  │  │  │       • 移动端通常 3-4 级                        │    │
  │  │  │                                                 │    │
  │  │  │  5.3  ToneMapping                               │    │
  │  │  │       • HDR → LDR 色调映射                       │    │
  │  │  │       • ACES / Filmic 曲线                      │    │
  │  │  │       • Color Grading                           │    │
  │  │  │       • 输出 sRGB / Gamma 2.2                   │    │
  │  │  │                                                 │    │
  │  │  │  5.4  FXAA                                      │    │
  │  │  │       • 后处理抗锯齿                             │    │
  │  │  │       • 在 ToneMapping 之后执行                  │    │
  │  │  │       • 可选（如果已用 MSAA 可跳过）              │    │
  │  │  └─────────────────────────────────────────────────┘    │
  │  ▼                                                         │
  │  ⑥ Upscale Resolve                                         │
  │  │  • 场景分辨率 → 屏幕分辨率                                │
  │  │  • 用于动态分辨率缩放                                     │
  │  ▼                                                         │
  │  ⑦ UI                                                      │
  │     • Slate / UMG 渲染                                      │
  │     • 在最终分辨率上渲染（不受动态分辨率影响）                 │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

### 2.2 InitView

```
═══ InitView 详细流程 ═══

  FMobileSceneRenderer::InitViews()
  │
  ├── ComputeViewVisibility()
  │   ├── FrustumCull()              // 视锥剔除
  │   ├── OcclusionCull()            // 遮挡剔除（可选，移动端通常关闭）
  │   └── DistanceCull()             // 距离剔除
  │
  ├── SortPrimitives()
  │   ├── Opaque: Front-to-Back      // 不透明从前到后（减少 Overdraw）
  │   ├── Masked: Front-to-Back      // Masked 同上
  │   └── Translucent: Back-to-Front // 半透明从后到前（正确混合）
  │
  ├── SetupMobileBasePassAfterShadowInit()
  │   ├── 收集影响每个物体的光源列表
  │   ├── 移动端限制: 每物体最多 4 个动态光源
  │   └── 选择最重要的光源（距离 × 强度 × 衰减）
  │
  └── CreateIndirectArguments()       // GPU Driven（UE5）

  ⚠️ 移动端 InitView 注意事项：
  • 遮挡剔除在移动端开销较大，通常使用软件遮挡或关闭
  • 排序对 TBR 架构的 Overdraw 优化至关重要
  • 光源裁剪直接影响 Forward Shading 的 Shader 复杂度
```

### 2.3 ShadowDepthMaps

```
═══ 阴影渲染 ═══

  ┌──────────────────────────────────────────────────────────┐
  │  方向光 CSM（Cascaded Shadow Map）                        │
  │                                                          │
  │  Cascade 0: 近距离（0-10m）  → 高分辨率                   │
  │  Cascade 1: 中距离（10-30m） → 中分辨率                   │
  │  Cascade 2: 远距离（30-80m） → 低分辨率                   │
  │                                                          │
  │  移动端优化：                                             │
  │  • 通常 2-3 个 Cascade（PC 用 4 个）                      │
  │  • Shadow Map 分辨率: 512-1024（PC 用 2048-4096）         │
  │  • 使用 Modulated Shadow 替代传统 Shadow Map              │
  │                                                          │
  │  Modulated Shadow（移动端特有）：                         │
  │  • 不需要 Shadow Map 采样                                 │
  │  • 使用 Stencil 标记阴影区域                              │
  │  • 直接调制颜色（乘以阴影颜色）                            │
  │  • 开销极低但效果有限                                     │
  └──────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────┐
  │  点光源 / 聚光灯阴影                                      │
  │                                                          │
  │  • 点光源: Cube Shadow Map（6 面）或 Dual Paraboloid      │
  │  • 聚光灯: 单张 Shadow Map                                │
  │  • 移动端通常限制动态阴影光源数量（1-2 个）                 │
  └──────────────────────────────────────────────────────────┘
```

### 2.4 RenderForward

```
═══ Forward Shading 光照模型 ═══

  在 Opaque And Mask Pass 中，每个像素的光照计算：

  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  FinalColor = Σ(每个光源的贡献)                           │
  │                                                          │
  │  对于每个光源 i：                                         │
  │  Light[i] = BRDF(N, V, L[i]) × LightColor[i]             │
  │           × Attenuation[i] × Shadow[i]                   │
  │                                                          │
  │  FinalColor = Emissive                                   │
  │             + AmbientLight (SH / IBL)                    │
  │             + Σ Light[i]                                 │
  │             + ReflectionProbe                            │
  │                                                          │
  │  移动端 BRDF 简化：                                       │
  │  • 使用 GGX 近似（而非完整 GGX）                           │
  │  • 环境光使用 SH（球谐）而非完整 IBL                       │
  │  • 反射使用 Reflection Capture（预计算）                  │
  │                                                          │
  └──────────────────────────────────────────────────────────┘

═══ 每物体光源限制 ═══

  移动端 Forward Shading 的光源处理：

  ┌──────────────────────────────────────────────────────────┐
  │  方向光: 1 个（全局主光源）                                │
  │  动态光: 最多 4 个/物体（可配置）                          │
  │  静态光: 通过 Lightmap 预烘焙                             │
  │                                                          │
  │  光源选择策略：                                           │
  │  Priority = Intensity × (1 / Distance²) × Radius         │
  │  选择 Priority 最高的 N 个光源                            │
  │                                                          │
  │  Shader 变体：                                            │
  │  • 0 动态光: 最简 Shader                                  │
  │  • 1 动态光: 中等 Shader                                  │
  │  • 2+ 动态光: 复杂 Shader                                 │
  │  → 不同光源数量生成不同 Shader Permutation                 │
  └──────────────────────────────────────────────────────────┘
```

### 2.5 Resolve

```
═══ MSAA Resolve ═══

  如果启用了 MSAA（2x/4x）：

  ┌──────────────────────────────────────────────────────────┐
  │  SceneColor (MSAA)  →  Resolve  →  SceneColor (非 MSAA)  │
  │                                                          │
  │  在 TBR 架构上：                                          │
  │  • Resolve 在 Tile 写回时自动完成                         │
  │  • 几乎零额外开销（Memoryless）                            │
  │  • MSAA RT 不占用显存（Metal Memoryless）                 │
  │                                                          │
  │  在 IMR 架构上：                                          │
  │  • 需要额外的 Resolve Pass                                │
  │  • MSAA RT 占用 N 倍显存                                  │
  │  • 有明显的带宽开销                                       │
  └──────────────────────────────────────────────────────────┘
```

### 2.6 PostProcess

```
═══ HDR PostProcess Chain ═══

  ┌──────────────────────────────────────────────────────────┐
  │  输入: SceneColor (HDR, FP16)                            │
  │                                                          │
  │  ① Distortion                                            │
  │  │  • 屏幕空间扭曲                                        │
  │  │  • 需要 SceneColor 的副本作为输入                      │
  │  │  • 移动端可选（开销较大）                               │
  │  ▼                                                       │
  │  ② Bloom                                                 │
  │  │  • 提取亮度 > 阈值的像素                               │
  │  │  • 多级降采样（1/2, 1/4, 1/8, 1/16）                   │
  │  │  • 每级高斯模糊                                        │
  │  │  • 逐级升采样合并                                      │
  │  │  • 移动端: 3-4 级，PC: 5-6 级                          │
  │  ▼                                                       │
  │  ③ ToneMapping + Color Grading                           │
  │  │  • HDR → LDR 映射                                     │
  │  │  • 应用 Color Grading LUT                             │
  │  │  • 输出 8-bit sRGB                                    │
  │  │  • 移动端使用 3D LUT（16³ 或 32³）                     │
  │  ▼                                                       │
  │  ④ FXAA（可选）                                           │
  │  │  • 后处理抗锯齿                                        │
  │  │  • 在 LDR 空间执行                                     │
  │  │  • 如果已用 MSAA 通常跳过                              │
  │  ▼                                                       │
  │  输出: FinalColor (LDR, RGBA8)                           │
  └──────────────────────────────────────────────────────────┘

  ⚠️ 移动端 PostProcess 注意：
  • 每个 PostProcess Pass = 一次全屏绘制
  • 在 TBR 上每次全屏绘制可能触发 Tile Flush
  • 尽量合并 Pass（如 Bloom + ToneMapping 合并）
  • 使用 FrameBufferFetch 减少带宽
```

### 2.7 Upscale & UI

```
═══ Upscale Resolve ═══

  场景渲染分辨率可以低于屏幕分辨率（动态分辨率）：

  ┌──────────────────────────────────────────────────────────┐
  │  Scene RT (e.g. 720p)  →  Upscale  →  Screen (1080p)    │
  │                                                          │
  │  Upscale 方法：                                           │
  │  • Bilinear: 最快，质量最低                              │
  │  • Lanczos: 中等质量                                     │
  │  • FSR 1.0: AMD 开源，质量较好                           │
  │  • 自定义: 项目特定的 Upscale Shader                     │
  └──────────────────────────────────────────────────────────┘

═══ UI 渲染 ═══

  • UI 在最终屏幕分辨率上渲染
  • 不受动态分辨率影响（保持清晰）
  • Slate / UMG Widget 渲染
  • 使用独立的 UI RT 或直接渲染到 BackBuffer
```

## 3. Forward LDR（Simple Forward）管线

### 3.1 完整流程

```
═══ Forward LDR 完整 Pass 顺序 ═══

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  ① InitView                                                 │
  │  │  • 与 HDR 相同的可见性判定和排序                           │
  │  ▼                                                          │
  │  ② ToneMap LUT 生成                                         │
  │  │  • 预计算 Color Grading LUT                              │
  │  │  • 在场景渲染之前生成                                     │
  │  │  • 后续在 Base Pass 中直接应用                            │
  │  ▼                                                         │
  │  ③ ShadowDepthMaps                                         │
  │  │  • 与 HDR 相同                                           │
  │  ▼                                                         │
  │  ④ RenderForward                                           │
  │  │  ┌─────────────────────────────────────────────────┐    │
  │  │  │  4.1  PrePass（Depth Only）                      │    │
  │  │  │                                                  │    │
  │  │  │  4.2  Opaque And Mask                            │    │
  │  │  │       • 光照计算 + ToneMapping 在同一 Shader 中   │    │
  │  │  │       • 直接输出 LDR（Gamma Space）               │    │
  │  │  │       • 无需后续 ToneMapping Pass                 │    │
  │  │  │                                                  │    │
  │  │  │  4.3  Xray                                       │    │
  │  │  │                                                  │    │
  │  │  │  4.4  Translucency PreDepth                      │    │
  │  │  │                                                  │    │
  │  │  │  4.5  Copy Depth / Color（条件触发）              │    │
  │  │  │                                                  │    │
  │  │  │  4.6  Decal                                      │    │
  │  │  │                                                  │    │
  │  │  │  4.7  Translucency                               │    │
  │  │  │       • 半透明也在 Gamma Space 中渲染             │    │
  │  │  │                                                  │    │
  │  │  │  4.8  Highlight                                  │    │
  │  │  │                                                  │    │
  │  │  │  4.9  OutLine                                    │    │
  │  │  │                                                  │    │
  │  │  │  4.10 FrameBufferFetch ToneMapping               │    │
  │  │  │       • 使用 FBF 读取当前像素颜色                  │    │
  │  │  │       • 应用最终的色彩校正                        │    │
  │  │  │       • 零额外带宽（TBR 上）                      │    │
  │  │  └─────────────────────────────────────────────────┘     │
  │  ▼                                                          │
  │  ⑤ Upscale Resolve                                          │
  │  │  • 场景分辨率 → UI 分辨率                                 │
  │  ▼                                                          │
  │  ⑥ UI                                                       │
  │     • Slate / UMG 渲染                                      │
  │                                                             │
  │  ⚠️ 注意: 没有独立的 PostProcess 阶段！                      │
  │     • 无 Bloom                                              │
  │     • 无 Distortion                                         │  
  │     • 无 FXAA（依赖 MSAA）                                   │
  │     • ToneMapping 内嵌在 Base Pass 或 FBF 中                 │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

### 3.2 与 HDR 的关键差异

```
═══ HDR vs LDR 关键差异对比 ═══

  ┌──────────────────────┬──────────────────────┬──────────────────────┐
  │  特性                │  Forward HDR          │  Forward LDR         │
  ├──────────────────────┼──────────────────────┼──────────────────────┤
  │  SceneColor 格式     │  FP16 (64-bit/px)     │  RGBA8 (32-bit/px)   │
  │  色彩空间             │  Linear (HDR)        │  Gamma (sRGB)        │
  │  ToneMapping         │  独立 PostProcess     │  内嵌在 Base Pass    │
  │  Bloom               │  ✅ 支持             │  ❌ 不支持           │
  │  DOF                 │  ✅ 支持             │  ❌ 不支持           │
  │  Distortion          │  ✅ 支持             │  ❌ 不支持           │
  │  FXAA                │  ✅ 支持             │  ❌ 不支持           │
  │  MSAA                │  ✅ 可选             │  ✅ 可选             │
  │  带宽开销             │  中等                │  最低                 │
  │  RT 内存              │  FP16 = 2× RGBA8     │  RGBA8 = 最小        │
  │  Shader 复杂度        │  中等                │  稍高（内嵌 TM）      │
  │  GlobalShader        │  HDR 版本             │  LDR 版本            │
  │  适用场景             │  高配移动端           │  低配移动端           │
  └──────────────────────┴──────────────────────┴──────────────────────┘

  ⚠️ 两套 GlobalShader：
  • HDR 和 LDR 使用不同的 GlobalShader 变体
  • 通过宏 MOBILE_HDR / MOBILE_LDR 隔开
  • 材质 Shader 中也有对应的宏分支
  • LDR 模式下材质渲染需要反转屏幕（Gamma 校正差异）
```

### 3.3 ToneMap LUT

```
═══ LDR 模式的 ToneMap LUT ═══

  在 LDR 模式中，ToneMapping 不是独立的后处理 Pass
  而是通过预计算的 LUT 在 Base Pass 中完成：

  ┌──────────────────────────────────────────────────────────┐
  │  渲染前：                                                 │
  │  ┌─────────────────────────────────────────────┐         │
  │  │  生成 3D LUT (16×16×16 或 32×32×32)          │         │
  │  │  • 编码 ToneMapping 曲线                     │         │
  │  │  • 编码 Color Grading 调整                   │         │
  │  │  • 编码 White Balance / Saturation 等        │         │
  │  └─────────────────────────────────────────────┘         │
  │                                                          │
  │  Base Pass 中：                                           │
  │  ┌─────────────────────────────────────────────┐         │
  │  │  HDR Color = Lighting(...)                  │         │
  │  │  LDR Color = Texture3DSample(LUT, HDR Color)│         │
  │  │  Output = LDR Color  // 直接输出 Gamma Space │         │
  │  └─────────────────────────────────────────────┘         │
  │                                                          │
  │  优势：                                                   │
  │  • 无需额外的全屏 ToneMapping Pass                         │
  │  • 节省一次全屏绘制的带宽                                  │
  │  • LUT 采样开销极低（一次 3D 纹理采样）                     │
  │                                                          │
  │  劣势：                                                   │
  │  • LUT 精度有限（16³ = 4096 个采样点）                     │
  │  • 极端 HDR 值可能被截断                                  │
  │  • 不支持空间变化的 ToneMapping                           │
  └──────────────────────────────────────────────────────────┘
```

### 3.4 FrameBufferFetch ToneMapping

```
═══ FrameBufferFetch（FBF）═══

  LDR 管线最后一步使用 FBF 进行最终色彩校正：

  ┌──────────────────────────────────────────────────────────┐
  │  FrameBufferFetch 原理：                                  │
  │                                                          │
  │  在 TBR 架构上：                                          │
  │  • 当前像素的颜色已经在 Tile SRAM 中                       │
  │  • FBF 直接从 SRAM 读取，无需额外带宽                      │
  │  • 等价于 "免费" 的读取操作                                │
  │                                                          │
  │  在 IMR 架构上：                                          │
  │  • FBF 需要从显存读取当前像素                              │
  │  • 有额外带宽开销                                         │
  │  • 但仍比独立的全屏 Pass 高效                              │
  │                                                          │
  │  LDR 中的用途：                                           │
  │  • 读取当前像素的 LDR 颜色                                 │
  │  • 应用最终的 Gamma 校正                                  │
  │  • 应用 Dithering（减少色带）                             │
  │  • 应用 Film Grain 等简单效果                             │
  └──────────────────────────────────────────────────────────┘

  API 支持：
  • Metal:  [[color(0)]] attribute + framebuffer fetch
  • Vulkan: VK_EXT_rasterization_order_attachment_access
            (subpass input attachment)
  • OpenGL ES: GL_EXT_shader_framebuffer_fetch
  • 不支持的平台: 回退到 Copy + Sample
```

### 3.5 LDR 的限制与优势

```
═══ LDR 管线的取舍 ═══

  ┌──────────────────────────────────────────────────────────┐
  │  ✅ 优势                                                 │
  │                                                          │
  │  • 带宽最低: RGBA8 = FP16 的一半                          │
  │  • 无后处理: 省去多次全屏绘制                              │
  │  • 内存最小: RT 占用最少                                  │
  │  • 功耗最低: 适合低端设备                                  │
  │  • Shader 编译快: 变体更少                                │
  │                                                          │
  │  ❌ 限制                                                 │
  │                                                          │
  │  • 无 Bloom: 没有泛光效果                                 │
  │  • 无 DOF: 没有景深效果                                   │
  │  • 无 Motion Blur: 没有运动模糊                           │
  │  • 无 Distortion: 没有屏幕扭曲                            │
  │  • 无 FXAA: 只能依赖 MSAA 抗锯齿                          │
  │  • 色彩精度低: 8-bit 可能出现色带                          │
  │  • Decal 需要改造: Forward LDR 中 Decal 行为不同          │
  │  • 材质渲染反转: Gamma Space 导致某些材质效果不同           │
  │                                                          │
  │  ⚠️ 关键注意事项：                                        │
  │  • 两套 GlobalShader 通过宏隔开                           │
  │  • 材质在 LDR 模式下渲染结果可能与 HDR 不同                │
  │  • 需要在两种模式下分别测试材质效果                         │
  └──────────────────────────────────────────────────────────┘
```

```
  │  │  │                                                 │    │
  │  │  │  8.3  Tiled / Clustered Deferred（可选）         │    │
```

### 4.2 GBuffer 布局

```
═══ UE4/UE5 默认 GBuffer 布局 ═══

  ┌──────────────────────────────────────────────────────────────────┐
  │  RT          │ 格式       │ 内容                                  │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  GBufferA    │ RGBA8      │ R: WorldNormal.X                     │
  │              │            │ G: WorldNormal.Y                     │
  │              │            │ B: WorldNormal.Z                     │
  │              │            │ A: PerObjectData                     │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  GBufferB    │ RGBA8      │ R: Metallic                          │
  │              │            │ G: Specular                          │
  │              │            │ B: Roughness                         │
  │              │            │ A: ShadingModelID + SelectiveOutput  │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  GBufferC    │ RGBA8      │ R: BaseColor.R                       │
  │              │            │ G: BaseColor.G                       │
  │              │            │ B: BaseColor.B                       │
  │              │            │ A: IndirectIrradiance (encoded)      │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  GBufferD    │ RGBA8      │ Custom Data（Shading Model 特定）     │
  │  (optional)  │            │ 如: Subsurface Color, Clear Coat      │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  GBufferE    │ RGBA16F    │ Pre-computed Shadow Factor           │
  │  (optional)  │            │ (Static Shadow + Lightmap)            │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  GBufferF    │ RGBA8      │ Velocity (Motion Vector)             │
  │  (optional)  │            │ 用于 TAA / Motion Blur                │
  ├──────────────┼────────────┼──────────────────────────────────────┤
  │  SceneDepth  │ D24S8 /    │ Depth + Stencil                      │
  │              │ D32F       │                                      │
  └──────────────┴────────────┴──────────────────────────────────────┘

  带宽分析（1080p，不含 MSAA）：
  • GBufferA-C (must)   : 3 × 1920×1080×4 = 24 MB
  • SceneDepth          : 1920×1080×4 = 8 MB
  • SceneColor (FP16)   : 1920×1080×8 = 16 MB
  • Total GBuffer Write : ~48 MB/Frame
  • 加上读取             : ~96 MB/Frame（Only GBuffer 相关）

  ⚠️ 这就是 Deferred 在移动端带宽开销大的原因
```

### 4.3 Base Pass

```
═══ Base Pass 详解 ═══

  Base Pass 是 Deferred 管线中最重要的 Pass：

  ┌──────────────────────────────────────────────────────────┐
  │  输入:                                                   │
  │  • 顶点数据（Position, Normal, UV, Tangent）              │
  │  • 材质参数（Textures, Constants）                        │
  │  • Lightmap UV + Lightmap Texture                        │
  │                                                          │
  │  Vertex Shader:                                          │
  │  • 世界空间变换                                           │
  │  • 计算 Lightmap UV                                      │
  │  • 输出 Clip Space Position                              │
  │                                                          │
  │  Pixel Shader:                                           │
  │  • 采样材质纹理（Diffuse, Normal, ORM 等）                 │
  │  • 计算材质属性                                           │
  │  • 采样 Lightmap（静态光照）                              │
  │  • 编码并输出到 GBuffer                                   │
  │  • ⚠️ 不计算动态光照                                      │
  │                                                          │
  │  输出（MRT - Multiple Render Targets）:                   │
  │  • GBufferA: World Normal                                │
  │  • GBufferB: Metallic / Specular / Roughness             │
  │  • GBufferC: BaseColor + Indirect Irradiance             │
  │  • SceneDepth: Depth + Stencil                           │
  └──────────────────────────────────────────────────────────┘

  SRP Batcher / Instancing 优化：
  • 相同材质的物体批处理
  • 减少 State Change
  • UE5 Nanite 物体不经过传统 Base Pass
```

### 4.4 Lighting Pass

```
═══ Lighting Pass 详解 ═══

  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  方向光（Directional Light）：                            │
  │  ┌────────────────────────────────────────────────┐      │
  │  │  全屏 Quad 绘制                                 │      │
  │  │  ↓                                             │      │
  │  │  采样 GBufferA → Normal                        │      │
  │  │  采样 GBufferB → Metallic/Roughness            │      │
  │  │  采样 GBufferC → BaseColor                     │      │
  │  │  采样 SceneDepth → 重建世界坐标                 │      │
  │  │  采样 ShadowMap → 阴影因子                      │      │
  │  │  ↓                                             │      │
  │  │  BRDF(N, V, L, Roughness, Metallic)            │      │
  │  │  × LightColor × Shadow × SSAO                  │      │
  │  │  ↓                                             │      │
  │  │  累加到 SceneColor                              │      │
  │  └────────────────────────────────────────────────┘      │
  │                                                          │
  │  点光源 / 聚光灯：                                        │
  │  ┌────────────────────────────────────────────────┐      │
  │  │  Step 1: Stencil 标记                          │      │
  │  │  • 渲染 Light Volume（球体/锥体）的背面          │      │
  │  │  • Depth Test: Greater                         │      │
  │  │  • 标记被光源影响的像素                          │      │
  │  │                                                │      │
  │  │  Step 2: 光照计算                               │      │
  │  │  • 只处理 Stencil 标记的像素                     │      │
  │  │  • 渲染 Light Volume 的正面                     │      │
  │  │  • 采样 GBuffer + 计算 BRDF                     │      │
  │  │  • 累加到 SceneColor                            │      │
  │  └────────────────────────────────────────────────┘      │
  │                                                          │
  │  Tiled Deferred（可选优化）：                             │
  │  ┌────────────────────────────────────────────────┐      │
  │  │  • 将屏幕分为 16×16 或 32×32 的 Tile            │      │
  │  │  • 每个 Tile 计算影响它的光源列表                │      │
  │  │  • Compute Shader 一次性处理所有光源             │      │
  │  │  • 减少 GBuffer 读取次数                        │      │
  │  │  • 适合大量光源场景（100+）                      │      │
  │  └────────────────────────────────────────────────┘      │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### 4.5 Reflection & SSR

```
═══ 反射系统 ═══

  ┌──────────────────────────────────────────────────────────┐
  │  反射来源（按优先级）：                                    │
  │                                                          │
  │  1. SSR（Screen Space Reflection）                       │
  │     • 屏幕空间光线步进                                    │
  │     • 使用 HZB 加速                                       │
  │     • 只能反射屏幕内可见的内容                             │
  │     • 开销: 中-高                                        │
  │                                                          │
  │  2. Lumen Reflection（UE5）                              │
  │     • 基于 Surface Cache 的光线追踪                       │
  │     • 支持屏幕外反射                                      │
  │     • 软件/硬件光追可选                                   │
  │                                                          │
  │  3. Reflection Capture（预计算）                          │
  │     • 预烘焙的 Cubemap                                    │
  │     • 放置在场景中的 Capture Actor                        │
  │     • 开销最低但不支持动态内容                             │
  │                                                          │
  │  4. Planar Reflection（可选）                             │
  │     • 平面反射（水面、镜子）                               │
  │     • 需要额外渲染一次场景                                 │
  │     • 开销极高，慎用                                      │
  │                                                          │
  │  混合策略：                                               │
  │  FinalReflection = SSR × SSR_Confidence                  │
  │                  + ReflectionCapture × (1 - SSR_Conf)    │
  └──────────────────────────────────────────────────────────┘
```

### 4.6 Translucency

```
═══ Deferred 管线中的半透明 ═══

  ⚠️ 关键: 半透明物体在 Deferred 管线中使用 Forward Shading！

  原因：
  • Deferred 的 GBuffer 不支持混合（Blend）
  • 半透明需要 Alpha Blending
  • 因此半透明物体回退到 Forward 路径

  ┌──────────────────────────────────────────────────────────┐
  │  半透明渲染顺序：                                         │
  │                                                          │
  │  1. Sort Back-to-Front                                   │
  │  2. 对每个半透明物体：                                     │
  │     • 采样 SceneColor（用于折射/透射）                     │
  │     • Forward Shading 计算光照                            │
  │     • Alpha Blend 到 SceneColor                          │
  │                                                          │
  │  Separate Translucency：                                 │
  │  • 某些半透明物体（如 UI 粒子）渲染到独立 RT                │
  │  • 在 DOF 之后合并                                        │
  │  • 避免 DOF 模糊 UI 元素                                  │
  │                                                          │
  │  半透明光照选项：                                         │
  │  • Volumetric Non-Directional: 最简单，均匀光照           │
  │  • Volumetric Directional: 带方向的体积光照               │
  │  • Volumetric Per-Vertex: 逐顶点光照                      │
  │  • Surface Forward Shading: 完整的 Forward 光照           │
  └──────────────────────────────────────────────────────────┘
```

### 8.1 PrePass (Depth Only)

```
═══ PrePass / Early-Z ═══
```

## 9. 面试高频问题

```
═══ Q1: Forward 和 Deferred 渲染的核心区别是什么？ ═══

A:
• Forward: 光照在 Base Pass 中计算，每个物体遍历所有影响它的光源
  - 优势: 支持 MSAA，带宽低，适合移动端
  - 劣势: 多光源时 Shader 复杂度高，每像素光源数有限

• Deferred: 先将材质属性写入 GBuffer，再在独立的 Lighting Pass 中计算光照
  - 优势: 光源数量不影响 Base Pass 复杂度，支持大量光源
  - 劣势: GBuffer 带宽高，不支持 MSAA，半透明需要回退到 Forward

═══ Q2: 为什么移动端优先使用 Forward 而不是 Deferred？ ═══

A:
1. 带宽: TBR 架构对带宽敏感，GBuffer 的多 RT 写入/读取开销大
2. MSAA: TBR 上 MSAA 几乎免费（Tile SRAM 内 Resolve），Deferred 不支持 MSAA
3. 光源数量: 移动端场景光源数量有限，Forward 的劣势不明显
4. 功耗: 更少的 RT 读写 = 更低的功耗 = 更少的发热

═══ Q3: Forward HDR 和 Forward LDR 的区别？什么时候选哪个？ ═══

A:
• HDR: SceneColor 使用 FP16，支持完整后处理（Bloom/DOF/FXAA）
  适合: 需要后处理效果的中高端移动设备

• LDR: SceneColor 使用 RGBA8，ToneMapping 内嵌在 Base Pass
  无独立后处理阶段，带宽最低
  适合: 低端设备，不需要后处理效果的项目

关键差异:
• 两套 GlobalShader 通过宏隔开
• LDR 在 Gamma Space 渲染，材质效果可能不同
• LDR 的 Decal 需要特殊处理

═══ Q4: Deferred 管线中半透明物体怎么处理？ ═══

A:
半透明物体在 Deferred 管线中使用 Forward Shading，因为:
• GBuffer 不支持 Alpha Blending（MRT 写入是覆盖式的）
• 半透明需要与背景混合

流程:
1. 不透明物体通过 Deferred 管线渲染完成
2. 半透明物体按 Back-to-Front 排序
3. 每个半透明物体使用 Forward Shading 计算光照
4. Alpha Blend 到已有的 SceneColor 上

═══ Q5: PrePass（Early-Z）在什么情况下有用？什么情况下反而有害？ ═══

A:
有用:
• IMR 架构: 提前写入深度，后续 Base Pass 利用 Early-Z 跳过被遮挡像素
• Deferred 管线: 生成 HZB 用于 GPU Occlusion Culling
• 大量 Overdraw 的场景

可能有害:
• TBR 架构: HSR（Hidden Surface Removal）已经在硬件层面做了类似优化
  额外的 PrePass 增加了顶点处理开销和 Tile 内存压力
• 简单场景: Overdraw 不严重时，PrePass 的额外 Draw Call 反而是负担

═══ Q6: Bloom 在移动端的开销为什么大？如何优化？ ═══

A:
Bloom 需要多次全屏绘制（降采样 + 模糊 + 升采样）:
• 标准 Bloom: 5-6 级 = 10-12 次全屏绘制
• 每次全屏绘制在 TBR 上可能触发 Tile Flush

优化:
• 减少 Bloom 级数（3-4 级）
• 合并降采样和模糊 Pass
• 使用半分辨率或更低分辨率
• 将 Bloom 最后一级与 ToneMapping 合并
• 使用 Compute Shader 替代全屏 Quad

═══ Q7: Virtual Shadow Map 相比 CSM 有什么优势？ ═══

A:
VSM 优势:
• 缓存友好: 只更新变化的 Page，静态场景几乎零开销
• 精度均匀: 不像 CSM 那样近处高精度远处低精度
• 支持更多光源: 每个光源都可以有高质量阴影
• 与 Nanite 集成: 利用 Nanite 的 GPU Driven 渲染

VSM 劣势:
• GPU 内存占用更大
• 需要 Nanite 支持才能发挥最大优势
• 移动端暂不支持

═══ Q8: 解释 Unreal 的多线程渲染架构 ═══

A:
三个线程流水线:
1. GameThread: 游戏逻辑、物理、动画更新
2. RenderThread: 可见性判定、生成渲染命令
3. RHIThread: 将渲染命令翻译为 GPU API 调用并提交

GameThread 领先 RenderThread 1 帧
总延迟 1-2 帧

优势: 三个线程并行工作，提高 CPU 利用率
劣势: 增加输入延迟，需要处理跨线程数据同步

═══ Q9: FrameBufferFetch 是什么？为什么在移动端重要？ ═══

A:
FrameBufferFetch 允许 Pixel Shader 读取当前像素已有的颜色值。

在 TBR 架构上:
• 当前像素数据已在 Tile SRAM 中
• FBF 直接从 SRAM 读取，零额外带宽
• 等价于 "免费" 的读取操作

用途:
• LDR 管线的最终 ToneMapping
• 自定义混合模式
• 减少需要 Copy SceneColor 的场景

支持:
• Metal: framebuffer fetch attribute
• Vulkan: subpass input attachment
• OpenGL ES: GL_EXT_shader_framebuffer_fetch

═══ Q10: 如何在运行时切换渲染管线？ ═══

A:
• Forward HDR ↔ LDR: 通过 r.MobileHDR 控制
  需要重新编译 Shader（两套 GlobalShader）
  通常在项目设置中固定，不在运行时切换

• Forward ↔ Deferred: 通过项目设置选择
  不同的 SceneRenderer 类
  不支持运行时切换

• 实际做法: 根据设备性能在启动时选择配置
  使用 Scalability Settings 调整质量级别
  而不是切换整个管线
```

## 10. 实践检查清单

```
═══ 管线选择 ═══
□ 确定目标平台和最低配置设备
□ 评估光源数量需求（Forward vs Deferred）
□ 评估后处理需求（HDR vs LDR）
□ 评估带宽预算（TBR 架构优先 Forward）
□ 在目标设备上实测三种管线的帧率

═══ Forward HDR 优化 ═══
□ 限制每物体动态光源数量（4 个以内）
□ 优化 CSM Cascade 数量和分辨率
□ 精简后处理LPass（关闭不必要的效果）
□ 合并后处理 Pass（Bloom + ToneMapping）
□ 使用 MSAA 替代 FXAA（TBR 上更高效）
□ 启用动态分辨率

═══ Forward LDR 优化 ═══
□ 确认所有材质在 Gamma Space 下效果正确
□ 测试 Decal 在 LDR 模式下的表现
□ 配置 ToneMap LUT 参数
□ 验证 FrameBufferFetch 在目标设备上可用
□ 使用 MSAA 2x（TBR 上几乎免费）

═══ Deferred 优化 ═══
□ 优化 GBuffer 布局（减少不必要的 RT）
□ 使用 Tiled/Clustered Deferred 处理大量光源
□ 配置 HZB 用于 GPU Occlusion Culling
□ 优化 Shadow Map 分辨率和 Cascade 设置
□ 控制后处理 Pass 数量
□ 考虑使用 VSM 替代 CSM（UE5）

═══ 通用优化 ═══
□ 使用 RenderDoc / Xcode GPU Capture 分析每个 Pass
□ 检查 Draw Call 数量（stat scenerendering）
□ 检查 Shader 复杂度（stat shaders）
□ 检查带宽使用（GPU Profiler）
□ 验证多线程渲染正常工作
□ 测试不同质量级别的 Scalability Settings
```
