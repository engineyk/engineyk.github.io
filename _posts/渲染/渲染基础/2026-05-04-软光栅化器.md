# 软光栅化器

---

## 1. 软光栅化器基础原理

### 1.1 什么是软光栅化器

```bash
硬件光栅化器（Hardware Rasterizer）：
  GPU 固定功能管线中的光栅化单元
  输入：裁剪后的三角形顶点（Clip Space）
  输出：覆盖的像素片段（Fragment）
  特点：固定管线，不可编程，以 2×2 Quad 为最小单位

软件光栅化器（Software Rasterizer）：
  用 Compute Shader 或 CPU 代码实现的可编程光栅化
  输入：三角形顶点坐标
  输出：逐像素写入 Buffer（通常是 Visibility Buffer / Depth Buffer）
  特点：完全可编程，无 Quad Overdraw，适合微小三角形

为什么需要软光栅化器？
  硬件光栅化器以 2×2 像素的 Quad 为最小处理单位
  当三角形面积 < 1 像素时，仍然要执行 4 个像素的 Fragment Shader
  → 产生严重的 Quad Overdraw（最高 4x 浪费）
  → 微小三角形越多，浪费越严重

  软光栅化器逐像素处理，无 Quad 概念
  → 面积 < 1 像素的三角形只执行 1 次写入
  → 对于 Nanite 的像素级三角形，效率远高于硬件光栅化
```

### 1.2 三角形光栅化核心算法

#### Edge Equation（边方程法）

```
原理：
  三角形由 3 条边定义，每条边是一条直线
  对于任意点 P，计算它相对于每条边的"哪一侧"
  如果 P 在所有 3 条边的"内侧"，则 P 在三角形内部

边方程：
  给定边 V0→V1，点 P 的边函数值为：
  E(P) = (V1.x - V0.x) * (P.y - V0.y) - (V1.y - V0.y) * (P.x - V0.x)

  E(P) > 0 → P 在边的左侧（内侧，CCW winding）
  E(P) = 0 → P 在边上
  E(P) < 0 → P 在边的右侧（外侧）

判断点在三角形内：
  E01(P) >= 0 && E12(P) >= 0 && E20(P) >= 0
  → P 在三角形内部（或边上）
```

#### 两种遍历策略

```
1. Bounding Box 遍历（矩形框法）：
   计算三角形的 AABB 包围盒
   遍历包围盒内所有像素
   对每个像素计算 3 个边方程，判断是否在三角形内

   优点：实现简单
   缺点：包围盒内很多像素不在三角形内，浪费计算
   适用：小三角形（包围盒小，浪费少）

2. Scanline 遍历（扫描线法）：
   逐行扫描，计算每行与三角形边的交点
   只遍历三角形实际覆盖的像素

   优点：无浪费像素
   缺点：需要计算交点，逻辑复杂
   适用：较大三角形

Nanite 的选择：
  小三角形（宽度 ≤ 4px）→ Bounding Box 法
  大三角形（宽度 > 4px）→ Scanline 法
  通过 WaveActiveAnyTrue 动态选择
```

#### 定点数精度

```
为什么使用定点数？
  浮点数在亚像素精度下有舍入误差
  可能导致相邻三角形之间出现裂缝（Crack）
  定点数保证精确的整数运算，无裂缝

Nanite 使用 SubpixelBits = 8：
  即 8 位亚像素精度（256 个子采样点/像素）
  坐标格式：整数部分.8位小数部分（如 16.8 定点数）
  最大三角形尺寸：受限于定点数位宽
    CPU 版本（24.8）：最大 2047×2047 像素
    GPU 版本（4.8）：最大 127×127 像素（足够，因为 Nanite 三角形很小）
SubPixel：两个物理像素之间还有像素，称之为Sub-Pixel
```

#### Fill Convention（填充规则）

```
问题：当像素恰好在三角形边上时，属于哪个三角形？
  如果两个相邻三角形都画这个像素 → 重复绘制
  如果两个都不画 → 出现裂缝

解决：Top-Left Rule（左上规则）
  像素属于"上边"或"左边"的三角形
  定义：
    Top Edge  : 水平边且方向向左（Edge.Y == 0 && Edge.X > 0）
    Left Edge : 非水平边且方向向上（Edge.Y < 0）

  实现：
    对于非 Top-Left 的边，将边方程常量 C 减 1
    → 边上的像素不满足 C >= 0 条件
    → 被排除出当前三角形
    → 但会被相邻三角形的 Top-Left 边包含

Nanite CPU 版本：
  C -= (Edge.Y < 0 || (Edge.Y == 0 && Edge.X > 0)) ? 0 : 1;

Nanite GPU 版本（避免分支）：
  C -= saturate(Edge.y + saturate(1.0f - Edge.x));
```

### 1.3 深度插值

```
三角形内任意点的深度通过梯度线性插值：

已知三角形 3 个顶点的深度 Z0, Z1, Z2
计算深度梯度：
  v01 = V1 - V0
  v02 = V2 - V0
  DetXY = v01.x * v02.y - v01.y * v02.x  （三角形面积×2）

  GradZ.x = (v01.z * v02.y - v01.y * v02.z) / DetXY  （X方向深度变化率）
  GradZ.y = (v01.x * v02.z - v01.z * v02.x) / DetXY  （Y方向深度变化率）

任意像素 (x, y) 的深度：
  Z(x, y) = Z0 + GradZ.x * (x - V0.x) + GradZ.y * (y - V0.y)

逐像素步进时只需加法：
  Z(x+1, y) = Z(x, y) + GradZ.x
  Z(x, y+1) = Z(x, y) + GradZ.y
```

### 1.4 背面剔除

```
通过三角形面积的符号判断正反面：
  DetXY = v01.x * v02.y - v01.y * v02.x

  DetXY < 0 → 正面（CCW winding，面向相机）
  DetXY >= 0 → 背面（CW winding，背对相机）→ 剔除

注意：这是在屏幕空间判断的
  顶点已经投影到屏幕坐标后，再计算 DetXY
  如果三角形在屏幕上是顺时针排列 → 背面
```

### 1.5 深度测试与原子写入

```
软光栅化器需要自己实现深度测试：

CPU 版本：
  直接比较并写入（单线程无竞争）
  Pixels[index] = max(Pixels[index], newValue);

GPU 版本（Compute Shader）：
  多个线程可能同时写入同一像素 → 需要原子操作
  
  Nanite 的做法：
    将 TriangleID 和 Depth 打包到一个 64-bit 值中
    使用 InterlockedMax 原子操作写入
    深度值放在高位 → InterlockedMax 自动实现深度测试

  打包格式（Visibility Buffer）：
    uint64 = (Depth << 32) | PixelValue
    PixelValue = (VisibleClusterIndex << 7) | TriangleID

  写入：
    InterlockedMax(OutVisBuffer64[pixelPos], packedValue);
    → 深度最大（最近）的三角形自动胜出
```

## 2. Nanite 中软光栅化器的使用

### 2.1 为什么 Nanite 需要软光栅化器

```
Nanite 的核心目标：渲染像素级大小的三角形（Micro-Polygon）

问题：硬件光栅化器的 Quad Overdraw
┌─────────────────────────────────────────────────────────────────┐
│  硬件光栅化器以 2×2 Quad 为最小执行单位                           │
│                                                                 │
│  情况1：三角形覆盖 4+ 像素                                        │
│  ┌──┬──┐                                                        │
│  │▓▓│▓▓│  4 个像素都在三角形内                                   │
│  ├──┼──┤  → 4 次 Fragment Shader，0 浪费                        │
│  │▓▓│▓▓│  效率 = 100%                                           │
│  └──┴──┘                                                        │
│                                                                 │
│  情况2：三角形只覆盖 1 个像素（Nanite 常见情况）                   │
│  ┌──┬──┐                                                        │
│  │▓▓│  │  只有 1 个像素在三角形内                                 │
│  ├──┼──┤  → 但 GPU 仍执行 4 次 Fragment Shader                   │
│  │  │  │  效率 = 25%（浪费 75%）                                 │
│  └──┴──┘                                                        │
│                                                                 │
│  情况3：三角形面积 < 1 像素（Nanite 极端情况）                     │
│  ┌──┬──┐                                                        │
│  │▓ │  │  三角形可能只覆盖半个像素                                │
│  ├──┼──┤  → GPU 仍执行 4 次 Fragment Shader                      │
│  │  │  │  效率 < 25%                                            │
│  └──┴──┘                                                        │
│                                                                 │
│  软光栅化器：逐像素处理，无 Quad 概念                              │
│  ┌──┐                                                           │
│  │▓▓│  只执行 1 次写入                                           │
│  └──┘  效率 = 100%                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

性能对比（Nanite 场景，大量像素级三角形）：
  硬件光栅化：平均 Quad 利用率 ~25-50%
  软光栅化：  Quad 利用率 = 100%（无 Quad 概念）
  → 对于微小三角形，软光栅化可快 2-4x
```

### 2.2 Nanite 的混合光栅化策略

```
Nanite 不是只用软光栅化，而是混合使用：

┌─────────────────────────────────────────────────────────────────┐
│  Nanite 混合光栅化策略                                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  GPU Culling 阶段输出两个列表：                                  │
│                                                                 │
│  ┌──────────────────────┐    ┌──────────────────────┐           │
│  │  SW Raster List      │    │  HW Raster List      │           │
│  │  (软光栅化列表)       │    │  (硬光栅化列表)       │           │
│  │                      │    │                      │           │
│  │  Cluster 面积小      │    │  Cluster 面积大       │           │
│  │  三角形 < 几个像素    │    │  三角形 > 几个像素    │           │
│  │                      │    │                      │           │
│  │  → Compute Shader    │    │  → 传统 VS/PS 管线    │           │
│  │    软件光栅化         │    │    硬件光栅化         │           │
│  └──────────┬───────────┘    └──────────┬───────────┘           │
│             │                           │                       │
│             ▼                           ▼                       │
│  ┌──────────────────────────────────────────────────┐           │
│  │         Visibility Buffer (uint64)               │           │
│  │         = Depth (高32位) | TriID (低32位)         │           │
│  │         通过 InterlockedMax 原子写入              │           │
│  └──────────────────────────────────────────────────┘           │
│                                                                 │
│  分类阈值：                                                      │
│    Cluster 投影到屏幕后的面积 < 阈值 → 软光栅化                    │
│    Cluster 投影到屏幕后的面积 >= 阈值 → 硬光栅化                   │
│    （阈值通常为 ~32-64 像素²）                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

为什么不全用软光栅化？
  大三角形时，硬件光栅化器的固定功能管线更快
  硬件光栅化器有专用的三角形设置（Triangle Setup）硬件
  大三角形的 Quad 利用率高，Overdraw 浪费小
  → 大三角形用硬件更划算

为什么不全用硬光栅化？
  小三角形时，Quad Overdraw 严重
  每个 Cluster 128 个三角形，需要 128×3 = 384 个顶点的 Draw Call
  大量小 Draw Call 的调度开销高
  → 小三角形用 Compute Shader 更划算
```

### 2.3 Nanite 软光栅化的 Compute Shader 实现


#### 执行模型

```
┌─────────────────────────────────────────────────────────────────┐
│  MicroPolyRasterize 执行模型                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Dispatch: N 个 Cluster → N 个 Thread Group                     │
│  每个 Thread Group: 128 个线程                                   │
│                                                                 │
│  Phase 1: 矩阵计算（1 个线程）                                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ Thread 0: 计算 LocalToSubpixel 矩阵                      │    │
│  │           存入 GroupShared Memory (LDS)                 │    │
│  └─────────────────────────────────────────────────────────┘    │
│  ─── GroupMemoryBarrierWithGroupSync() ───                      │
│                                                                 │
│  Phase 2: 顶点变换（128 个线程并行）                              │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ Thread 0~127: 各变换 1~2 个顶点                          │    │
│  │   DecodePosition → MVP 变换 → 透视除法 → 亚像素坐标       │    │
│  │   结果存入 GroupShared: GroupVerts[256] 或 [384]         │    │
│  └─────────────────────────────────────────────────────────┘    │
│  ─── GroupMemoryBarrierWithGroupSync() ───                      │
│                                                                 │
│  Phase 3: 三角形光栅化（128 个线程并行）                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ Thread 0~NumTris: 各光栅化 1 个三角形                    │    │
│  │   读取 3 个顶点 → 背面剔除 → 包围盒 → 逐像素遍历           │    │
│  │   → InterlockedMax 写入 Visibility Buffer               │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  关键数据：                                                      │
│    每个 Cluster 最多 128 个三角形、256 个顶点                     │
│    GroupShared Memory: GroupVerts[256] × 12 bytes = 3KB         │
│    （USE_CONSTRAINED_CLUSTERS 时）                              │
│    或 GroupVerts[384] × 12 bytes = 4.5KB                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 入口函数 MicropolyRasterize

```c
// Engine\Shaders\Private\Nanite\Rasterizer.usf

[numthreads(128, 1, 1)]
void MicropolyRasterize(
    uint VisibleIndex : SV_GroupID,       // 每个 Group 处理一个 Cluster
    uint GroupIndex   : SV_GroupIndex)    // 128 个线程
{
  // 1. 获取 Cluster 数据
  FVisibleCluster VisibleCluster = GetVisibleCluster(VisibleIndex);
  FInstanceSceneData InstanceData = GetInstanceData(VisibleCluster.InstanceId);
  FNaniteView NaniteView = GetNaniteView(VisibleCluster.ViewId);
  FCluster Cluster = GetCluster(VisibleCluster.PageIndex, VisibleCluster.ClusterIndex);

  // Phase 1: 矩阵计算（1 个线程）  
  // 2. 计算 LocalToSubpixel 变换矩阵（Group 共享，只算一次）       
  if (GroupIndex == 0) {
      LocalToSubpixelLDS = CalculateLocalToSubpixel(NaniteView, InstanceData);
  }
  GroupMemoryBarrierWithGroupSync();

  // Phase 2: 顶点变换（128 个线程并行）
  // 3. 变换所有顶点到亚像素空间（并行，每线程处理 1-2 个顶点）
  for (uint i = 0; i < 2; i++) {
      uint VertIndex = GroupIndex + i * 128;
      if (VertIndex < Cluster.NumVerts) {
          float3 PointLocal = DecodePosition(VertIndex, Cluster);
          float4 PointClip = mul(float4(PointLocal, 1), LocalToSubpixel);
          float3 Subpixel = PointClip.xyz / PointClip.w;
          GroupVerts[VertIndex] = float3(floor(Subpixel.xy), Subpixel.z);
      }
  }
  GroupMemoryBarrierWithGroupSync();

  // Phase 3: 三角形光栅化（128 个线程并行）  
  // 4. 每个线程光栅化一个三角形
  if (GroupIndex < Cluster.NumTris) {
      uint3 TriIndices = ReadTriangleIndices(Cluster, GroupIndex);
      float3 Vertices[3] = {
          GroupVerts[TriIndices.x],
          GroupVerts[TriIndices.y],
          GroupVerts[TriIndices.z],
      };
      uint PixelValue = ((VisibleIndex + 1) << 7) | GroupIndex;
      
      // 调用软光栅化函数
      RasterizeTri(NaniteView, ViewRect, PixelValue, Vertices, ...);
  }
}
```

#### RasterizeTri GPU 版本核心逻辑

```c
void RasterizeTri(FNaniteView NaniteView, int4 ViewRect, uint PixelValue,
                  float3 Verts[3], bool bUsePageTable)
{
    // 1. 背面剔除
    float3 v01 = Verts[1] - Verts[0];
    float3 v02 = Verts[2] - Verts[0];
    float DetXY = v01.x * v02.y - v01.y * v02.x;
    if (DetXY >= 0.0f) return;  // 背面，剔除

    // 2. 计算深度梯度
    float InvDet = rcp(DetXY);
    float2 GradZ;
    GradZ.x = (v01.z * v02.y - v01.y * v02.z) * InvDet;
    GradZ.y = (v01.x * v02.z - v01.z * v02.x) * InvDet;

    // 3. 计算包围盒并裁剪
    int2 MinPixel = ...; // 三角形 AABB 最小像素
    int2 MaxPixel = ...; // 三角形 AABB 最大像素
    MinPixel = max(MinPixel, ViewRect.xy);
    MaxPixel = min(MaxPixel, ViewRect.zw - 1);
    if (any(MinPixel > MaxPixel)) return;  // 完全在视口外
    MaxPixel = min(MaxPixel, MinPixel + 63);  // 限制最大光栅化范围

    // 4. 计算边方程初始值
    float2 Edge01 = -v01.xy;
    float2 Edge12 = Vert1 - Vert2;
    float2 Edge20 = v02.xy;
    
    float C0 = Edge01.y * Vert0.x - Edge01.x * Vert0.y;
    float C1 = Edge12.y * Vert1.x - Edge12.x * Vert1.y;
    float C2 = Edge20.y * Vert2.x - Edge20.x * Vert2.y;

    // 5. Fill Convention (Top-Left Rule, 无分支版本)
    C0 -= saturate(Edge01.y + saturate(1.0f - Edge01.x));
    C1 -= saturate(Edge12.y + saturate(1.0f - Edge12.x));
    C2 -= saturate(Edge20.y + saturate(1.0f - Edge20.x));

    // 6. 选择遍历策略
    bool bScanLine = WaveActiveAnyTrue(MaxPixel.x - MinPixel.x > 4);

    if (bScanLine) {
        // === 扫描线算法（大三角形）===
        // 逐行计算边与扫描线的交点，只遍历覆盖的像素
        for (y = MinPixel.y; y <= MaxPixel.y; y++) {
            float x0 = ...; // 左交点
            float x1 = ...; // 右交点
            for (x = x0; x <= x1; x++) {
                WritePixel(OutVisBuffer64, PixelValue, uint2(x,y), ZX, ...);
                ZX += GradZ.x;
            }
            // 步进 Y
        }
    } else {
        // === 矩形框算法（小三角形）===
        // 遍历包围盒内所有像素，检测是否在三角形内
        for (y = MinPixel.y; y <= MaxPixel.y; y++) {
            for (x = MinPixel.x; x <= MaxPixel.x; x++) {
                if (min3(CX0, CX1, CX2) >= 0) {  // 在三角形内
                    WritePixel(OutVisBuffer64, PixelValue, uint2(x,y), ZX, ...);
                }
                // 步进 X（只需加法）
                CX0 -= Edge01.y;
                CX1 -= Edge12.y;
                CX2 -= Edge20.y;
                ZX += GradZ.x;
            }
            // 步进 Y
            CY0 += Edge01.x;
            CY1 += Edge12.x;
            CY2 += Edge20.x;
            ZY += GradZ.y;
        }
    }
}
```

### 2.4 Nanite 硬件光栅化的对比实现

```bash
硬件光栅化路径（大三角形 Cluster）：
  使用传统的 VS → Rasterizer → PS 管线
  VS: CommonRasterizerVS — 解压顶点，变换到裁剪空间
  PS: HWRasterizePS — 写入 PixelValue + Depth 到 Visibility Buffer

  两种 VS 模式：
  1. Primitive Shader 模式（AMD RDNA2+）
     → 类似 Mesh Shader，一次提交整个 Cluster
     → 128 线程处理 128 三角形 + 256 顶点
     → 支持 Vertex-to-Triangle Mask 优化

  2. 传统 VS 模式
     → VertexID / 3 = TriIndex, VertexID % 3 = 顶点在三角形中的索引
     → 每个 Cluster 作为一个 Instance（SV_InstanceID = VisibleIndex）
     → 每个 Instance 绘制 128×3 = 384 个顶点

  PS 输出：
    与软光栅化相同的 Visibility Buffer
    WritePixel(OutVisBuffer64, PixelValue, PixelPos, DeviceZ, ...);
    → 硬件深度测试 + 原子写入
```

### 2.5 Visibility Buffer 技术

```
Nanite 光栅化（无论软硬）输出的不是传统 GBuffer，而是 Visibility Buffer：

┌─────────────────────────────────────────────────────────────────┐
│  Visibility Buffer (uint64 per pixel)                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  高 32 位：Depth（深度值）                                       │
│  低 32 位：PixelValue                                           │
│    = (VisibleClusterIndex + 1) << 7 | TriangleID                │
│                                                                 │
```

### 2.7 Nanite 软光栅化的关键优化

```
1. GroupShared Memory 顶点共享
   → 每个 Cluster 的顶点只变换一次，存入 LDS
   → 128 个三角形共享 256 个顶点（平均每三角形 2 个新顶点）
   → 避免硬件管线中顶点重复变换的问题

2. 动态选择遍历策略
   → WaveActiveAnyTrue(MaxPixel.x - MinPixel.x > 4)
   → 如果 Wave 中任何三角形宽度 > 4 像素 → 全部用扫描线
   → 否则用矩形框法（小三角形更高效）
   → Wave 级别统一分支，避免 divergence

3. 限制最大光栅化范围
   → MaxPixel = min(MaxPixel, MinPixel + 63)
   → 防止异常大三角形阻塞 CS 线程
   → 超大三角形应该走硬件光栅化路径

4. 无分支 Fill Convention
   → C -= saturate(Edge.y + saturate(1.0f - Edge.x))
   → 避免 if/else 分支，GPU 友好

5. InterlockedMax 深度测试
   → 将 Depth 放在高位，TriID 放在低位
   → 一次原子操作同时完成深度测试和像素写入
   → 无需额外的 Depth Buffer

6. 1 Dispatch 处理所有软光栅 Cluster
   → 不像硬件光栅化需要逐 Cluster 发 Draw Call
   → GPU 调度开销极低
   → Indirect Dispatch：数量由 GPU Culling 阶段动态决定
```

# 算法

## 概述
软光栅化器是计算机图形学中将矢量图形转换为像素表示的核心组件。本文档详细介绍常用的光栅化算法及其实现原理。

## 1. 直线光栅化算法

### 1.1 DDA算法 (Digital Differential Analyzer)

DDA算法是最基础的直线绘制算法，通过数值微分的方式生成直线。

**算法原理：**
```cpp
// 假设直线从(x1,y1)到(x2,y2)，斜率m = (y2-y1)/(x2-x1)
float m = (y2 - y1) / (x2 - x1);
float y = y1;

for(int x = x1; x <= x2; x++) {
    write_pixel(x, round(y), line_color);
    y += m;  // 每次x增加1，y增加斜率m
}
```

**特点：**
- ✅ 实现简单，易于理解
- ❌ 每生成一个像素都需要一次浮点数加法运算
- ❌ 效率较低，存在浮点精度问题
- ❌ 需要四舍五入操作，可能产生锯齿

### 1.2 Bresenham算法 (布雷森汉姆算法)

Bresenham算法通过整数运算优化直线绘制，是硬件和软件光栅处理器的标准算法，由Jack Elton Bresenham于1962年发明。

#### 算法数学原理

**基本假设：**直线从(x0,y0)到(x1,y1)，斜率为0 ≤ m ≤ 1

**误差项推导：**
设当前像素位置为(x, y)，下一个候选像素为(x+1, y)或(x+1, y+1)
真实直线在x+1处的y坐标为：y_real = m*(x+1 - x0) + y0
误差项 d1 = (y + 1) - y_real = y + 1 - m*(x+1 - x0) - y0
误差项 d2 = y_real - y = m*(x+1 - x0) + y0 - y

令决策参数p = d1 - d2 = 2m(x+1 - x0) - 2(y - y0) - 1

**整数优化：**
为避免浮点运算，令Δx = x1 - x0, Δy = y1 - y0
则m = Δy/Δx
p = 2Δy(x+1 - x0) - 2Δx(y - y0) - Δx

令p' = p * Δx，得到整数决策参数：
p' = 2Δy(x+1 - x0) - 2Δx(y - y0) - Δx

#### 算法实现

**基本Bresenham算法（斜率0 ≤ m ≤ 1）：**
```cpp
void bresenham_line(int x0, int y0, int x1, int y1, Color color) {
    int dx = x1 - x0;
    int dy = y1 - y0;
    
    // 处理斜率大于1的情况
    bool steep = abs(dy) > abs(dx);
    if (steep) {
        std::swap(x0, y0);
        std::swap(x1, y1);
        std::swap(dx, dy);
    }
    
    // 处理反向绘制
    if (x0 > x1) {
        std::swap(x0, x1);
        std::swap(y0, y1);
        dx = -dx;
        dy = -dy;
    }
    
    int y = y0;
    int p = 2 * dy - dx;  // 初始决策参数
    int y_step = (dy > 0) ? 1 : -1;
    
    for (int x = x0; x <= x1; x++) {
        if (steep) {
            set_pixel(y, x, color);  // 交换x,y坐标
        } else {
            set_pixel(x, y, color);
        }
        
        if (p >= 0) {
            y += y_step;
            p -= 2 * dx;
        }
        p += 2 * dy;
    }
}
```

**通用Bresenham算法（支持所有斜率）：**

```cpp
void bresenham_line_general(int x0, int y0, int x1, int y1, Color color) {
    int dx = abs(x1 - x0);
    int dy = abs(y1 - y0);
    int sx = (x0 < x1) ? 1 : -1;
    int sy = (y0 < y1) ? 1 : -1;
    int err = dx - dy;
    
    while (true) {
        set_pixel(x0, y0, color);
        
        if (x0 == x1 && y0 == y1) break;
        
        int e2 = 2 * err;
        if (e2 > -dy) {
            err -= dy;
            x0 += sx;
        }
        if (e2 < dx) {
            err += dx;
            y0 += sy;
        }
    }
}
```

#### 算法特点分析

**优势：**
- ✅ 仅使用整数加法、减法和移位操作
- ✅ 效率极高，适合硬件实现
- ✅ 避免了浮点运算和四舍五入
- ✅ 生成的直线质量优秀
- ✅ 支持所有斜率方向的直线绘制

**数学特性：**
- 决策参数p的初始值：p0 = 2Δy - Δx
- 当p0 ≥ 0时，选择(x+1, y+1)，更新p = p + 2Δy - 2Δx
- 当p0 < 0时，选择(x+1, y  )，更新p = p + 2Δy

**硬件友好特性：**
- 无乘法运算，只有加法和移位
- 状态机简单，适合流水线处理
- 内存访问模式具有空间局部性

#### 应用扩展

**圆形绘制（Bresenham圆算法）：**
```cpp
void bresenham_circle(int xc, int yc, int r, Color color) {
    int x = 0, y = r;
    int p = 3 - 2 * r;
    
    while (x <= y) {
        // 绘制八分对称点
        set_pixel(xc + x, yc + y, color);
        set_pixel(xc - x, yc + y, color);
        set_pixel(xc + x, yc - y, color);
        set_pixel(xc - x, yc - y, color);
        set_pixel(xc + y, yc + x, color);
        set_pixel(xc - y, yc + x, color);
        set_pixel(xc + y, yc - x, color);
        set_pixel(xc - y, yc - x, color);
        
        if (p < 0) {
            p = p + 4 * x + 6;
        } else {
            p = p + 4 * (x - y) + 10;
            y--;
        }
        x++;
    }
}
```

**椭圆绘制：**
类似原理，但需要处理长短轴不同的情况。

## 2. 多边形填充算法

### 2.1 奇偶填充规则 (Even-Odd Rule)

从待填充点向任意方向发射射线，计算与多边形边界的交点数量：
- 交点为奇数：点在多边形内部
- 交点为偶数：点在多边形外部

### 2.2 种子填充算法 (Seed Fill)

从多边形内部的一个种子点开始，递归或迭代地填充相邻像素，直到遇到边界。

### 2.3 扫描线填充算法 (Scanline Fill)

逐行扫描屏幕，计算每行与多边形的交点，填充交点之间的像素段。

## 3. 三角形填充算法

### 3.1 Edge-Walking 算法

Edge-Walking是经典的三角形填充算法，通过遍历三角形的边来计算扫描线覆盖范围。

**算法伪代码：**
```cpp
void edge_walking(Vertex vertices[3]) {
    // 对三角形顶点按y坐标排序
    sort_vertices_by_y(vertices);
    
    // 初始化左右边的x坐标和斜率
    float x_left, x_right;
    float dx_left, dx_right;
    
    // 计算各边的斜率
    for (int i = 0; i < 2; i++) {  // 处理每对边
        initialize_edge_parameters(vertices, i, &x_left, &dx_left, &x_right, &dx_right);
        
        // 遍历扫描线
        for (int y = min_y; y <= max_y; y++) {
            // 填充当前扫描线
            for (int x = (int)x_left; x <= (int)x_right; x++) {
                set_pixel(x, y, fill_color);
            }
            
            // 更新左右边的x坐标
            x_left += dx_left;
            x_right += dx_right;
        }
    }
}
```

**算法特点：**
- ✅ 实现相对简单
- ✅ 效率较高
- ❌ 需要处理特殊情况（水平边、退化三角形等）

### 3.2 Edge-Equation 算法
基于边方程计算像素是否在三角形内部，通常与重心坐标结合使用。

## 4. 渲染模式

### 4.1 线框模式 (Wireframe Mode)
只绘制几何体的边，使用Bresenham算法生成高质量的线条。

**应用场景：**
- 调试和开发阶段
- 技术演示
- 风格化渲染

### 4.2 填充模式 (Fill Mode)
填充几何体的整个面片，使用三角形填充算法。

**实现方式：**
- Edge-Walking算法
- Edge-Equation算法
- 基于GPU的光栅化

## 5. 性能优化考虑

### 5.1 算法选择
- **移动设备**：优先使用整数运算的Bresenham算法
- **高质量要求**：考虑使用Edge-Equation算法
- **实时渲染**：结合硬件加速光栅化

### 5.2 内存访问优化
- 使用扫描线连贯性减少重复计算
- 预计算斜率等参数
- 批量处理像素

## 6. 实际应用建议

1. **游戏开发**：使用硬件光栅化，在Shader中实现自定义光栅化效果
2. **软件渲染器**：Bresenham算法 + Edge-Walking组合
