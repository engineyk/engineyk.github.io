# 1. 核心概念：什么是半透明 Overdraw

## 1.1 Overdraw 的本质

```
Overdraw = 同一个像素被 Fragment Shader 执行的次数

For opaque objects:
  Early-Z / Z-Prepass can reject occluded fragments BEFORE pixel shader
  → Overdraw ≈ 1x (ideal)

For translucent objects:
  NO depth write → NO Early-Z rejection
  Every overlapping translucent layer MUST execute pixel shader
  → Overdraw = N (N = number of overlapping translucent layers)

Key insight:
  Overdraw is a PIXEL-LEVEL metric, not a draw call metric
  Reducing draw calls does NOT necessarily reduce overdraw
```

## 1.2 为什么半透明 Overdraw 是性能杀手

```
Cost per pixel = Shader ALU + Texture Fetch + Blend Operation

For N overlapping translucent layers:
  Total cost = N × (Shader ALU + Texture Fetch) + N × Blend ROP

Example: 5 overlapping particle layers, each with 20 texture samples
  = 5 × 20 = 100 texture fetches per pixel
  = 5 × full lighting calculation
  = 5 × alpha blend operations

On mobile (TBR architecture):
  Each blend requires read-modify-write to tile memory
  Bandwidth = N × pixel_size × screen_area
  This is the #1 GPU bottleneck for particle-heavy scenes
```

## 1.3 Overdraw vs DrawCall — 关键区分

```
┌─────────────────────────────────────────────────────────────────┐
│  Metric     │ What it measures        │ Reduced by              │
├─────────────┼─────────────────────────┼─────────────────────────┤
│  DrawCall   │ CPU→GPU command count   │ Batching, Instancing    │
│  SetPass    │ PSO/Shader state switch │ Material sorting, PSO   │
│  Overdraw   │ Pixel shader executions │ Culling, sorting, LOD   │
│  Bandwidth  │ Memory read/write       │ Compression, resolution │
└─────────────────────────────────────────────────────────────────┘

CRITICAL: These are INDEPENDENT metrics!
  - Merging draw calls → reduces DrawCall, does NOT reduce Overdraw
  - Merging PSO → reduces SetPass, does NOT reduce Overdraw
  - Only reducing VISIBLE OVERLAPPING AREA reduces Overdraw
```

# 2. 半透合并后 Overdraw 更高还是更低？

## 2.1 直接结论

> **半透明合并（Mesh Merge / Static Batching / Dynamic Batching）不会改变 Overdraw。**
> Overdraw 取决于屏幕上重叠的半透明像素层数，与 DrawCall 数量无关。

```
Before merge: 10 translucent meshes → 10 draw calls
After merge:  1 merged mesh        → 1 draw call

But if the same pixels are covered by the same geometry:
  Overdraw BEFORE merge = N layers per pixel
  Overdraw AFTER merge  = N layers per pixel (UNCHANGED!)

The GPU still processes every triangle that covers each pixel.
Merging only reduces CPU overhead (draw call submission).
```

## 2.2 合并可能导致 Overdraw 变高的情况

```
Scenario: Mesh merge expands bounding box

Before merge:
  ┌──┐
  │A │  ┌──┐
  └──┘  │B │
        └──┘
  A and B don't overlap → Overdraw = 1x everywhere

After merge into single mesh (with shared material):
  The GPU processes the ENTIRE merged mesh as one draw
  If the merged bounding box is larger:
    - Frustum culling becomes less effective
    - More triangles survive culling
    - Potentially MORE pixels are shaded

  ┌─────────┐
  │ A ... B │  ← merged bounding box is larger
  └─────────┘

  If A and B were far apart, merging HURTS culling efficiency
  → More triangles rendered → potentially HIGHER overdraw
```

## 2.3 合并可能间接降低 Overdraw 的情况

```
Scenario: Reduced CPU overhead allows better frame budget

Before merge:
  CPU: 8ms (draw call submission)
  GPU: 6ms (actual rendering)
  Total: 14ms → must reduce quality → more aggressive LOD

After merge:
  CPU: 2ms (fewer draw calls)
  GPU: 6ms (same rendering)
  Total: 8ms → budget headroom → can afford better culling

Indirect benefit:
  Freed CPU time can be used for:
  - Better occlusion culling (Hi-Z, software occlusion)
  - Per-particle distance culling
  - Dynamic particle count reduction
  → These ACTUALLY reduce overdraw
```

## 2.4 总结表格

```
┌──────────────────────────────────────────────────────────────────┐
│  Merge Type          │ DrawCall │ Overdraw │ Notes               │
├──────────────────────┼──────────┼──────────┼─────────────────────┤
│  Static Batching     │ ↓↓       │ = or ↑   │ Larger bbox hurts   │
│  Dynamic Batching    │ ↓        │ =        │ Same geometry       │
│  GPU Instancing      │ ↓↓↓      │ =        │ Same mesh repeated  │
│  SRP Batcher         │ ↓↓       │ =        │ Only reduces SetPass│
│  Mesh Merge (artist) │ ↓        │ = or ↑   │ Depends on layout   │
│  Material Merge      │ ↓        │ = or ↑   │ Atlas may waste area│
└──────────────────────────────────────────────────────────────────┘

Key takeaway:
  合并优化的是 CPU 端的 DrawCall/SetPass 开销
  Overdraw 是 GPU 端的像素级问题，合并无法解决
```

# 3. 合并 PSO 渲染后 Overdraw 会降低吗？

## 3.1 什么是 PSO（Pipeline State Object）

```
PSO = Pipeline State Object (D3D12 / Vulkan / Metal)

A PSO encapsulates:
  - Vertex Shader
  - Pixel Shader
  - Blend State (alpha blend mode)
  - Depth/Stencil State
  - Rasterizer State (cull mode, fill mode)
  - Render Target Format
  - Input Layout (vertex format)

In UE5, PSO is managed by FGraphicsPipelineStateInitializer:
  - Each unique combination of above states = 1 PSO
  - Switching PSO = expensive GPU state change ("SetPass Call")
```

## 3.2 PSO 合并/排序的作用

```
PSO "merging" is actually PSO SORTING:
  Sort draw calls by PSO to minimize state switches

Before sorting:
  Draw 1: PSO_A (material_red, translucent)
  Draw 2: PSO_B (material_blue, translucent)
  Draw 3: PSO_A (material_red, translucent)  ← state switch back!
  Draw 4: PSO_B (material_blue, translucent)
  = 4 PSO switches

After sorting:
  Draw 1: PSO_A (material_red, translucent)
  Draw 3: PSO_A (material_red, translucent)  ← no switch!
  Draw 2: PSO_B (material_blue, translucent)
  Draw 4: PSO_B (material_blue, translucent)  ← no switch!
  = 2 PSO switches (reduced by 50%)
```

## 3.3 PSO 合并对 Overdraw 的影响

> **PSO 合并/排序不会降低 Overdraw。**

```
PSO sorting reduces:
  ✅ SetPass Call count (GPU state switch overhead)
  ✅ Driver overhead (fewer pipeline state changes)
  ✅ CPU command buffer size

PSO sorting does NOT reduce:
  ❌ Overdraw (same pixels still shaded same number of times)
  ❌ Triangle count (same geometry submitted)
  ❌ Texture fetch count (same shaders executed)

Why?
  PSO sorting changes the ORDER of draw calls, not the CONTENT.
  The same triangles cover the same pixels regardless of order.
  Every translucent fragment still executes its pixel shader.
```

## 3.4 PSO 排序对半透明的特殊问题

```
CRITICAL CONFLICT: PSO sorting vs Back-to-Front sorting

Translucent objects REQUIRE back-to-front sorting for correct blending:
  Alpha blend: FinalColor = SrcColor × SrcAlpha + DstColor × (1 - SrcAlpha)
  This is ORDER-DEPENDENT!

PSO sorting groups by material:
  All red objects first, then all blue objects
  → BREAKS back-to-front order!
  → Incorrect visual result (blending artifacts)

UE5 solution (from RenderTranslucency source):
  Translucent objects are sorted BACK-TO-FRONT, not by PSO
  This means MORE PSO switches for translucent rendering
  This is an intentional trade-off: correctness > performance
```

```cpp
// UE5 source: FBasePassMeshProcessor::ShouldDraw
// Translucent objects are routed to specific passes:
//   TPT_TranslucencyStandard
//   TPT_TranslucencyAfterDOF
//   TPT_TranslucencyAfterMotionBlur
// Within each pass, objects are sorted by distance (back-to-front)
// NOT by PSO/material
```

## 3.5 PSO Precaching 的真正作用

```
UE5 PSO Precaching (r.PSOPrecaching):
  Pre-creates PSO objects during loading
  Avoids runtime PSO compilation hitches

From UE5 source (CollectPSOInitializers):
  - Collects all possible PSO combinations for materials
  - Pre-compiles them on background threads
  - Stores in PipelineStateCache

This eliminates HITCHES, not OVERDRAW:
  Without precaching: first use of PSO → 50-200ms hitch
  With precaching: PSO already compiled → 0ms hitch
  Overdraw is identical in both cases
```

## 3.6 总结

```
┌─────────────────────────────────────────────────────────────────┐
│  PSO Operation      │ SetPass │ Overdraw │ Hitch │ Visual       │
├─────────────────────┼─────────┼──────────┼───────┼──────────────┤
│  PSO Sorting        │ ↓↓      │ =        │ =     │ May break!   │
│  PSO Precaching     │ =       │ =        │ ↓↓↓   │ =            │
│  PSO Deduplication  │ ↓       │ =        │ ↓     │ =            │
│  Material Merging   │ ↓↓      │ = or ↑   │ ↓     │ =            │
└─────────────────────────────────────────────────────────────────┘

Answer: PSO 合并/排序不会降低 Overdraw
  - PSO 优化的是 GPU 状态切换开销（SetPass）
  - PSO Precaching 优化的是首次使用时的编译卡顿
  - Overdraw 只能通过减少重叠的半透明像素面积来降低
```

# 4. 单个半透材质重叠位置 Overdraw 分析

## 4.1 核心问题

> 同一个半透明材质的多个 Mesh 在屏幕上重叠时，Overdraw 是什么情况？

## 4.2 逐层分析

```
Scenario: 3 translucent quads with SAME material, overlapping at pixel P

  Camera
    │
    ▼
  ┌─────┐
  │Quad1│ (z=10, alpha=0.5)
  ├─────┤
  │Quad2│ (z=20, alpha=0.5)
  ├─────┤
  │Quad3│ (z=30, alpha=0.5)
  └─────┘
  ─────── Opaque Background (z=50)

At pixel P (where all 3 overlap):
  Step 1: Render background → FrameBuffer = BG_Color
  Step 2: Render Quad3 (farthest) → PS executes, blend with BG
  Step 3: Render Quad2 (middle)   → PS executes, blend with result
  Step 4: Render Quad1 (nearest)  → PS executes, blend with result

  Overdraw at pixel P = 3 (three PS executions for translucent)
  Total PS executions = 1 (BG) + 3 (translucent) = 4

At pixel Q (only Quad1 covers):
  Overdraw = 1

At pixel R (Quad1 + Quad2 overlap):
  Overdraw = 2
```

## 4.3 同材质 vs 不同材质 — Overdraw 完全相同

```
IMPORTANT: Overdraw does NOT depend on whether materials are same or different!

Case A: 3 quads, SAME material (Material_Red)
  Pixel P overdraw = 3

Case B: 3 quads, DIFFERENT materials (Red, Blue, Green)
  Pixel P overdraw = 3

The pixel shader runs once per triangle per pixel, regardless of material.
Same material only helps with:
  - DrawCall batching (GPU Instancing)
  - PSO state switches (fewer SetPass calls)
  - SRP Batcher efficiency

But overdraw is IDENTICAL.
```

## 4.4 自身重叠（Self-Overlap）

```
Special case: A SINGLE mesh with self-overlapping triangles

Example: A translucent sphere viewed from outside
  Front-facing triangles overlap back-facing triangles
  If two-sided rendering is enabled:
    Front triangles: 1 layer
    Back triangles: 1 layer
    Overlap area: 2 layers → Overdraw = 2

Example: A translucent ribbon/trail particle
  The ribbon curves back on itself
  Where it overlaps: Overdraw = number of overlapping segments

  ╭──────╮
  │      │  ← 1 layer (overdraw = 1)
  │  ╭───┤
  │  │   │  ← 2 layers (overdraw = 2, self-overlap!)
  ╰──┤   │
     ╰───╯

This is a SINGLE draw call, SINGLE material, but overdraw > 1
Proves: Overdraw is purely geometric, not material/drawcall related
```

## 4.5 GPU 如何处理半透明像素

```
GPU Rasterization Pipeline for Translucent:

  Vertex Shader → Rasterizer → Fragment Shader → Blend → FrameBuffer

For each triangle covering pixel P:
  1. Rasterizer generates fragment at P
  2. Depth test: READ-ONLY (no depth write for translucent)
     - If fragment is behind opaque geometry → DISCARD (Early-Z works!)
     - If fragment is in front of opaque geometry → CONTINUE
  3. Fragment Shader executes (texture fetch, lighting, etc.)
  4. Alpha Blend with current FrameBuffer value
  5. Write blended result back to FrameBuffer

Key observations:
  ✅ Early-Z CAN reject translucent fragments behind OPAQUE geometry
  ❌ Early-Z CANNOT reject translucent fragments behind OTHER translucent
  ❌ No mechanism to skip overlapping translucent layers
```

## 4.6 数学模型：Overdraw 计算

```
For a screen with resolution W × H:

Total pixel shader invocations for translucent:
  PSI_total = Σ(for each pixel P) layers(P)

Average overdraw:
  Overdraw_avg = PSI_total / (W × H)

For N translucent quads of size S × S pixels, randomly placed:
  Expected overlap at any pixel ≈ N × (S²) / (W × H)
  Overdraw_avg ≈ N × S² / (W × H)

Example:
  Screen: 1920 × 1080 = 2,073,600 pixels
  100 particles, each 200 × 200 pixels = 40,000 pixels
  Expected overdraw = 100 × 40,000 / 2,073,600 ≈ 1.93x

  If particles cluster in center (50% of screen):
  Expected overdraw in cluster = 100 × 40,000 / 1,036,800 ≈ 3.86x
  → Clustering DOUBLES the overdraw!
```

## 4.7 可视化 Overdraw

```
UE5 Visualization:
  Viewport → Optimization Viewmodes → Shader Complexity
  Or: r.ShaderComplexityMode 1

  Color coding:
    Green  = low overdraw (1-2 layers)
    Yellow = medium (3-4 layers)
    Red    = high (5-8 layers)
    White  = extreme (8+ layers) ← performance disaster

Unity Visualization:
  Scene View → Overdraw mode
  Or: Frame Debugger → check pixel shader invocation count

RenderDoc:
  Pixel History → shows every draw call that touched a pixel
  Count the translucent draws = exact overdraw at that pixel
```

# 5. UE5 半透明渲染管线源码分析

## 5.1 RenderTranslucency 调度流程

```
From UE5 source (TranslucentRendering.cpp):

FDeferredShadingSceneRenderer::RenderTranslucency()
│
├─ Check ViewsToRender flags (UnderWater | AboveWater)
│
├─ Copy SceneColor (for refraction/distortion)
│
├─ Create/reuse depth texture
│   ├─ If scaling: create downsampled depth
│   └─ Else: reuse scene depth
│
├─ If AllowTranslucencyAfterDOF():
│   ├─ RenderTranslucencyInner(TPT_TranslucencyStandard)
│   ├─ RenderTranslucencyInner(TPT_TranslucencyStandardModulate)
│   ├─ [Optional] RenderHairComposition()
│   ├─ RenderTranslucencyInner(TPT_TranslucencyAfterDOF)
│   ├─ RenderTranslucencyInner(TPT_TranslucencyAfterDOFModulate)
│   └─ RenderTranslucencyInner(TPT_TranslucencyAfterMotionBlur)
│
└─ Else:
    └─ RenderTranslucencyInner(TPT_AllTranslucency)

Each pass renders ALL translucent objects in that category
Objects within each pass are sorted BACK-TO-FRONT
→ Every overlapping layer executes its pixel shader
→ Overdraw = number of overlapping layers (unavoidable)
```

## 5.2 半透明排序 vs PSO 排序的冲突

```cpp
// From FBasePassMeshProcessor::ShouldDraw (BasePassRendering.cpp)
// UE5 routes translucent objects to specific passes:

case ETranslucencyPass::TPT_TranslucencyStandard:
    bShouldDraw = !Material.IsTranslucencyAfterDOFEnabled()
                && !Material.IsTranslucencyAfterMotionBlurEnabled();
    break;

case ETranslucencyPass::TPT_TranslucencyAfterDOF:
    bShouldDraw = Material.IsTranslucencyAfterDOFEnabled()
                && BlendMode != BLEND_ColoredTransmittanceOnly;
    break;

case ETranslucencyPass::TPT_TranslucencyAfterMotionBlur:
    bShouldDraw = Material.IsTranslucencyAfterMotionBlurEnabled();
    break;

// Within each pass: sorted by distance, NOT by PSO
// This means frequent PSO switches within translucent rendering
// Trade-off: visual correctness > PSO efficiency
```

## 5.3 UE5 混合模式与 Overdraw

```
From SetupMaterialEnvironment (MaterialShared.cpp):

UE5 Blend Modes:
  BLEND_Opaque          → No overdraw (Early-Z)
  BLEND_Masked          → No overdraw (Early-Z, but discard cost)
  BLEND_Translucent     → Full overdraw (alpha blend)
  BLEND_Additive        → Full overdraw (additive blend)
  BLEND_Modulate        → Full overdraw (multiply blend)
  BLEND_AlphaComposite  → Full overdraw (premultiplied alpha)
  BLEND_AlphaHoldout    → Full overdraw (holdout mask)

ALL translucent blend modes cause overdraw.
The blend mode affects HOW pixels combine, not WHETHER they execute.
```

# 6. 优化策略与最佳实践

## 6.1 真正降低 Overdraw 的方法

```
│  Strategy              │ Overdraw Reduction │ Complexity         │
├────────────────────────┼────────────────────┼────────────────────┤
```

## 6.2 粒子 Overdraw 优化（UE5 源码验证）

```
From UE5 source (Niagara/Cascade):

1. SubUV Cutout Geometry:
   "optimal bounding geometry for each SubUV frame will be used
    when rendering the sprites instead of full quads"
   "This reduction in overdraw can reduce the GPU cost by 2x or 3x"

   Instead of:          Use:
   ┌──────────┐         ╭──╮
   │  ╭──╮    │         │  │
   │  │▓▓│    │    →    │▓▓│
   │  ╰──╯    │         ╰──╯
   └──────────┘
   Full quad (100%)     Tight bounds (~30-50%)
   Overdraw: high       Overdraw: 2-3x lower

2. Cutout Texture Warning:
   "Missing Cutout Texture for Sub UV Image,
    which might lead to overdraw and reduce performance"
   → Always set cutout texture for particle sprites!

3. Distance-based particle count:
   Far particles → reduce MaxParticles
   Very far → replace with impostor or disable
```

## 6.3 半分辨率半透明渲染

```
UE5 supports separate translucency at reduced resolution:

From RenderTranslucency source:
  SeparateTranslucencyDimensions.Scale can be < 1.0
  → Renders translucent at lower resolution
  → Upscales and composites with full-res scene

Performance impact:
  Scale = 0.5 → 75% fewer pixels to shade
  Scale = 0.75 → 44% fewer pixels to shade

  If overdraw = 5x at full res:
    Full res:  5 × 1920×1080 = 10,368,000 PS invocations
    Half res:  5 × 960×540  =  2,592,000 PS invocations (75% reduction!)

Trade-off: slight blur on translucent edges
Recommended for: particle-heavy scenes, mobile
```

## 6.4 移动端特殊优化

```
Mobile (TBR/TBDR) specific concerns:

1. Tile Memory Pressure:
   Each translucent blend = read-modify-write in tile memory
   High overdraw → tile memory thrashing → bandwidth spike

2. Optimization strategies:
   ✅ Limit particle count aggressively (mobile: max 50-100 visible)
   ✅ Use Additive blend instead of Alpha blend where possible
      (Additive is order-independent → can sort by PSO!)
   ✅ Use half precision (half4) for all translucent shaders
   ✅ Minimize texture samples in translucent shaders
   ✅ Use pre-multiplied alpha to reduce blend operations
   ✅ Avoid reading SceneColor in translucent shaders
      (forces tile resolve → extremely expensive on TBR)

3. Budget guidelines:
   ┌────────────────────────────────────────────────┐
   │  Platform    │ Max Overdraw │ Max Particles    │
   ├──────────────┼──────────────┼──────────────────┤
   │  High-end PC │ 8-10x        │ 1000+            │
   │  Console     │ 4-6x         │ 500              │
   │  Mobile High │ 2-3x         │ 100              │
   │  Mobile Low  │ 1-2x         │ 30               │
   └────────────────────────────────────────────────┘
```

# 7. 面试高频问答

## Q1: 半透合并后 Overdraw 更高还是更低？

```
Answer:
  半透明合并（Mesh Merge / Batching）不改变 Overdraw。

  Overdraw 是像素级指标，取决于屏幕上重叠的半透明层数。
  合并只减少 DrawCall（CPU 端开销），不减少像素着色器执行次数。

  特殊情况：
  - 合并后包围盒变大 → 剔除效率降低 → Overdraw 可能反而增加
  - 合并释放 CPU 预算 → 可用于更好的剔除 → 间接降低 Overdraw

  核心结论：要降低 Overdraw，必须减少重叠的半透明像素面积，
  而不是减少 DrawCall。
```

## Q2: 合并 PSO 渲染后 Overdraw 会降低吗？

```
Answer:
  不会。PSO 合并/排序降低的是 SetPass Call（GPU 状态切换开销），
  不是 Overdraw。

  PSO 排序改变的是 DrawCall 的执行顺序，不改变渲染内容。
  同样的三角形覆盖同样的像素，像素着色器执行次数不变。

  而且对于半透明渲染，PSO 排序还有一个致命问题：
  半透明必须按距离从后向前排序（Back-to-Front）才能正确混合，
  如果按 PSO 排序会破坏混合顺序，导致视觉错误。
  UE5 源码中半透明渲染明确按距离排序，而非按 PSO 排序。

  PSO Precaching 的作用是消除首次使用时的编译卡顿（hitch），
  同样不影响 Overdraw。
```

## Q3: 单个半透材质重叠位置 Overdraw 是什么情况？

```
Answer:
  Overdraw = 该像素位置上重叠的半透明三角形层数。

  与材质是否相同无关：
  - 3 个相同材质的 Mesh 重叠 → Overdraw = 3
  - 3 个不同材质的 Mesh 重叠 → Overdraw = 3
  - 1 个 Mesh 自身重叠（如弯曲的飘带）→ Overdraw = 重叠段数

  GPU 对每个覆盖该像素的三角形都会执行一次 Fragment Shader，
  然后进行 Alpha Blend。这是半透明渲染的本质要求，无法避免。

  唯一能被跳过的情况：
  半透明片段在不透明几何体后面 → Early-Z 可以剔除
  但半透明片段之间无法互相剔除（因为不写深度）
```

## Q4: 如何有效降低半透明 Overdraw？

```
Answer (by priority):
  P0: 限制粒子大小和数量（最直接有效）
  P0: 使用 SubUV Cutout Geometry（UE5 实测降低 2-3 倍）
  P1: 距离 LOD（远处减少粒子数/简化 Shader）
  P1: 半分辨率渲染半透明（降低 75% 像素着色开销）
  P2: 屏幕空间粒子密度限制
  P2: 简化半透明 Shader（减少纹理采样和 ALU）
  P3: 使用 Additive 替代 Alpha Blend（可按 PSO 排序）
  P3: OIT（Order-Independent Transparency）消除排序问题

  不有效的方法：
  ❌ 合并 DrawCall（不影响 Overdraw）
  ❌ PSO 排序（不影响 Overdraw，还可能破坏混合顺序）
  ❌ 增加 GPU Instancing（不影响 Overdraw）
```

## Q5: TBR 架构下半透明 Overdraw 为什么特别严重？

```
Answer:
  TBR (Tile-Based Rendering) 架构（iOS/大部分 Android）：
  - 屏幕分为 16×16 或 32×32 像素的 Tile
  - 每个 Tile 在片上缓存（On-Chip Memory）中完成渲染
  - 最终结果写回主存

  半透明 Overdraw 在 TBR 上的问题：
  1. 每次 Alpha Blend = 读取 Tile Memory + 计算 + 写回
  2. 高 Overdraw → Tile Memory 频繁读写 → 带宽压力
  3. 如果半透明 Shader 读取 SceneColor（如折射）：
     → 强制 Tile Resolve（将整个 FrameBuffer 写回主存）
     → 极其昂贵，相当于一次全屏 Blit

  对比 IMR (Immediate Mode Rendering, PC GPU)：
  - 直接读写 FrameBuffer（显存）
  - 带宽本身就高，Overdraw 影响相对较小
  - 但仍然是 ALU 和纹理采样的浪费

  结论：移动端对半透明 Overdraw 的敏感度是 PC 的 3-5 倍
```

# 8. 延迟渲染管线中的 Overdraw

## 8.1 延迟渲染 vs 前向渲染 — Overdraw 对比总览

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Rendering Stage     │ Forward Rendering    │ Deferred Rendering         │
├──────────────────────┼──────────────────────┼────────────────────────────┤
│  Opaque BasePass     │ Overdraw × N lights  │ Overdraw only GBuffer      │
│  Lighting            │ Per-object per-light │ Per-pixel (fullscreen)     │
│  Translucent         │ Same (back-to-front) │ Same (back-to-front)       │
│  Total Overdraw      │ HIGH (O × L)         │ LOW for opaque, same trans │
│  Bandwidth           │ Lower (single RT)    │ Higher (MRT GBuffer)       │
└──────────────────────────────────────────────────────────────────────────┘

Key insight:
  Deferred rendering was INVENTED to solve the overdraw problem
  in multi-light scenarios. It decouples geometry from lighting.
```

## 8.2 延迟渲染管线各阶段 Overdraw 分析

```
UE5 Deferred Rendering Pipeline:

  ┌────────────────────────────────────────────────────────────────┐
  │ Stage 1: Z-PrePass (Depth Only)                                │
  │   Purpose: "Draw a depth pass to avoid overdraw in the other   │
  │            passes" (UE5 source comment)                        │
  │   Overdraw: YES, but CHEAP (depth-only, no pixel shader)       │
  │   Cost per pixel: ~0 (only depth write, no color/texture)      │
  ├────────────────────────────────────────────────────────────────┤
  │ Stage 2: BasePass (GBuffer Fill)                               │
  │   Purpose: Write material properties to GBuffer (MRT)          │
  │   Overdraw: MINIMAL (Early-Z rejects occluded fragments)       │
  │   With Z-PrePass: Overdraw ≈ 1x (ideal)                        │
  │   Without Z-PrePass: Overdraw depends on draw order            │
  ├────────────────────────────────────────────────────────────────┤
  │ Stage 3: Deferred Lighting                                     │
  │   Purpose: Calculate lighting per-pixel from GBuffer           │
  │   Overdraw: 1x per light (fullscreen or light volume)          │
  │   Key advantage: Light count does NOT multiply geometry cost   │
  ├────────────────────────────────────────────────────────────────┤
  │ Stage 4: Translucent Pass                                      │
  │   Purpose: Render translucent objects                          │
  │   Overdraw: SAME as forward (N layers per pixel)               │
  │   This is the ONLY stage where overdraw is still a problem     │
  └────────────────────────────────────────────────────────────────┘
```

## 8.3 Z-PrePass 如何消除不透明 Overdraw

```
UE5 Z-PrePass modes (from CVarEarlyZPass source):
  r.EarlyZPass = 0: OFF (no depth prepass)
  r.EarlyZPass = 1: Good occluders only (large, non-masked)
  r.EarlyZPass = 2: All opaque (including masked)
  r.EarlyZPass = 3: Heuristic (default, engine decides)

From UE5 source (DepthRendering.cpp):
  ShouldRenderPrePass():
    return EarlyZPassMode == DDM_MaskedOnly
        || EarlyZPassMode == DDM_AllOpaque
        || EarlyZPassMode == DDM_AllOpaqueNoVelocity;

  RenderPrePass():
    // "Draw a depth pass to avoid overdraw in the other passes."
    if (DepthPass.EarlyZPassMode != DDM_None)
        RenderDepthPass(EMeshPass::DepthPass);

How it eliminates overdraw:

  Without Z-PrePass:
    BasePass renders objects in arbitrary order
    Object A (z=10) renders first → PS executes, writes GBuffer
    Object B (z=5) renders second → PS executes, OVERWRITES GBuffer
    Object A's PS execution was WASTED → Overdraw!

  With Z-PrePass:
    Step 1: Depth-only pass fills Z-Buffer (very cheap)
    Step 2: BasePass with depth test = EQUAL
    Only fragments at EXACT depth pass → Overdraw = 1x

  ┌──────────────────────────────────────────────────────────┐
  │  Z-PrePass Mode    │ BasePass Overdraw │ PrePass Cost    │
  ├────────────────────┼───────────────────┼─────────────────┤
  │  OFF (r.EarlyZ=0)  │ 1.5-3x (random)  │ 0               │
  │  Occluders (=1)    │ ~1.2x             │ Low (few objs)  │
  │  All Opaque (=2)   │ ≈1.0x (ideal)     │ Medium          │
  │  Heuristic (=3)    │ ~1.1x             │ Adaptive        │
  └──────────────────────────────────────────────────────────┘
```

## 8.4 BasePass GBuffer 写入的 Overdraw 代价

```
Deferred BasePass writes to MULTIPLE render targets (MRT):

From UE5 source (CalculateDerivedMaterialParameters):
  USES_GBUFFER = (FEATURE_LEVEL >= SM4 && (SOLID || MASKED) && !FORWARD_SHADING)

  If USES_GBUFFER:
    MRT0: SceneColor (Emissive + Indirect)
    MRT1: WorldNormal (encoded)
    MRT2: Metallic / Specular / Roughness
    MRT3: BaseColor / ShadingModelID / SelectiveOutputMask
    MRT4: Velocity (optional)
    MRT5: CustomData (SSS, ClearCoat, etc.)
    MRT6: PrecomputedShadowFactor (optional)

  Total: 4-7 render targets written per pixel!

Overdraw cost in BasePass (per wasted pixel):
  Forward: 1 RT write × shader cost
  Deferred: 4-7 RT writes × shader cost

  → GBuffer overdraw is MORE EXPENSIVE per pixel than forward!
  → This is WHY Z-PrePass is critical for deferred rendering
  → Without Z-PrePass, deferred BasePass overdraw wastes
     massive bandwidth writing to 4-7 MRTs

Bandwidth calculation:
  GBuffer per pixel ≈ 4 × RGBA8 + Depth = 20 bytes
  At 1080p: 1920 × 1080 × 20 = ~40 MB per frame
  With 2x overdraw: 80 MB (doubled!)
  With Z-PrePass (1x): 40 MB + 8 MB (depth prepass) = 48 MB
  → Z-PrePass SAVES bandwidth even though it adds an extra pass
```

## 8.5 延迟光照 Pass 的 Overdraw

```
Deferred Lighting: Per-light fullscreen/volume passes

From UE5 source (RenderDeferredSinglePass):
  // Subpass 1: BasePass (GBuffer fill)
  // Subpass 2: Decals
  // Subpass 3: Deferred Lighting ← lighting happens HERE
  MobileDeferredShadingPass(RHICmdList, ...)

Lighting Overdraw:

  Directional Light:
    Fullscreen quad → every pixel lit once
    Overdraw = 1x per directional light
    N directional lights → N fullscreen passes

  Point/Spot Light:
    Light volume (sphere/cone) rendered
    Only pixels INSIDE volume are lit
    Overdraw = number of overlapping light volumes at each pixel

  ┌──────────────────────────────────────────────────────────┐
  │  Light Type      │ Overdraw Pattern                      │
  ├──────────────────┼───────────────────────────────────────┤
  │  Directional     │ 1x per light (fullscreen)             │
  │  Point Light     │ Pixels inside sphere volume           │
  │  Spot Light      │ Pixels inside cone volume             │
  │  Rect Light      │ Pixels inside projected rect          │
  └──────────────────────────────────────────────────────────┘

  Forward vs Deferred lighting overdraw:

  Forward (N objects, L lights):
    Each object rendered L times (multi-pass) or
    Each object evaluates L lights in shader (single-pass)
    Total PS invocations = N_pixels × L (worst case)

  Deferred (N objects, L lights):
    BasePass: N_pixels × 1 (with Z-PrePass)
    Lighting: Screen_pixels × L (but only lit pixels)
    Total = N_pixels + Screen_pixels × L

  When N_objects >> Screen_pixels (dense scenes):
    Deferred wins MASSIVELY on overdraw
    Forward: 10000 objects × 50 lights = 500,000 PS per pixel
    Deferred: 10000 objects × 1 + 1 × 50 = 10,051 PS per pixel
```

## 8.6 延迟渲染中半透明仍然有 Overdraw

```
CRITICAL: Deferred rendering does NOT help translucent overdraw!

Why?
  Translucent objects CANNOT write to GBuffer:
    - GBuffer stores ONE material per pixel (winner-takes-all)
    - Translucent needs to BLEND multiple layers
    - Alpha blending is inherently order-dependent

  From UE5 source (CalculateDerivedMaterialParameters):
    USES_GBUFFER = (SOLID || MASKED) && !FORWARD_SHADING
    // Translucent materials do NOT use GBuffer!

  Translucent objects in deferred pipeline:
    - Rendered AFTER deferred lighting pass
    - Use FORWARD shading (even in deferred pipeline!)
    - Sorted back-to-front
    - Each layer executes full pixel shader
    - Overdraw = N layers (same as pure forward)

  From UE5 source (RenderDeferredSinglePass):
    // Subpass 1: BasePass (opaque → GBuffer, deferred)
    // Subpass 2: Decals
    // Subpass 3: Deferred Lighting
    // Subpass 4: Fog
    // Subpass 5: RenderTranslucency ← FORWARD shading!

  This means:
    ┌────────────────────────────────────────────────────────┐
    │  Object Type   │ Deferred Overdraw │ Forward Overdraw  │
    ├────────────────┼───────────────────┼───────────────────┤
    │  Opaque        │ ≈1x (Z-PrePass)   │ 1.5-3x (random)   │
    │  Opaque+Lights │ 1x + L lights     │ N × L (disaster)  │
    │  Translucent   │ N layers (same!)  │ N layers (same!)  │
    │  Particles     │ N layers (same!)  │ N layers (same!)  │
    └────────────────────────────────────────────────────────┘

  Conclusion:
    延迟渲染解决了不透明物体的 Overdraw 问题
    但半透明 Overdraw 在延迟和前向管线中完全相同
    半透明仍然是延迟渲染管线中的性能瓶颈
```

## 8.7 延迟渲染特有的 Overdraw 来源

```
Beyond standard translucent overdraw, deferred has unique sources:

1. Decal Overdraw:
   Deferred decals are rendered as screen-space passes
   Each decal = 1 draw call covering its projected area
   Overlapping decals → overdraw on GBuffer

   From UE5 source (RenderDeferredSinglePass):
     RenderDecals(RHICmdList, View, DecalViewInfo, ...)
     // Decals write to SceneColor + GBuffer
     // Multiple overlapping decals = GBuffer overdraw

2. Light Volume Overdraw:
   Point/Spot lights render their bounding volumes
   Overlapping light volumes → lighting overdraw
   100 overlapping point lights at one pixel = 100x lighting PS

3. Shadow Projection Overdraw:
   Each shadow-casting light projects shadows
   From UE5 source (RenderDeferredShadowProjections):
     RenderShadowProjections(...)
     ApplyVirtualShadowMapProjectionForLight(...)
     RenderCapsuleDirectShadows(...)
   Each shadow projection = fullscreen or volume pass

4. Post-Process Overdraw:
   SSR, SSAO, SSS, Bloom, DOF...
   Each = fullscreen pass reading/writing screen-size textures
   Not "overdraw" in traditional sense, but similar bandwidth cost

Summary of deferred-specific overdraw:
  ┌───────────────────────────────────────────────────────────┐
  │  Source              │ Overdraw Type    │ Mitigation      │
  ├──────────────────────┼──────────────────┼─────────────────┤
  │  GBuffer fill        │ MRT bandwidth    │ Z-PrePass       │
  │  Deferred decals     │ GBuffer rewrite  │ Limit count     │
  │  Light volumes       │ Lighting PS      │ Tiled/Clustered │
  │  Shadow projections  │ Screen-space     │ VSM caching     │
  │  Post-process        │ Fullscreen       │ Half-res, skip  │
  │  Translucent         │ Same as forward  │ Same strategies │
  └───────────────────────────────────────────────────────────┘
```

## 8.8 Tiled/Clustered Deferred — 解决光照 Overdraw

```
Problem: Standard deferred renders each light as separate pass
  100 lights = 100 fullscreen/volume passes = massive overdraw

Solution: Tiled Deferred / Clustered Deferred

Tiled Deferred:
  1. Divide screen into tiles (e.g., 16×16 pixels)
  2. For each tile, find which lights affect it
  3. In ONE pass, evaluate ALL lights per tile
  → Overdraw = 1x regardless of light count!

  ┌────┬────┬────┬────┐
  │ L1 │L1,2│ L2 │    │
  │ L3 │L2,3│    │ L4 │
  ├────┼────┼────┼────┤
  │ L3 │    │ L4 │ L4 │
  │    │ L5 │L4,5│    │
  └────┴────┴────┴────┘
  Each tile evaluates only its relevant lights
  No redundant fullscreen passes

Clustered Deferred (UE5 approach):
  Extends tiling to 3D (depth slices)
  Better for varying depth complexity
  Used by UE5 for local lights

  Performance comparison:
  ┌───────────────────────────────────────────────────────────┐
  │  Method              │ 10 lights │ 100 lights│ 1000 lights│
  ├──────────────────────┼───────────┼───────────┼────────────┤
  │  Standard Deferred   │ 10 passes │ 100 passes│ 1000 passes│
  │  Tiled Deferred      │ 1 pass    │ 1 pass    │ 1 pass     │
  │  Clustered Deferred  │ 1 pass    │ 1 pass    │ 1 pass     │
  │  Forward+ (Tiled Fwd)│ 1 pass    │ 1 pass    │ 1 pass     │
  └───────────────────────────────────────────────────────────┘
```

## 8.9 移动端延迟渲染的 Overdraw 特殊处理

```
From UE5 source (MobileShadingRenderer.cpp):

Mobile Deferred uses SUBPASSES to minimize bandwidth:

  RenderDeferredSinglePass():
    Subpass 1: Depth PrePass + BasePass (GBuffer fill)
    Subpass 2: Decals (SceneColor + GBuffer write)
    Subpass 3: Deferred Lighting (SceneColor write)
    → All in ONE render pass using framebuffer fetch!

  PassParameters->RenderTargets.SubpassHint = ESubpassHint::DeferredShadingSubpass;

Why subpasses matter for overdraw:
  Without subpasses (PC):
    GBuffer written to VRAM → read back for lighting
    Each pass = full VRAM read/write cycle
    Bandwidth = GBuffer_size × 2 × num_passes

  With subpasses (Mobile TBR):
    GBuffer stays in tile memory (on-chip)
    Lighting reads GBuffer from tile memory (FREE!)
    Bandwidth = GBuffer_size × 1 (final write only)

  This effectively ELIMINATES the bandwidth cost of
  GBuffer overdraw on mobile TBR architectures!

  But translucent overdraw is STILL expensive:
    Translucent is rendered AFTER all subpasses
    Each blend = tile memory read-modify-write
    High translucent overdraw = tile memory thrashing
```

## 8.10 延迟渲染 Overdraw 面试总结

```
Q: 延迟渲染管线中 Overdraw 是什么情况？

Answer:
  延迟渲染通过 Z-PrePass + GBuffer 架构，将不透明物体的 Overdraw
  从 Forward 的 N×L（物体数×灯光数）降低到接近 1x。

  具体各阶段：
  1. Z-PrePass: 只写深度，极低开销，为后续 BasePass 提供 Early-Z
  2. BasePass (GBuffer): 有 Z-PrePass 时 Overdraw ≈ 1x
     但每个像素写 4-7 个 MRT，单次 Overdraw 的带宽代价比 Forward 高
  3. Deferred Lighting: 每个灯光一个 Pass（或 Tiled/Clustered 合并为 1 Pass）
     光照 Overdraw = 重叠灯光体积数
  4. Translucent: 与 Forward 完全相同，仍然是 N 层 Overdraw
     延迟渲染无法解决半透明 Overdraw

  延迟渲染特有的 Overdraw 来源：
  - Deferred Decals（GBuffer 重写）
  - Light Volume 重叠
  - Shadow Projection Pass
  - Post-Process 全屏 Pass

  核心结论：
  延迟渲染解决了不透明物体的几何 Overdraw 和多灯光 Overdraw，
  但半透明 Overdraw 在任何管线中都无法避免，
  这也是为什么半透明优化（粒子限制、半分辨率渲染等）
  在延迟渲染管线中同样至关重要。
```
