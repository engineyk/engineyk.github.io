## 2. Houdini 生成地形

### 2.1 Houdini Heightfield 工作流

```
Houdini 地形生成核心概念：

  Heightfield = 2D Volume（高度场）
  不同于传统 Mesh，Heightfield 是基于体素的 2D 网格
  每个体素存储一个高度值（float）

  ┌─────────────────────────────────────────────────────────┐
  │  Houdini Heightfield 工作流                              │
  │                                                         │
  │  1. 创建基础地形                                         │
  │     Heightfield 节点 → 设置尺寸（如 2048×2048）           │
  │     Grid Spacing = 1m（每米一个采样点）                   │
  │                                                         │
  │  2. 叠加地形特征                                         │
  │     Noise → 基础起伏                                     │
  │     Distort → 扭曲变形                                   │
  │     Terrace → 阶梯状地形                                 │
  │     Erode → 侵蚀模拟（水力/热力）                         │
  │                                                         │
  │  3. 绘制遮罩层（Mask Layer）                             │
  │     Paint → 手动绘制区域                                 │
  │     By Feature → 基于坡度/高度/曲率自动生成               │
  │                                                         │
  │  4. 材质分配                                             │
  │     基于遮罩层分配地表材质权重                            │
  │     输出为 Splatmap / Weightmap                         │
  │                                                         │
  │  5. 导出                                                │
  │     Heightmap → R16 / EXR                               │
  │     Splatmap → RGBA PNG                                 │
  │     Normalmap → RGB PNG（可选）                          │
  └─────────────────────────────────────────────────────────┘
```

### 2.3 导出格式与规范

```
═══ Heightmap 导出 ═══

  推荐格式：R16 RAW 或 16-bit PNG
  
  分辨率选择：
  ┌─────────────────┬──────────────────┬──────────────────┐
  │ 地图尺寸         │ 推荐分辨率        │ 文件大小（R16）   │
  ├─────────────────┼──────────────────┼──────────────────┤
  │ 1km × 1km       │ 1025 × 1025      │ ~2 MB            │
  │ 2km × 2km       │ 2049 × 2049      │ ~8 MB            │
  │ 4km × 4km       │ 4097 × 4097      │ ~32 MB           │
  │ 8km × 8km       │ 8193 × 8193      │ ~128 MB          │
  └─────────────────┴──────────────────┴──────────────────┘

  ⚠️ 分辨率注意事项：
  • Unity Terrain: 推荐 2^N + 1（如 513, 1025, 2049, 4097）
  • Unreal Landscape: 必须匹配 (ComponentQuadSize × NumComponents) + 1
  • 通用规则：宽高必须相等（正方形）

  高度范围映射：
  • Houdini 高度值 → 归一化到 [0, 1]
  • 导出时记录 MinHeight 和 MaxHeight
  • 引擎导入时反映射：height = value * (Max - Min) + Min

═══ Splatmap / Weightmap 导出 ═══

  格式：RGBA 8-bit PNG（每张存 4 层权重）
  
  多层导出策略：
  • 4 层以内：1 张 RGBA Splatmap
  • 5~8 层：2 张 RGBA Splatmap
  • 8 层以上：考虑使用 Virtual Texture 或 Texture Array

  权重归一化：
  • 所有层权重之和 = 1.0（或 255）
  • Houdini 中使用 Heightfield Layer Property 设置

═══ 法线贴图导出（可选）═══

  格式：RGB 8-bit PNG 或 16-bit
  • 可以在引擎中从 Heightmap 实时计算，不一定需要导出
  • 预烘焙法线可节省运行时计算开销
  • 注意坐标系差异（Houdini Y-up vs Unity Y-up vs Unreal Z-up）
```

### 2.4 自动化管线（HDA / PDG）

```
═══ HDA（Houdini Digital Asset）═══

  将地形生成流程封装为 HDA，在引擎中直接调用：

  ┌─────────────────────────────────────────────────────────┐
  │  Houdini Engine for Unity / Unreal                      │
  │                                                         │
  │  工作流：                                                │
  │  1. 在 Houdini 中创建地形生成网络                         │
  │  2. 封装为 HDA（.hda / .otl 文件）                       │
  │  3. 在引擎中拖入 HDA                                     │
  │  4. 暴露参数（种子、尺寸、侵蚀强度等）                     │
  │  5. 引擎中调参 → Houdini 后台计算 → 自动导入结果           │
  │                                                         │
  │  优势：                                                  │
  │  • 美术可在引擎中直接调参，无需打开 Houdini                │
  │  • 参数化生成，快速迭代                                   │
  │  • 可批量生成不同变体                                     │
  │                                                         │
  │  限制：                                                  │
  │  • 需要 Houdini Engine 许可证                            │
  │  • 计算在 CPU 端，大地形较慢                              │
  │  • 运行时不建议使用（仅编辑器工具）                        │
  └─────────────────────────────────────────────────────────┘

═══ PDG（Procedural Dependency Graph）═══

  用于大规模地形的批量处理：

  场景：8km × 8km 地图，分为 64 个 1km × 1km 的 Tile
  
  PDG 工作流：
  1. 定义 Tile 网格（8×8）
  2. 每个 Tile 独立生成地形（可并行）
  3. 相邻 Tile 边界混合（避免接缝）
  4. 批量导出所有 Tile 的 Heightmap + Splatmap
  5. 引擎中按 Tile 流式加载

  优势：
  • 分布式计算（多核 / 多机并行）
  • 自动处理 Tile 边界
  • 增量更新（只重新生成修改的 Tile）
```

## 3. Unity 导入地形数据

### 3.1 Unity 内置 Terrain 导入

```csharp
// Method 1: Import RAW heightmap to Unity Terrain
public void ImportHeightmapToTerrain(Terrain terrain, string rawFilePath,
                                      int resolution, float heightScale)
{
    byte[] rawData = File.ReadAllBytes(rawFilePath);
    TerrainData terrainData = terrain.terrainData;
    
    // Set terrain resolution
    terrainData.heightmapResolution = resolution;
    terrainData.size = new Vector3(
        resolution, heightScale, resolution);
    
    // Parse R16 RAW data
    float[,] heights = new float[resolution, resolution];
    for (int y = 0; y < resolution; y++)
    {
        for (int x = 0; x < resolution; x++)
        {
            int index = (y * resolution + x) * 2; // 2 bytes per pixel (R16)
            ushort rawHeight = (ushort)(rawData[index] | (rawData[index + 1] << 8));
            heights[y, x] = rawHeight / 65535f; // Normalize to [0, 1]
        }
    }
    
    terrainData.SetHeights(0, 0, heights);
}

// Method 2: Import 16-bit PNG heightmap
public void ImportPNGHeightmap(Terrain terrain, Texture2D heightmapTex,
                                float heightScale)
{
    TerrainData terrainData = terrain.terrainData;
    int res = heightmapTex.width;
    
    terrainData.heightmapResolution = res;
    terrainData.size = new Vector3(res, heightScale, res);
    
    Color[] pixels = heightmapTex.GetPixels();
    float[,] heights = new float[res, res];
    
    for (int y = 0; y < res; y++)
    for (int x = 0; x < res; x++)
    {
        // For R16 texture, height is in the red channel
        heights[y, x] = pixels[y * res + x].r;
    }
    
    terrainData.SetHeights(0, 0, heights);
}
```

### 3.2 自定义地形系统导入

```csharp
// Custom terrain system: load heightmap as GPU texture directly
public class TerrainDataImporter
{
    /// <summary>
    /// Load heightmap as RenderTexture for GPU-side terrain rendering
    /// No CPU readback needed — vertex shader samples directly
    /// </summary>
    public static RenderTexture LoadHeightmapToGPU(string path, int resolution)
    {
        // Load raw data
        byte[] rawData = File.ReadAllBytes(path);
        
        // Create R16 texture (16-bit single channel)
        Texture2D cpuTex = new Texture2D(
            resolution, resolution,
            TextureFormat.R16, false);
        cpuTex.LoadRawTextureData(rawData);
        cpuTex.Apply(false, true); // makeNoLongerReadable = true (save memory)
        
        // Blit to RenderTexture for GPU sampling
        RenderTexture gpuTex = new RenderTexture(
            resolution, resolution, 0,
            RenderTextureFormat.RHalf); // 16-bit float
        gpuTex.filterMode = FilterMode.Bilinear;
        gpuTex.wrapMode = TextureWrapMode.Clamp;
        Graphics.Blit(cpuTex, gpuTex);
        
        // Release CPU texture (data now on GPU only)
        Object.Destroy(cpuTex);
        
        return gpuTex;
    }
    
    /// <summary>
    /// Load splatmap for terrain texture blending
    /// </summary>
    public static Texture2D LoadSplatmap(string path)
    {
        byte[] pngData = File.ReadAllBytes(path);
        Texture2D splatmap = new Texture2D(2, 2, TextureFormat.RGBA32, false);
        splatmap.LoadImage(pngData);
        splatmap.filterMode = FilterMode.Bilinear;
        splatmap.wrapMode = TextureWrapMode.Clamp;
        return splatmap;
    }
}
```

### 3.3 Heightmap 格式与精度

```
═══ 格式对比 ═══

┌──────────────┬───────────┬──────────────┬───────────────────────────┐
│ 格式         │ 精度       │ 每像素字节    │ 适用场景                  │
├──────────────┼───────────┼──────────────┼───────────────────────────┤
│ R8 (8-bit)   │ 256 级    │ 1 byte       │ 低端机、小地图             │
│ R16 (16-bit) │ 65536 级  │ 2 bytes      │ 主流方案（推荐）           │
│ R32F (32-bit)│ 浮点精度   │ 4 bytes      │ 超大地图、高精度需求       │
│ RHalf (16F)  │ 半精度浮点 │ 2 bytes      │ GPU 采样友好              │
└──────────────┴───────────┴──────────────┴──────────────────────────┘

═══ 精度计算 ═══

  地形高度范围 = 512m（-256m ~ +256m）
  
  R8:  512m / 256 = 2m 精度 → 明显阶梯感 ❌
  R16: 512m / 65536 = 0.0078m ≈ 0.78cm 精度 → 足够 ✅
  R32F: 浮点精度 → 远超需求 ⚠️（浪费内存）

  结论：R16 是性价比最高的选择
```

### 3.4 Splatmap / Weightmap 导入

```csharp
// Import splatmap and assign terrain layers
public void ImportSplatmap(Terrain terrain, Texture2D splatmap,
                            TerrainLayer[] layers)
{
    TerrainData data = terrain.terrainData;
    data.terrainLayers = layers;
    
    int res = splatmap.width;
    data.alphamapResolution = res;
    
    // Unity alphamap: [y, x, layerIndex]
    float[,,] alphamap = new float[res, res, layers.Length];
    Color[] pixels = splatmap.GetPixels();
    
    for (int y = 0; y < res; y++)
    for (int x = 0; x < res; x++)
    {
        Color c = pixels[y * res + x];
        // RGBA → 4 layers
        if (layers.Length > 0) alphamap[y, x, 0] = c.r;
        if (layers.Length > 1) alphamap[y, x, 1] = c.g;
        if (layers.Length > 2) alphamap[y, x, 2] = c.b;
        if (layers.Length > 3) alphamap[y, x, 3] = c.a;
    }
    
    data.SetAlphamaps(0, 0, alphamap);
}
```

## 4. 分块加载（Chunk Streaming）

### 4.1 Chunk 划分策略

```
═══ 为什么需要分块？ ═══

  一个 4km × 4km 的地形，如果用单个 Mesh：
  • 顶点数 = 4097 × 4097 ≈ 1680 万 → 无法一次渲染
  • 无法做 LOD（整体只有一个精度）
  • 无法做视锥剔除（要么全画，要么不画）
  • 无法流式加载（必须全部加载到内存）

  分块后（64m × 64m Chunk）：
  • 总 Chunk 数 = 64 × 64 = 4096
  • 每个 Chunk 顶点数 = 65 × 65 = 4225（LOD0）
  • 可见 Chunk ≈ 100~200（视锥剔除后）
  • 每个 Chunk 独立 LOD
  • 按需加载/卸载

═══ Chunk 尺寸选择 ═══

┌──────────────┬──────────────┬───────────────┬──────────────────────┐
│ Chunk 尺寸   │ 优势          │ 劣势          │ 适用场景              │
├──────────────┼──────────────┼───────────────┼──────────────────────┤
│ 32m × 32m    │ LOD 精细      │ Chunk 数量多  │ 小地图、高精度需求     │
│ 64m × 64m    │ 平衡          │ 平衡          │ 主流选择（推荐）      │
│ 128m × 128m  │ DrawCall 少   │ LOD 粗糙      │ 超大地图、移动端      │
│ 256m × 256m  │ DrawCall 极少 │ 剔除效率低    │ 极大地图              │
└──────────────┴──────────────┴───────────────┴──────────────────────┘
```

### 4.2 四叉树空间管理

```
═══ 四叉树结构 ═══

  四叉树（Quadtree）是地形空间管理的经典数据结构：

  Level 0 (Root): 整个地图 4096m × 4096m
       ┌───────┬───────┐
       │       │       │
       │  NW   │  NE   │
       │       │       │
       ├───────┼───────┤
       │       │       │
       │  SW   │  SE   │
       │       │       │
       └───────┴───────┘

  Level 1: 4 个 2048m × 2048m
  Level 2: 16 个 1024m × 1024m
  Level 3: 64 个 512m × 512m
  Level 4: 256 个 256m × 256m
  Level 5: 1024 个 128m × 128m
  Level 6: 4096 个 64m × 64m (叶节点 = Chunk)

═══ 四叉树的用途 ═══

  1. 视锥剔除加速
     → 从根节点开始，如果父节点不在视锥内，跳过所有子节点
     → 复杂度从 O(N) 降到 O(log N)

  2. LOD 选择
     → 根据相机距离，决定在哪一层停止细分
     → 近处展开到叶节点（最高精度）
     → 远处在中间层停止（低精度大块）

  3. 流式加载管理
     → 按四叉树层级决定加载优先级
     → 近处优先加载高精度数据
     → 远处只加载低精度数据
```

```csharp
// Quadtree node for terrain management
public class TerrainQuadNode
{
    public Bounds Bounds;
    public int Level;
    public int LOD;
    public TerrainQuadNode[] Children; // 4 children (NW, NE, SW, SE)
    public bool IsLeaf => Children == null;
    
    /// <summary>
    /// Recursively select LOD based on camera distance
    /// </summary>
    public void SelectLOD(Vector3 cameraPos, float[] lodDistances)
    {
        float dist = Vector3.Distance(cameraPos, Bounds.center);
        
        // Find appropriate LOD level
        LOD = lodDistances.Length - 1; // Start with lowest LOD
        for (int i = 0; i < lodDistances.Length; i++)
        {
            if (dist < lodDistances[i])
            {
                LOD = i;
                break;
            }
        }
        
        // If not leaf and need higher detail, recurse into children
        if (!IsLeaf && LOD < Level)
        {
            foreach (var child in Children)
                child.SelectLOD(cameraPos, lodDistances);
        }
    }
    
    /// <summary>
    /// Frustum culling with early-out at parent level
    /// </summary>
    public void FrustumCull(Plane[] frustumPlanes, List<TerrainQuadNode> visible)
    {
        if (!GeometryUtility.TestPlanesAABB(frustumPlanes, Bounds))
            return; // Parent outside frustum → skip all children
        
        if (IsLeaf || ShouldRenderAtThisLevel())
        {
            visible.Add(this);
            return;
        }
        
        foreach (var child in Children)
            child.FrustumCull(frustumPlanes, visible);
    }
}
```

```csharp
// Async chunk loading with priority queue
public class TerrainStreamingManager
{
    private Dictionary<Vector2Int, TerrainChunk> _loadedChunks = new();
    private PriorityQueue<ChunkLoadRequest> _loadQueue = new();
    private ObjectPool<TerrainChunk> _chunkPool;
    
    // Max chunks to load per frame (avoid frame spike)
    private const int MAX_LOADS_PER_FRAME = 2;
    private const int MAX_UNLOADS_PER_FRAME = 4;
    
    public void UpdateStreaming(Vector3 cameraPos)
    {
        Vector2Int cameraChunk = WorldToChunkCoord(cameraPos);
        
        // 1. Determine which chunks should be loade 检查哪些块需要加载
        HashSet<Vector2Int> desiredChunks = GetDesiredChunks(cameraChunk);
        
        // 2. Queue chunks to load (not yet loaded)
        foreach (var coord in desiredChunks)
        {
            if (!_loadedChunks.ContainsKey(coord))
            {
                float priority = Vector2Int.Distance(coord, cameraChunk);
                _loadQueue.Enqueue(new ChunkLoadRequest(coord, priority));
            }
        }
        
        // 3. Queue chunks to unload (no longer needed) 卸载不需要的加载
        var toUnload = new List<Vector2Int>();
        foreach (var (coord, chunk) in _loadedChunks)
        {
            if (!desiredChunks.Contains(coord))
                toUnload.Add(coord);
        }
        
        // 4. Process load queue (limited per frame)
        int loaded = 0;
        while (_loadQueue.Count > 0 && loaded < MAX_LOADS_PER_FRAME)
        {
            var request = _loadQueue.Dequeue();
            LoadChunkAsync(request.Coord);
            loaded++;
        }
        
        // 5. Process unload (limited per frame)
        for (int i = 0; i < Math.Min(toUnload.Count, MAX_UNLOADS_PER_FRAME); i++)
        {
            UnloadChunk(toUnload[i]);
        }
    }
    
    private async void LoadChunkAsync(Vector2Int coord)
    {
        TerrainChunk chunk = _chunkPool.Get();
        
        // Async load heightmap tile
        var heightData = await Addressables.LoadAssetAsync<Texture2D>(
            $"Terrain/Height_{coord.x}_{coord.y}").Task;
        
        // Async load splatmap tile
        var splatData = await Addressables.LoadAssetAsync<Texture2D>(
            $"Terrain/Splat_{coord.x}_{coord.y}").Task;
        
        chunk.Initialize(coord, heightData, splatData);
        _loadedChunks[coord] = chunk;
    }
    
    private void UnloadChunk(Vector2Int coord)
    {
        if (_loadedChunks.TryGetValue(coord, out var chunk))
        {
            chunk.Release();
            _chunkPool.Return(chunk);
            _loadedChunks.Remove(coord);
        }
    }
}
```

## 5. 地形细分（Tessellation）

### 5.1 硬件细分管线

```
═══ GPU 细分管线阶段 ═══

  Vertex Shader
       ↓
  Hull Shader (HS)          ← 控制细分级别
       ↓
  Tessellator (Fixed-Func)  ← 硬件生成新顶点
       ↓
  Domain Shader (DS)        ← 计算新顶点位置（采样 Heightmap）
       ↓
  Geometry Shader (可选)
       ↓
  Fragment Shader

═══ 细分的优势 ═══

  传统方案：CPU 预生成多级 LOD Mesh → 切换时有突变
  细分方案：GPU 动态生成顶点 → 平滑过渡，无突变

  ┌──────────────────────────────────────────────────────┐
  │  无细分：                                             │
  │  远处 ▲───▲───▲───▲  （固定低精度）                    │
  │  近处 ▲───▲───▲───▲  （同样低精度 → 锯齿明显）          │
  │                                                      │
  │  有细分：                                             │
  │  远处 ▲───▲───▲───▲  （低细分 → 少顶点）               │
  │  近处 ▲▲▲▲▲▲▲▲▲▲▲▲▲  （高细分 → 多顶点 → 平滑）        │
  └──────────────────────────────────────────────────────┘
```

```hlsl
// Hull Shader: Adaptive tessellation based on camera distance
// Terrain Tessellation Shader

[domain("quad")]
[partitioning("fractional_even")]
[outputtopology("triangle_cw")]
[outputcontrolpoints(4)]
[patchconstantfunc("PatchConstantFunction")]
HullOutput TerrainHull(InputPatch<VertexOutput, 4> patch, uint id : SV_OutputControlPointID)
{
    HullOutput o;
    o.positionWS = patch[id].positionWS;
    o.uv = patch[id].uv;
    return o;
}

// Patch constant function: determine tessellation factors
TessFactors PatchConstantFunction(InputPatch<VertexOutput, 4> patch)
{
    TessFactors f;
    
    // Calculate edge midpoints
    float3 edge0Mid = (patch[0].positionWS + patch[1].positionWS) * 0.5;
    float3 edge1Mid = (patch[1].positionWS + patch[2].positionWS) * 0.5;
    float3 edge2Mid = (patch[2].positionWS + patch[3].positionWS) * 0.5;
    float3 edge3Mid = (patch[3].positionWS + patch[0].positionWS) * 0.5;
    float3 center   = (patch[0].positionWS + patch[1].positionWS +
                        patch[2].positionWS + patch[3].positionWS) * 0.25;
    
    // Distance-based tessellation factor
    f.edge[0] = CalcTessFactor(edge0Mid);
    f.edge[1] = CalcTessFactor(edge1Mid);
    f.edge[2] = CalcTessFactor(edge2Mid);
    f.edge[3] = CalcTessFactor(edge3Mid);
    f.inside[0] = CalcTessFactor(center);
    f.inside[1] = CalcTessFactor(center);
    
    return f;
}

float CalcTessFactor(float3 worldPos)
{
    float dist = distance(worldPos, _WorldSpaceCameraPos);
    
    // Remap distance to tessellation factor
    // Near: maxTess (e.g., 16), Far: minTess (e.g., 1)
    float factor = saturate(1.0 - (dist - _TessMinDist) / (_TessMaxDist - _TessMinDist));
    return lerp(_TessMinFactor, _TessMaxFactor, factor);
}

// Domain Shader: displace tessellated vertices using heightmap
[domain("quad")]
DomainOutput TerrainDomain(TessFactors factors,
                            float2 uv : SV_DomainLocation,
                            const OutputPatch<HullOutput, 4> patch)
{
    DomainOutput o;
    
    // Bilinear interpolation of patch corners
    float3 worldPos = lerp(
        lerp(patch[0].positionWS, patch[1].positionWS, uv.x),
        lerp(patch[3].positionWS, patch[2].positionWS, uv.x),
        uv.y);
    
    float2 texUV = lerp(
        lerp(patch[0].uv, patch[1].uv, uv.x),
        lerp(patch[3].uv, patch[2].uv, uv.x),
        uv.y);
    
    // Sample heightmap and displace vertex
    float height = _HeightMap.SampleLevel(sampler_HeightMap, texUV, 0).r;
    worldPos.y = height * _HeightScale;
    
    o.positionCS = TransformWorldToHClip(worldPos);
    o.uv = texUV;
    o.positionWS = worldPos;
    
    return o;
}
```

### 5.3 Displacement Mapping

```
═══ Displacement vs Tessellation ═══

  Tessellation = 增加几何细节（生成更多三角形）
  Displacement = 移动顶点位置（基于纹理偏移）
  
  地形中两者结合使用：
  1. Tessellation 生成足够密度的顶点网格
  2. Displacement 根据 Heightmap 移动顶点高度
  3. 可选：额外的 Detail Displacement（岩石纹理凹凸）

  │  Base Mesh (低精度)                                   │
  │  ────────────────────                                │
  │                                                      │
  │  + Tessellation (增加顶点)                            │
  │  ▲▲▲▲▲▲▲▲▲▲▲▲▲▲▲▲▲▲▲▲                                │
  │                                                      │
  │  + Heightmap Displacement (移动高度)                  │
  │  ▲  ▲▲  ▲▲▲▲  ▲▲  ▲  ▲▲▲  ▲▲  ▲                      │
  │   ▲▲  ▲▲    ▲▲  ▲▲ ▲▲   ▲▲  ▲▲                       │
  │                                                      │
  │  + Detail Displacement (微观凹凸)                     │
  │  ≈ 最终地形表面                                       │
  └──────────────────────────────────────────────────────┘
```

### 5.4 移动端替代方案

```
═══ 移动端不支持 Tessellation 怎么办？ ═══

  问题：
  • OpenGL ES 3.1 / Metal 支持 Tessellation，但性能开销大
  • 低端 Android 设备不支持 Tessellation
  • 移动端 GPU 对额外顶点处理敏感

  替代方案：

  1. 预生成多级 LOD Mesh（最常用）
     → CPU 预生成 LOD0~LOD3 的 Mesh
     → 根据距离切换 Mesh
     → 零 GPU 额外开销
     → 缺点：LOD 切换有突变（需 Morphing 缓解）

  2. Vertex Texture Fetch（VTF）
     → 使用低精度 Mesh + Heightmap 顶点采样
     → 在 Vertex Shader 中采样 Heightmap 偏移顶点
     → 不增加顶点数，但提高表面精度
     → 需要 GPU 支持 VTF（大部分现代移动 GPU 支持）

  3. Geometry Instancing + LOD
     → 同一 LOD 的 Chunk 合并为一次 DrawCall
     → 减少 CPU 开销
     → 配合 VTF 提高精度

  移动端推荐组合：
  预生成 LOD Mesh + VTF + GPU Instancing
```

## 6. 地形缝合（Stitching）

### 6.1 LOD 接缝问题

```
═══ 为什么会出现接缝？ ═══

  相邻 Chunk 使用不同 LOD 级别时，边界顶点数量不匹配：

  LOD0 (高精度)          LOD1 (低精度)
  ▲─▲─▲─▲─▲─▲─▲─▲─▲     ▲───────▲───────▲
                    ↕ 裂缝（Gap）
  
  LOD0 边界有 9 个顶点
  LOD1 边界只有 3 个顶点
  中间的顶点没有对应 → 产生裂缝

═══ 接缝的表现 ═══

  • 地面出现黑色裂缝线（可以看到天空盒/背景）
  • 光照不连续（法线突变）
  • 阴影异常（裂缝处阴影穿透）
```

### 6.2 Skirt（裙边）方案

```
═══ 原理 ═══

  在每个 Chunk 的边缘，向下延伸一圈额外的顶点（裙边）
  裙边遮挡住裂缝，从上方看不到缝隙

  侧视图：
  
  LOD0 Chunk              LOD1 Chunk
  ▲─▲─▲─▲─▲─▲─▲─▲─▲     ▲───────▲───────▲
                    │     │
                    ▼     ▼  ← Skirt（向下延伸）
                    │     │
                    └─────┘  ← 裙边底部（低于最低地形高度）

═══ 实现 ═══
```

```hlsl
// Skirt vertex generation in vertex shader

    float height = tex2Dlod(_HeightMap, float4(heightUV, 0, 0)).r;
```

```
═══ Skirt 优缺点 ═══

  ✅ 优点：
  • 实现简单，不需要修改相邻 Chunk 的 Mesh
  • 性能开销极低（只增加边缘一圈顶点）
  • 对所有 LOD 组合都有效

  ❌ 缺点：
  • 裙边在侧面观察时可能可见（悬崖边缘）
  • 不是真正的"缝合"，只是遮挡
  • 裙边高度需要足够大以覆盖最大高度差
```

### 6.3 Vertex Morphing（顶点形变）

```
═══ 原理 ═══

  在 LOD 切换时，不是突然替换 Mesh，而是平滑过渡：
  
  LOD0 → LOD1 过渡过程：
  
  t=0.0 (LOD0):  ▲─▲─▲─▲─▲─▲─▲─▲─▲  （所有顶点在原位）
  t=0.3:         ▲─▲─▲─▲─▲─▲─▲─▲─▲  （奇数顶点开始向偶数邻居移动）
  t=0.7:         ▲──▲──▲──▲──▲──▲──▲  （奇数顶点接近中点）
  t=1.0 (LOD1):  ▲───────▲───────▲    （奇数顶点完全合并，安全切换）

  关键：奇数顶点 morph 到相邻偶数顶点的中点
  当 morphFactor = 1.0 时，LOD0 和 LOD1 的边界完全一致
```

```c
// Vertex morphing in vertex shader
float3 MorphVertex(float3 posLOD0, float3 posLOD1, float morphFactor)
{
    return lerp(posLOD0, posLOD1, morphFactor);
}

// Calculate morph factor based on distance
float CalcMorphFactor(float dist, float lodStartDist, float lodEndDist)
{
    // Smooth transition in the overlap zone
    return saturate((dist - lodStartDist) / (lodEndDist - lodStartDist));
}

VertexOutput TerrainVertMorph(VertexInput v)
{
    VertexOutput o;
    
    float3 worldPos = TransformObjectToWorld(v.positionOS);
    float dist = distance(worldPos, _WorldSpaceCameraPos);
    
    // Sample height at current LOD position
    float2 uvCurrent = v.uv;
    float heightCurrent = tex2Dlod(_HeightMap, float4(uvCurrent, 0, 0)).r;
    
    // Sample height at next LOD position (snapped to lower LOD grid)
    float2 uvSnapped = SnapToLowerLODGrid(v.uv, _CurrentLOD);
    float heightSnapped = tex2Dlod(_HeightMap, float4(uvSnapped, 0, 0)).r;
    
    // Morph between current and snapped position
    float morphFactor = CalcMorphFactor(dist, _LODMorphStart, _LODMorphEnd);
    float finalHeight = lerp(heightCurrent, heightSnapped, morphFactor);
    
    worldPos.y = finalHeight * _HeightScale;
    o.positionCS = TransformWorldToHClip(worldPos);
    o.uv = uvCurrent;
    
    return o;
}
```

### 6.4 T-Junction 修复

```
═══ T-Junction 问题 ═══

  当高 LOD 和低 LOD 的边界相邻时，高 LOD 的边界顶点
  落在低 LOD 的边上（而非顶点上），形成 T 形交叉：

  LOD0:  ▲─▲─▲─▲─▲
                    ↕ T-Junction
  LOD1:  ▲─────▲─────▲

  LOD0 的中间顶点 ▲ 落在 LOD1 的边 ─── 上
  由于浮点精度，这个顶点可能不在边上 → 产生微小裂缝

═══ 修复方案 ═══

  方案 1：边界顶点退化（Degenerate）
  → 高 LOD 边界的多余顶点强制 snap 到低 LOD 的边上
  → 需要知道相邻 Chunk 的 LOD 级别

  方案 2：预生成多套 Index Buffer
  → 为每种邻接 LOD 组合预生成不同的 Index Buffer
  → 4 条边 × 每边可能的 LOD 差 → 组合数较多
  → Unreal Landscape 使用此方案

  方案 3：Skirt + Morphing 组合
  → Skirt 遮挡大裂缝
  → Morphing 消除小裂缝
  → 最简单的实现方式
```

```csharp
// Pre-generate index buffers for different neighbor LOD combinations
// Each edge can have: same LOD, LOD+1, LOD+2, etc.
public class TerrainIndexBufferCache
{
    // Key: (myLOD, neighborLOD_North, neighborLOD_East, neighborLOD_South, neighborLOD_West)
    private Dictionary<(int, int, int, int, int), int[]> _cache = new();
    
    public int[] GetIndexBuffer(int myLOD, int lodN, int lodE, int lodS, int lodW)
    {
        var key = (myLOD, lodN, lodE, lodS, lodW);
        if (!_cache.TryGetValue(key, out var indices))
        {
            indices = GenerateIndices(myLOD, lodN, lodE, lodS, lodW);
            _cache[key] = indices;
        }
        return indices;
    }
    
    private int[] GenerateIndices(int myLOD, int lodN, int lodE, int lodS, int lodW)
    {
        int gridSize = 64 >> myLOD;
        var indices = new List<int>();
        
        for (int z = 0; z < gridSize - 1; z++)
        for (int x = 0; x < gridSize - 1; x++)
        {
            bool isNorthEdge = (z == gridSize - 2);
            bool isSouthEdge = (z == 0);
            bool isEastEdge  = (x == gridSize - 2);
            bool isWestEdge  = (x == 0);
            
            // If on edge and neighbor has lower LOD, use degenerate triangles
            if (isNorthEdge && lodN > myLOD)
                AddDegenerateTriangles(indices, x, z, Direction.North, lodN - myLOD);
            else if (isSouthEdge && lodS > myLOD)
                AddDegenerateTriangles(indices, x, z, Direction.South, lodS - myLOD);
            else
                AddNormalQuad(indices, x, z, gridSize);
            
            // Similar for East/West edges...
        }
        
        return indices.ToArray();
    }
}
```

### 6.5 Chunk 边界共享顶点

```
═══ 共享顶点策略 ═══

  相邻 Chunk 的边界顶点使用完全相同的位置和高度值
  确保边界处没有任何间隙

  Chunk A          Chunk B
  ▲─▲─▲─▲─▲ ← 共享 → ▲─▲─▲─▲─▲
  
  实现方式：
  1. Heightmap 边界重叠 1 个像素
     → Chunk A 的右边界 = Chunk B 的左边界
     → 采样同一个 Heightmap 像素

  2. 顶点位置精确对齐
     → 使用整数坐标计算，避免浮点误差
     → 或使用 snapping（对齐到网格）

  3. 法线边界处理
     → 边界顶点的法线需要考虑两侧的高度差
     → 使用 3×3 或 5×5 采样核计算法线
     → 确保边界法线连续
```

## 7. 贴图的加载管理

### 7.1 Splatmap 多层混合

```
═══ Splatmap 原理 ═══

  Splatmap = 权重贴图，每个通道存储一层地表纹理的权重
  
  RGBA Splatmap:
  ┌──────────────────────────────────────────────────┐
  │  R 通道: 草地权重    (0.0 ~ 1.0)                  │
  │  G 通道: 岩石权重    (0.0 ~ 1.0)                  │
  │  B 通道: 沙地权重    (0.0 ~ 1.0)                  │
  │  A 通道: 雪地权重    (0.0 ~ 1.0)                  │
  │                                                  │
  │  约束: R + G + B + A = 1.0（权重归一化）           │
  └──────────────────────────────────────────────────┘

  多于 4 层：使用多张 Splatmap
  • Splatmap 0: Layer 0~3 (RGBA)
  • Splatmap 1: Layer 4~7 (RGBA)
  • 最多 8 层 = 2 张 Splatmap
```

```c
// 4-layer terrain blending with splatmap
half4 TerrainBlend4Layer(float2 uv, float2 terrainUV)
{
    // Sample splatmap (terrain-space UV)
    half4 splat = tex2D(_SplatMap, terrainUV);
    
    // Sample each layer texture (tiled UV)
    half4 layer0 = tex2D(_Layer0Tex, uv * _Layer0Tiling);
    half4 layer1 = tex2D(_Layer1Tex, uv * _Layer1Tiling);
    half4 layer2 = tex2D(_Layer2Tex, uv * _Layer2Tiling);
    half4 layer3 = tex2D(_Layer3Tex, uv * _Layer3Tiling);
    
    // Weighted blend
    half4 result = layer0 * splat.r
                 + layer1 * splat.g
                 + layer2 * splat.b
                 + layer3 * splat.a;
    
    return result;
}

// 8-layer terrain blending (2 splatmaps)
half4 TerrainBlend8Layer(float2 uv, float2 terrainUV)
{
    half4 splat0 = tex2D(_SplatMap0, terrainUV);
    half4 splat1 = tex2D(_SplatMap1, terrainUV);
    
    half4 result = tex2D(_Layer0Tex, uv * _Layer0Tiling) * splat0.r
                 + tex2D(_Layer1Tex, uv * _Layer1Tiling) * splat0.g
                 + tex2D(_Layer2Tex, uv * _Layer2Tiling) * splat0.b
                 + tex2D(_Layer3Tex, uv * _Layer3Tiling) * splat0.a
                 + tex2D(_Layer4Tex, uv * _Layer4Tiling) * splat1.r
                 + tex2D(_Layer5Tex, uv * _Layer5Tiling) * splat1.g
                 + tex2D(_Layer6Tex, uv * _Layer6Tiling) * splat1.b
                 + tex2D(_Layer7Tex, uv * _Layer7Tiling) * splat1.a;
    
    return result;
    // ⚠️ 8 层 = 10 次纹理采样（2 splatmap + 8 layer）→ 移动端很重！
}
```

### 7.2 Texture Array

```
═══ Texture Array vs 传统多纹理 ═══

  传统方式：每层一个 Texture2D → N 个 Sampler → Sampler 数量受限
  Texture Array：所有层打包到一个 Texture2DArray → 1 个 Sampler

  ┌──────────────────┬──────────────────┬──────────────────┐
  │ 特性              │ 传统多纹理        │ Texture Array   │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Sampler 数量     │ N 个（受限 16）   │ 1 个              │
  │ 层数上限          │ ~4~8 层          │ 理论无限          │
  │ 纹理尺寸          │ 可以不同          │ 必须相同         │
  │ Mipmap           │ 独立              │ 共享             │
  │ 压缩格式          │ 可以不同          │ 必须相同         │
  │ 硬件要求          │ 所有 GPU         │ GLES 3.0+        │
  └──────────────────┴──────────────────┴──────────────────┘
```

```c
// Texture Array based terrain blending
TEXTURE2D_ARRAY(_TerrainLayers);
SAMPLER(sampler_TerrainLayers);

half4 TerrainBlendTextureArray(float2 uv, float2 terrainUV, int layerCount)
{
    half4 splat0 = tex2D(_SplatMap0, terrainUV);
    half4 splat1 = tex2D(_SplatMap1, terrainUV);
    half weights[8] = { splat0.r, splat0.g, splat0.b, splat0.a,
                         splat1.r, splat1.g, splat1.b, splat1.a };
    
    half4 result = 0;
    for (int i = 0; i < layerCount; i++)
    {
        if (weights[i] > 0.01) // Skip layers with negligible weight
        {
            half4 layerColor = SAMPLE_TEXTURE2D_ARRAY(
                _TerrainLayers, sampler_TerrainLayers,
                uv * _LayerTilings[i], i);
            result += layerColor * weights[i];
        }
    }
    return result;
}
```

### 7.3 Triplanar Mapping

```
═══ 为什么需要 Triplanar？ ═══

  地形纹理在陡峭表面（悬崖）会严重拉伸：
  
  平坦地面：纹理正常 ✅
  陡峭悬崖：纹理被拉伸成条状 ❌

  Triplanar Mapping：从三个轴向分别投影纹理，按法线方向混合

═══ 原理 ═══

  对每个像素：
  1. 用世界坐标的 XY 平面采样（顶部/底部视角）
  2. 用世界坐标的 XZ 平面采样（前/后视角）
  3. 用世界坐标的 YZ 平面采样（左/右视角）
  4. 按法线方向加权混合三个采样结果
```

```c
half4 TriplanarSample(sampler2D tex, float3 worldPos, float3 worldNormal, float tiling)
{
    // Three projections
    half4 projX = tex2D(tex, worldPos.yz * tiling); // YZ plane
    half4 projY = tex2D(tex, worldPos.xz * tiling); // XZ plane (top-down)
    half4 projZ = tex2D(tex, worldPos.xy * tiling); // XY plane
    
    // Blend weights based on normal direction
    float3 blend = abs(worldNormal);
    blend = pow(blend, 4.0); // Sharpen the blend (reduce transition zone)
    blend /= (blend.x + blend.y + blend.z); // Normalize
    
    return projX * blend.x + projY * blend.y + projZ * blend.z;
}

// Usage in terrain shader:
// Flat areas → mostly projY (top-down)
// Cliffs → mostly projX or projZ (side projection)
// ⚠️ 3x texture samples per layer → expensive! Use only for cliff areas
```

### 7.4 距离混合与 Macro Variation

```
═══ Tiling 重复问题 ═══

  地形纹理在远处会出现明显的重复 Tiling 图案
  
  解决方案：

  1. Macro Texture（宏观纹理）
     → 覆盖整个地形的低分辨率纹理
     → 与 Tiling 纹理相乘，打破重复感
     → 额外 1 次纹理采样

  2. Distance Blend（距离混合）
     → 近处：高频 Detail 纹理（Tiling 大）
     → 远处：低频 Macro 纹理（Tiling 小）
     → 基于距离 lerp

  3. Stochastic Tiling（随机 Tiling）
     → 随机旋转/偏移 UV
     → 消除规律性重复
     → 需要额外 Shader 计算

  4. World-Space Noise
     → 基于世界坐标的噪声函数调制颜色
     → 无额外纹理开销
```

```c
// Distance-based texture blending to reduce tiling
half4 DistanceBlendTerrain(float2 uv, float3 worldPos, float viewDist)
{
    // Near: high-frequency detail texture
    half4 detailColor = tex2D(_DetailTex, uv * _DetailTiling);
    
    // Far: low-frequency macro texture (covers entire terrain)
    half4 macroColor = tex2D(_MacroTex, uv * _MacroTiling);
    
    // Blend based on distance
    float blendFactor = saturate((viewDist - _BlendStartDist) / _BlendRange);
    half4 baseColor = lerp(detailColor, macroColor, blendFactor);
    
    // Optional: multiply with world-space noise to break repetition
    float noise = tex2D(_NoiseTex, worldPos.xz * 0.001).r;
    baseColor.rgb *= lerp(0.8, 1.2, noise); // Subtle variation
    
    return baseColor;
}
```

### 7.5 Height-Based Blending

```
═══ 基于高度的混合（比线性混合更自然）═══

  线性混合：weight = splatmap value → 过渡区域模糊
  高度混合：考虑纹理自身的高度信息 → 过渡区域更自然

  原理：
  • 每层纹理附带一张高度图（存在 Alpha 通道或单独纹理）
  • 混合时，高度值高的纹理"凸出"，低的"凹下"
  • 岩石从草地中"露出"，而非简单的半透明混合
```

```c
// Height-based blending for more natural terrain transitions
half4 HeightBlend(half4 color0, float height0, float weight0,
                   half4 color1, float height1, float weight1,
                   float blendSharpness)
{
    // Add height to weight
    float h0 = height0 + weight0;
    float h1 = height1 + weight1;
    
    // Find the maximum height
    float maxHeight = max(h0, h1) - blendSharpness;
    
    // Clamp and normalize
    float b0 = max(h0 - maxHeight, 0);
    float b1 = max(h1 - maxHeight, 0);
    float total = b0 + b1 + 0.001; // Avoid division by zero
    
    return (color0 * b0 + color1 * b1) / total;
}

// Multi-layer height blend
half4 TerrainHeightBlend4Layer(float2 uv, float2 terrainUV)
{
    half4 splat = tex2D(_SplatMap, terrainUV);
    
    // Sample albedo + height (height stored in alpha channel)
    half4 layer0 = tex2D(_Layer0Tex, uv * _Layer0Tiling); // .a = height
    half4 layer1 = tex2D(_Layer1Tex, uv * _Layer1Tiling);
    half4 layer2 = tex2D(_Layer2Tex, uv * _Layer2Tiling);
    half4 layer3 = tex2D(_Layer3Tex, uv * _Layer3Tiling);
    
    // Height-weighted blend
    float4 heights = float4(layer0.a, layer1.a, layer2.a, layer3.a);
    float4 weights = splat * heights; // Combine splatmap weight with texture height
    
    // Sharpen the blend
    float maxWeight = max(max(weights.x, weights.y), max(weights.z, weights.w));
    weights = max(weights - maxWeight + _BlendSharpness, 0);
    weights /= dot(weights, 1.0); // Normalize
    
    half4 result;
    result.rgb = layer0.rgb * weights.x
               + layer1.rgb * weights.y
               + layer2.rgb * weights.z
               + layer3.rgb * weights.w;
    result.a = 1.0;
    
    return result;
}
```

## 8. Virtual Texture（VT / VRT）

### 8.1 Virtual Texture 原理

```
═══ 核心问题 ═══

  大世界地形纹理总量远超显存：
  • 4km × 4km 地形，8 层材质
  • 每层 Diffuse + Normal + ORM = 3 张纹理
  • 8 × 3 = 24 张纹理 × 2048² × 4 bytes ≈ 384 MB
  • 加上 Mipmap ≈ 512 MB → 超出移动端显存预算

═══ Virtual Texture 解决方案 ═══

  将所有纹理合并为一张超大虚拟纹理（如 128K × 128K）
  运行时只加载相机可见区域的纹理页面（Page）

  ┌─────────────────────────────────────────────────────────┐
  │  Virtual Texture (128K × 128K)                          │
  │  ┌──┬──┬──┬──┬──┬──┐                                    │
  │  │P0│P1│P2│P3│P4│..│  ← 虚拟页面（大部分不在显存中）      │
  │  ├──┼──┼──┼──┼──┼──┤                                    │
  │  │P6│P7│P8│P9│..│..│                                    │
  │  └──┴──┴──┴──┴──┴──┘                                    │
  │         ↓ Page Table 映射                               │
  │  ┌──────────────────┐                                   │
  │  │  Physical Cache  │  固定大小（如 4K × 4K）            │
  │  │  ┌────┬────┬────┐│                                   │
  │  │  │ P0 │ P2 │ P7 ││  ← 只缓存当前可见的 Page            │
  │  │  ├────┼────┼────┤│                                   │
  │  │  │ P1 │ P8 │ P9 ││                                   │
  │  │  └────┴────┴────┘│                                   │
  │  └──────────────────┘                                   │
  └─────────────────────────────────────────────────────────┘

  运行时流程：
  1. 渲染时采样 Page Table → 获取物理缓存地址
  2. Cache Hit → 直接采样物理缓存
  3. Cache Miss → 请求加载该 Page → 下一帧可用
```

### 8.2 Runtime Virtual Texture（RVT）

```
═══ RVT 核心概念 ═══

  RVT = 运行时渲染结果缓存到虚拟纹理

  传统方式：
  每帧 → 采样 N 层纹理 → 混合 → 输出
  开销: N × (Diffuse + Normal + ORM) 次采样/像素

  RVT 方式：
  预渲染 → 混合结果写入 VT Cache → 运行时只采样 Cache
  开销: 1 次 VT 采样/像素（+ 偶尔的 Cache Miss 更新）

═══ RVT 在地形中的应用 ═══

  1. 降低材质开销
     • 8 层混合 → 1 次 VT 采样
     • Pixel Shader 指令数大幅减少
     • 移动端尤其重要

  2. 物体与地形融合
     • 岩石底部采样地形 RVT → 颜色融合
     • 建筑地基与地面过渡
     • 无需手动绘制 Decal

  3. 远景材质简化
     • 远处使用 RVT 低 Mip
     • 近处使用完整材质
     • 基于距离自动切换

═══ RVT 工作流程 ═══

  ┌──────────┐    ┌──────────────┐    ┌──────────────┐
  │ 相机移动  │ →  │ 确定可见 Page│ →   │ 检查 Cache   │
  └──────────┘    └──────────────┘    └──────┬───────┘
                                             │
                                     ┌───────┴───────┐
                                     │               │
                                 ┌───▼───┐      ┌────▼────┐
                                 │ Cache │      │ Cache   │
                                 │ Hit   │      │ Miss    │
                                 └───┬───┘      └────┬────┘
                                     │               │
                                     │          ┌────▼────────┐
                                     │          │ 渲染该 Page  │
                                     │          │ 到 VT Cache │
                                     │          └────┬────────┘
                                     │               │
                                 ┌───▼───────────────▼───┐
                                 │  采样 VT Cache 渲染    │
                                 └───────────────────────┘
```

### 8.3 Streaming Virtual Texture（SVT）

```
═══ SVT vs RVT ═══

┌──────────────────┬──────────────────┬──────────────────┐
```

### 8.4 Feedback Buffer 机制

```
═══ Feedback Buffer 原理 ═══

  问题：GPU 怎么知道哪些 Page 需要加载？
  答案：Feedback Buffer（反馈缓冲区）

  流程：
  1. 渲染一个低分辨率的 Feedback Pass
     → 每个像素输出它需要的 VT Page ID + Mip Level
  2. CPU 回读 Feedback Buffer
     → 收集所有需要的 Page 列表
  3. 加载缺失的 Page 到物理缓存
  4. 更新 Page Table

  ┌──────────────────────────────────────────────────────┐
  │  Feedback Pass (低分辨率，如 屏幕/16)                 │
  │                                                      │
  │  Fragment Shader 输出：                               │
  │  ┌──────────────────────────────────┐                │
  │  │  R: Page X 坐标                  │                │
  │  │  G: Page Y 坐标                  │                │
  │  │  B: Mip Level                    │                │
  │  │  A: VT ID（多个 VT 时区分）       │                │
  │  └──────────────────────────────────┘                │
  │                                                      │
  │  CPU 回读 → 去重 → 排序（按优先级）→ 加载 Page          │
  └──────────────────────────────────────────────────────┘
```

```c
// Feedback buffer shader (simplified)
half4 VTFeedbackFrag(Varyings i) : SV_Target
{
    // Calculate which VT page this pixel needs
    float2 vtUV = i.uv * _VTSize; // Virtual texture UV
    float mipLevel = CalcMipLevel(i.uv, _VTSize);
    
    // Page coordinates at this mip level
    float pageSize = _PageSize * pow(2, mipLevel);
    float2 pageCoord = floor(vtUV / pageSize);
    
    // Encode page request
    return half4(
        pageCoord.x / 255.0,
        pageCoord.y / 255.0,
        mipLevel / 15.0,    // Max 15 mip levels
        _VTID / 255.0       // VT identifier
    );
}
```

### 8.5 VT 性能分析与优化

```
═══ 关键性能指标 ═══

┌──────────────────────┬──────────────────────────────────┐
│ 指标                  │ 说明                             │
├──────────────────────┼──────────────────────────────────┤
│ Page Fault Rate      │ Cache Miss 频率（理想 < 5%/帧）   │
│ Cache Size           │ 物理缓存大小（默认 4096²）         │
│ Feedback Resolution  │ 反馈缓冲区分辨率（屏幕/16）        │
│ Page Load Latency    │ Page 加载延迟（1~3 帧）           │
│ Memory Usage         │ Cache + Page Table 内存占用       │
└──────────────────────┴──────────────────────────────────┘

═══ 优化策略 ═══

  1. 增大 Cache Size → 减少 Page Fault（但增加内存）
  2. 预加载 → 相机移动方向预测，提前加载 Page
  3. 优先级排序 → 近处 Page 优先加载
  4. Mip Bias → 远处使用更低 Mip，减少 Page 需求
  5. 异步加载 → Page 加载不阻塞主线程
  6. Page 压缩 → 使用 ASTC/BC 压缩减少传输带宽

═══ VT 内存预算参考 ═══

┌──────────────────┬──────────────────┬──────────────────┐
│ Cache Size        │ 内存占用          │ 适用场景          │
├──────────────────┼──────────────────┼──────────────────┤
│ 2048 × 2048      │ ~16 MB           │ 移动端            │
│ 4096 × 4096      │ ~64 MB           │ 主流 PC           │
│ 8192 × 8192      │ ~256 MB          │ 高端 PC / 主机    │
└──────────────────┴──────────────────┴──────────────────┘
```

## 9. LOD 系统

### 9.1 基于距离的 LOD

```
═══ 最基础的 LOD 方案 ═══

  根据 Chunk 中心到相机的距离，选择不同精度的 Mesh：

  LOD0: 64×64 顶点（距离 < 100m）
  LOD1: 32×32 顶点（距离 100~200m）
  LOD2: 16×16 顶点（距离 200~400m）
  LOD3:  8×8  顶点（距离 > 400m）

  距离阈值计算（对数分布）：
  D_i = D_near × (D_far / D_near) ^ (i / N)

  例：D_near=50, D_far=800, N=4
  LOD0: 50m
  LOD1: 100m
  LOD2: 200m
  LOD3: 400m
```

### 9.2 Clipmap LOD

```
═══ Clipmap 原理 ═══

  Clipmap = 以相机为中心的同心环形网格
  每一环的网格密度递减（类似 Mipmap 的空间版本）

  俯视图：
  ┌─────────────────────────────────────┐
  │  Level 3 (最低精度，最大范围)         │
  │  ┌─────────────────────────────┐    │
  │  │  Level 2                    │    │
  │  │  ┌─────────────────────┐    │    │
  │  │  │  Level 1            │    │    │
  │  │  │  ┌─────────────┐    │    │    │
  │  │  │  │  Level 0    │    │    │    │
  │  │  │  │  (最高精度)  │    │    │    │
  │  │  │  │  [Camera]   │    │    │    │
  │  │  │  └─────────────┘    │    │    │
  │  │  └─────────────────────┘    │    │
  │  └─────────────────────────────┘    │
  └─────────────────────────────────────┘

  特点：
  • 每一环的顶点数相同（如 64×64）
  • 但覆盖的世界空间面积翻倍
  • 相机移动时，环形网格跟随移动
  • 不需要 Chunk 概念，整个地形是连续的

  优势：
  • 无 Chunk 边界 → 无接缝问题
  • LOD 过渡自然（环形渐变）
  • 顶点总数固定 → 性能可预测

  劣势：
  • 实现复杂
  • 不适合流式加载（整体更新）
  • 相机快速移动时更新开销大
```

### 9.3 CDLOD（Chunked Distance LOD）

```
═══ CDLOD 原理 ═══

  结合 Chunk 分块和 Clipmap 思想：
  • 使用四叉树管理 Chunk
  • 每个 Chunk 的 LOD 基于距离
  • 相邻 Chunk 的 LOD 差最多为 1（约束条件）
  • 使用 Morphing 平滑过渡

  这是目前最主流的地形 LOD 方案

  四叉树 LOD 选择：
  ┌───────────────────────────────────────┐
  │  从根节点开始递归：                     │
  │  if (距离 < 当前层级的阈值)             │
  │      → 继续细分（展开子节点）           │
  │  else                                  │
  │      → 在当前层级渲染（不再细分）       │
  │                                        │
  │  约束：相邻节点 LOD 差 ≤ 1              │
  │  → 如果违反，强制提升低 LOD 节点        │
  └───────────────────────────────────────┘
```

### 9.4 Geometry Clipmap

```
═══ Geometry Clipmap（GPU Gems 2, 2005）═══

  经典的 GPU 地形渲染方案：

  核心思想：
  • 预生成一组同心环形 Mesh（L 形条带）
  • 每一环的世界空间尺寸翻倍
  • 顶点着色器中采样 Heightmap 设置高度
  • 相机移动时，Mesh 跟随移动（Toroidal Update）

  Mesh 结构：
  ┌─────────────────────────────────┐
  │  ┌───────────────────────────┐  │
  │  │  ┌───────────────────┐    │  │
  │  │  │  ┌─────────────┐  │    │  │
  │  │  │  │  Level 0    │  │    │  │
  │  │  │  │  (Full Grid)│  │    │  │
  │  │  │  └─────────────┘  │    │  │
  │  │  │  Level 1 (L-shape)│    │  │
  │  │  └───────────────────┘    │  │
  │  │  Level 2 (L-shape)        │  │
  │  └───────────────────────────┘  │
  │  Level 3 (L-shape)              │
  └─────────────────────────────────┘

  每一环只渲染 L 形区域（不包含内部已被更高精度覆盖的部分）

  性能特点：
  • 顶点总数固定：N_levels × (GridSize² - InnerSize²)
  • DrawCall 固定：每层 1~4 个 DrawCall
  • 带宽固定：每帧只更新移动方向的一行/列
```

## 10. 地形法线与光照

### 10.1 法线生成方式

```
═══ 方式 1：从 Heightmap 实时计算 ═══

  采样相邻 4 个高度值，计算梯度：
```

```c
float3 CalcTerrainNormal(sampler2D heightMap, float2 uv, float texelSize, float heightScale)
{
    float hL = tex2D(heightMap, uv + float2(-texelSize, 0)).r * heightScale;
    float hR = tex2D(heightMap, uv + float2( texelSize, 0)).r * heightScale;
    float hD = tex2D(heightMap, uv + float2(0, -texelSize)).r * heightScale;
    float hU = tex2D(heightMap, uv + float2(0,  texelSize)).r * heightScale;
    
    // Cross product of tangent vectors
    float3 normal = normalize(float3(hL - hR, 2.0 * texelSize, hD - hU));
    return normal;
}
// ✅ Pros: No extra texture, always up-to-date
// ❌ Cons: 4 extra texture samples per pixel, lower quality at low mip
```

```
═══ 方式 2：预烘焙 Normal Map ═══

  离线计算法线，存储为 RGB 纹理
  运行时直接采样，无需额外计算

  ✅ Pros: 高质量，无运行时开销
  ❌ Cons: 额外内存，修改 Heightmap 后需重新烘焙

═══ 方式 3：混合方案（推荐）═══

  近处：采样预烘焙 Normal Map（高质量）
  远处：从 Heightmap 实时计算（节省内存）
  过渡区域：lerp 混合
```

### 10.2 地形阴影

```
═══ 地形阴影的特殊挑战 ═══

  1. 面积大 → Shadow Map 分辨率不足
     → 使用 CSM（Cascaded Shadow Maps）
     → 近处高精度，远处低精度

  2. 自阴影 → 地形自身遮挡自身
     → 需要正确的 Shadow Bias
     → Normal Offset Bias 对地形效果好

  3. 远距离阴影 → 超出 CSM 范围
     → 使用 Distance Shadow（预烘焙远景阴影）
     → 或 Heightmap 自阴影（Ray March）

═══ Heightmap 自阴影（高级）═══

  在 Heightmap 上做 Ray March，计算每个点是否被遮挡：
```

```c
// Heightmap-based self-shadowing (ray march along light direction)
float TerrainSelfShadow(sampler2D heightMap, float2 uv, float3 lightDir,
                         float heightScale, int steps)
{
    float2 lightUVDir = normalize(lightDir.xz) * _TexelSize;
    float heightStep = lightDir.y / length(lightDir.xz) * _TexelSize * heightScale;
    
    float currentHeight = tex2D(heightMap, uv).r * heightScale;
    float2 currentUV = uv;
    float rayHeight = currentHeight;
    
    for (int i = 1; i <= steps; i++)
    {
        currentUV += lightUVDir;
        rayHeight += heightStep;
        
        float sampledHeight = tex2D(heightMap, currentUV).r * heightScale;
        
        if (sampledHeight > rayHeight)
            return 0.0; // In shadow
    }
    
    return 1.0; // Not in shadow
}
```

## 11. 性能优化总览

```
═══ 地形渲染性能优化 Checklist ═══

  CPU 优化：
  □ 视锥剔除使用 Burst Job（并行化）
  □ LOD 计算使用脏标记（只在相机移动时更新）
  □ Chunk 加载使用异步 + 对象池
  □ 每帧限制加载/卸载数量（避免帧率尖峰）
  □ DrawCall 合批（同 LOD 的 Chunk 合并）

  GPU 优化：
  □ Heightmap 使用 R16 格式（不要 R32F）
  □ Splatmap 层数控制（移动端 ≤ 4 层）
  □ 远处 Chunk 使用简化 Shader（减少纹理采样）
  □ 使用 Texture Array 替代多个独立纹理
  □ LOD 过渡使用 Morphing（避免突变）

  内存优化：
  □ 流式加载（只加载可见区域）
  □ 纹理压缩（ASTC / ETC2）
  □ Heightmap Read/Write 关闭
  □ 对象池复用 Chunk 资源
  □ 远处使用低精度数据

  带宽优化：
  □ Virtual Texture 减少纹理采样
  □ Mipmap Streaming
  □ 半精度（half）计算
  □ 减少 RenderTexture 切换

═══ 性能预算参考 ═══
```

## 13. 面试高频问题

```
═══ Q1: 地形为什么要分块（Chunk）？分块大小怎么选？ ═══

A:
分块原因：
• 视锥剔除：不在视野内的 Chunk 不渲染
• LOD：每个 Chunk 独立 LOD，近处高精度远处低精度
• 流式加载：按需加载/卸载，控制内存
• 合批：同 LOD 的 Chunk 可以 GPU Instancing 合并

分块大小选择：
• 太小（32m）：Chunk 数量多，管理开销大
• 太大（256m）：LOD 粒度粗，剔除效率低
• 推荐 64m × 64m（平衡点）
• 移动端可用 128m（减少 Chunk 数量）

═══ Q2: 地形 LOD 切换时如何避免裂缝和突变？ ═══

A:
裂缝修复（3 种方案）：
1. Skirt（裙边）：边缘顶点向下延伸遮挡裂缝，最简单
2. T-Junction 修复：高 LOD 边界顶点退化到低 LOD 网格上
3. 共享边界顶点：相邻 Chunk 边界使用相同顶点位置

突变修复：
• Vertex Morphing：LOD 切换时顶点平滑过渡
  奇数顶点逐渐 morph 到偶数邻居中点
  morphFactor 从 0→1 时安全切换 LOD

═══ Q3: Heightmap 用什么格式？精度怎么选？ ═══

A:
• R8 (8-bit): 256 级精度，512m 范围 → 2m 精度 → 有阶梯感 ❌
• R16 (16-bit): 65536 级精度，512m 范围 → 0.78cm 精度 → 推荐 ✅
• R32F (32-bit): 浮点精度 → 超大地图需要，一般浪费 ⚠️

GPU 采样格式：RHalf (16-bit float) 最佳
存储格式：R16 RAW 或 16-bit PNG

═══ Q4: Virtual Texture 在地形中解决什么问题？ ═══

A:
核心问题：多层材质混合导致纹理采样过多
• 8 层材质 × 3 张纹理/层 = 24 次采样/像素 → 太重

VT 解决方案：
• RVT：将混合结果缓存到虚拟纹理，运行时只需 1 次采样
• SVT：超大纹理按需流式加载，不全部驻留显存

工作流程：
Feedback Buffer 收集 Page 需求 → Cache Miss 时渲染 Page → 缓存到物理纹理

═══ Q5: Houdini 生成地形的工作流是什么？ ═══

A:
1. Heightfield 节点创建基础地形（设置尺寸、分辨率）
2. Noise 叠加基础起伏
3. Erode 水力侵蚀（河谷、冲积扇）
4. Slump 热力侵蚀（崩塌、滑坡）
5. Mask by Feature 基于坡度/高度生成遮罩
6. 遮罩驱动材质分配（Splatmap）
7. 导出 R16 Heightmap + RGBA Splatmap
8. 引擎导入

自动化：封装为 HDA，在引擎中直接调参
批量处理：PDG 分 Tile 并行生成

═══ Q6: 地形细分（Tessellation）的原理？移动端怎么办？ ═══

A:
原理：
• Hull Shader 控制细分级别（基于距离）
• Tessellator 硬件生成新顶点
• Domain Shader 采样 Heightmap 设置高度（Displacement）
• 近处高细分，远处低细分 → 自适应精度

移动端替代：
• 预生成多级 LOD Mesh（最常用）
• Vertex Texture Fetch（VTF）在 VS 中采样 Heightmap
• 两者结合 + GPU Instancing

═══ Q7: Splatmap 多层混合的性能问题怎么优化？ ═══

A:
问题：N 层 = N 次纹理采样 + 2 次 Splatmap 采样

优化方案：
1. Texture Array：所有层打包到一个 Texture2DArray，1 个 Sampler
2. Virtual Texture：混合结果缓存，运行时 1 次采样
3. Height-Based Blending：比线性混合更自然，不增加采样
4. 距离降级：远处减少层数（8→4→2→1）
5. 跳过零权重层：if (weight > 0.01) 才采样
6. 移动端限制 4 层以内

═══ Q8: Triplanar Mapping 是什么？什么时候用？ ═══

A:
问题：陡峭表面（悬崖）UV 映射拉伸严重
解决：从 XY/XZ/YZ 三个平面分别投影纹理，按法线方向混合

用途：悬崖、洞穴、垂直岩壁
代价：3× 纹理采样 → 只在需要的区域使用
优化：只对坡度 > 阈值的区域启用 Triplanar

═══ Q9: Geometry Clipmap 和 CDLOD 的区别？ ═══

A:
Geometry Clipmap：
• 同心环形 Mesh，每环精度递减
• 无 Chunk 概念，整体连续
• 顶点总数固定，性能可预测
• 不适合流式加载

CDLOD（Chunked Distance LOD）：
• 四叉树管理 Chunk
• 每个 Chunk 独立 LOD
• 支持流式加载
• 相邻 LOD 差约束 ≤ 1
• 目前最主流的方案

═══ Q10: 地形渲染的主要性能瓶颈在哪里？ ═══

A:
1. CPU: DrawCall 数量 → GPU Instancing 合批
2. VS: 顶点数量 → LOD + 视锥剔除
3. PS: 多层材质混合（最常见瓶颈）→ VT / Texture Array
4. 带宽: 大量纹理采样 → 纹理压缩 / VT
5. 内存: Heightmap + Splatmap + Collision → 流式加载 / 降精度

移动端最大瓶颈通常是 PS（Fragment Bound）
PC 端最大瓶颈通常是带宽或 DrawCall
```
