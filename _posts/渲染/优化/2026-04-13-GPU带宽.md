## 1. 带宽基础概念

### 1.1 什么是带宽

带宽（Bandwidth）是 GPU 与显存（DRAM/LPDDR）之间的数据传输速率，是移动端渲染的**第一性能瓶颈**。

```
带宽的本质：
  GPU Core ←→ DRAM 之间每秒能传输的数据量

  带宽 = 总线位宽 × 时钟频率 × 通道数
  例: LPDDR5 4266MHz × 32bit × 2通道 = 34.1 GB/s

为什么带宽是移动端第一瓶颈：
  1. 功耗：一次 DRAM 访问 ≈ 100-200 次 ALU 运算的功耗
  2. 延迟：DRAM 访问延迟 100-300 个 GPU 时钟周期
  3. 发热：带宽消耗是移动端发热的主要来源
  4. 降频：过高的带宽使用导致热降频，帧率断崖式下降
```

### 1.2 带宽 vs 延迟 vs 吞吐量

```
三个关键概念的区别：

  带宽 (Bandwidth):
    定义: 单位时间内的最大数据传输量
    类比: 高速公路的车道数 × 限速
    单位: GB/s
    影响: 决定了每帧能处理多少数据

  延迟 (Latency):
    定义: 单次数据请求从发出到返回的时间
    类比: 从出发到目的地的单程时间
    单位: ns (纳秒) 或 GPU clock cycles
    影响: 决定了 GPU 等待数据的时间

  吞吐量 (Throughput):
    定义: 单位时间内实际完成的有效数据传输量
    类比: 实际通过高速公路的车辆数
    单位: GB/s (通常 < 理论带宽)
    影响: 受 Cache 命中率、访问模式影响

  关系:
    实际吞吐量 ≤ 理论带宽
    吞吐量 = 带宽 × 利用率
    利用率受延迟、Cache 命中率、访问模式影响
```

### 1.3 功耗模型

```
各类操作的相对功耗（以一次 ALU 运算为基准 = 1）：

  操作类型                     相对功耗       绝对功耗 (pJ)
  ─────────────────────────────────────────────────────
  ALU (FP32 add/mul)           1×            ~1 pJ
  ALU (FP16 add/mul)           0.25-0.5×     ~0.25-0.5 pJ
  Register File Read           1-2×          ~1-2 pJ
  L1 Cache Read (On-chip)      5-10×         ~5-10 pJ
  L2 Cache Read (On-chip)      20-50×        ~20-50 pJ
  Tile Memory Read (TBR)       5-10×         ~5-10 pJ
  DRAM Read (Off-chip)         100-200×      ~100-200 pJ
  DRAM Write (Off-chip)        100-200×      ~100-200 pJ
  ─────────────────────────────────────────────────────

  关键洞察:
    一次 DRAM 访问的功耗 = 100-200 次 ALU 运算
    减少 1 MB 的 DRAM 访问 ≈ 节省数百万次 ALU 运算的功耗
    这就是为什么"带宽优化"本质上是"功耗优化"
```

## 3. 一帧的带宽消耗解剖

### 3.1 带宽消耗公式

```
单个 Render Target 的带宽消耗:

  BW_read  = Width × Height × BytesPerPixel × LoadAction
  BW_write = Width × Height × BytesPerPixel × StoreAction
  BW_total = BW_read + BW_write

  其中:
    LoadAction  = 1 (Load) 或 0 (DontCare/Clear)
    StoreAction = 1 (Store) 或 0 (DontCare)

  例: 1080p RGBA8 Render Target
    Size = 1920 × 1080 × 4 bytes = 8.29 MB
    Load + Store = 8.29 × 2 = 16.59 MB
    DontCare + Store = 8.29 × 1 = 8.29 MB (save 50%)

  例: 1080p RGBA16F (HDR) Render Target
    Size = 1920 × 1080 × 8 bytes = 16.59 MB
    Load + Store = 16.59 × 2 = 33.18 MB
```

### 3.2 典型一帧的带宽消耗明细

```
典型 URP Forward 渲染一帧的带宽消耗 (1080p):

  ┌─────────────────────────────────────────────────────────────────┐
  │ Pass                    │ RT Format   │ Load  │ Store │ BW (MB) │
  ├─────────────────────────────────────────────────────────────────┤
  │ Shadow Map (2048²)      │ D16         │ Clear │ Store │  8.0    │
  │ Depth Prepass           │ D32F        │ Clear │ Store │  8.3    │
  │ Opaque Color            │ RGBA16F     │ Clear │ Store │ 16.6    │
  │ Opaque Depth (r/w)      │ D32F        │ Load  │ Store │ 16.6    │
  │ Copy Depth              │ R32F        │ DC    │ Store │  8.3    │
  │ Copy Color (opaque tex) │ RGBA16F     │ DC    │ Store │ 16.6    │
  │ Transparent Color       │ RGBA16F     │ Load  │ Store │ 33.2    │
  │ Transparent Depth (r)   │ D32F        │ Load  │ DC    │  8.3    │
  │ Post Process Pass 1     │ RGBA16F     │ DC    │ Store │ 16.6    │
  │ Post Process Pass 2     │ RGBA16F     │ Load  │ Store │ 33.2    │
  │ Final Blit to Screen    │ RGBA8       │ DC    │ Store │  8.3    │
  ├─────────────────────────────────────────────────────────────────┤
  │ Subtotal: Render Targets                       │      │ 174.0   │
  ├─────────────────────────────────────────────────────────────────┤
  │ Texture Sampling (est.)                        │      │  30-80  │
  │ Vertex Buffer Reads (est.)                     │      │  5-15   │
  │ CBuffer / Uniform Reads                        │      │  1-3    │
  │ Overdraw (1.5x average)                        │      │ +50%    │
  ├─────────────────────────────────────────────────────────────────┤
  │ TOTAL (estimated)                              │      │ 300-450 │
  └─────────────────────────────────────────────────────────────────┘

  At 60 FPS:
    300-450 MB/frame × 60 = 18-27 GB/s

  Budget check:
    High-end mobile (25 GB/s available): TIGHT ⚠️
    Mid-range mobile (10 GB/s available): OVER BUDGET ❌
    Low-end mobile (5 GB/s available): FAR OVER BUDGET ❌❌
```

**RGBA8 1080p 带宽计算**

```
RGBA8 格式:
  R = 8 bit, G = 8 bit, B = 8 bit, A = 8 bit
  总计 = 32 bit = 4 bytes per pixel

1080p 分辨率:
  1920 × 1080 = 2,073,600 pixels

单次访问大小:
  2,073,600 × 4 bytes = 8,294,400 bytes ≈ 8.29 MB ≈ 7.91 MiB 
```

### 3.3 Overdraw 对带宽的乘数效应

```
Overdraw 的带宽影响:

  Overdraw = 实际着色的像素数 / 屏幕像素数

  理想值: 1.0x (每个像素只着色一次)
  可接受: 1.5-2.0x
  危险:   2.0-3.0x
  灾难:   3.0x+

  带宽乘数效应:
    每增加 1x Overdraw:
      - Fragment Shader 的纹理采样带宽 × 2
      - Color RT 的读写带宽 × 2 (if blending)
      - 不影响 Vertex 带宽和 Shadow Map 带宽

  例: 1080p RGBA16F, 2x Overdraw
    无 Overdraw: 16.6 MB (color write)
    2x Overdraw: 16.6 × 2 = 33.2 MB (color write)
    加上纹理采样: 额外 20-40 MB
    总增加: ~35-55 MB per frame

  Overdraw 的主要来源:
    1. 透明物体 (UI, 粒子, 半透明特效)
    2. 不正确的渲染顺序 (不透明物体未 front-to-back)
    3. 全屏后处理 Pass (每个 Pass = 1x Overdraw)
    4. 大面积 Skybox 被不透明物体遮挡但仍渲染
```

## 4. 移动端 TBR 架构的带宽模型

### 4.1 TBR vs IMR 带宽对比

```
IMR (Immediate Mode Rendering) — 桌面端:

  每个三角形立即处理完整管线:
    Vertex → Rasterize → Fragment → Write to DRAM

  带宽模式:
    每个 Fragment 直接读写 DRAM 中的 Color/Depth Buffer
    Overdraw 直接导致 DRAM 读写翻倍
    依赖大 L2 Cache 缓解带宽压力

  ┌──────────┐      ┌──────────┐
  │ Fragment │────▶│  DRAM    │  每个 Fragment 都访问 DRAM
  │ Shader   │◀────│ Color/Z  │  带宽消耗 = 像素数 × Overdraw × BPP
  └──────────┘      └──────────┘


TBR (Tile-Based Rendering) — 移动端:

  Phase 1: Geometry (Binning)
    所有顶点处理完成，三角形分配到 Tiles
    写入 Tile List 到 DRAM (一次性)

  Phase 2: Fragment (per Tile)
    每个 Tile (16×16 or 32×32 pixels):
      1. Load Tile from DRAM → Tile Memory (if needed)
      2. Process ALL fragments for this tile ON-CHIP
      3. Store Tile from Tile Memory → DRAM (if needed)

  ┌──────────┐      ┌──────────────┐     ┌──────────┐
  │ Fragment │────▶│ Tile Memory  │────▶│  DRAM    │
  │ Shader   │◀────│ (On-chip)    │     │ (最终结果)│
  └──────────┘      └──────────────┘     └──────────┘
                   ↑ 片上高速缓存                ↑ 只在 Tile 开始/结束时访问
                   ↑ Overdraw 不增加 DRAM 带宽!

  TBR 带宽优势:
    - Fragment 的 Color/Depth 读写在 Tile Memory 完成 (片上)
    - Overdraw 只增加 Tile Memory 带宽 (片上, 几乎免费)
    - DRAM 带宽 = Tile 数量 × Tile 大小 × (Load + Store)
    - 与 Overdraw 无关!
```

### 4.2 TBR Tile 生命周期与带宽

```
单个 Tile 的生命周期:

  ┌────────────────────────────────────────────────────────┐
  │ Step 1: Load (DRAM → Tile Memory)                      │
  │   LoadAction.Load:     Read tile data from DRAM        │
  │   LoadAction.Clear:    Initialize tile to clear value  │
  │   LoadAction.DontCare: Skip (undefined content)        │
  │                                                        │
  │   Bandwidth cost:                                      │
  │     Load:     TileWidth × TileHeight × BPP             │
  │     Clear:    0 (hardware clear, no DRAM access)       │
  │     DontCare: 0 (skip entirely)                        │
  ├────────────────────────────────────────────────────────┤
  │ Step 2: Render (all on Tile Memory)                    │
  │   All fragments for this tile are processed            │
  │   Color read/write: Tile Memory (on-chip, fast)        │
  │   Depth read/write: Tile Memory (on-chip, fast)        │
  │   Blending: Tile Memory (on-chip, fast)                │
  │                                                        │
  │   Bandwidth cost: 0 DRAM bandwidth                     │
  │   (all on-chip, regardless of Overdraw!)               │
  ├────────────────────────────────────────────────────────┤
  │ Step 3: Store (Tile Memory → DRAM)                     │
  │   StoreAction.Store:    Write tile data to DRAM        │
  │   StoreAction.Resolve:  MSAA resolve + write to DRAM   │
  │   StoreAction.DontCare: Skip (discard content)         │
  │                                                        │
  │   Bandwidth cost:                                      │
  │     Store:    TileWidth × TileHeight × BPP             │
  │     Resolve:  TileWidth × TileHeight × BPP (resolved)  │
  │     DontCare: 0 (skip entirely)                        │
  └────────────────────────────────────────────────────────┘

  最优情况: LoadAction.DontCare + StoreAction.DontCare
    = 0 DRAM bandwidth (transient attachment, stays on-chip)

  最差情况: LoadAction.Load + StoreAction.Store
    = 2 × Width × Height × BPP (full round-trip)
```

### 4.3 什么会打破 TBR 的带宽优势

```
⚠️ TBR 带宽优势被破坏的场景:

  1. 中途读取 Render Target (Texture Fetch from active RT)
     ─────────────────────────────────────────────────────
     问题: 如果 Shader 需要采样当前正在写入的 RT
     后果: GPU 必须将所有 Tile 写回 DRAM，然后重新加载
     这叫做 "Tile Flush" 或 "Resolve"

     触发条件:
       - 在 Opaque Pass 中采样 _CameraColorTexture
         (如果它就是当前 Color RT)
       - 在同一个 Pass 中同时读写同一个 RT

     解决方案:
       - 使用 CopyColor/CopyDepth Pass 复制到单独的纹理
       - 使用 Vulkan/Metal Subpass (framebuffer fetch)
       - 分离读写到不同的 Pass

  2. 过多的 Render Pass 切换
     ─────────────────────────────────────────────────────
     问题: 每次切换 Render Target = 所有 Tile 必须 Store + Load
     后果: N 次 RT 切换 = N × (Store + Load) × 全屏像素

     例: 5 个全屏 Post Process Pass (1080p RGBA16F)
       5 × (16.6 + 16.6) = 166 MB 仅 RT 切换!

     解决方案:
       - 合并 Post Process Pass (Uber Shader)
       - 使用 Vulkan Subpass 合并相邻 Pass
       - 减少不必要的全屏 Pass

  3. MSAA 未正确 Resolve
     ─────────────────────────────────────────────────────
     问题: MSAA 4x 意味着每个像素存储 4 个样本
     后果: Tile Memory 需求 × 4，可能溢出到 DRAM
            Store 时如果不 Resolve，写入量 × 4

     解决方案:
       - 使用 StoreAction.Resolve (Resolve + Store 合一)
       - 不要 Store MSAA 数据再单独 Resolve
       - 移动端推荐 MSAA 2x 或 4x (不要 8x)

  4. 不正确的 Load/Store Actions
     ─────────────────────────────────────────────────────
     问题: 默认 LoadAction.Load 会读取上一帧的数据
     后果: 每个 RT 多一次全屏读取

     常见错误:
       ❌ Clear RT 但 LoadAction 设为 Load (先读再清)
       ❌ 临时 RT 的 StoreAction 设为 Store (写回但不再使用)
       ❌ Depth RT 在最后一个 Pass 后仍 Store

     解决方案:
       - 如果要 Clear: 使用 LoadAction.Clear 或 DontCare
       - 如果 RT 不再使用: 使用 StoreAction.DontCare
       - 审查每个 Pass 的 Load/Store 配置
```

### 4.4 各厂商 TBR 实现差异

```
ARM Mali:
  架构: TBR (Tile-Based Rendering)
  Tile 大小: 16×16 pixels
  特有技术:
    - FPK (Forward Pixel Kill): 在 Fragment Shader 执行前剔除被遮挡像素
    - AFBC (Arm Frame Buffer Compression): 无损帧缓冲压缩
    - Transaction Elimination: 跳过未变化的 Tile 写回
  带宽优化:
    - FPK 减少无效 Fragment 的纹理采样带宽
    - AFBC 可节省 30-50% RT 带宽
    - Transaction Elimination 在静态场景可节省 50-90% 写回带宽

Qualcomm Adreno:
  架构: Flex Render (可切换 TBR/IMR)
  Tile 大小: 动态 (通常 256×256 到 512×512)
  特有技术:
    - LRZ (Low Resolution Z): 低分辨率深度预测试
    - UBWC (Universal Bandwidth Compression): 通用带宽压缩
    - FlexRender: 根据场景自动选择 TBR 或 Direct 模式
  带宽优化:
    - UBWC 可节省 25-50% 带宽 (纹理 + RT)
    - LRZ 在 Binning 阶段剔除不可见像素
    - 大 Tile 减少 Tile List 开销

Apple GPU:
  架构: TBDR (Tile-Based Deferred Rendering)
  Tile 大小: 32×32 pixels
  特有技术:
    - HSR (Hidden Surface Removal): 硬件级完美像素剔除
    - Lossless Compression: 无损 RT 压缩
    - Memoryless RT: 完全不分配 DRAM 的临时 RT
  带宽优化:
    - HSR 保证每个像素只执行一次 Fragment Shader (0 Overdraw)
    - Memoryless RT: 0 DRAM 带宽 (完全片上)
    - Programmable Blending: 片上读取 Color (无需 DRAM)

  带宽效率排名 (同等场景):
    Apple GPU > Adreno > Mali
    (Apple 的 HSR + Memoryless 组合最强)
```

## 6. 带宽优化技术与量化收益

### 6.1 Load/Store Action 优化

```
技术: 正确配置每个 Pass 的 Load/Store Actions

  优化前 (默认):
    每个 RT: LoadAction.Load + StoreAction.Store
    1080p RGBA16F: 16.6 + 16.6 = 33.2 MB per RT per pass

  优化后:
    Shadow Map:    Load=Clear,  Store=Store     → 8.0 MB (save 8.0 MB)
    Depth Prepass: Load=Clear,  Store=Store     → 8.3 MB (save 8.3 MB)
    Opaque Color:  Load=Clear,  Store=Store     → 16.6 MB (save 16.6 MB)
    Opaque Depth:  Load=Load,   Store=Store     → 16.6 MB (no save)
    Temp PostFX:   Load=DC,     Store=DC        → 0 MB (save 33.2 MB!)
    Final Depth:   Load=Load,   Store=DontCare  → 8.3 MB (save 8.3 MB)

  量化收益: 节省 50-100 MB/帧 (30-50% RT 带宽)
```

```csharp
// URP ScriptableRenderPass: configure load/store actions
public override void OnCameraSetup(CommandBuffer cmd, ref RenderingData renderingData) {
    // Shadow map: clear on load, store result
    ConfigureTarget(shadowmapRT);
    ConfigureClear(ClearFlag.All, Color.clear);
    // Internally sets LoadAction.Clear, StoreAction.Store
}

// For passes that don't need previous content:
public override void OnCameraSetup(CommandBuffer cmd, ref RenderingData renderingData) {
    ConfigureTarget(tempRT, RenderBufferLoadAction.DontCare, RenderBufferStoreAction.Store);
    // Skip loading previous content = save one full-screen read
}

// For transient RTs (content not needed after pass):
public override void OnCameraCleanup(CommandBuffer cmd) {
    // Mark depth as not needed after transparent pass
    ConfigureTarget(depthRT, RenderBufferLoadAction.Load, RenderBufferStoreAction.DontCare);
    // Skip writing back = save one full-screen write
}
```

### 6.2 Vulkan/Metal Subpass 合并

```
技术: 使用 Subpass 将多个 Pass 合并为一个 Render Pass

  原理:
    Subpass 允许后续 Pass 直接从 Tile Memory 读取前一个 Pass 的输出
    无需 Store → DRAM → Load 的往返

  优化前 (3 个独立 Pass):
    Pass 1 (Opaque):     Write Color → DRAM (16.6 MB)
    Pass 2 (Lighting):   Read Color ← DRAM (16.6 MB) + Write → DRAM (16.6 MB)
    Pass 3 (Composite):  Read Color ← DRAM (16.6 MB) + Write → DRAM (16.6 MB)
    Total: 83 MB

  优化后 (1 个 Render Pass, 3 个 Subpass):
    Subpass 1 (Opaque):     Write Color → Tile Memory (0 DRAM)
    Subpass 2 (Lighting):   Read Color ← Tile Memory (0 DRAM) + Write → Tile Memory
    Subpass 3 (Composite):  Read Color ← Tile Memory (0 DRAM) + Write → DRAM (16.6 MB)
    Total: 16.6 MB

  量化收益: 节省 66.4 MB (80% 减少!)

  URP 中的 Subpass:
    - Native Render Pass (Unity 2020.2+)
    - 自动合并相邻 Pass 为 Subpass
    - 需要 Vulkan/Metal 后端
    - 通过 FrameDebugger 验证合并是否生效
```

### 6.3 Memoryless Render Target (Apple/Vulkan)

```
技术: 声明 RT 为 Memoryless，完全不分配 DRAM

  原理:
    Memoryless RT 只存在于 Tile Memory
    不分配 DRAM 存储
    Load/Store 都是 DontCare
    适用于中间结果 RT (depth, stencil, MSAA resolve source)

  适用场景:
    ✅ Depth attachment (if not sampled later)
    ✅ MSAA color attachment (resolved to non-MSAA target)
    ✅ Stencil attachment (if not sampled later)
    ❌ 需要在后续 Pass 中采样的 RT

  量化收益:
    1080p D32F Memoryless: 节省 8.3 MB/帧 (Load) + 8.3 MB/帧 (Store) = 16.6 MB
    1080p MSAA 4x Color Memoryless: 节省 66.4 MB/帧

  Unity 配置:
    RenderTextureMemoryless.Depth
    RenderTextureMemoryless.Color
    RenderTextureMemoryless.MSAA
```

```csharp
// Create a memoryless depth RT
RenderTextureDescriptor desc = new RenderTextureDescriptor(width, height);
desc.colorFormat = RenderTextureFormat.Depth;
desc.depthBufferBits = 32;
desc.memorylessMode = RenderTextureMemoryless.Depth;
// This RT exists ONLY in Tile Memory — 0 DRAM allocation, 0 DRAM bandwidth
```

### 6.4 帧缓冲压缩 (AFBC / UBWC)

```
技术: 硬件级无损帧缓冲压缩

  ARM Mali — AFBC (Arm Frame Buffer Compression):
    压缩率: 30-50% (场景依赖)
    适用: Color RT, Depth RT, Textures
    条件: RT 格式必须支持 AFBC (大部分标准格式支持)
    注意: 某些操作会破坏 AFBC (如 CPU readback)

  Qualcomm Adreno — UBWC (Universal Bandwidth Compression):
    压缩率: 25-50%
    适用: Color RT, Depth RT, Textures, YUV surfaces
    条件: 需要正确的内存布局 (UBWC layout)
    注意: 非 UBWC 布局的纹理无法享受压缩

  Apple GPU — Lossless Compression:
    压缩率: 30-50%
    适用: 所有 RT 和纹理
    条件: 自动启用，无需配置

  量化收益 (1080p RGBA16F, 假设 40% 压缩率):
    未压缩: 16.6 MB per RT access
    压缩后: 16.6 × 0.6 = 9.96 MB
    节省: 6.64 MB per RT access
    一帧 10 次 RT 访问: 节省 ~66 MB

  开发者注意:
    - 这些压缩是硬件自动完成的
    - 开发者需要做的是"不要破坏它"
    - 避免: CPU readback, 不支持的格式, 错误的内存布局
    - 使用 GPU profiler 验证压缩是否生效
```

### 6.5 Transaction Elimination (Mali)

```
技术: 跳过未变化的 Tile 写回

  原理:
    Mali GPU 在 Store 阶段比较 Tile 内容与 DRAM 中的旧内容
    如果 Tile 内容未变化，跳过写回
    使用 CRC 校验快速比较

  适用场景:
    - 静态 UI 区域 (大部分帧不变)
    - 天空盒区域 (相机不动时不变)
    - 阴影贴图 (光源不动时不变)

  量化收益:
    静态场景: 可节省 50-90% 的 Store 带宽
    动态场景: 通常节省 10-30%
    UI 界面: 可节省 70-90%

  开发者注意:
    - 自动启用，无需配置
    - 确保 LoadAction 正确 (Load, not Clear/DontCare)
    - 如果每帧 Clear 整个 RT，Transaction Elimination 无效
      (因为 Clear 后内容一定"变化"了)
```

### 7.2 Mipmap 与带宽

```
Mipmap 对带宽的影响:

  没有 Mipmap:
    远处物体采样高分辨率纹理
    Cache 命中率极低 (相邻像素采样纹理中相距很远的 texel)
    带宽浪费 + 画面闪烁 (aliasing)

  有 Mipmap:
    远处物体采样低分辨率 mip level
    Cache 命中率高 (相邻像素采样相邻 texel)
    带宽节省 + 画面平滑

  Mipmap 内存开销:
    额外 33% 内存 (1 + 1/4 + 1/16 + ... = 4/3)
    但带宽节省远超内存开销

  量化收益:
    无 Mipmap: 每次采样可能触发 Cache Miss → 128-256 bytes from DRAM
    有 Mipmap: 大部分采样 Cache Hit → 0 bytes from DRAM
    典型场景带宽节省: 50-80%

  ⚠️ 必须开启 Mipmap 的情况:
    - 3D 场景中的所有纹理
    - 会在不同距离观察的物体

  ⚠️ 不需要 Mipmap 的情况:
    - UI 纹理 (固定大小显示)
    - 全屏后处理纹理
    - 精确像素对应的纹理 (如 LUT)
```

### 7.3 纹理 Streaming

```
纹理 Streaming 与带宽:

  问题: 加载所有纹理的最高 mip level 会:
    - 占用大量内存
    - 初始加载时间长
    - 不可见纹理浪费内存和带宽

  Unity Texture Streaming:
    - 根据相机距离动态加载/卸载 mip levels
    - 远处物体只加载低 mip (节省内存和带宽)
    - 靠近时异步加载高 mip

  带宽影响:
    - 减少常驻内存中的纹理数据量
    - 减少 GPU 纹理 Cache 压力
    - Mip 切换时有短暂的额外加载带宽

  配置:
    QualitySettings.streamingMipmapsActive = true;
    QualitySettings.streamingMipmapsMemoryBudget = 256; // MB
    QualitySettings.streamingMipmapsMaxLevelReduction = 2;
```

## 8. Render Target 带宽优化

### 8.1 RT 格式选择

```
Render Target 格式与带宽:

  格式              BPP    1080p 大小   适用场景
  ──────────────────────────────────────────────────────
  RGBA8             32     8.3 MB       LDR 最终输出
  RGB565            16     4.1 MB       低质量 LDR
  RGBA16F           64     16.6 MB      HDR 渲染 (标准)
  RG16F             32     8.3 MB       法线/速度缓冲
  R16F              16     4.1 MB       单通道 (AO, 深度)
  R11G11B10F        32     8.3 MB       HDR 无 Alpha ★
  RGB9E5            32     8.3 MB       HDR 共享指数
  D16               16     4.1 MB       Shadow Map ★
  D32F              32     8.3 MB       场景深度
  D24S8             32     8.3 MB       深度 + 模板
  ──────────────────────────────────────────────────────

  ★ 推荐优化:

  1. HDR Color: RGBA16F → R11G11B10F
     节省: 16.6 - 8.3 = 8.3 MB per access
     代价: 没有 Alpha 通道
     适用: 不需要 Alpha 的 HDR 场景

     节省: 16.6 - 8.3 = 8.3 MB per access
     代价: 需要编码/解码

  4. 速度缓冲: RGBA16F → RG16F
     节省: 16.6 - 8.3 = 8.3 MB per access
     代价: 无 (只需要 2 个通道)
```

## 9. Shader 采样与带宽

### 9.1 纹理采样的带宽成本

```
单次纹理采样的带宽成本:

  采样模式          Cache Hit    Cache Miss     说明
  ──────────────────────────────────────────────────────
  Point (Nearest)   0 bytes      64-128 bytes   1 texel           点
  Bilinear          0 bytes      128-256 bytes  4 texels          二线性
  Trilinear         0 bytes      256-512 bytes  8 texels (2 mips) 三线性
  Anisotropic 2x    0 bytes      512-1024 bytes 16 texels
  Anisotropic 4x    0 bytes      1-2 KB         32 texels
  Anisotropic 8x    0 bytes      2-4 KB         64 texels
  Anisotropic 16x   0 bytes      4-8 KB         128 texels
  ──────────────────────────────────────────────────────

  Cache Hit 时: 0 DRAM 带宽 (从 L1/L2 Cache 读取)
  Cache Miss 时: 需要从 DRAM 加载整个 Cache Line

  Cache 命中率的影响因素:
    1. Mipmap: 有 Mipmap → 高命中率
    2. UV 连续性: 连续 UV → 高命中率
    3. 纹理大小: 小纹理 → 更容易 fit in cache
    4. 同时使用的纹理数量: 越少 → 越高命中率
    5. 纹理压缩: 压缩纹理 → 更多 texel fit in cache

  典型 Cache 命中率:
    良好场景: 90-95% → 实际带宽 = 理论带宽 × 5-10%
    一般场景: 70-85% → 实际带宽 = 理论带宽 × 15-30%
    差场景:   50-70% → 实际带宽 = 理论带宽 × 30-50%
```

### 9.2 减少采样次数

```
减少纹理采样的技术:

  1. 合并纹理通道 (Channel Packing)
     ─────────────────────────────────────
     优化前: 4 次采样
       float metallic  = tex2D(_MetallicMap, uv).r;
       float roughness = tex2D(_RoughnessMap, uv).r;
       float ao        = tex2D(_AOMap, uv).r;
       float height    = tex2D(_HeightMap, uv).r;

     优化后: 1 次采样
       float4 packed = tex2D(_MaskMap, uv);
       // R=Metallic, G=AO, B=Height, A=Roughness

     节省: 3 次纹理采样 = 潜在 384-1536 bytes/pixel

  2. 使用 half 精度采样
     ─────────────────────────────────────
     half4 color = SAMPLE_TEXTURE2D(_MainTex, sampler_MainTex, uv);
     // half 精度采样可能使用更窄的数据路径
     // 某些 GPU 上可以提高采样吞吐量

  3. 避免 dependent texture reads
     ─────────────────────────────────────
     // BAD: dependent read (UV depends on another texture)
     float2 distortedUV = uv + tex2D(_DistortionMap, uv).rg;
     float4 color = tex2D(_MainTex, distortedUV);
     // GPU cannot prefetch — guaranteed cache miss

     // BETTER: compute UV in vertex shader if possible
     // Pass distorted UV from vertex shader

  4. 降低各向异性过滤级别
     ─────────────────────────────────────
     移动端推荐:
       地面/墙面: Aniso 2x-4x (视角倾斜明显)
       角色/道具: Bilinear 或 Aniso 2x
       UI: Point 或 Bilinear
       特效: Bilinear
```

### 9.3 Texture Atlas 与带宽

```
Texture Atlas 的带宽优势:

  问题: 多个小纹理 → 频繁切换纹理 → Cache 抖动

  解决: 合并为 Atlas → 单次绑定 → Cache 友好

  带宽影响:
    多纹理: 每个纹理占用独立的 Cache Line
             切换纹理时 Cache 被刷新
             Cache 命中率: 60-80%

    Atlas:   单个大纹理，连续的 Cache Line
             不需要切换纹理
             Cache 命中率: 85-95%

  注意事项:
    - Atlas 过大可能超出 Cache 容量
    - UV 边缘需要 padding 防止 bleeding
    - Mipmap 在 Atlas 边缘可能出问题
    - 推荐 Atlas 大小: 2048×2048 或 4096×4096
```

```
      - Limiter: Bandwidth:     是否带宽受限

  通用工具:
    - Unity Frame Debugger: 查看每个 Pass 的 RT 配置
    - RenderDoc:            查看 API 调用和资源绑定
    - Unity Profiler:       GPU Usage 模块
```

### 10.2 带宽瓶颈诊断流程

```
诊断流程:

  Step 1: 确认是否带宽受限
  ─────────────────────────────────────
    方法: 降低分辨率 50%
    如果帧时间显著下降 → 带宽/填充率受限
    如果帧时间不变 → 顶点/CPU 受限

  Step 2: 定位带宽消耗来源
  ─────────────────────────────────────
    方法: 逐个禁用 Pass，观察带宽变化
    a. 禁用 Post Processing → 带宽下降多少?
    b. 禁用 Shadow → 带宽下降多少?
    c. 替换所有材质为纯色 → 纹理采样带宽
    d. 减少透明物体 → Overdraw 带宽

  Step 3: 量化各项消耗
  ─────────────────────────────────────
    使用 GPU Profiler 读取硬件计数器:
    - RT Load/Store 带宽
    - 纹理采样带宽
    - 顶点数据带宽
    - 总 DRAM 读写

  Step 4: 对比预算
  ─────────────────────────────────────
    实际带宽 vs 目标设备可用带宽
    如果超出 → 按优先级优化

  Step 5: 验证优化效果
  ─────────────────────────────────────
    每次优化后重新测量
    确认带宽下降且画面质量可接受
```

### 10.3 常见带宽问题模式

```
模式 1: "RT 乒乓球"
  症状: 大量 RT 切换，每次都 Load + Store
  诊断: Frame Debugger 中看到频繁的 SetRenderTarget
  解决: 合并 Pass，使用 Subpass

模式 2: "未压缩纹理"
  症状: 纹理采样带宽异常高
  诊断: 检查纹理导入设置，是否使用了 RGBA32
  解决: 使用 ASTC 压缩

模式 3: "Mipmap 缺失"
  症状: 纹理 Cache 命中率低
  诊断: L2 Cache Miss Rate 高，远处物体纹理闪烁
  解决: 开启 Mipmap

模式 4: "Overdraw 爆炸"
  症状: 透明物体区域带宽飙升
  诊断: Overdraw 可视化显示大面积红色
  解决: 减少透明层数，使用 Alpha Test 替代 Alpha Blend

模式 5: "全屏 Pass 过多"
  症状: Post Processing 占用 50%+ 带宽
  诊断: 计算全屏 Pass 数量 × RT 大小
  解决: 合并 Pass，降低后处理分辨率

模式 6: "Shadow Map 过大"
  症状: Shadow Pass 带宽占比高
  诊断: Shadow Map 分辨率 4096×4096
  解决: 降低到 1024-2048，使用级联阴影
```

---

> **核心原则**: 带宽优化的本质是功耗优化。每减少 1 MB 的 DRAM 访问，不仅提升帧率，更重要的是降低功耗和发热，让设备能在更长时间内维持目标帧率。移动端渲染的终极目标不是"跑到最高帧率"，而是"在热预算内持续稳定运行"。
