# 2. 打包

## 2.1 打包规则

```
打包配置（正则表达式匹配，优先级从上到下）：

格式：资源路径正则 => 输出包名 : [options]

示例：
Assets/Characters/(.+)/(.+)\.prefab  => char_{1}_{2}  : immortal
Assets/Effects/(.+)\.prefab          => effect_{1}
Assets/UI/Panels/(.+)\.prefab        => ui_panel_{1}  : bigbundle
Assets/Scenes/(.+)\.unity            => scene_{1}

Options:
  immortal   - 常驻内存，不参与卸载
  bigbundle  - 大包标记，合并小资源
  alone      - 单独打包，不被其他资源引用
  alonedep   - 仅被引用一次时合并到引用方
```

## 2.2 依赖管理

```
依赖类型：
  Alone     - 单独资源，所有引用不拆分
  AloneDep  - 仅被引用一次时合并到引用方，否则单独打包
  Shared    - 被多个Bundle引用，单独打包避免冗余

冗余检查流程：
  1. 获取所有Bundle的依赖资源列表
  2. 统计每个资源被引用的次数
  3. 被引用 > 1 次的资源单独剔除打包
  4. 未被使用的资源不打包（减少包体）
```

## 2.3 Bundle 粒度建议

| 资源类型    | 建议粒度    | 参考大小 | 说明               |
| ----------- | ----------- | -------- | ------------------ |
| 场景 Bundle | 按场景      | 8MB      | 场景独立，按需加载 |
| 角色模型    | 按角色      | 2-5MB    | 含骨骼、动画、贴图 |
| 特效        | 按特效组    | 1-3MB    | 同类特效合并       |
| UI 面板     | 按模块      | 1-2MB    | 同模块UI合并       |
| 公共贴图    | 按图集      | 2-4MB    | Atlas合并          |
| 音效        | 按场景/模块 | 2-5MB    | 避免过细粒度       |

> **原则**：粒度越细越利于最小更新，但IO次数增多；大粒度减少IO寻址，但更新代价高。
> 建议使用配置文件管理，不要在代码中写死。

# 3. 文件系统

## 3.1 SFP (Streamable File Package) 文件系统

```
文件结构：
┌──────────────┐
│  FlagCode    │  ← 文件头加密，生成MD5码
├──────────────┤
│  引导块       │  ← 启动时读取，生成FileIndex
├──────────────┤
│  Bitmap      │  ← 数据块使用情况
├──────────────┤
│  目录项       │  ← 节点号 + 文件名（数字ID代替字符串路径）
├──────────────┤
│  String块    │  ← 数字路径映射表
├──────────────┤
│  数据块       │  ← 数据大小对齐，实际资源数据
└──────────────┘
```

## 3.2 IO 池与复用

```csharp
// IO池：复用FileStream，改文件句柄+偏移量
class IOPool {
    // 启动时预热200个FileStream，运行时保持30个缓冲
    Queue<FileStream> _pool = new Queue<FileStream>(200);

    FileStream Acquire(string path, long offset, long size) {
        if (_pool.Count > 0) {
            var fs = _pool.Dequeue();
            // 复用：修改偏移量
            return new OffsetStream(fs, offset, size);
        }
        return new FileStream(path, FileMode.Open, FileAccess.Read, FileShare.Read);
    }

    void Release(FileStream fs) {
        if (_pool.Count < 30)
            _pool.Enqueue(fs);
        else
            fs.Dispose();
    }
}
```

## 3.3 APK 读取（Android）

```
APK ZIP格式结构：
  0x04034b50  Local File Header  - 文件头，记录压缩信息
  0x02014b50  Central Directory  - 核心目录，记录所有文件偏移量
  0x06054b50  EOCD               - 目录结束标识

读取流程：
  1. 解析EOCD，获取Central Directory偏移量
  2. 遍历Central Directory，建立文件名→偏移量映射
  3. 运行时通过偏移量直接读取，无需解压
  4. 使用 baseFilePath + baseOffset + baseSize 定位资源
```

```csharp
// Android APK内资源读取
// 加入baseFilePath，baseOffset、baseSize，读取时累加偏移量
var stream = new OffsetStream(apkFileStream, baseOffset, baseSize);
var bundle = AssetBundle.LoadFromStream(stream);
```

# 4. 加载流程

## 4.1 完整调用链

```
业务调用
    │
    ▼
AssetManager.LoadGameObject(path, instance, callback)
    │
    ▼
ResLoadManager.LoadAssetAsync(path, type, priority, action)
    │
    ├── 检查缓存 → 命中 → 直接回调
    │
    ▼
添加到优先级队列 PriorityQueue
    │
    ▼
Tick() → TickLoading()
    │
    ├── 1. DoHttpReq()     网络下载（CDN/平台API）
    ├── 2. DoBundleLoad()  Bundle加载
    └── 3. DoAssetLoad()   Asset加载
            │
            ▼
        Addressables.LoadAssetAsync<T>(path)
            │
            ▼
        ResourceManager.ProvideResource(location)
            │
            ├── 生成依赖加载 ProvideResourceGroupCached()
            └── 递归加载依赖Bundle
                    │
                    ▼
                handle.Completed → OnHandleCompleted
                    │
                    ▼
                Tick() → TickCallback() → 分帧回调业务层
```

## 4.3 Addressables 内部流程

```csharp
// 1. 从key找到ResourceLocation[]
public AsyncOperationHandle<TObject> LoadAssetAsync<TObject>(string key) {
    foreach (var locatorInfo in m_ResourceLocators) {
        if (locator.Locate(key, out ResourceLocation[] locs)) { }
    }
    return LoadAssetAsync(locs);
}

// 2. 生成依赖加载树
AsyncOperationHandle<TObject> LoadAssetAsync(ResourceLocation location) {
    // 递归生成依赖加载
    AsyncOperationHandle<List<AsyncOperationHandle>> depOp = ProvideResourceGroupCached() {
        foreach (var loc in locations)
            ops.Add(ResourceManager.ProvideResource(loc, desiredType)); // 递归
    };
    // 依赖完成后加载本体
    handle.Completed += m_dependencyCompleteAction;
}

// 3. 完成回调，加入结果缓存
void OnHandleCompleted(AsyncOperationHandle handle) {
    if (handle.Status == AsyncOperationStatus.Succeeded) {
        if (!m_resultToHandle.ContainsKey(handle.Result)) {
            handle.Destroyed += m_OnHandleDestroyedAction;
            m_resultToHandle.Add(handle.Result, handle);
        }
    }
}
```

// 内部实现
public void LoadGameObject(string path, bool instance, in ActionClosure onLoadComplete) {
    var action = ActionClosure.Create(static (bool ins, ActionClosureWrapper cb, string path, GameObject obj) => {
        GameObject go = Object.Instantiate(obj);

# 5. 运行时管理

## 5.1 ResLoadManager 主循环

```csharp
class ResLoadManager {
    void Tick() {
        // 1. 异常处理（强制返回登录场景）
        HandleException();

        // 2. 资源清除
        TickClean();

        // 3. 回调分帧（超过10ms下帧继续）
        TickCallback();

        // 4. 加载优先级控制
        TickLoading();

        // 5. 卸载策略
        TickUnload();
    }

    void TickCallback() {
        foreach (callbackList) {
            if (elapsedTimeMs > 10) break;  // 超时下帧
            DoCallBack();
        }
    }

    void TickLoading() {
        foreach (priorityLoadedDic) {
            if (curLoadingCount > 10) break;  // 限制并发数
            DoHttpReq();
            DoBundleLoad();
            DoAssetLoad();
        }
    }

    void TickUnload() {
        if (MemoryWarning) { Clean(); }
        if (idleBundle.Count > 30) { RealBundle(); }
    }
}
```

# 6. 资源释放

## 6.1 释放方式

```csharp
// 方式1：Release Handle
Addressables.Release(_op);

// 方式2：Handle完成后自动释放
_op.ReleaseHandleOnCompletion();

// 方式3：通过IResLoad统一释放（推荐）
class AddressableResLoad : IResLoad {
    Dictionary<string, AsyncOperationHandle> refCountDic = new();

    void LoadAssetAsync(string path, ...) {
        var op = Addressables.LoadAssetAsync<UnityEngine.Object>(path);
        refCountDic.Add(path, op);
    }

    void Dispose() {
        foreach (var kvp in refCountDic)
            Addressables.Release(kvp.Value);
        refCountDic.Clear();
    }
}
```

# 7. 注意事项

## 7.1 加载慢问题排查

```
加载慢（主角模型最后显示）排查步骤：
1. 检查网络问题（CDN延迟、带宽）
2. 检查平台接口API是否正常（微信WX.GetCachePath）
3. 检查Bundle数量和大小（是否过多小Bundle）
4. 检查Asset对象大小（贴图分辨率、Mesh面数）
5. 检查优先级设置（主角是否设置Critical优先级）
6. 检查是否有同步加载阻塞主线程
```

## 7.2 回调卡顿

```
原因：同一帧实例化大量对象，耗时较高
解决：
  1. 回调分帧处理（超过10ms下帧继续）
  2. 实例化分帧（每帧限制实例化数量）
  3. 对象池预创建（避免运行时实例化）

单个回调卡顿原因：
  1. Awake逻辑过重（大量Dictionary初始化）
  2. CreateGPUProgram（Shader变体编译）→ 需要Shader预热
```

## 7.3 卸载卡顿

```
战斗时卸载卡顿原因：
  1. AssetBundle.Unload(true) 同步卸载大量资源
  2. Resources.UnloadUnusedAssets() 耗时长

解决：
  1. 分帧卸载（每帧卸载有限数量）
  2. 延迟卸载（战斗结束后空闲时卸载）
  3. 内存警告时才触发强制卸载
```

## 7.4 UniTask GC 问题

```csharp
// 问题：async UniTask方法每次调用都会new状态机对象，产生GC
// 频繁调用（如每帧加载）会导致GC卡顿

// 生成的状态机代码（有GC）：
AddressableLoader.<LoadAddressablesAssetAsync>d__3<T> d__ =
    new AddressableLoader.<LoadAddressablesAssetAsync>d__3<T>(); // GC!

// 解决方案：
// 1. 使用ActionClosure减少upvalue开销
// 2. 避免频繁调用async方法
// 3. 使用对象池缓存状态机（UniTask.CompletedTask）
// 4. 批量加载代替单个加载
```

## 7.5 LoadFromStream 注意事项

```
1. Stream必须支持 CanRead && CanSeek
2. 每个AssetBundle必须有独立的Stream实例（不能共享）
3. Stream在AB整个生命周期内必须保持打开
4. 关闭顺序：AB.Unload() → Stream.Close()
5. 加密流必须正确实现Seek（推荐XOR或AES-CTR）
6. 无加密需求优先使用LoadFromFile（性能最优）
```

## 7.6 引用计数管理

```csharp
// 同一资源多处引用时，需要引用计数
class RefCountedAsset {
    int refCount = 0;
    AsyncOperationHandle handle;

    void AddRef() { refCount++; }
    void Release() {
        refCount--;
        if (refCount <= 0)
            Addressables.Release(handle);
    }
}

// 注意：Addressables内部已有引用计数
// 每次LoadAssetAsync引用+1，每次Release引用-1
// 引用归零时才真正卸载
```

# 8. 优化方案

## 8.1 IO 优化

```
1. IO池复用：预热FileStream，避免频繁创建销毁
2. 合并包（Pack File）：多个小Bundle合并为一个大文件
   - 减少文件句柄数量
   - 减少文件系统开销
   - 使用OffsetStream为每个Bundle提供独立视图
3. Buffer Size调优：
   - < 1MB Bundle：4096（默认）
   - 1-10MB Bundle：16384
   - > 10MB Bundle：65536
4. 异步加载：先轮询10个异步，补2个同步
5. 预读策略：根据玩家位置预读周围场景
```

## 8.2 内存优化

```
1. 按需加载：不加载当前不需要的资源
2. 对象池：复用GameObject，避免频繁Instantiate/Destroy
3. 资源池：缓存已加载的Asset，避免重复加载
4. 延迟卸载：空闲Bundle放入idleList，超过阈值再卸载
5. 内存警告响应：收到MemoryWarning立即清理空闲资源
6. Texture压缩：使用ASTC/ETC2等压缩格式
7. Mipmap管理：远处物体使用低Mip
```

## 8.3 加载速度优化

```
1. 优先级加载：关键资源（主角、UI）优先加载
2. 分组加载：按场景/模块分组，按需加载
3. 预下载：进入场景前预下载依赖Bundle
4. 并发控制：限制同时加载数量（避免IO竞争）
5. 缓存命中：已加载资源直接返回，不重复加载
6. Shader预热：启动时预编译Shader变体
7. 资源预热：进入战斗前预加载常用资源
```

# 9. 平台适配

## 9.1 Android

```
问题：
  1. StreamingAssets在APK内，不能直接用FileStream
  2. 低端机IO慢，资源过多时加载极慢
  3. 内存限制严格，需要积极卸载

解决方案：
  1. 使用APK内偏移量直接读取（不解压）
     - 解析ZIP Central Directory获取文件偏移量
     - 使用OffsetStream + baseOffset读取
  2. 或先解压到可写目录再加载
  3. 使用VFS文件系统管理资源
  4. 开启FileShare.Read允许多个句柄同时读取

代码示例：
  // Android APK内资源读取
  var stream = new OffsetStream(apkStream, baseOffset, baseSize);
  var bundle = AssetBundle.LoadFromStream(stream);
```

void DoBattle() {
    // 创建战斗专用加载器
    IResLoad battleResLoad = ResLoadManager.Create();

    // 加载战斗资源
    battleResLoad.LoadAssetAsync("Assets/Skill.Prefab", typeof(GameObject), 
        LoadingPriority.High, onSkillLoaded);
    battleResLoad.LoadAssetAsync("Assets/Icon.Prefab", typeof(Sprite), 
        LoadingPriority.Normal, onIconLoaded);

    // 退出战斗时统一释放
    OnBattleEnd += () => {
        battleResLoad.Dispose();  // 释放所有战斗资源
        ResLoadManager.CheckLeak();  // 检查泄露
    };
}

// 加载管理器完整实现
class AddressableResLoad : IResLoad {
    Dictionary<string, AsyncOperationHandle> refCountDic = new();

    void LoadAssetAsync(string path, Type assetType, short priority, CallBackObject action) {
        if (refCountDic.ContainsKey(path)) {
            // 已加载，直接回调
            action.Invoke(refCountDic[path].Result);
            return;
        }
        var op = Addressables.LoadAssetAsync<UnityEngine.Object>(path);
        refCountDic.Add(path, op);
        op.Completed += handle => action.Invoke(handle.Result);
    }

    void Dispose() {
        foreach (var kvp in refCountDic)
            Addressables.Release(kvp.Value);
        refCountDic.Clear();
    }
}
